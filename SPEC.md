## 0. Intent and scope

### 0.1 What this is

An operating shell for Claude Code that runs on top of the standard GitHub workflow. Target repos are cloned into the shell's `workspace/` directory and worked on from there.

The shell covers two layers of the same workflow:

- **Engineering execution** — issue → branch → draft PR → checklist commits → ready PR → merge. Per the GitHub-standard flow.
- **Setting direction** — `MISSION.md` → Directive Issue → Execution Issue, where a Directive is a medium-term directional context that scopes several issues — a feature with subsystems, a refactor, a migration alike — without being directly executable.

Both layers use the same generate → review → mode-based-approval → audit pattern (§1.5). See §1.7 for the eng-mode / dir-mode split, §2.1 for the Directive lifecycle, §4.9 for the dir-mode reviewer.

The shell guarantees:

1. **Adherence to the GitHub-standard flow** — issue → branch → draft PR → checklist commits → ready PR → merge.
2. **Doc → Test → Code work order** — every code change goes through docs first, then tests, then implementation.
3. **Active SSOT maintenance** — documents are updated alongside code. Code-only changes with stale docs are prevented.
4. **Aggressive use of subagents** — output quality doesn't rely on a single model and single context. Specialized per-phase agents are composed.
5. **Hook enforcement of engineering discipline** — direct commits to protected branches, force pushes, secret exposure, malformed commits, untested merges, etc. are actually blocked.
6. **Type-aware separation of dir-mode and eng-mode artifacts** — Directives and Execution Issues share storage but follow distinct workflows; hooks, commands, and listings discriminate by the `Item Type` custom field (§1.7).

### 0.2 What this is not

- Not a general dotfiles framework. The shell is scoped to `workspace/`.
- Not an absolute enforcer. No policy is set in stone. A conscious escape hatch is always open, and every bypass is recorded in the audit log.
- Doesn't own the target repo's knowledge SSOT. Updates are made by people / Claude inside the target.

### 0.3 Assumptions

- GitHub-standard, but **upstream-only**. Not a fork workflow. All branches are created on upstream; PRs go branch → main within the same repo.
- Hosted in a git repo, shareable across a team/org. Changes ship via PR.

### 0.4 Deferred capabilities

The following directional-layer capabilities are deferred — listed so current design choices preserve the option to add them later, not because they are scheduled work. Operating experience motivates each addition; see MISSION's "Success looks like" → orchestrator criterion for the framing.

- Orchestrator / automatic switching between eng-mode and dir-mode workflows.
- `/derive-next-directive`, `/reassess` commands. (`/block-directive` and `/revise-directive` were promoted from this list under Directive #75 — see §5.16 and §5.17.)
- Auto-confidence calculation for Directives.
- Auto-detection of Directive completion based on success-signal evaluation.
- Auto-revision of Directives based on engineering outcomes.
- Directive dependency graph or visualization.
- Multi-MISSION support (multiple canonical-direction docs per shell instance).
- External Director-tool integration interface (e.g., webhook receivers).
- Stop conditions / kill switches / budget controls (deferred with the orchestrator).
- `roadmap-reviewer` subagent (deferred until operating experience surfaces real friction).
- Session-level cleanup: a `SessionEnd` hook (where supported) and/or a user-invoked `/wrap` for branch tidy + temp-file cleanup (referenced from §6.3; not yet specified or built).

## 1. Core principles

These principles must be visible throughout the shell. CLAUDE.md, agents, commands, and hook messages should all point to the same principles.

### 1.1 GH-standard is the backbone

The flow comes before the principles. Each principle is an attribute attached to some stage of the flow.

### 1.2 Doc → Test → Code order

A code change has three phases:

**Phase A — Doc**: write the behavior / contract / decision into the doc first. Whichever of the target SSOTs applies (MISSION, SPEC, README, CLAUDE.md, ARCHITECTURE, ADR, API doc) — for a contract/behaviour change the **SPEC** is the primary surface (§1.3).

**Phase B — Test**: translate the doc from A into a test. The test must fail at this stage. Confirm the failure message is the intended picture.

**Phase C — Code**: minimal implementation to make B pass. Run adjacent regression checks afterward.

Each phase should be a separate commit when possible. The PR's commit graph should show the doc → test → code arc.

Effect of this order:
- Intent is externalized before code, enabling self-consistency checking.
- Tests stay closer to spec than when written alongside code.
- Re-entering the work later, you immediately know "which phase am I in?"

**Discard-boundary = Execution** (Directive #477). The Code phase is where the most working churn accrues — file reads, abandoned approaches, lint/smoke iterations — none of which is durable. The **discard boundary sits at the Execution unit**: that within-Execution churn can be discarded once the Code commit lands, while *directive-level* memory (the cross-Execution context a Directive carries) is preserved. The `implementer` subagent (§4.12) + `/implement` (§5.28) are the action-form of this — they run Phase C in an ephemeral context and return only the structured result (commit + plan-deviations + discoveries), so the churn never re-enters the main assistant's context. This is the **default Code-phase route** (Directive #477 signal-4: the formal A/B measurement was descoped and the context-narrowing hypothesis adopted as operating policy on circumstantial evidence — the convergent mainstream pattern + the native subagent context-discard), with a **documented opt-out** for trivial / one-line / glue edits and the orchestrator's own glue (Non-goal 2) that stay in the main loop, and **fail-open reversibility** (a missing implementer path degrades to main-loop authoring) as the safety valve; §3.7 records the same boundary against the flush → clear → reconstruct loop.

**Exception categories** — the following types relax the phase order. Note the reason in one line in the PR Plan.

| Type | Relaxation | Rationale |
|------|-----------|-----------|
| `fix` | failing test reproducing → code → (if needed) doc | Bug reproduction plays the spec role |
| `refactor` | when external behavior is unchanged, doc can be skipped; existing tests suffice | New doc/tests would be fake commits |
| `perf` | measure → code → doc (report results) | Hypothesis validation comes first |
| spike | free on a separate branch, restructured into a proper PR before merge | Protect exploration cost |
| one-line typo / obvious fix | single commit, keep the phase prefix | Splitting cost > benefit |

`feat`, `docs`, and changes to external contracts / API / schema apply the phase order strictly.

### 1.3 Active SSOT maintenance

Two SSOTs coexist:

| Layer | Location | Question it answers |
|-------|----------|---------------------|
| Process SSOT | claude-eng-shell git | **How** we work |
| Project SSOT | each target repo git | **What** / **why** we're building |

Target SSOT structure:
- `MISSION.md` — the project's ground truth. Why it exists, success criteria, explicit non-goals.
- `SPEC.md` — the project's **behavioural contract / design spec**: what the system does and the contracts it upholds. The primary Doc-phase surface for any contract/API/schema/behaviour change.
- `README.md` — for external readers.
- `CLAUDE.md` — directory index, run commands, domain vocabulary, conventions.
- `docs/ARCHITECTURE.md` — module boundaries, data flow.
- `docs/ADRs/NNNN-*.md` — record of irreversible decisions.

**MISSION + SPEC are the frequently-consulted pair** — the *why/what* (MISSION) and the *contract* (SPEC), the half of the durable memory the shell selectively injects from on nearly every task. README / CLAUDE.md / ARCHITECTURE / ADRs are reference memory, consulted on demand. This mirrors how the shell dogfoods itself (its own `MISSION.md` + `SPEC.md`), now imposed on targets too.

**Requiredness keys on code, not on an external surface.** `MISSION.md` is always required — every project has a direction. `SPEC.md` is **required for any project that carries code**: SPEC is the *behavioural* SSOT (*what the system does* and the contracts it upholds, per the definition above), and any codebase has behaviour worth specifying — so requiredness keys on the presence of code, **not** on whether an *external* contract (CLI, API, schema, protocol) exists. (This re-bases an earlier framing that gated SPEC on an external contract; that framing under-counted internal-only systems — like this shell itself — whose behaviour is exactly what a SPEC records.) A brand-new project with no code yet legitimately has none, but the shell treats SPEC as expected from the first code. The onboard tier decides how the slot is surfaced (§5.1): the shell scaffolds the SPEC slot from a lightweight template, it never authors the target's contract content (the MISSION boundary — the shell owns the *how*, not the *what/why*).

**SPEC navigation is single-file + line-indexed ToC.** A SPEC is one file with a machine-generated Table of contents between `<!-- TOC START -->` / `<!-- TOC END -->` markers (`scripts/build_toc.sh`), so readers consult the ToC then `Read --offset` the targeted section instead of loading the whole file. Two norms make this robust: (a) **ToC line numbers are a hint, the heading text is the truth** — if an offset lands on the wrong section (the SPEC was edited), re-Grep the heading rather than trust the number; (b) **regenerate the ToC in the same commit that edits a SPEC heading** (not deferred to `/ship`), so the index is never stale across the rest of the task. Single-file is chosen for **no-divergence** (one place a contract lives cannot contradict itself), not for speed; splitting a SPEC into a per-subsystem directory is a **scale escape hatch** for when a single file grows too large to full-read — each split file then being a single-file SSOT for its domain.

A code change commits together with the SSOT items it invalidates or updates. If a doc is intentionally omitted, the commit body states `Docs: n/a — <short reason>`.

### 1.4 PR-as-living-doc

The PR body simultaneously serves three roles:
- **The work item's ground truth** — what this PR is for.
- **Cross-session memory** — Claude's context is volatile but the PR body is durable. The next session auto-loads it to restore context.
- **Execution progress display** — the checklist renders as a GitHub progress bar.

Principles:
- The PR body is **editorial, not append-only**. Curate right after each commit. Tidy stale items; if history is needed, keep one line per entry in a separate "Changelog" section.
- **commit → PR sync** order. Fact (commit) first, then reflection (PR body). Reversing it creates false states.
- Before updating, refetch the remote body once. If it differs from what the shell last saw — checked against the persistent cache at `.claude/state/pr-cache/<owner>__<repo>__pr-<n>.json` via `helpers/pr_cache.sh` (§5.4) — abort the automatic update and inform the user. Don't build a merge logic beyond that — conflicts are rare, and a human should be in the loop when they happen.
- **No stale unchecked items at merge.** A merged PR body must reflect truth at the moment of merge. Every `- [ ]` line is in one of three terminal states: ticked (`[x]`) because it was done, marked `[~] N/A — <reason>` because it intentionally won't be done, or removed from the body. "Pending, but oh well" is not acceptable — closed/merged PRs are loaded by SessionStart (§6.5) into future context, so stale uncheckedness propagates as misinformation. The same rule applies to **issue bodies on auto-close**: every acceptance-criterion item is ticked, N/A'd with reason, or struck through with a closing comment that names the disposition.

### 1.5 Aggressive use of subagents

Quality benefits from not loading every burden onto a single context.

| Situation | Subagent |
|-----------|---------|
| Wide read-only exploration | `explorer` |
| 3+ file changes, API/schema/migration | `planner` |
| Doc writing / updating | `doc-writer` |
| Test writing (Phase B) | `test-writer` |
| Pre-commit / pre-PR review | `code-reviewer` |
| Auth / input / deps / crypto changes | `security-reviewer` |
| Rationale check on a proposed issue body | `issue-reviewer` |
| Approach / alternatives check on a planner output | `plan-reviewer` |
| Quality check on a proposed Directive or completion claim | `activation-reviewer` (§4.9) |

Independent investigations that can run in parallel: call multiple agents at once.

You don't need to call all nine on a single PR. Use them per the table; the main assistant shouldn't re-run in `explorer` what it already explored. No call-count limits or tracking.

**Working-tree isolation** (#285): the read-only-by-intent subagents — `code-reviewer`, `security-reviewer`, `issue-reviewer`, `plan-reviewer`, `activation-reviewer`, `explorer` — share the parent session's working tree unless invoked with worktree isolation, and they carry `Bash`, so a tree-mutating git command inside one (`git checkout <base> -- <path>`, `restore`, `stash`, `reset`, `add`, `commit`) silently reverts or stages the parent's uncommitted work — observed once in the v0.3 run. Two layers prevent it: (a) **canonical** — invoke these subagents with **worktree isolation** so their git cannot touch the parent tree; and (b) **belt-and-suspenders** — each such agent's prompt constrains it to **read-only git only** (`git diff`/`show`/`log`/`status`; compare a base via `git diff <base>...HEAD` or `git show <ref>:<path>`, never `git checkout <base> -- <path>`). The structural backstop is that merges build from the *pushed commit*, not the working tree, so a post-commit reversion cannot ship; still, run `git status` before each commit/merge to catch a pre-commit reversion.

**Operating-mode coupling**: `code-reviewer`, `security-reviewer`, `issue-reviewer`, `plan-reviewer`, and `activation-reviewer` all substitute for human review at different points in the flow. In `attended` mode they surface findings to the user, who decides; in `unattended` mode their verdict gates the next step directly (`ship` → proceed; `refine` → re-produce with feedback; `block` → stop with the reason and either ask the user, or — when no human is available — park the work). (`activation-reviewer` uses the equivalent `pass` / `revise` / `reject` vocabulary, §4.9.1.) This pattern is what makes the `unattended` mode operationally complete: every human-confirm checkpoint has a subagent fallback.

**Filer-aware invariants** (added by Directive #92 — Issue #95): a layer of protections **above** the attended/unattended axis, keyed on the Issue's `authorAssociation` per GitHub's role taxonomy (`OWNER` / `MEMBER` / `MAINTAINER` / `COLLABORATOR` → trusted; everything else → untrusted). Independent of operating mode — both attended and unattended honor the same filer-aware rules. Two protections enforced by the `trusted-filer-mutate` hook matcher (§6.1):

- **AI never auto-closes-as-not-planned on a trusted-filer Issue.** Closing a trusted-filer Issue with `--reason completed` is allowed (evidence-backed); closing as `not planned` or `duplicate` requires human confirm. Untrusted-filer Issues continue to follow the regular attended/unattended close rules.
- **Removing the `directive` label declassifies a Directive — human confirm required always, regardless of filer trust tier.** The `directive` label is the type-awareness key (§1.7, §6.1 `proposed-protect`); silently dropping it would bypass dir-mode review for an Issue still positioned as a Directive.

Both invariants escape via `SKIP_HOOKS=trusted-filer-mutate SKIP_REASON='<why>'` (§7). Helper `is_trusted_filer <issue#>` lives at `.claude/hooks/helpers/issue_filer.sh`; caches per-session under `.claude/state/issue-filer-cache/`. Bot-filed Issues (Dependabot, GitHub Actions) carry `authorAssociation="NONE"` and are currently treated as untrusted — documented limitation per the helper's notes (Directive #92 brief §11).

### 1.6 GitHub standard (upstream-only)

- No forks. The clone's origin is always upstream.
- All branches are created on upstream.
- PRs are branch → **base** within the same repo. **Default base is `main`**; `/work-on --base <branch>` (§5.3) opts into an alternate base for experimental work or long-lived feature branches. See §10.5 "Topic-branch workflow."
- Every change starts from an issue. issue → PR is 1:N (intermediate PRs use `Refs #N`, final uses `Closes #N`). For PRs whose base is non-`main`, `Refs` is the right default — the issue closes only when work reaches `main`.
- Branch naming: `<gh-username>/<type>/[<issue#>-]<slug>`. The required-issue-# group (§6.1.1) always includes `<issue#>-`; the optional group (`chore`/`style`/`build`/`ci`/`test`/`revert`) may omit it. Examples: `ilgyu/feat/42-oidc-login`, `ilgyu/chore/typo-readme`.

### 1.7 Artifact tiers and the subagent shape they share

The shell operates over two **artifact tiers** above the commit/PR layer: a Directive Issue (medium-term directional context) and an Execution Issue (a concrete unit of work). Both tiers follow the same workflow pattern from §1.5 (generate → review → approval → audit); they differ in which subagents author and which review at each step. The terms `eng-mode` and `dir-mode` name the workflow each tier exercises, but the substrate, hooks, audit log, mode toggle (§5.7.1), and state directory are shared.

**Shared code, per-project state** (the MISSION-tier principle §3.2.2 builds on, and the reason "the substrate … are shared" above holds across projects): there is exactly **one** canonical code/SPEC/norms source, and every bound project resolves it **live** — a project binds to the canonical root (§3.2.1) rather than copying it, so a friction-fix propagates to *all* bound projects and none is pinned to a stale version (MISSION § "The mechanism" → "Isolation model"). What isolates per project is only **ephemeral state**: the audit log, caches, and scope **registry** under each project's `eng-state/` (§3.2.2), plus the session and Claude Code's native per-project memory. Shared substrate = the durable memory selective injection draws from; per-project state = the narrowed surface. Code/version isolation (per-project pinning) is an explicit non-goal — it would break propagation.

| Tier (workflow name) | Artifact type | Generate-subagent(s) | Review-subagent | Approval rule |
|---------------------|---------------|----------------------|------------------|---------------|
| **Execution Issue** (eng-mode) | Execution Issue, PR, Commit | `planner`, `doc-writer`, `test-writer`, `explorer` | `code-reviewer`, `security-reviewer`, `issue-reviewer`, `plan-reviewer` | attended: human / unattended: reviewer verdict (§1.5) |
| **Directive Issue** (dir-mode) | Directive Issue | `planner`, `doc-writer`, `explorer` | `activation-reviewer` (§4.9) | attended: human / unattended: `activation-reviewer` verdict |

**Hierarchy of artifacts** that the two modes act on:

```
MISSION.md                 ← human-defined long-term anchor (doc-as-code)
  │            ↓
  │       Initiative        ← planning-tier strategic commitment, consumed from
  │          │                upstream — never authored by the shell (live; the
  │          │                tier the shell consumes — see below)
  └──→ Directive Issue ←─────┘  directing context (parent: a MISSION section OR an
       │                         Initiative), revisable, medium-term (dir-mode)
       ↓
     Execution Issue        ← concrete unit of work (eng-mode)
       ↓
     Pull Request           ← implementation evidence (eng-mode)
       ↓
     Reflection             ← engineering result summary
```

Dir-mode operates on the Directive layer; eng-mode operates on Execution Issue and below. The Initiative tier above the Directive is consumed from upstream, never authored by the shell — see "Initiatives — the planning tier the shell consumes" below.

**Workflow selection is manual.** Users invoke `/file-directive`, `/activate`, `/complete-directive`, `/list-directives`, `/link-directive`, `/revise-directive`, `/block-directive`, `/consume-initiative`, `/initiative-feedback` (§5.10–§5.22) for directing work and `/file-issue`, `/work-on`, `/ship` for engineering work. (`/activate-directive` and `/triage` are deprecated aliases of `/activate`, retained one release cycle — §5.12/§5.18.) No orchestrator decides between them; an automatic selector is a future addition that operating experience may motivate (see MISSION's "Success looks like" → orchestrator criterion).

**Strategic judgment remains external.** AI may generate, draft, propose, and review at any level. Activation, completion, and revision decisions pass through either a human (attended) or `activation-reviewer` (unattended) — the shell never autonomously decides direction.

**Canonical direction.** The repo's `MISSION.md` is the canonical long-term anchor. Directives reference its sections in their `## MISSION fit` field; `activation-reviewer` checks alignment with MISSION rather than a separate Goal artifact. MISSION.md as doc-as-code beats an Issue-or-Project Goal artifact because the latter creates two representations of the same canonical direction — revising one without the other surfaces drift.

**Substrate.** All dir-mode state lives on GitHub Issues. The rationale for Issues-as-SSOT over Project-fields-as-SSOT: Project access is owner-scoped while repo access is broader (storing state on Project fields gates writes by access the contributor may not have); dual-recording across labels, fields, and body invites silent sync drift; and Issues are the OSS-standard proposal channel that external contributors expect.

- **`directive`** label — marks an Issue as a Directive. The single Type-awareness key.
- **`initiative`** label (#249) — marks an Issue as an Initiative (the planning tier above a Directive, consumed from upstream — see "Initiatives — the planning tier the shell consumes" below). The single Type-awareness key for that tier, symmetric to and mutually exclusive with `directive`; resolved by the `is_initiative_issue` predicate (`helpers/issue_type.sh`).
- **`status:proposed`** label — Status=Proposed (filed but not yet activated).
- **`status:blocked`** label — Status=Blocked (active but cannot proceed without external input).
- Open + `directive` + no `status:*` label → Status=**Active**.
- Closed + `directive` + state-reason=completed → Status=**Completed**.
- **`P0`/`P1`/`P2`/`P3`** label — Priority.
- **`execution`** label (#186) — positive type label for an Execution Issue (a unit of work parented under a Directive). Makes Execution a first-class queryable type rather than label-absence; the `Parent Directive: #N` marker is then purely the parent pointer. Distinct from `task` (standalone, not parented) and `bug`.
- **Issue body** — Objective, Success signals, Non-goals, Constraints, MISSION fit (Confidence is documented in body, not a field).
- **Body line 1 `Parent Directive: #N`** — Parent reference for Execution Issues; consumed by `/reflect`, the dir-mode-post-merge workflow, AND the issues-to-project-mirror workflow.

The GitHub Project named `<repo-name> roadmap` (override via `$CLAUDE_ENG_PROJECT_NAME`) is a **derived view**, not the SSOT. Setup script `scripts/setup_project.sh` ensures the four mirrored fields exist: `Item Type / Status / Priority / Parent` (SINGLE_SELECT for the first three, TEXT for the last; the discriminator field is named `Item Type` rather than `Type` because GitHub reserves `Type` as a built-in Projects-v2 field name — #342). `Iteration` is owner-managed via the GH UI; the mirror workflow does NOT mirror it (Iteration is owner-level cadence, not repo-level deliverable grouping — those concerns split between Project Iteration and GitHub Milestone). Confidence and Success Signals live in the Issue body, not Project fields. The Project Item is populated by `.github/workflows/issues-to-project-mirror.yml` on every `issues.{opened,edited,labeled,unlabeled,closed,reopened,milestoned,demilestoned}` event. Dir-mode commands write to Issues only; they never write to the Project (the mirror handles the derivation). The two-layer idempotency contract on `setup_project.sh` (field existence + SINGLE_SELECT option-set correctness, from issue #76) carries forward to the 4-state Status schema: missing declared options are added via `updateProjectV2Field` GraphQL mutation; user-added options outside the declared set are preserved.

**Filer-aware invariants** (§1.5, cluster C / issue #95): the `trusted-filer-mutate` hook matcher protects trusted-filer (OWNER / MEMBER / MAINTAINER / COLLABORATOR per `authorAssociation`) Issues from AI auto-close-as-not-planned and protects the `directive` label from declassification on any filer. Mode-independent — applies in both `attended` and `unattended`.

**Audit-line format guard** (issue #87): `audit_log` in `.claude/hooks/hookrt.sh` enforces strict format on `directive-file/created` lines pre-write — malformed calls are rejected and written as `decision=format-error` warn lines instead. Smoke §50a remains as defense-in-depth + historical-line grandfathering.

**Initiatives — the planning tier the shell consumes** *(fully live as of milestone `v2 consuming initiatives`: the **recognition layer** (the `initiative` label, the `is_initiative_issue` predicate, `Parent Initiative: #N` marker reading — #249, M1.1), the **enforcement layer** (label mutual-exclusivity, the Directive parent-XOR, Initiative read-only-except-comments — #251, M1.2), the **reviewer layer** (`activation-reviewer` parent-kind branching + the contract-evaluability and extraction-faithfulness checks — #254, M2), the **consumption** flow (`/consume-initiative` extraction — #257, M3), the **feedback** flow (`/initiative-feedback` structured challenge + completion — #260, M4), and the **derived-view** integration (the issues-to-project-mirror workflow populates an Initiative-parented Directive's Project `Parent` field — #262) are all live.)* An **Initiative** is a planning-tier strategic commitment sitting one tier above a Directive. It arrives from **outside the shell** as a GitHub Issue bearing the `initiative` label — the single Type-awareness key for the tier, symmetric to `directive` and mutually exclusive with it. The shell **consumes** Initiatives and never authors them; an Initiative is **read-only to the shell except for appended comments** (the shell never edits its body, labels, or state). Consequently an Initiative is **outside the 4-state Directive lifecycle and the `status:*` taxonomy entirely** (§2.1): the shell never applies `status:proposed`/`status:blocked`/`status:*` to it, it is never "Activated" or "Completed" or "Blocked," and `/activate` refuses it (§5.12). Its real progress lives upstream and surfaces only through `/initiative-feedback` (§5.22); the Project mirror may render an open Initiative as `Active` purely as a derived-view artifact of the shared Status field (§2.1), not a lifecycle claim. The origin of an Initiative (a person, a planning tool) is immaterial — the shell relies only on the contract below, not on how the Initiative was produced.

- **Parent generalization.** A Directive has exactly one parent, of one of two mutually-exclusive kinds: a **MISSION section** (named in the Directive's `## MISSION fit` body field per §2.1 — the default) or an **Initiative** (a `Parent Initiative: #N` marker on body line 1, the same line-1 `<kind>: #N` *shape* as `Parent Directive: #N` — but **not** consumed by the same workflow set; see "Derived-view integration" below). The two parent kinds have **different shapes** — a body *field* naming a section vs. a line-1 *marker* — so the parent-XOR check (below) tests "a MISSION-fit field and no Initiative marker" XOR "an Initiative marker and no MISSION-fit field," not two parallel markers. An Initiative is introduced only when a strategic commitment genuinely bundles several Directives and is worth reviewing and evolving in its own right; the default stays MISSION-parented, so the tier does not become ceremony. This **generalizes** the hierarchy above rather than replacing it — every existing MISSION-parented Directive remains valid unchanged.
- **Derived-view integration.** The line-1 `Parent Initiative: #N` marker is consumed by **exactly one** of the three `Parent Directive` consumers (§5.2): the **issues-to-project-mirror** workflow, which derives a Directive's Project `Parent` field from its own line-1 marker and recognizes **both** `Parent Directive: #N` and `Parent Initiative: #N` (#262) — so an Initiative-parented Directive's Project `Parent` points to its Initiative (the `Parent` field is free-text `#N`, so the referent's type is immaterial). It is **not** read by `/reflect` or the dir-mode-post-merge workflow: those consume the **closing Execution Issue's** `Parent Directive` marker to reflect onto a Directive, and an Execution Issue is structurally always Directive-parented (the parent-XOR + `label-parent-consistency` restrict `Parent Initiative` to Directives, and a Directive never branches nor closes via a PR `Closes`, so it never becomes a `closingIssuesReference` — §6.1, §10.5). A Directive's `Parent Initiative` marker therefore never reaches the reflection path; **Directive→Initiative upward feedback flows through `/initiative-feedback` (§5.22)**, deliberately a structured comment channel, not auto-reflection. This is why "symmetric" above is scoped to marker *shape*, not consumption.
  - **Create-path split.** The two parent kinds are born from **different commands**: `/file-directive` (§5.10) authors a `## MISSION fit` field only, so it produces **MISSION-parented** Directives exclusively; `/consume-initiative` (§5.21) is the **sole** producer of `Parent Initiative`-marked Directives. There is deliberately no `/file-directive` path to an Initiative parent — a hand-added `Parent Initiative` marker on a `/file-directive` body would carry a MISSION-fit field too and trip the parent-XOR. To parent a Directive under an Initiative, consume the Initiative; do not hand-author the marker.
- **The contract (load-bearing).** An Initiative must carry a **termination condition evaluable without knowledge of the code.** The shell extracts Directives by working backward from it. A *complete* Directive needs code knowledge — its Success signals and Constraints reference code reality, which is why the shell, not the upstream, extracts Directives — but the Initiative's termination condition sits above code and so can and must be stated without it. A condition that cannot be (one that smuggles in execution detail) marks the Initiative under-specified; the shell surfaces that upstream rather than guessing. This is the planning/execution boundary: above the point where a termination condition can be set without the code is the upstream's job; below it is the shell's.
- **Feedback is upward and non-deciding.** The shell sends findings to the Initiative as comments, a structured **challenge** when execution reality contradicts the Initiative, and a **completion** signal when the extracted Directives land — carrying code-derived information the planning layer structurally cannot have (real dependencies, measurability, implementation cost). It never rejects, retires, or asserts completion of an Initiative: discarding or closing a strategic commitment is a strategic judgment that stays upstream. This mirrors the reviewer pattern — a `block`/challenge escalates rather than auto-discards (§4.9.1, §5.7.1).
- **Location.** An Initiative is a GitHub Issue **in the same target repo**, located by the `initiative` label. The mechanism of its arrival is out of scope; **cross-repo Initiatives are out of scope** (they would reintroduce the cross-repo-trust surface of §6.1's `trusted-filer-mutate` close arm).
- **Mechanisms.** **Live (M1.1, #249):** the `is_initiative_issue` predicate (symmetric to `is_directive_issue`, `helpers/issue_type.sh`; cached, fail-open per §6.1) and `Parent Initiative: #N` marker reading (the `issue_has_initiative_parent_marker` tri-state resolver, distinct from the `Parent Directive` resolver that backs `label-parent-consistency`). **Live (M1.2, #251):** three enforcement matchers — the `initiative`/`directive` label mutual-exclusivity and the Directive parent-XOR (exactly one parent kind present: a `## MISSION fit` field **or** a line-1 `Parent Initiative:` marker, never both and never neither (different shapes — see Parent generalization), enforced via `issue_has_mission_fit_field`/`issue_has_initiative_parent_marker`), both folded into the `label-parent-consistency` matcher on the `--add-label` path; and `initiative-readonly` (mutating `gh issue edit`/`close`/`reopen` on an `initiative` Issue blocked, `gh issue comment` allowed). All fail-open per §6.1. **Live (M2, #254):** reviewer branching — `activation-reviewer` (§4.9, extended in place rather than via a new subagent, per the §4.9.3 caveat) recognizes the `initiative` label, branches a Directive's alignment check on its parent kind (MISSION section vs. parent Initiative's termination condition), and gains a contract-evaluability check (Initiative intake) + an extraction-faithfulness check (post-extraction), both invoked by the M3 consume flow. **Live (M3, #257):** the `/consume-initiative <N>` command (§5.21) — the consume flow: contract gate (I1) → working-backward extraction → faithfulness gate (I2) → file the extracted Directives as `status:proposed` `Parent Initiative`-parented proposals (never activated; the only Initiative write is a plain under-specification comment). **Live (M4, #260):** the `/initiative-feedback <N>` command (§5.22) — the upward-feedback flow: a `## Initiative challenge` comment (code-derived contradiction → request re-evaluation, escalate-not-reject) and a `## Initiative completion` comment (merged-Directive evidence → request termination assessment, never assert-done/close); comments only, never mutating the Initiative. **Live (#262):** the **derived-view** integration — the issues-to-project-mirror workflow recognizes a Directive's line-1 `Parent Initiative: #N` marker and populates its Project `Parent` field (the one `Parent Directive` consumer that legitimately reads a Directive's own marker; `/reflect` + dir-mode-post-merge read the Execution child's marker and are correctly unchanged — see Derived-view integration). **The milestone-`v2` model is now fully live** — recognition + enforcement + reviewer + consume + feedback + derived-view.

**Out of scope** (Initiative tier): authoring Initiatives; knowing their origin; tracking a challenge's outcome once escalated; rejecting/retiring an Initiative from within the shell; nested Initiative tiers (the tier is flat above the Directive); automatic orchestration between planning-mode and execution-mode work; cross-repo Initiatives.

#### Substrate-in-target contract (Issue #114; Directive #107)

Dir-mode's substrate is installed into target repos under `workspace/` via the `/onboard-dir-mode` skill. The boundary broadening — shell now mutates target `.github/` files + labels in addition to its `.claude/` symlinks — is acceptable only because each install step is detectable, individually revertable, and additive (never replaces target-owned artifacts). The reversibility contract below makes adoption low-risk: a target can fully unwind the shell via one PR plus a few `gh` commands, with no load-bearing dependency on installed files.

**Three-tier feature model.** Targets adopt the shell at one of three tiers; each is a strict superset of the previous and tier transitions are mechanical:

- **Tier 1: eng-mode** — no substrate required. `/file-issue`, `/work-on`, `/ship`, secret scan, AC closeout, conventional-commit enforcement.
- **Tier 2: dir-mode-with-labels** — tier 1 + the 12-label dir-mode set (`directive`, `initiative`, `status:proposed`, `status:blocked`, `awaiting-author`, `execution`, `task`, `discussion`, `P0`, `P1`, `P2`, `P3`), installed via `scripts/ensure_v3_labels.sh` + inline `directive` and `initiative` creates. (Count note: these 12 enumerate the dir-mode lifecycle/priority/type labels; the tier-2 *installer* reports **15 total** because `ensure_v3_labels.sh` creates 13 labels — the 11 lifecycle/priority labels (incl. the release-backbone `skip-changelog` #133) plus the two initiative-feedback projection labels `initiative:challenged` / `initiative:completion-requested` (#359) — but not the two tier type-keys, and `onboard_target.sh` then adds `directive` and `initiative` (#249) inline — i.e. 13 ensure + 2 inline = 15, traceable to `onboard_target.sh`'s count message and smoke §63g.) Unlocks `/file-directive`, `/activate`, `/complete-directive` writing to Issues directly. No Project mirror, no Issue templates. (`awaiting-author` — #172 — marks the handoff state after a `revise` or trusted-filer `reject`; it is an action-signal label without a `status:` prefix, orthogonal to the lifecycle stage. The former raw-filing triage label was retired by #179 — raw filings now receive `status:proposed`+`task` directly.)
- **Tier 3: full substrate** — tier 2 + `.github/ISSUE_TEMPLATE/*.yml` + `.github/workflows/{auto-status-proposed,auto-clear-awaiting-author,issues-to-project-mirror,dir-mode-post-merge,check-changelog,check-toc,initiative-feedback-label}.yml` (7 workflows, matching the canonical source `target-substrate/workflows/` and smoke §63a; `initiative-feedback-label.yml` #359 projects `/initiative-feedback`'s `## Initiative challenge` / `## Initiative completion` comment markers into the `initiative:challenged` / `initiative:completion-requested` labels — applied by the target-repo CI Action on `issue_comment`, never by eng's runtime, preserving the `initiative-readonly` boundary) — plus the sourced helpers the workflows read after `actions/checkout` (`resolve_parent_directive.sh` #335, `detect_bare_refs_directive.sh` #337; §5.15, §10.5) and the SPEC ToC-freshness checker `build_toc.sh` that `check-toc.yml` runs (#347, §1.3 — byte-identical to the canonical `scripts/build_toc.sh`, smoke-guarded), which ship alongside the workflows into the target — + GitHub Project (`<repo> roadmap`) with the dir-mode field schema. Unlocks the template chooser for external contributors and the Project-as-derived-view. **Blank issues are disabled** (`blank_issues_enabled: false`, #186) — every UI filing must use a typed template (directive/execution/task/bug/discussion), so it is always classified and (for the four work-types) `status:proposed`-gated; the `discussion` template (§5.19) is the sanctioned low-friction path, so blank issues are not needed. (`auto-status-proposed.yml` — repurposed by #179 — now serves as **defense-in-depth** for deliberate non-UI label-free creations (API / `gh issue create --no-label`): it stamps `status:proposed`+`task`. `/triage` is a deprecated alias for `/activate`, §5.12/§5.18.)

**Bootstrap path.** `/onboard-dir-mode` installs the tier-N artifacts via a **PR** to the target — never direct-push. Labels + Project install directly (not git-tracked); maintainer reverts via `gh label delete` / `gh project delete` per the reversibility contract below. **Stage-0 precedes all of this**: a brand-new target with no default branch (empty repo / unborn HEAD) is taken to a seedable default branch by `/bootstrap-repo` (§5.0) *before* `/onboard` (§5.1) can run — a PR-based substrate install presupposes a default branch to target, so stage-0 owns the no-default-branch starting state.

**Graceful-degradation principle.** Dir-mode commands check their substrate prerequisites before acting. **Hard-refusal only when the SPECIFIC dependency for the command is missing AND the command's contract cannot be met without it** — otherwise run in degraded mode with a one-line warning. Per-command judgment, not global. Test anchors: `/file-issue` works at tier 1; `/file-directive` hard-refuses at tier 1 (no `directive` label), degrades-with-warning at tier 2 (no Project mirror), works fully at tier 3; `/activate` (which `/triage` now aliases, §5.12/§5.18) requires tier 2 (the `status:proposed` label); `activation-reviewer` at tier 2 falls back to the existing bootstrap allowance for repos without MISSION.md.

**Reversibility contract.** Every install step is detectable + revertable: files via `git rm` in a normal PR; labels via `gh label delete`; Project via out-of-band `gh project delete` (owner-state, not scripted). The shell can be fully unwound from a target via one PR (file removals) + one-liner per label + an out-of-band Project decision. The per-command **substrate preflight** has landed (Directive #107, children #114/#118 merged): each dir-mode command (`/file-directive`, `/activate`, `/complete-directive`, `/list-directives`, `/link-directive`, `/consume-initiative`, `/initiative-feedback`, `/block-directive`) carries a step-0 preflight that aborts with `"target lacks dir-mode substrate; run /onboard-dir-mode --tier 2 first"` when `gh label list | grep -qx directive` (or `initiative`) finds nothing, fail-open on `gh` network errors.

### 1.8 In-session narrowing levers (clear-free)

A context **clear** is the bluntest narrowing tool (§3.7) — it discards the whole working context and pays the cost of reconstructing from durable memory. The levers below keep the working context task-relevant **without** a clear, so a `/clear` is a rarer last resort rather than a routine necessity. This section inventories them as one named set with the contract each enforces and its current **enforcement face** (guidance vs. hook, in the §6.0 sense). It is the complement of §3.7: §3.7 makes a clear *safe* (`active → archived`), §1.8 makes a clear *less necessary*.

| Lever | Contract it enforces | Current face |
|-------|----------------------|--------------|
| **Subagent isolation** (§1.5) | Wide or context-dirtying work (broad search, multi-file review) runs in a separate agent window; only the narrowed result returns to the parent context. | **Guidance** — the CLAUDE.md subagent table + "don't re-run an exploration in `explorer`"; worktree-isolation is hook-adjacent (#285) but delegation itself is advised, not enforced. |
| **Doc → Test → Code phasing handoff** (§1.2) | Each phase is its own commit; the next phase consumes the prior phase's *artifact*, not the accumulated conversation — context resets across the arc. | **Mixed** — guidance for the order; the PostToolUse commit reminder (§6.2) and the `phase:`/`next:` status projection (§6.4) partially mechanize it. |
| **Artifact-only reviewers** (§1.5, §4.5–§4.9) | Reviewers judge from the diff / PR body / issue / MISSION, never from the main assistant's conversation — so review context stays task-scoped regardless of how long the authoring conversation grew. | **Mixed** — guidance + the worktree-isolation backstop (#285). |
| **Targeted SPEC / file reads** (§1.3) | Consult the ToC (or a symbol search) then `Read --offset/--limit` the specific section, rather than loading a whole large file into context. | **Guidance only** — SPEC §1.3 + CLAUDE.md prose; nothing observes a whole-file load. ← the lever reinforced below. |
| **Plan-as-manifest** (§4.1, §6.4) | The planner's Plan is the task-scoped manifest of what the work touches; the per-turn surface stays `status_compact`, not a re-read of everything. | **Mixed** — guidance + the UserPromptSubmit `status_compact` injection (§6.4). |

**Advisory → affordance migration (this section's reinforcement).** Per §6.0 P3, a guidance-only lever is hardened into an affordance once the pattern proves out. The **targeted-reads** lever — guidance-only above — gains a positive **PostToolUse `Read` nudge** (§6.2): when a `Read` loads a whole file (no `offset`/`limit`) above a line threshold, the hook emits a one-line, non-blocking suggestion to use a targeted `Read --offset/--limit` or delegate the search to `explorer`. The face is **positive, not a block**: a missed narrowing nudge is ignorable at no cost (§6.0 P1 cost-asymmetry), so the lever guides rather than refuses. The other levers stay at their current faces; this section describes the present state, it does not reclassify them.

### 1.9 Harness-overlap classification

The shell runs on top of a Claude Code harness that **improves under use** and natively absorbs more of the generic agent-support machinery over time (context compaction, deferred/lazy tool loading, excerpt-reading search agents, native subagents, plan mode, task tracking, file-based memory). Treat the harness as a **rising floor** (MISSION "The mechanism"): a shell mechanism that merely re-does what the harness now does natively is redundant cost, while the shell's durable, non-redundant value concentrates in what the harness — which must stay general — does *not* encode. This section records, as standing doctrine, each mechanism's **posture** toward that floor. It is a classification, **not a removal mandate**: ceding is a forward-discipline signal (retire via a separate Issue, weighed against safety value), never an in-place deletion (#449).

**Posture vocabulary** (exactly one per mechanism):
- `cede-to-harness` — the harness now does this natively; the shell's version is redundant and is a candidate for retirement (forward discipline, not deleted here).
- `keep-as-policy` — opinionated GitHub-engineering policy the harness does not encode (and should not, staying general); the shell is the only source of it.
- `keep-as-safety-redundancy` — a deliberately redundant backstop for a high-cost-of-wrong (irreversible / shared-history-corrupting) action, kept even where the harness offers a partial native guard, per the §6.0 cost-asymmetry principle.

Coverage is **parity-guarded** (smoke §116): this section must classify every mechanism enumerated in §1.8 (narrowing levers), §4 (subagents), §5 (slash commands), and §6.1 (hook matchers) — a new mechanism in any of those families fails smoke until it gains a row here. The classification references each mechanism's home section by anchor; it adds no parallel contract content (§9).

**Narrowing levers (§1.8 — 5):**

| Lever | Posture | Rationale |
|-------|---------|-----------|
| Subagent isolation (§1.8 / §1.5) | `cede-to-harness` | The harness ships native subagents (Agent/Task, fork, worktree, parallel) and an excerpt-reading search agent; running wide work in a separate window is now a harness primitive. |
| Doc → Test → Code phasing handoff (§1.8 / §1.2) | `keep-as-policy` | The harness has no task-phase model; consuming the prior phase's artifact instead of the conversation is engineering policy the shell imposes. |
| Artifact-only reviewers (§1.8 / §4.5–§4.9) | `keep-as-policy` | "Judge from the diff / PR / issue / MISSION, never the conversation" is a review discipline, not a harness capability. |
| Targeted SPEC / file reads (§1.8 / §1.3) | `cede-to-harness` | The harness now narrows input natively — deferred/lazy tool loading, excerpt-reading Explore, targeted reads — so the generic form of this lever is harness-absorbed. |
| Plan-as-manifest (§1.8 / §4.1) | `keep-as-policy` | The persistent, task-scoped manifest is the PR body / GitHub artifact (constraint-isomorphic durable memory), not the harness's ephemeral plan-mode buffer. |

**Slash commands (§5 — 27):**

### 2.1 Directive lifecycle

A Directive moves through **four states** (Proposed / Active / Blocked / Completed). All transitions are entered via explicit dir-mode commands (§5.10–§5.22). Status is encoded as **labels on the Issue**; the Project Item's Status field is mirrored by `.github/workflows/issues-to-project-mirror.yml`. A transient revision state was considered and dropped — the audit-log entry + the archived prior-body comment are the durable evidence of revision (see `/revise-directive`, §5.16).

**Scope of the 4-state machine.** The four states + the `Blocked`/`Completed` transitions are **Directive-specific**. Execution Issues (`execution`/`task`/`bug`) share only the `Proposed → Active` gate (the full-symmetry lifecycle below), not `Blocked`/`Completed`. **`discussion`** Issues (§5.19) and **`initiative`** Issues (§1.7) are **outside this lifecycle entirely** — a discussion closes via `/resolve-discussion`, and an Initiative is read-only/consumed (the shell never sets its `status:*`). Because the mirror derives the Project `Status` field from the open/no-`status:*` label shape, an open `discussion`/`initiative` Issue surfaces in the Project as `Active` by default; that is a derived-view artifact of the shared field, **not** a claim that they run the Directive 4-state machine — read their real state from their own path (resolve/consume), not the mirrored `Status`.

```
   /file-directive
        │  (creates Issue with [directive, status:proposed])
        ▼
   ┌────────┐        /activate        ┌────────┐
   │Proposed│ ──────────────────────▶ │ Active │
   └────────┘   (reviewer-gated;      └────────┘
                removes status:           │
                proposed label)           │  /complete-directive
                                          │  (reviewer-gated; closes
                                          │   Issue --reason completed)
                                          ▼
                                     ┌──────────┐
                                     │Completed │
                                     └──────────┘

       /block-directive (§5.17)
   Active ──────────────▶ Blocked
                          (adds status:blocked label;
                           Issue stays open;
                           unblock via /activate)

   /revise-directive (§5.16) — body replace + audit log only;
   no Status change (the audit-log entry + the archived prior-body
   comment ARE the durable evidence of revision).
```

**State semantics** (label-encoded on the Issue; mirrored to the Project Item by `issues-to-project-mirror.yml`):

| State | Encoding on Issue | Entry transition | Reviewer-gated? |
|-------|-------------------|------------------|-----------------|
| `Proposed` | open + `directive` label + `status:proposed` label | `/file-directive` (creates with both labels) | Yes — `activation-reviewer` on filing body |
| `Active` | open + `directive` label + no `status:*` label | `/activate` (removes `status:proposed`; `/activate-directive` is a deprecated alias, §5.12) | Yes — re-reviews body in case it was edited since filing |
| `Blocked` | open + `directive` label + `status:blocked` label | `/block-directive` (adds `status:blocked` + `## Blocked: <reason>` comment) | No on block (annotation-only); unblock via `/activate` (treats `status:blocked` as a re-activatable precondition; on `pass` removes the label, §5.12) |
| `Completed` | closed + `directive` label + close-reason=completed | `/complete-directive` (posts closing-comment evidence + closes `--reason completed`) | Yes — `activation-reviewer` evaluates evidence sufficiency |

**Lifecycle symmetry for Execution Issues** (#172, Directive #167). Under **full symmetry**, the `Proposed → Active` gate applies to **all** Issues, not just Directives: `/file-issue` stamps `status:proposed` on every new Execution Issue, and `/activate <N>` (§5.12) removes it after an `activation-reviewer` pass — the same generate → review → gate shape the Directive path uses. `status:proposed` Execution Issues are Issue-ops-only and cannot enter the PR/branch flow (the `proposed-protect` hook, §6.1, blocks branch creation). `issue-reviewer` (author-side, at `/file-issue`) and `activation-reviewer` (observer-side, at `/activate`) are complementary, mirroring how `/file-directive` and `/activate` both gate the Directive path. The `awaiting-author` label (§1.7) marks the handoff state when the reviewer returned `revise` or a trusted-filer `reject` and the author's action is pending.

**Reviewer-gating contract** (per §1.5 operating-mode coupling):

- `attended` mode: human reviews `activation-reviewer`'s output and decides.
- `unattended` mode: `activation-reviewer`'s VERDICT line (`pass` / `revise` / `reject`) gates directly. `reject` on activation applies the filer-aware handling (§5.12: trusted → keep open + `awaiting-author`; untrusted + draft → close + auto-discussion; untrusted no-draft → close); `reject`/insufficient-evidence on completion leaves the Directive `Active`; `reject` on revision aborts the body replacement (no Status change, no audit `reconciled` line). `revise` retains `status:proposed` + `awaiting-author`, escalating to `reject` after N=3 rounds.

**Escape hatch.** `SKIP_HOOKS=directive-review SKIP_REASON='<why>' /complete-directive <id>` (and the equivalent on `/file-directive`, `/activate`, `/revise-directive`) bypasses the reviewer; the bypass is audit-logged per §7. Use is reserved for cases where the reviewer's verdict is wrong and a human accepts the recorded responsibility — not a normalized routing. `/block-directive` is not reviewer-gated (annotation-only — no body change) and therefore has no review-skip variant.

**Audit categories** for dir-mode events: `directive-file`, `directive-activate`, `activation` (the `/activate` category, generalized from `directive-activate`; the log retains historical `directive-activate` entries), `directive-complete`, `directive-revise`, `directive-block`, `directive-link`, `directive-reflect` (`/reflect` posts/skips), `directive-exec-count` (`/ship`'s post-merge directive-aware audit, §5.7), `directive-review`, `proposed-protect` (generalized from `directive-protect` by #171; the append-only log retains historical `directive-protect` entries), `triage` (decisions: `decided` with action=accept|reject, `refiled`, `rejected-skip`), `trusted-filer-mutate` (decisions: `deny`, `allow`), `label-parent-consistency` (decisions: `block`, `allow`; enforces label↔parent-marker consistency on `gh issue edit --add-label` — the `Parent Directive` marker for execution/task/bug (#199), plus the `initiative`/`directive` tier-key mutual-exclusivity and the Directive parent-XOR (#251), §6.1), `initiative-readonly` (decisions: `block`, `allow`; blocks mutating `gh issue edit`/`close`/`reopen` on an `initiative` Issue — the shell consumes Initiatives, never mutates them; #251, §6.1), `directive-close` (decisions: `block`, `allow`; blocks a close keyword + Directive `#N` in a PR body or commit message — auto-close would bypass `/complete-directive`; #490, §6.1), `initiative-consume` (decisions: `extracted` with count=N, `under-specified`, `revise`; emitted by `/consume-initiative` §5.21, #257), `initiative-feedback` (decisions: `challenge`, `completion`; emitted by `/initiative-feedback` §5.22, #260), `mirror-stale-warn` (reserved for `issues-to-project-mirror.yml` failures; currently emits nothing). All append to `.claude/audit/audit.jsonl` per §6/§7.

### 3.2 Injection mechanism

`scripts/clone-into.sh` runs the following right after `git clone`. **The target's existing `.claude/` content is never overwritten.** If a same-named asset already exists, that asset is not injected — only a warning is printed. The target's choice wins.

```bash
mkdir -p "$TARGET/.claude" "$TARGET/.claude/agents" "$TARGET/.claude/commands"

# Per-project binding symlink (no global env): hooks resolve the canonical
# shell root via $CLAUDE_PROJECT_DIR/.claude/eng-shell-root (see §3.2.1).
ln -sfn "$SHELL_ROOT" "$TARGET/.claude/eng-shell-root"

# settings.local.json: symlink to the shell's settings.injected.json (the
# target-facing settings whose hook commands resolve via the binding symlink;
# the shell's own settings.json stays $CLAUDE_ENG_SHELL_ROOT-based for dogfood).
if [ -e "$TARGET/.claude/settings.local.json" ] && [ ! -L "$TARGET/.claude/settings.local.json" ]; then
  echo "WARN: $TARGET/.claude/settings.local.json already exists. Shell settings not injected; user merge needed." >&2
else
  ln -sfn "$SHELL_ROOT/.claude/settings.injected.json" "$TARGET/.claude/settings.local.json"
fi

# agents/commands: skip and warn if a same-named asset exists.
for kind in agents commands; do
  for src in "$SHELL_ROOT/.claude/$kind"/*.md; do
    [ -e "$src" ] || continue
    dest="$TARGET/.claude/$kind/$(basename "$src")"
    if [ -e "$dest" ] && [ ! -L "$dest" ]; then
      echo "WARN: $dest already exists. Shell asset injection skipped." >&2
    else
      ln -sfn "$src" "$dest"
    fi
  done
done

# Add the shell injections (symlinks) to the target's .git/info/exclude so target git doesn't see them.
for entry in .claude/settings.local.json .claude/eng-shell-root; do
  grep -qxF "$entry" "$TARGET/.git/info/exclude" 2>/dev/null \
    || printf '%s\n' "$entry" >> "$TARGET/.git/info/exclude"
done
```

Rules:
- If the target has a same-named asset, the shell's version is not injected (conflict avoidance). If you want to use both, rename or move your own asset.
- Post-hoc registration of a directory the user already cloned: `scripts/register.sh <path>` runs the same inject logic.
- The target's `CLAUDE.md`, `MISSION.md`, `docs/` are **not touched**. The shell's CLAUDE.md (process norms) is loaded whole by Claude Code as project instructions each session — not hook-injected; UserPromptSubmit injects only `status_compact` (§6.4). As an always-on surface CLAUDE.md is held to the §9 thin-pointer + size budget.

#### 3.2.1 Hook command path resolution

There are **two** consumers of hook settings, and they resolve the shell root differently (#312, Directive #311):

1. **The shell repo itself (dogfood)** reads its own tracked `.claude/settings.json`, whose `command` entries use **`$CLAUDE_ENG_SHELL_ROOT/.claude/hooks/<e>.sh`**. `$CLAUDE_ENG_SHELL_ROOT` is exported by the `bin/claude-eng` wrapper (or a dev/CI env export). This path is unchanged and stays env-based.

2. **Injected targets** symlink `settings.local.json → shell/.claude/settings.injected.json`, whose `command` entries use **`${CLAUDE_PROJECT_DIR}/.claude/eng-shell-root/.claude/hooks/<e>.sh`**. Claude Code guarantees `$CLAUDE_PROJECT_DIR` (the project root) for hook commands; `.claude/eng-shell-root` is the per-project **binding symlink** (created at inject time, §3.2) pointing at the canonical shell root. So the target resolves the shell **without any global env var** — a plain `claude` launched in the target works. This **supersedes** the prior rule (which required `$CLAUDE_ENG_SHELL_ROOT` and noted that bare `$CLAUDE_PROJECT_DIR` "would resolve to an empty location and fail") — the binding symlink is exactly what makes `$CLAUDE_PROJECT_DIR`-relative resolution succeed.

```json
// shell/.claude/settings.injected.json (target-facing)
{ "hooks": { "PreToolUse": [{ "matcher": "Bash",
  "hooks": [{ "type": "command",
    "command": "${CLAUDE_PROJECT_DIR}/.claude/eng-shell-root/.claude/hooks/pre_tool_use.sh" }] }] } }
```

**Hook self-location.** Independent of which consumer invoked them, the hook entry scripts derive their root as `SHELL_ROOT="${CLAUDE_ENG_SHELL_ROOT:-$(cd "$(dirname "${BASH_SOURCE[0]}")/../.." && pwd -P)}"` — the env var wins when set (back-compat); otherwise the script self-locates from `${BASH_SOURCE[0]}`, and `pwd -P` resolves the `eng-shell-root` symlink through to the real canonical root so helpers/`hookrt.sh`/audit still resolve. This is what lets the target path in (2) find the shell's code.

`$CLAUDE_ENG_SHELL_ROOT` is exported by the `bin/claude-eng` wrapper on entry:

```bash
#!/usr/bin/env bash
set -euo pipefail
export CLAUDE_ENG_SHELL_ROOT="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd -P)"
exec claude "$@"
```

Inside hook scripts, shell-side **code** (helpers, hooks) is accessed via `$CLAUDE_ENG_SHELL_ROOT`, while target files and **per-project ephemeral state** are accessed via `$CLAUDE_PROJECT_DIR` — keep code (shared) and state (per-project) cleanly separated (§3.2.2).

#### 3.2.2 Per-project ephemeral state (`eng-state/`)

Ephemeral runtime assets — the **audit log**, the `.claude/state` **caches** (issue-type, issue-filer, status, pr; the park log), and the scope-guard **registry** — resolve per project under `eng-state/` (#314 + #316, Directive #311). The audit log and caches resolve via `eng_state_dir()`; the registry resolves via `eng_registry_file [project_dir]`; both live in `hookrt.sh`.

`eng_state_dir()` resolution order (`set -u`-safe, no external calls): `ENG_STATE_DIR_OVERRIDE` (test seam) → `${CLAUDE_PROJECT_DIR}/.claude/eng-state` when `CLAUDE_PROJECT_DIR` is set (Claude Code guarantees it for hook commands) → otherwise **empty**, in which case callers fall back to the legacy shared `$CLAUDE_ENG_SHELL_ROOT/.claude/{audit,state}`.

`eng_registry_file [project_dir]` serves two execution contexts with a single definition (#316):
- **Argless** (hook context — `cwd_guard.in_scope`/`path_in_scope`): defers to `eng_state_dir()` → `${esd}/registry.txt`, else the legacy shared `${CLAUDE_ENG_SHELL_ROOT:-}/.claude/state/registry.txt`.
- **With an explicit project-dir argument** (launcher / CLI context — `bin/claude-eng`, `register.sh`/`inject`, `self_register`, `dr_check_registry_guard` — where `CLAUDE_PROJECT_DIR` is unset because the call *precedes* the Claude session): `${1}/.claude/eng-state/registry.txt`.

The caller passes the project root it already holds (the `<dir>` argument or `pwd -P`), so registry discovery becomes "does `<dir>` carry its own `eng-state/registry.txt`?" — a **self-describing per-project registry**, not a lookup in a shared cross-project index. Under Directive #311's model there is no longer a central authority listing every project.

**Back-compat read-floor.** Every *reader* — `cwd_guard.in_scope`/`path_in_scope` (hook context) and `dr_check_registry_guard`/`bin/claude-eng` (CLI context) — when the per-project registry file is **absent**, falls back to *reading* the legacy shared `$CLAUDE_ENG_SHELL_ROOT/.claude/state/registry.txt`. This keeps a target registered before #316 (legacy entry only, no `eng-state/registry.txt` yet) enforcing on its next session, honoring the #311 "existing setups keep working" constraint. The fallback is **read-only** and triggers **only when the per-project file is absent** — once a target is registered the new way (per-project file present), the shared registry is never consulted for it, so a migrated target gets strict isolation. Writers (`inject`, `self_register`) always write per-project, so write-isolation (success signal 2) is unaffected; and because `in_scope` only ever matches the *current* path against the registry, reading the shared registry as a floor cannot leak another project's scope into the current session.

Consequence: in hook context two projects bound to the same canonical code source have **mutually-invisible** audit/caches **and registries** (writing audit/registry/state in project A leaves project B's `eng-state/` unchanged); in plain-Bash context (env unset, no arg) behavior is unchanged (back-compat — the legacy shared paths remain the floor). This realizes the §1.7 / MISSION principle that the **codebase is shared** while **ephemeral state isolates per project**. `eng-state/` is added to the target's `.git/info/exclude` (the shell repo excludes its own `eng-state/` via its tracked `.gitignore`).

**Fail posture (security).** The registry gates the `out-of-scope` enforcement matcher (§6.1). A missing/empty registry yields `in_scope=false` → hooks pass through transparently (**fail-open**), unchanged from the pre-#316 shared-registry behavior — the move changes only *which path* is read, never the `[ -f "$registry" ] || return 1` posture or the #218 trailing-slash normalization. Because the resolver is `set -u`-safe, an unset `CLAUDE_ENG_SHELL_ROOT` (the #312 self-located case) cannot abort the guard. Dogfood coherence: `self_register` writes through `eng_registry_file "$SHELL_ROOT"`, so the shell's registration write-target equals the path `cwd_guard` reads under `CLAUDE_PROJECT_DIR=$SHELL_ROOT` — a mismatch would silently disable enforcement inside the shell repo (§3.6).

### 3.3 Scope guard — cwd-based

All hook scripts prelude:

```bash
# Match physical PWD against registered (physical) paths in the registry.
# The registry is written by register.sh / clone-into.sh; clones inside workspace/ are recorded the same way.
# It resolves per-project via eng_registry_file (hookrt.sh, #316): argless = hook
# context (CLAUDE_PROJECT_DIR), set -u-safe, legacy shared path as the floor.
REGISTRY=$(eng_registry_file)
PWD_REAL=$(cd "$PWD" 2>/dev/null && pwd -P) || exit 0
[ -f "$REGISTRY" ] || exit 0
in_scope=0
while IFS= read -r entry; do
  [ -z "$entry" ] && continue
  case "$PWD_REAL/" in "$entry"/*) in_scope=1; break ;; esac
  [ "$PWD_REAL" = "$entry" ] && { in_scope=1; break; }
done < "$REGISTRY"
[ "$in_scope" = "1" ] || exit 0
```

→ If `claude-eng` is invoked from an unregistered directory, hooks pass through transparently. Works regardless of logical vs physical `$PWD` (we use PWD_REAL). External path registration is covered by the same mechanism.

**Fail-open contract.** All three guard exits — an unresolvable `PWD_REAL` (`cd … || exit 0`), an absent or empty registry (`[ -f "$REGISTRY" ] || exit 0`), and an out-of-scope cwd (`[ "$in_scope" = 1 ] || exit 0`) — fail **open**: a missing/empty/unreadable registry or an unregistered cwd means enforcement is **off**, never on. This is the guardrail-not-sandbox framing (§6.1): the shell narrows what a careful engineer would not do *inside a registered project*, and is silent everywhere else; a registry that cannot be resolved can only under-enforce (pass through), never lock the user out. (The escape hatch, §7, and the helper fail-policy table, §6.1, follow the same fail-open posture.)

The registry (per-project `.claude/eng-state/registry.txt`, or the legacy shared `.claude/state/registry.txt` outside hook context) format: one physical path (`pwd -P` output) per line. Comments and blank lines are ignored.

### 3.4 User-global isolation

The shell's installation and operation leave no changes in the user-global environment. The only way to "disable" the shell is to stop invoking `claude-eng`. No deactivation procedure or cleanup script is needed.

**Surfaces never touched**:
- `~/.claude` (user-global Claude settings)
- shell rc/login files (`~/.zshrc`, `~/.bashrc`, `~/.bash_profile`, `~/.profile`, `~/.config/zsh/*`, `~/.config/fish/*`, etc.)
- global git config (`~/.gitconfig`, `~/.config/git/*`)
- PATH auto-registration — the user adds `<shell-root>/bin` to PATH or uses absolute path / alias themselves. The shell never injects an export line into rc files.
- LaunchAgents, systemd user units, login items, or any auto-start registration

`bootstrap.sh` boundary:
- Allowed: check dependencies (`gh`, `git`, `python3`, `jq`), run `gh auth status`, create `.claude/audit/` and `.claude/state/` inside the shell repo.
- Forbidden: any modification of the surfaces above, package manager invocations, sudo.

Target repos are the shell's work — they fall outside this isolation rule (the injection in §3.2 is intended behavior).

### 3.5 Entry point

`scripts/setup.sh <local-path | repo-url>` is the single host-side orchestrator: it runs `scripts/bootstrap.sh` (deps check) → `scripts/register.sh` or `scripts/clone-into.sh` (the register/clone arg contract below) → `scripts/lib/onboard_checks.sh` (onboard pre-flight) → an optional dir-mode gate (`scripts/onboard_target.sh`, default N), then prints the next command (or, with `--enter`, execs `claude` directly). Reachable by relative path, so the `claude-eng` PATH export is optional — only needed to invoke `claude-eng` from an arbitrary directory.

`claude-eng` is meant to start at the root of `workspace/<repo>/`. Subdirectories work too (walk-up), but starting at the root avoids edge cases in monorepos where subprojects may carry their own `.claude/`.

**External path support**: a target repo can live outside the shell's `workspace/` (e.g. `~/code/<repo>`). `scripts/register.sh <abs-path>` (1) creates a `workspace/<basename>` symlink, (2) records the target's physical path (`pwd -P`) in the target's per-project `.claude/eng-state/registry.txt` via `eng_registry_file "$target"` (#316), (3) runs the inject logic of §3.2. The shell's `.gitignore` whitelists `workspace/*` so the shell git tree doesn't pick up target trees even in the external-path case. Direct clones into `workspace/` are also recorded in the registry — the guard is always registry-based.

**Auto-register prompt**: when `claude-eng <path>` is called with an unregistered external path, prompt once before entering the session:
```
$ claude-eng ~/code/foo
? '/Users/.../code/foo' is not registered. Register as workspace/foo and enter? [Y/n]
```
yes → run `scripts/register.sh` → inject logic (§3.2) → start session. no → exit.

**Monorepo subprojects**: clone-into injects **only at the repo root**. Subprojects with their own `.claude/` are not recursively injected. When the user starts `claude-eng` in a subproject, the walk-up finds the root's shell injection. The subproject's `.claude/CLAUDE.md` is merged via Claude Code's default walk-up behavior.

**Discovery surface** (Issue #318, Directive #311): `claude-eng list` prints the registered target projects, one resolved physical path per line, then exits 0. Like `--version` it short-circuits **before** the cwd registry guard (§3.3), so it works from any cwd including unregistered ones, and has no side effects (no audit, no registry mutation, no cwd change). Enumeration is **advisory, not authoritative** — post-#316 the registry is self-describing per-project (§3.2.2), so there is no single global index. `list` unions two sources and dedups by resolved absolute path: (1) the `workspace/*` symlinks (`register.sh` creates one per registration — the most complete set), resolving each and **skipping dangling** links; (2) the legacy shared `$CLAUDE_ENG_SHELL_ROOT/.claude/state/registry.txt` (catches pre-#316 + dogfood entries). A target injected without a `workspace/` symlink, or whose symlink was removed, is **invisible to `list` by design** — this incompleteness is inherent to the shared-code/per-project-state model, not a bug.

**Binding-health surface** (Issue #334, Directive #311): when `claude-eng <dir>` enters an injected target (`<dir>/.claude/settings.local.json` is a symlink) whose binding `<dir>/.claude/eng-shell-root` is **missing or dangling**, it emits one non-blocking stderr warning naming the repair (`scripts/register.sh <dir>`) before `cd`-ing in and `exec claude`. This is the launch-time detector for the broken-binding silent-no-op that the #318-removed SessionStart banner structurally could not surface (the SessionStart hook is itself invoked *through* the binding — §6.5(c)); `claude-eng` runs *before* the session and resolves the dir directly, so it is the one place that can warn. Healthy bindings and non-injected dirs stay silent; the check has no side effects beyond the warning.

**Version surface** (Issue #123, Directive #122): `claude-eng --version` reads the top-level `VERSION` file (single line, semver 0.x format — `MAJOR.MINOR.PATCH` with `MAJOR=0` per Directive #122 constraint #1) and exits 0 with that string on stdout. If `VERSION` is missing or empty, the handler falls back to `git -C "$SHELL_ROOT" describe --tags --always`; if both fail, a one-line diagnostic is printed to stderr and the process exits 1. The `--version` short-circuit runs immediately after `SHELL_ROOT` resolution and before the cwd-based registry guard (§3.3), so it works from any cwd including unregistered paths. The flag has no side effects: no audit log, no registry mutation, no cwd change. Tags follow `v` + semver (e.g., `v0.2.0`) and are pushed manually post-merge — `gh pr merge` does not push tags, and the merge gate does not assert tag-state freshness (explicit Directive #122 non-goal).

### 3.6 Shell self-registration

The shell repo itself must appear in its own registry so that `cwd_guard.in_scope` (§3.3) returns true when `claude-eng` is invoked from the shell root. Without it, the shell's own hooks short-circuit on `in_scope || exit 0` and the dogfooding promise of §17 is empty — the shell can't enforce its own discipline on itself. Self-registration writes through `eng_registry_file "$SHELL_ROOT"` → `$SHELL_ROOT/.claude/eng-state/registry.txt` (#316), which is exactly the path `cwd_guard` reads in a dogfood session (`CLAUDE_PROJECT_DIR=$SHELL_ROOT` → argless `eng_registry_file` → the same file). Write-target == read-target; a divergence would silently fail-open the shell's own scope guard.

`scripts/bootstrap.sh` (and `scripts/register.sh "$SHELL_ROOT"`) ensure `$SHELL_ROOT` is present in the registry (idempotent — repeated runs do not duplicate). Self-registration is **registry-only**: no `workspace/<basename>` symlink and no `inject_into` are performed against the shell, because the shell IS the injection source and the symlink would loop into `workspace/` contained within `$SHELL_ROOT`.

`scripts/register.sh "$SHELL_ROOT"` (or any path that physical-resolves to the shell root) recognizes this case and:
- skips the `workspace/<basename>` symlink (would otherwise create a directory loop);
- skips `inject_into` (the source files are the destinations);
- still ensures the registry entry exists.

For target repos this is unchanged: §3.2 inject + registry entry, both performed by `clone-into.sh` / `register.sh` as before.

### 3.7 Context lifecycle: flush → clear → reconstruct

The injection mechanism (§3.2) and its runtime surfaces (§6.4 per-turn status, §6.5(b) session restore) are the *inbound* half of the MISSION dual — selective injection pulls durable memory into the working context. This section names the full loop they presuppose: **flush → clear → reconstruct**, the operational form of MISSION's `active → archived, not active → deleted` invariant.

A context **clear** is the bluntest narrowing tool: it discards the entire working context. Done naively it is `active → delete` — task-scoped state that lived only in the conversation (decisions taken, the current phase, what the next step was) is lost. The MISSION invariant requires it instead be `active → archived`: the active surface contracts, but the artifact persists in durable memory for later selective re-injection. Three steps make that hold:

1. **Flush** (the gap this section closes) — *before* a clear, distill task-scoped session state into durable memory. The flush is **explicit / skill-invoked** (`/flush`, §5.24), not automatic. What it distills is the small, reconstructable core: the current branch, issue, PR, phase, and the next intended step — the same field set `status_compact` (`helpers/status.sh`) projects. Where it writes (durable-memory targets, in priority order):
   - the **PR body** — curated per the PR-as-living-doc rule (§1.4), the richest task-scoped artifact when a PR exists;
   - a record under **`.claude/state/`** — the always-available target (a branch may have no PR yet), carrying the `status_compact` field set;
   - **GitHub artifacts** generally (issue/PR comments) for anything narrative that belongs on the durable record.

   The flush writes *at least one* durable artifact, so that after the clear the state is recoverable rather than gone.

2. **Clear** — the working context is discarded. The shell does **not** and **cannot** intercept Claude Code's native `/clear` (it is outside the shell boundary, §3.4); `/flush` therefore *precedes* a clear and makes it safe, rather than replacing or wrapping it. The clear-trigger contract is **explicit only** in this revision: the operator (or a skill) decides when to clear. **Threshold-based / automatic triggering is deferred** (§0.4) — both because the shell cannot hook the native clear, and because a wrong automatic clear has a high cost-asymmetry (§6.0): a clear is irreversible within the session.

3. **Reconstruct** — on the next session, **SessionStart session restore (§6.5(b))** re-injects the durable memory: the current branch's full PR body, the referenced issue body, the MISSION summary, and one `/status` snapshot (CLAUDE.md is natively loaded whole, not re-injected here); §6.4 then keeps the per-turn surface narrow. This is the *already-built* inbound half — this section adds no new reconstruct path, deliberately, to avoid a parallel injection surface that could drift from §6.5(b).

**Known gap (noted, not closed here):** §6.5(b) restore re-injects the PR body, the issue body, and a `/status` snapshot — it does **not** currently read the `.claude/state/` flush record written in step 1. So when a PR exists, reconstruct is complete; on a branch with no PR yet, the `.claude/state/` record is written (the archive exists) but is not auto-re-injected at SessionStart. Closing that gap — teaching §6.5(b) restore to read the flush record — is a follow-up, kept out of this revision to hold the change to one section + one affordance.

**Discard-boundary = Execution** (Directive #477). The flush → clear → reconstruct loop is the *session-level* application of the MISSION `active → archived` invariant; the **discard boundary at the Execution unit** (§1.2) is the *within-task* application. The within-Execution authoring churn (the Code phase's file reads, abandoned approaches, lint/smoke iterations) is `active → delete`-able once the Code commit lands — it was never durable — while **directive-level memory is preserved** (`active → archived`, the cross-Execution context a Directive carries). The `implementer` subagent (§4.12) + `/implement` (§5.28) enforce the boundary structurally: Phase C runs in the subagent's ephemeral context and only the structured result (commit + plan-deviations + discoveries) crosses back, so the churn is discarded at the Execution edge rather than flushed. This is the **default Code-phase route** (Directive #477 signal-4: the formal A/B measurement was descoped, the narrowing hypothesis adopted as operating policy on circumstantial evidence), with a **documented opt-out** for trivial / glue edits that stay in the main loop and **fail-open reversibility** as the safety valve; preserving directive-level memory across Executions is the EI-2 coding-memory loop (signal 3, closed).

## 4. Subagent specifications

Each agent is defined in `.claude/agents/<name>.md` as YAML frontmatter + body.

### 4.1 planner

**When**: before any code edit. Required for 3+ file changes, migrations, schema changes, API changes, or external contract changes.

**Input**: user request, linked issue body, MISSION.md, target CLAUDE.md.

**When MISSION is absent**: planner proceeds with `issue body + README + CLAUDE.md` alone. The "How this serves the mission" section is labeled `(MISSION.md absent — review /onboard stub suggestion)`. Don't synthesize a MISSION via inference.

**Output** (markdown that drops straight into the PR body):

```
## How this serves the mission
<Which MISSION.md item this contributes to>

## Plan
<Why this approach, trade-offs, decision rationale>

## Target base
<branch name; default `main`. For non-`main`, name the topic-branch / feature-branch the PR will merge into. See SPEC §10.5.>

## Alternatives considered
- <Alternative 1>: <one-line why-not>
- <Alternative 2>: <one-line why-not>
(If the choice is forced — single viable approach — say so explicitly with a one-line justification rather than fabricating alternatives.)

## Key context
<related file:line>

## Checklist
- [ ] **Doc**: <doc update item 1>
- [ ] **Doc**: <doc update item 2>
- [ ] **Test**: <failing test item>
- [ ] **Code**: <implementation item>
- [ ] **Code**: <implementation item>

## Out of scope
<What this PR will NOT do>

## Risks
<Compatibility, performance, data, security>

## Open questions
<To be decided before proceeding>
```

Checklist order: **Doc first, Test next, Code last**. Granularity: one item ≈ one commit ≈ 30 min to 2 hours.

**Alternatives section is mandatory** — even when the choice feels obvious. A "no alternatives — forced choice" entry with a one-line justification is acceptable; an empty section is not. The section exists so reviewers see what was rejected and why, and so future-you can revisit the trade-off without re-running the analysis.

### 4.2 explorer

**When**: locating code, symbol lookups, "where is X defined", "who calls Y" — read-only wide search. Protects the main context.

**Constraints**: no code edits. No dumping large files. Always summarize.

**Output**:
- Definition location (`path:line`)
- Up to 5 main references (`path:line` + one-line context)
- Related modules
- If nothing found: say "not found" and suggest next searches.

### 4.3 doc-writer

**When**: Phase A. Any change that alters external surface.

**Responsibilities**:
- Identify affected SSOT docs (MISSION, README, CLAUDE.md, ARCHITECTURE, ADR, API doc).
- Patch each doc preserving its existing format and tone.
- For absent docs, propose a minimal stub only after user confirmation.
- When an irreversible decision is detected, recommend writing an ADR (`docs/ADRs/NNNN-title.md`).

**Forbidden**: inventing target SSOT content (humans must review and sign off). Only stub proposals when absent.

### 4.4 test-writer

**When**: Phase B, right after doc.

**Responsibilities**:
- Take Phase A docs / spec as input; write a failing test.
- Reuse existing test patterns, tools, fixtures.
- Cover intended behavior + at least one boundary / exception / error path.
- Run it and confirm it **fails** (intentionally). If it passes, either Phase A is wrong or the test is too weak.

**Principles**:
- Suspect over-mocking.
- One test = one assertion.
- If tooling is absent (e.g. pytest not installed), report and stop. Don't guess.

### 4.5 code-reviewer

**When**: before commit / PR. Auto-invoked by `/review`, `/ship`.

**Premise**: assume no knowledge of the main assistant's discussion or reasoning. Input is:
- The full PR diff. Prefer `gh pr diff <num>` (or `--web` equivalent) over a literal `git diff main...HEAD` — the PR's `baseRefName` may be non-`main` for topic-branch / experimental work (§10.5), and `gh pr diff` honors that automatically. The principle is "review what the PR actually changes against its declared base," not "everything since main."
- Read-only adjacent context for changed files (for call-graph tracing, as needed)
- Target `MISSION.md`, referenced issue body (for MISSION fit + acceptance criteria evaluation)
- Full PR body (for declared plan + checklist verification)

Nothing outside this input.

**Check**:
- Consistency (coherent changes, missing adjacent callers)
- Tests (alignment with Phase B, regression risk)
- Error handling (at boundaries only, no defensive code creep)
- Security surface
- Obvious performance traps
- Readability, naming
- Scope (extraneous changes?)
- **Doc sync (Phase A reflected?)**
- **MISSION fit (contributing to or violating which item?)**
- **Issue acceptance criteria met?**
- **Enforcement-style (§6.0)** — *only for diffs that add/change a hook, gate, matcher, or standing guidance*: a new block must name its positive alternative and the negative/positive face must match the cost-asymmetry (§6.0 P1 + P4); a bare block or gateless guidance is a one-sided regression. (n/a for diffs touching no enforcement surface.)

**Output**: ship / ship after fix / block. Each finding cites `path:line`.

### 4.6 security-reviewer

**When**: PRs touching auth / authz / sessions / external input / new deps / crypto / hashing / randomness / new IO boundaries.

**Check areas**: authentication / authorization, injection (SQL / shell / HTML / path), sensitive data exposure, weak crypto, CSRF / CORS / headers, dependency risk.

**Output**: High / Medium / Low / Info. Each finding: risk + exploit scenario + remediation.

### 4.7 issue-reviewer

**When**: invoked by `/file-issue` before `gh issue create`. Validates the rationale triad of SPEC §5.2 — every issue passes through this gate.

**Input**:
- Proposed issue body (title + body + label).
- Target `MISSION.md` (if absent, label as `(MISSION.md absent — review /onboard stub suggestion)`).
- Open-issues list (`gh issue list --state open`).

**Check**:
- **MISSION fit**: does the body cite a specific MISSION item? "It's general improvement" or "tidy" answers fail.
- **Why now**: does the body explain what changes if this waits? Vague "would be nice" fails.
- **Existing-coverage**: scan open-issues / open-PRs; does another issue cover this, or would a comment on an open PR suffice? If yes → suggest the cheaper path.
- **Acceptance criteria**: explicit, testable, ≥1 item.
- **Enforcement-style fit (§6.0)** — *only when the proposed issue adds/changes a hook, gate, matcher, or standing guidance* (else n/a): the negative/positive face must match the cost of being wrong (§6.0 P1) and a proposed block must name its alternative / proposed guidance must have a gate behind it (§6.0 P4); a mismatch or one-sided proposal → `refine`.

**Output**: one of three verdicts on stdout's last line, each with a one-line reason:
- `ship` — file as-is.
- `refine: <one-line why>` — return for body revision before filing.
- `block: <one-line why>` — don't file at all (existing-coverage hit, scope problem, or non-MISSION work).

Inside `attended` mode the verdict surfaces to the user; inside `unattended` mode it gates the next step directly. Hook into `attended` is non-skippable — Auto mode does not suppress this gate (SPEC §5.2).

### 4.8 plan-reviewer

**When**: invoked by `/work-on` between `planner` and the plan-approval step. Validates that the alternatives section is real and that the chosen approach actually beats them.

**Input**:
- `planner` output (markdown).
- Issue body (the one the planner read).
- Target `MISSION.md`.

**Check**:
- **Alternatives section is real**: ≥1 alternative with a substantive why-not, or an explicit "no alternatives — forced choice" with a justification. "Just because" or empty fails.
- **Chosen approach beats alternatives**: read the "Plan" rationale; does it engage with the alternatives it lists, or just describe the chosen path?
- **Scope hygiene**: items in `Checklist` map to `Acceptance criteria` of the issue; anything orphaned belongs in `Out of scope` or a follow-up issue.
- **Phase-order sanity**: Doc → Test → Code as per SPEC §1.2; relaxation reasons noted if applicable.
- **Target base fit**: the `Target base` field (§4.1) names a branch. If non-`main`, the plan should engage with the topic-branch's constraints (see §10.5). A `main`-default plan that quietly proposes work belonging on an experimental base — or vice versa — fails this check.
- **Enforcement-style fit (§6.0)** — *only when the plan introduces/changes a hook, gate, matcher, or standing guidance* (else n/a): the negative/positive face must match the cost of being wrong (§6.0 P1), a planned block must name its alternative / planned guidance must have a gate (§6.0 P4), and an advisory-skip where §6.0 P3 calls for proving-out is itself a `refine`.

**Output**: same three verdicts as `issue-reviewer`. `refine` causes `/work-on` to re-invoke `planner` with the feedback; `block` stops with a comment on the issue naming the structural problem.

### 4.9 Director subagents

The dir-mode workflow (§1.7) uses one reviewer subagent (`activation-reviewer`, §4.9.1). A second (`roadmap-reviewer`, §4.9.2) is documented for future activation.

#### 4.9.1 activation-reviewer

**When**: invoked by `/file-directive` (proposed Directive body, §5.10), `/activate` (re-check on the possibly-edited body before Proposed→Active, and on unblock, §5.12), `/revise-directive` (the new body before replacement, §5.16), and `/complete-directive` (completion claim + linked-Execution-Issue evidence, §5.13); and — via the Initiative rulebook — by `/consume-initiative` (contract-evaluability at Initiative intake, extraction-faithfulness on the proposed Directive set, §5.21). Mirrors the structure of `issue-reviewer` (§4.7) and `plan-reviewer` (§4.8) — review only, never authors content.

**Type-label dispatch**: `activation-reviewer` is **type-neutral** — it resolves the reviewed Issue's type label and applies the matching rulebook. The `directive` label present → the Directive checks below. The `initiative` label present (#254, M2) → the **Initiative rulebook** (contract-evaluability — the termination condition must be evaluable without code knowledge; and extraction-faithfulness — extracted Directives decompose the termination condition without smuggling strategy); the reviewer never activates or mutates an Initiative (the shell consumes them), it judges the Initiative/extraction as input for the M3 consume flow. Absent (an Execution Issue, `task`/`bug`) → an Execution rulebook (schema completeness, acceptance-criteria verifiability, scope clarity, MISSION fit, duplicate coverage; **parent-fit is gated by label** — applied only to the `execution` type, while `task`/`bug` are standalone per the §1.7 `execution`/`task`/`bug` type-label definitions and skip the parent checks and the Parent-mismatch matrix). For a Directive, the **alignment check branches on parent kind** (#254): a `## MISSION fit` MISSION-section parent is judged against MISSION; a line-1 `Parent Initiative: #N` parent is judged against that Initiative's termination condition (the parent-XOR guarantees exactly one). `activation-reviewer`'s verdict vocabulary is `pass` / `revise` / `reject` (Issue #172; these replaced the former `ship` / `refine` / `block` 1:1), uniform across both rulebooks — only the *checks* differ by type. The structured refile fields and the type-mismatch / parent-mismatch matrices apply on `reject`. (`issue-reviewer` §4.7 and `plan-reviewer` §4.8 retain `ship` / `refine` / `block` — their contracts were out of #172's scope.)

**Input**:

- **For proposal review:** the proposed Directive body (objective, success signals, non-goals, constraints, and the parent — a `## MISSION fit` field **or** a line-1 `Parent Initiative: #N` marker, §1.7), plus the list of currently `Status=Active` Directives (via `gh issue list --label directive --label '-status:proposed' --state open` — Issues are the SSOT; the Project is a lagging derived view and is NOT consulted, §1.7/§5.11).
- **For completion review:** the Directive's body (success signals as written at activation), the list of linked Execution Issues (Issues with `Parent` = this Directive), and their close states / AC ticks.

**Checks**:

- **Schema completeness** — does the body cover objective / success signals / non-goals / constraints / MISSION fit (or, for an Initiative-parented Directive, a `Parent Initiative: #N` marker)? Missing any → `revise`.
- **Parent-kind alignment** (#254) — a Directive has exactly one parent kind; judge alignment against it: a `## MISSION fit` parent against the named MISSION section, a line-1 `Parent Initiative: #N` parent against that Initiative's termination condition. Both-or-neither parent kind → `revise` (the parent-XOR, §1.7). The full per-check detail is in the contract SSOT `.claude/agents/activation-reviewer.md` (check 1.5).
- **Success-signal verifiability** — are signals objectively testable, or vague aspiration? "Engineering reviews go faster" without a metric → `revise`.
- **Scope clarity** — is the objective bounded? "Improve the codebase" → `reject`.
- **Non-goal clarity** — are exclusions explicit or implied? Vague exclusions invite scope creep.
- **Active-Directive conflict** — does the proposed Directive overlap with an existing `Status=Active` Directive? If yes → `revise` or `reject` with a pointer.
- **Evidence sufficiency (completion only)** — do the linked Execution Issues collectively satisfy each success signal? A signal with no linked Execution Issue → `revise` ("need engineering evidence"). A signal whose Execution Issues are open or stalled → `revise`.

**Output** — three verdicts: `pass` / `revise` / `reject` (introduced by #172; these replaced `ship` / `refine` / `block` 1:1). On `reject` the verdict additionally carries the structured refile fields (`verdict` / `reason` / `refile-target-type` / `refile-target-parent` / `refile-body-draft`) and follows the type-mismatch / parent-mismatch matrices. The full output contract — verdict meanings, the `revise`-vs-`reject` boundary rule, the structured fields, both matrices, the `<!-- activation-verdict: <verdict> -->` marker, and the N=3 `revise`→`reject` escalation — is specified in `.claude/agents/activation-reviewer.md` (the contract SSOT). Callers apply the post-verdict handling per §2.1 (Directive lifecycle) and §5.12 (`/activate`).

**Rules**: do not author body content for a `revise` (matches §4.7 rule; the one exception is the `reject` `refile-body-draft` field — drafting there IS the preserve signal). Verdict is content-only; filer identity affects only the caller's post-verdict handling. Judge from the inputs above only. Auto-approving without scanning active-Directive conflicts is a regression against the §1.5 reviewer-pattern integrity.

#### 4.9.2 roadmap-reviewer (deferred)

Cross-Directive coherence check — MISSION alignment, Directive sequencing, drift detection. Deferred until operating experience surfaces the friction; see MISSION's "Success looks like" → orchestrator criterion.

#### 4.9.3 Session-restart caveat for new reviewer subagents

Claude Code enumerates available `subagent_type` values **at session start** by scanning `.claude/agents/*.md`. A reviewer added mid-session is **not routable** as a first-class subagent until the next session start: `Agent({subagent_type: "<new-name>"})` falls back to `general-purpose`, which loses the Type-aware specialized prompt.

This caveat is the parallel of §6.1's "Session-restart caveat for hook-spec changes" — Claude Code binds its routing surfaces (hook configuration AND subagent enumeration) at session start. Both bindings are stable for the session's duration; both are picked up automatically at the next session.

Empirical confirmation (Directive #62 / PR #67-followup): in the dogfood session, `.claude/agents/directive-reviewer.md` landed via PR #50 mid-session (this reviewer was later renamed to `activation-reviewer` by Issue #170 under Directive #167; the historical filename is retained here because it records the actual PR #50 event). Every subsequent `Agent({subagent_type: "directive-reviewer"})` invocation routed through `general-purpose` because the harness's `subagent_type` enumeration had been frozen against the pre-#50 state. A session restart resolves this automatically — no `.claude/settings.json` registration needed, no harness-config change. The same mechanism applies to the Issue #170 rename itself: until a session restart, `subagent_type: "activation-reviewer"` falls back to `general-purpose`.

**Mitigation for the in-session window**: the falling-back-to-`general-purpose` path is functionally complete — the agent's prompt instructs `general-purpose` to behave as the new reviewer. The cost is the loss of Type-aware specialization (the harness's per-subagent prompt + tool restriction). Acceptable for one session; restart at session end is canonical.

**Hook-side note**: the smoke `§42` structural assertions (PR #50) lock the agent file's shape, so the routing-layer regression is the only failure mode this caveat names. The behavioral surface that catches the routing regression is **smoke §42e** (issue #69, under Directive #62; extended for the type-neutral reviewer by #170): three assertions gated behind `CLAUDE_ENG_BEHAVIORAL_SMOKE=1` invoke `activation-reviewer` with a synthetic minimal-but-valid Directive body (asserting `^VERDICT: pass`), a synthetic Directive body missing the `## Success signals` section (asserting `^VERDICT: (revise|reject)`), and a synthetic minimal-but-valid Execution Issue body exercising the Execution-rulebook dispatch (asserting `^VERDICT: pass`). Env-var unset → §42e is a no-op so default smoke stays deterministic and offline; env-var set → the block exercises the live `subagent_type: "activation-reviewer"` route end-to-end. A regression in either the agent file or the routing surface flips at least one of the three verdicts.

> **§4.10 triage classifier — retired (#173, Directive #167).** The former triage classifier agent was deleted: its `status:proposed` substantive gate is now `activation-reviewer` (§4.9), and its template-shape classification is absorbed by `/activate`'s type-mismatch matrix. `/triage` survives only as a one-cycle deprecation alias (see §5.12). The `triage` audit category is retained in §2.1 as append-only history. (The §4.10 number is retired and not reused; the high-asymmetry reviewer tier below is **§4.11**.)

### 4.11 High-asymmetry reviewer tier (Issue #428)

A cross-cutting reviewer **tier**, not a subagent: where a wrong *approve* is irreversible or corrupts shared history, a single reviewer's verdict is replaced by an **N-way independent vote**. This is the reviewer-side counterpart of §6.0 P1 — the same cost-asymmetry that makes the protected-branch / secret matchers hard negative gates makes these *approvals* warrant redundancy. It is **blast-radius-gated, never a default**: N-way everywhere would bloat the acting context (the narrowing principle, MISSION) and correlate votes, so the common path stays single-reviewer. It extends the existing conditional dispatch (`/ship` step 2 already spawns `security-reviewer` only when the diff touches a security surface).

**Enumerated high-asymmetry decision set** (closed — `is_high_asymmetry <kind>` in `helpers/blast_radius.sh` returns rc 0 for exactly these, rc 1 otherwise):
- `merge-security-surface` — `gh pr merge` to the default branch of a PR whose diff touches a security surface (the §4.6 trigger), at `/ship`.
- `force-push` — a force-push / history-rewrite judgment surfaced *inside the `/ship` motion* (the §13 rebase-pull tail or a named-branch escape). This is the **reviewer judgment** when a force-push is legitimately part of shipping — distinct from the autopilot force-push **negative gate** (§6.1), which still blocks unreviewed force-pushes independently.
- `directive-completion` — the `/complete-directive` evidence evaluation (§5.13).
- `irreversible-adr` — merging a PR carrying an ADR (`docs/ADRs/`) to the default branch, at `/ship`. ADR *authoring* via `/adr` is a reversible doc-write (the file is editable until merged); the irreversible point is the merge, so the tier fires there, not in `/adr`.

**Mechanism — N=3, majority=2, independent.** For a high-asymmetry decision the invoking skill spawns the relevant reviewer type (`security-reviewer` for `merge-security-surface`/`force-push`, `activation-reviewer` for `directive-completion`, `code-reviewer` for `irreversible-adr`) **three times as independent fresh subagent invocations** — worktree-isolated, artifact-only (the §4.5 "no knowledge of the main assistant's discussion" premise), with **no shared context** between the three. **Independence is load-bearing**: redundancy de-risks only if the votes don't correlate through a shared prime — a serial debate where each reviewer sees the others' verdicts is explicitly *not* this (it is the rejected design). A **majority (2 of 3)** approve-verdict is required to proceed; a non-majority is a block.

This **parallel** N=3 is distinct from §4.9.1's **serial** N=3 (the `revise`→`reject` escalation, which counts *consecutive revise verdicts on one reviewer across rounds*). Here the three are simultaneous, independent, and tally to one gate decision; there one reviewer is re-consulted over time.

**Mode coupling** (§5.7.1): in `unattended`, the majority tally gates directly (it substitutes for the human exactly as a single reviewer's verdict does). A non-majority on a high-asymmetry decision is a **`hard` blocker** — unfixable without a product decision — so `/ship` parks (§5.7.1 hard-blocker path) for human adjudication; the park is the named next action (§6.0 P4), not a bare refusal.

**Fail-open** (§4.9.3 / §6.1 session-restart caveat): `is_high_asymmetry` is sourced via `safe_source`; on a miss (the helper absent in a pre-#428 session) the gated decision **degrades to a single reviewer** with an `audit_log warn reviewer-tier` record — never a lockout. A wrong *degrade* costs the redundancy on one decision (recoverable); a lockout would block all shipping (not) — the asymmetry that picks fail-open.

### 4.12 implementer (Issue #486; Directive #477)

**When**: Phase C (Code), **by default** — `/work-on` routes the Code phase here via `/implement` (§5.28). The **opt-out** is for trivial / one-line / glue edits and the orchestrator's own glue (Directive #477 Non-goal 2), which the main assistant authors directly in its own loop. The default-flip is Directive #477 **signal-4**: the formal A/B measurement was descoped and the context-narrowing hypothesis (§1.2 / §3.7) adopted as operating policy on circumstantial evidence, hedged by fail-open reversibility (a missing implementer path degrades to main-loop authoring) rather than a measured result.

**Premise**: the write-capable counterpart of the artifact-only reviewers (§4.5). It implements strictly from a supplied **manifest** and has **no knowledge of the main assistant's conversation** — the same "judge/act from the artifact, not the discussion" discipline, applied to authoring rather than review.

**Input = manifest** (the caller assembles it, §5.28): the **Plan** (PR-body / planner output), the **failing Phase-B test** + how to run it, the **named relevant file paths**, and the **directive-level learnings** — the distilled `### Learnings for the next Execution` blocks `/work-on` (§5.3 step 4.5) reads back from the parent Directive's `/reflect` comments (§5.15) and injects as **advisory context** (closes Directive #477 signal 3). If the Plan or the failing test is missing, the manifest is incomplete — the subagent reports and stops rather than inferring the intent; the directive-level learnings field is **optional** (advisory; its absence does not make the manifest incomplete).

**Output = structured return** (and nothing else): exactly (a) the commit / diff ref(s) authored, (b) plan-deviations (where/why it diverged from the Plan), and (c) discoveries (signal worth surfacing). The file reads, abandoned approaches, and lint/smoke iterations stay in the subagent's **ephemeral context** and are not returned — that churn-discard is the point (the within-Execution authoring churn never re-enters the parent context).

**Norms**: follows Doc → Test → Code (it is the Code phase — Doc/Test already exist, do not re-author them), commit-format (§10.2), and authors its commit in the **work language** (§5.7.2). Working-tree discipline (#285): write-capable, but git is constrained to its own `add`/`commit`; never a tree-mutating git command that touches the parent's uncommitted work.

**§4.9.3 caveat**: a freshly-added `subagent_type: implementer` falls back to `general-purpose` until the next session restart; `implementer.md`'s self-describing prompt makes that fallback functionally complete (the Type-aware tool restriction is the only loss), restart is canonical.

## 5. Slash commands

### 5.0 `/bootstrap-repo`

**When**: once, on a brand-new target that has **no default branch** — an empty GitHub repo (zero commits) or a greenfield local directory freshly `git init`'d (unborn HEAD). This is **stage-0**: it precedes `/onboard` (§5.1). The adoption lifecycle (`clone-into.sh` → `register.sh` → `/onboard` → `/onboard-dir-mode`, §2) presupposes a default branch with at least one commit already exists; `/bootstrap-repo` is the step that produces it, so there is an owner for the no-default-branch starting state rather than ad-hoc manual git.

**Precondition**: the target has no commits / no default branch. If a default branch already exists (`git rev-parse --verify HEAD` succeeds), `/bootstrap-repo` **refuses** — the repo is already past stage-0 and `/onboard` is the right entry point.

**Outputs**: a default branch (`main`) carrying a single seed commit — `MISSION.md` from `.claude/templates/mission.md` and `README.md` from `.claude/templates/readme_for_target.md` — pushed to the remote so the repo is ready for `/onboard`. Stage-0 seeds only these two SSOT files; `.github/` templates, labels, and substrate remain `/onboard` / `/onboard-dir-mode` concerns (§5.1, §1.7).

**The bootstrap exception (target repos).** The protected-branch gate is name-based (`PROTECTED_BRANCH_PATTERN`, §6.1): on an unborn HEAD, `git symbolic-ref --short HEAD` returns `main` (rc=0) while `git rev-parse --verify HEAD` fails, so `is_protected_branch()` matches by name and the seed commit (and the Edit/Write that authors `MISSION.md`/`README.md` on that branch) is blocked — correct, desirable behavior for the general case. `/bootstrap-repo` owns the **narrowly-scoped** exception for *this one seed commit only* — it does **not** weaken the gate. **In-agent escape (§7):** the two in-command forms are stripped before the hook in the live Bash tool, so they cannot disarm the seed commit. `/bootstrap-repo` instead writes a `branch`-category **file-based skip token** immediately before the seed commit — `scripts/eng_skip.sh branch '<seed-commit-subject>' '<reason>'` — and the seed commit then succeeds **in-agent** (the hook reads the token at fire time, audits it, and consumes it). The seed-commit subject is the high-entropy fingerprint:

```
scripts/eng_skip.sh branch 'chore: seed first commit (MISSION + README)' 'stage-0-bootstrap-seed-on-unborn-HEAD'
git commit -m "chore: seed first commit (MISSION + README)"
```

Running the seed commit in a **real terminal** (no hook in the path, the commit succeeds directly) remains the fallback. Either way the bypass routes through `should_skip branch` and is recorded in `audit.jsonl`. This generalizes the **shell's own-repo** first-commit exception (§16 item 15) to *target* repos: both are the same chicken-and-egg case (the first commit cannot ride a flow that presupposes a prior commit).

**Steps**:

1. **Preflight** — refuse if `git rev-parse --verify HEAD` succeeds (already has commits) with `"target already has a default branch — run /onboard, not /bootstrap-repo"`. Refuse if not inside a git work tree.
2. **Seed SSOT** — copy `MISSION.md` from `.claude/templates/mission.md` (a draft for the user to complete) and `README.md` from `.claude/templates/readme_for_target.md`, **via a Bash file copy** (`cp`), not the Edit/Write tool. The Edit/Write protected-branch arm (§6.1) blocks writes on the unborn-HEAD `main` and cannot be disarmed by a trailing sentinel (there is no command string to carry it); a `cp` into the registered target path carries no protected-branch check and is in-scope, so it is the correct seeding path.
3. **Seed commit** — `git add MISSION.md README.md`, write the `branch`-category file token (`scripts/eng_skip.sh branch '<seed-commit-subject>' '<reason>'`, §7), then run the seed commit above **in-agent** (the hook reads, audits, and consumes the token at fire time). Running it in a **real terminal** is the fallback. The seed commit message is a `chore` (no issue # required — there is no issue tracker state yet).
4. **Publish** — `git push -u origin main` (GitHub adopts the first pushed branch as the repo default).
5. **Handoff** — print `Next: /onboard` so the operator continues into the read-only onboarding check.

**Forbidden**:
- Running on a repo that already has a default branch (preflight refuses).
- Bypassing the protected-branch gate for anything other than the single seed commit — the exception is scoped to stage-0, not a general escape.
- Mutating `/onboard`'s responsibilities (`.github/`, labels, substrate) — stage-0 seeds only `MISSION.md` + `README.md`.

### 5.1 `/onboard`

**When**: once, right after a target repo is first cloned. **Precedes**: stage-0 `/bootstrap-repo` (§5.0) must already have produced a default branch — `/onboard` is a read-only check and assumes a seedable, cloned, registered repo; a brand-new target with no default branch goes through §5.0 first.

**Steps**:

1. **Upstream check** — `gh repo view --json isFork,parent`. Fork → block.
2. **Permission check** — push permission required. Missing → advise and block.
3. **Target SSOT check**:
   - `MISSION.md` present? Absent → strong warning + draft proposal from `.claude/templates/mission.md` (no auto-create).
   - `SPEC.md` present? `SPEC.md` is **required for any project that carries code** (§1.3) — it is the behavioural SSOT, so requiredness keys on the presence of code, not on whether an external contract (CLI, API, schema, protocol) exists. Absent on a code-carrying project → scaffold the slot from `.claude/templates/spec.md` (no auto-authoring of contract content — the MISSION boundary).
   - Report `README.md`, `CLAUDE.md`, `docs/ARCHITECTURE.md` presence.
4. **`.github/` check**:
   - `ISSUE_TEMPLATE/` absent → suggest shell template.
   - `PULL_REQUEST_TEMPLATE.md` absent → suggest shell template.
   - `CODEOWNERS` absent → recommend.
5. **Branch protection check** — `gh api repos/{owner}/{repo}/branches/main/protection`. Report missing items (PR required, review required, status checks required, force push blocked). Setup requires admin, so just print commands.
6. **CI presence** — check `.github/workflows/`. Absent → recommend appropriate workflows.

No automatic changes. All recommendations require human review and execution.

### 5.2 `/file-issue [--quick] <description>`

**Steps**:
- Confirm title, body, label with the user.
- Force MISSION reference and acceptance criteria in the body.
- **Rationale check** (before `gh issue create`): surface (a) which MISSION item this serves, (b) why now vs later, (c) whether an existing issue or an open-PR follow-up note would suffice. If any of (a)–(c) is weak, refine the issue or drop the request instead of filing. The check is mandatory — Auto / unattended mode does not skip it; a poorly-justified issue propagates downstream into planner output and merged code, where the cost of redirection is highest.
- **Reviewer gate**: invoke `issue-reviewer` (§4.7) on the proposed body before `gh issue create`. In `attended` mode the verdict surfaces to the user (who may override); in `unattended` mode the verdict gates the next step (`ship` → proceed; `refine` → re-author the body with the feedback and re-invoke; `block` → stop and surface the reason). The reviewer fills the human-confirm slot when no human is present. On a **non-pass** verdict the invoking flow emits a categorized reject-audit record (`issue-review`/`plan-review`/`activation` category + a reason-class token; the contract lives in §6.1, #361) — this applies to every reviewer gate in the command family (issue-reviewer here, plan-reviewer §5.3, activation-reviewer §5.12/§5.13), feeding the `scripts/promotion_candidates.sh` consumer.
- `--quick`: one-line issue (label `bug`) for fast creation. Rationale check + reviewer gate still required, in compressed form (one-line "serves X mission item, blocks Y" suffices). The `bug` label matches the `bug-report.yml` Issue Form template.
- **Directive parenting** (added by tracking #41 child #6, issue #47): when the new Issue is parented to a `Status=Active` Directive (via `--parent <directive-id>` or interactive prompt listing active Directives, §1.7), prepend the `Parent Directive: #N` body marker (line 1); the Project `Parent` field is **derived** from that marker by `issues-to-project-mirror.yml` — `/file-issue` writes the Issue only, never the Project directly (§1.7 Issues-as-SSOT). `--no-parent` opts out. Rationale check + `issue-reviewer` gate apply unchanged — parenting is metadata, not part of the rationale triad.
- **Full-symmetry stamp** (#172): `gh issue create` applies the `status:proposed` label so every new Execution Issue enters the lifecycle as Proposed and requires `/activate <N>` (§5.12) before it is actionable. This is why `issue-reviewer` (author-side, here) and `activation-reviewer` (observer-side, at `/activate`) are complementary, not redundant (§2.1). The output `Next:` line points at `/activate <N>`. `/work-on` against a `status:proposed` Issue is blocked by `proposed-protect` (§6.1) until activation.
- **Priority** (#291; parity with `/file-directive`, §5.10): `/file-issue` captures a `P0`/`P1`/`P2`/`P3` priority and applies it as a label at create time, so an Issue never lands priority-less (a priority-less backlog can't be triaged). Source order: an explicit `--priority P<N>` flag → else **ask the user** in `attended` mode → else **default `P2`** in `unattended` mode (the same default `/file-directive` uses). The `P<N>` label is **graceful-degradation guarded** — applied only if present on the target (tier ≥ 2), else a warn and file-without-it — and the priority is also recorded in the `issue-file` audit line so it survives a degraded label. An Execution Issue carries its own priority (it does not silently inherit the parent Directive's).
- Call `gh issue create`, return the number.

### 5.3 `/work-on <issue#> [--base <branch>]`

**Steps**:

1. `gh issue view <#> --json title,body,labels,comments` — read the issue.
2. Validate body — if acceptance criteria are missing or vague, ask the user to fix it. **Don't start work with an ambiguous goal.**
3. Resolve the **target base** (default `main`; override via `--base <branch>`). `git fetch origin && git checkout <base> && git pull --ff-only`. If `<base>` is not `main` and is not reachable locally or remotely, fail with a clear message — `/work-on` does NOT auto-create alternate bases (see §10.5).
4. Create branch:
   ```
   USER=$(gh api user --jq .login)
   TYPE=<detected type>
   SLUG=<kebab of issue title>
   git checkout -b "${USER}/${TYPE}/${ISSUE}-${SLUG}"
   ```
   If the branch already exists on remote: `git pull --ff-only origin <branch>`.
4.5. **Read back directive-level learnings** (closes Directive #477 signal 3 — the directive-level coding-memory loop). When the Issue body carries a `Parent Directive: #N` marker (line 1, §5.2), read the accumulated `### Learnings for the next Execution` blocks from #N's enriched `/reflect` comments (§5.15) and inject them as **advisory context** into (a) the planner input (step 5) and (b) the `directive-level learnings` field of the `/implement` manifest (§4.12). Advisory only — context the next Execution starts from, never a gate; absent/empty learnings (no parent Directive, or no enriched reflections yet) is a normal no-op. This is the *read* half of the loop `/reflect`'s Learnings section (§5.15) writes.
5. Invoke `planner` agent (with issue body + MISSION + target CLAUDE.md context + the resolved `<base>` + the step-4.5 directive-level learnings when present). The planner's output must include the mandatory `## Alternatives considered` AND `## Target base` sections (§4.1). The `Target base` value must equal the `<base>` passed in.
6. **Reviewer gate**: invoke `plan-reviewer` (§4.8) on the planner output. Parse the verdict line (`^VERDICT: (ship|refine|block)`). In `attended` mode the verdict surfaces to the user; in `unattended` mode it gates the next step (`ship` → step 7; `refine` → re-invoke `planner` with the feedback, then re-review; `block` → stop with a comment on the issue naming the structural problem). The reviewer substitutes for the human reviewer when no human is present.
7. **Wait for user approval of the plan.** Approval requires an **approach check**: confirm with the user that the chosen approach beats the alternatives the planner surfaced, not just that the plan exists. If the alternatives section is weak (single option without a forced-choice justification), refine before approval. In `unattended` mode, a clean `plan-reviewer` verdict from step 6 counts as approval.
8. **Write and commit Phase A (Doc)** — the first commit on the branch is real Doc work, never an empty seed. The branch graph begins with content the PR Plan describes, not a placeholder.
   ```
   # …edit the SSOTs identified by the plan (MISSION, README, CLAUDE.md, ARCHITECTURE, ADR)…
   git add <changed SSOTs>
   git commit -m "<type>(#<#>): <subject>

   {Closes|Refs} #<#>"
   git push -u origin HEAD
   gh pr create --draft --base "<base>" --title "<type>(#<#>): <subject>" --body "<template>"
   ```
   First-line trailer: use `Closes #<#>` when `<base>` is the repo's **default branch** (resolved via `gh repo view --json defaultBranchRef`, not a hardcoded `main` — #504); otherwise use `Refs #<#>` (per §10.4 — the issue closes only when work reaches the default branch, which is `main` on most repos but `master`/other elsewhere).

   If the planner's Phase A isn't ready to commit (e.g. blocked on clarification), defer the PR — keep the issue + plan in conversation. **Never fall back to an empty seed commit just to open the draft PR.**
9. Insert plan + checklist + ship gate sections in the PR body.
10. Sync internal TaskList with the checklist.
11. Continue with the next phase (Test).

### 5.4 `/sync-pr`

Explicit entry point for updating the PR body. The main assistant does this automatically; this command lets the user trigger it on demand. After a successful sync, `/sync-pr` references `/replan-check` (§5.26) as a post-sync advisory step — the body was just curated, the natural cadence to compare it against the actual diff for structural divergence.

Refetch the remote body once before updating. The shell stores the SHA-256 of the last body it saw at `.claude/state/pr-cache/<owner>%2F<repo>__pr-<n>.json` (the `/` between owner and repo is URL-encoded as `%2F` so `<owner>/<repo>` slug boundaries survive the filename); the file holds at least `last_seen_body_sha256`, `last_synced_head`, and `last_synced_at`. If the remote body's SHA differs from `last_seen_body_sha256`, an external edit happened (reviewer, parallel session, GitHub UI) — the automatic update aborts with a stderr message naming the conflict, and merging is the human's call. After a successful sync the cache is rewritten. A corrupt cache file (unparseable / missing key) also aborts the sync rather than silently treating as "first sync."

The cache lives under `.claude/state/`, which is gitignored — it's per-clone state, not part of the repo's history. The helper `.claude/hooks/helpers/pr_cache.sh` exports `pr_cache_read`, `pr_cache_write`, and `pr_cache_check`. The repo slug is derived from `gh repo view` of the cwd, overridable via `$PR_CACHE_REPO` (used by the smoke test suite). `/ship` reuses the cache transitively by routing its body curation through `/sync-pr` (§5.7 step 5). `/work-on` doesn't update the body after PR creation, so it doesn't touch the cache directly; later edits go through `/sync-pr`.

### 5.5 `/status`

`/status` and the `UserPromptSubmit` hook both delegate to the canonical helper at `.claude/hooks/helpers/status.sh`. Two output modes:

- `status_compact` — multi-line plaintext block, the standard turn-context surface.
- `status_json` — same fields as stable JSON for diff-comparison or piping.

Fields (both modes):
- Branch + dirty status
- Linked issue (#, title) — extracted from `Closes #N` / `Refs #N` in the PR body
- PR (#, draft/ready, checklist progress `[x/y]`, CI status) — `gh pr view`, `gh pr checks`
- Next unchecked checklist item
- Current phase (Doc / Test / Code / Review / Ship), inferred from the next unchecked item's section heading
- Active mode (attended / unattended) per §5.7.1
- **Bound canonical root** (#318) — the shared shell root this project resolves to, surfacing the §1.7 shared-code model. Source: `$CLAUDE_ENG_SHELL_ROOT` (back-filled by the hooks, §3.2.1) or `readlink $CLAUDE_PROJECT_DIR/.claude/eng-shell-root`.

`UserPromptSubmit` (§6.4) emits `status_compact` instead of a hand-rolled branch summary. Keeping the source single ensures `/status` and per-turn injection never drift.

The UserPromptSubmit hook reuses this output for the short per-turn summary.

Hits skip every `gh` round-trip (today: `gh pr view`, `gh issue view`, `gh pr checks` — 3 calls per uncached invocation). The 5-second TTL is short enough that real PR/issue/CI changes surface within one user turn (turns typically exceed 5 s) and long enough to dedupe a flurry of prompts. Cache files live under `.claude/state/` (gitignored).

### 5.7 `/ship`

Runs the ship sequence from §2 (steps 0–8), plus the pre-ready sub-step gates 7.5–7.7 detailed below. `/ship` runs the full test suite (step 4) before transitioning to ready (step 8). Tests are `/ship`'s responsibility; the hook does not add a separate test gate on ready transition (§6.1).

**High-asymmetry reviewer tier** (§4.11): when a ship decision is high-asymmetry — a `merge-security-surface` merge to the default branch, a `force-push` judgment inside the ship motion, or an `irreversible-adr` merge (a PR carrying `docs/ADRs/`) — `/ship` classifies it via `is_high_asymmetry` and replaces the single-reviewer verdict with an N=3 independent majority vote. A non-majority is a `hard` blocker → park (§5.7.1). The common path (no high-asymmetry kind) stays single-reviewer.

CI monitoring: snapshot `gh pr checks --json state` right before ready transition — already failing → block. Right after the transition, run `gh pr checks --watch` in background and return immediately. On failure, the PostToolUse hook notifies the user.

**Checklist audit (mandatory step 7.5, between CI snapshot and `gh pr ready`)**: scan the PR body for any `- [ ]` line. Each unchecked item must be resolved before transition — ticked if done, marked `[~] N/A — <reason>` if intentionally skipped, or removed if obsolete. The audit applies to every checklist in the body (Plan, Test plan, Docs touched, Ship gate). Per SPEC §1.4, stale uncheckedness at merge propagates as misinformation into future context.

**Issue AC closeout (mandatory step 7.6, before `gh pr ready`)**: for every issue in the PR's `closingIssuesReferences`, run `scripts/ac_closeout.sh <pr-num>` to post the canonical `## AC closeout (resolved by PR #N)` comment with each acceptance-criterion item ticked or `[~] N/A`'d. The helper is idempotent (skips an issue that already carries a `^## AC closeout` comment). The PreToolUse `ac-closeout` gate (§6.1) blocks `gh pr merge` when an unchecked-AC closing issue lacks this marker, so running step 7.6 here satisfies the merge gate by construction. Steps 7.5 (PR-body side) and 7.6 (issue side) together are the symmetric halves of "PR-as-living-doc" (§1.4) across the `Closes #N` linkage. Skipped (all-Directive closing set) when Directives close via `/complete-directive`, not AC checkboxes.

**Changelog gate (mandatory step 7.7, before `gh pr ready`)**: the PR must satisfy the §18.1 fragment contract (CI-enforced per §18.6) *before* ready, not after a red CI check — exactly one of {a fragment exists for the PR number or one of its `closingIssuesReferences`, the `skip-changelog` label is present} must hold. This is a **presence** check; fragment *shape* stays CI's job (§18.5). When neither holds, `/ship` invokes `/changelog` (§5.23) to author the fragment or apply the label — `attended` proposes for confirmation, `unattended` decides automatically (§5.7.1) — so the flow self-heals instead of failing at CI. The skip decision follows §18.7.

When mode is `unattended` (see §5.7.1), `/ship` continues past step 8 into the CI-wait → merge / park branch.

### 5.7.1 Operating modes (attended / unattended)

`/ship` has two terminal behaviors selected by **mode**:

- **`attended`** (default): `/ship` stops at "ready" and hands off to a human reviewer.
- **`unattended`**: after "ready", `/ship` waits on CI / required reviews and either merges or parks the PR for human attention.

**Resolution priority** (high → low):

1. `--mode=<attended|unattended>` flag on `/ship`
2. `$CLAUDE_ENG_SHELL_MODE` env var
3. `.claude/state/mode` per-target file (one of `attended` / `unattended`, no other content)
4. default `attended`

Unknown values fail closed to `attended` and emit a stderr warning naming the offending source. `settings.json` is intentionally **not** a layer — that file is symlinked across targets (§3.2) and a mode key would leak between unrelated work.

**Blocker taxonomy** — `ship_classify_blocker` returns one of:

| State | Criterion |
|-------|-----------|
| `clean` | All required CI checks green; required reviewers (if any) satisfied; branch protection rules satisfied. Eligible for auto-merge. |
| `soft` | CI is `pending` / `in_progress` / `queued`, **or** there is a CI failure that is attributable to the agent's own last commit and is fixable without new product decisions. One self-fix attempt is permitted (see below). |
| `hard` | Required-review with no available reviewer, undiagnosable / non-attributable CI failure, branch-protection rule the agent cannot satisfy, or a `soft` state that has already consumed its one self-fix attempt. |

**Soft-blocker behavior**: at most **one** self-fix-and-push attempt per `/ship` invocation. The retry pushes a `fix(#N): ...` commit and returns to the CI-wait state. A second `soft` outcome immediately escalates to `hard` — no further fix attempts.

**Hard-blocker behavior** (parking) — all four actions, in order, and idempotent:

1. Post a `gh pr comment` with a deterministic state summary (blocker reason + current SHA + CI snapshot).
2. Apply label `unattended-parked` (create the label on demand if missing). If the label is already present from a previous park on the same PR, `ship_park_pr` writes a single `park-suppressed: <reason>` line to the log instead of reposting the comment, and emits no comment to stdout. The original park comment remains canonical; the label remains applied.
3. Append one line to `${SHIP_PARK_LOG_PATH:-$CLAUDE_ENG_SHELL_ROOT/.claude/state/unattended-park.log}` (shell-side, gitignored — not the target's MEMORY.md). The override exists for smoke-test isolation. On suppressed repeat, only the `park-suppressed: <reason>` line is appended. **Additionally, on a fresh park** (not the suppressed-repeat arm), `ship_park_pr` emits one `audit_log warn unattended-park parked "reason=<token>"` record (guarded by `command -v audit_log` — best-effort, so a standalone invocation without the runtime still writes the park-log unconditionally). This is **additive** (the human-readable park-log is unchanged, not a schema change): it bridges overnight park frequency — the highest-value unattended friction signal — into `audit.jsonl`, the aggregate the §6.5(d) friction advisory reads, so parks stop being invisible to the friction-observability loop (#398, Directive #391). The `unattended-park` audit category records park-as-friction-signal; it is read **directly** by the §6.5(d) advisory (neither `narrowing_candidates.sh` nor `promotion_candidates.sh` matches a `warn`/`parked` record).
4. Stop. Do not toggle draft or self-assign.

**Classifier robustness**: `ship_classify_blocker` fails closed to `hard` in two failure modes and emits a one-line stderr warning naming the cause:

- empty stdin → `ship_classify_blocker: no input on stdin — defaulting to hard`
- missing `jq` dependency → `ship_classify_blocker: jq not found — defaulting to hard`

The intent is to make overnight runs debuggable without changing the safe fail-closed default. Either condition resolves the classification to `hard` and the dispatch path proceeds to park.

**Merge mechanism**: `gh pr merge --auto --merge --delete-branch` (no-fast-forward merge commit; PR branch's commits are preserved on `main`). This requires repo-level auto-merge to be enabled at GitHub. If the API call fails because auto-merge is disabled, fall back to the park path (log + label + comment) with reason `auto-merge-disabled`.

**Why `--merge` rather than `--squash`**: SPEC §1.2 makes the Doc → Test → Code commit graph load-bearing — the arc is the artifact that proves the discipline happened. Squash erases the arc on merge (the PR branch is deleted; only a single new commit lands on `main`). `--merge` keeps both: the individual phase commits stay on `main`, plus a merge commit anchors the PR-as-a-unit so `git log --first-parent main` gives a clean PR-level view. The visible cost is a busier `git log`; the audit-trail philosophy in §6/§7 trades that for the preserved arc.

**Composition with branch protection**: server-side rules in GitHub are always authoritative. `unattended` does not bypass them. A required-reviews rule with no available reviewer resolves to `hard` and parks — the shell never self-approves or auto-requests review against policy.

**Composition with Auto mode**: Auto mode (single-conversation clarification suppression) is orthogonal to operating mode. Any combination is valid: `attended + Auto`, `unattended + non-Auto`, etc. Mode controls the `/ship` terminal behavior; Auto controls in-turn prompting.

Note: §6.1's "direct push to protected branch blocked" hook is **not** tripped by the merge step — `gh pr merge --auto` is a GitHub API call, not a `git push` to `main`.

**Directive-aware audit** (added by tracking #41 child #6, issue #47): when the merged PR's linked Issue carries a `Parent Directive: #N` marker (§5.2), `/ship` appends one audit line `directive-exec-count directive=#N pr=#M issue=#K` to `audit.jsonl`. The audit reads `closingIssuesReferences` — so in topic-branch flow (§10.5), sub-task PRs (which use `Refs #N`) emit no audit line, and the audit fires only on the **consolidator PR** that closes each parented Execution Issue at the moment work reaches `main`. Informational only — `/ship` does **not** auto-mark the Directive `Completed`. Completion is `/complete-directive`'s job, gated by `activation-reviewer` (§4.9). This preserves the §1.5 invariant: generation is open, decision is gated.

**Canonical hook-enforced path** (Directive #61): `.github/workflows/dir-mode-post-merge.yml` (installed from `.claude/templates/target-substrate/workflows/dir-mode-post-merge.yml`) fires on every `pull_request.closed && merged == true` event, regardless of who performed the merge (Claude-in-loop, human via UI, CI auto-merge). The Action emits an equivalent reflection comment on the parent Directive — the comment surface IS the audit surface for hook-enforced events, since the workflow runner cannot write to a per-clone `audit.jsonl`. The Markdown procedure above stays valid for Claude-in-loop runs (defense in depth); the workflow posts a marker-stamped **stub** and `/reflect` enriches it in place, the two paths deduping on the `reflect-stub`/`reflect-enriched` **content markers** rather than the shared PR URL (#329 — see §5.15).

### 5.7.2 Communication language vs work language

The shell separates two independent language channels (#322):

- **Communication language** — the human↔agent conversation. Whatever language the user writes in; the agent's chat replies stay in it. The shell never detects, configures, or translates it.
- **Work language** — the single language of **durable repository artifacts**. Resolved by `resolve_work_lang` (`.claude/hooks/helpers/work_lang.sh`), defaulting to `en`.

**Per-surface classification** — every shell-authored artifact is one or the other:

| Work language (durable, in the repo) | Communication language (ephemeral, in the chat) |
|---|---|
| commit messages; PR titles/bodies; issue / directive / execution bodies; acceptance criteria; changelog fragments; shell-authored code comments; audit `reason` text | the agent's conversational replies to the user |

**Pre-work context-recast step.** After the human conversation concludes and **before authoring any durable artifact**, the agent re-organizes the task context into the work language — it does not transliterate the chat verbatim but re-expresses intent, decisions, and plan in the work language, then authors every work-language surface from that recast. This is the "narrowing/normalization" half of the MISSION mechanism applied to language: durable memory and the artifact-only review surface stay legible in one consistent language regardless of the conversation language.

**Resolution priority** (high → low), mirroring §5.7.1:

1. `$CLAUDE_ENG_WORK_LANG` env var
2. `.claude/state/work-lang` per-target file (a single language code, cwd-relative — read exactly as `resolve_mode` reads `.claude/state/mode`)
3. default `en`

Any non-empty language code is accepted **verbatim** (BCP-47-style tokens such as `en`, `ko`, `fr`, `ja`, `pt-BR`) — there is no closed enum, so the capability generalizes to any (communication, work) pair and is never `ko`/`en`-hardcoded. An empty / whitespace-only value resolves to `en` (the surface is named in a stderr note). `resolve_work_lang` is `set -u`-safe and makes no external calls beyond `head`/`tr`.

**Config placement** — `.claude/state/work-lang`, **not** `eng-state/`. Work language is user-set, set-once **configuration** (the same tier as the `mode` toggle §5.7.1 and the `coauthor` toggle §10.2), not regenerable **ephemeral state** (`eng-state/` holds the audit log, caches, and registry — §3.2.2). Directive #322's body says "eng-state" but cites "the mode toggle" as the analog; the mode toggle lives at `.claude/state/mode`, so `work-lang` follows that concrete precedent (config tier, cwd-relative) rather than the mislabeled category word. Like §5.7.1, `settings.json` is intentionally **not** a layer — it is symlinked across targets (§3.2) and a work-language key would leak between unrelated projects.

Back-compat: unset → `en`, i.e. today's behavior; nothing changes for a target that never sets a work language. **Surfacing** is delivered (#325): `status_compact` emits a `work-lang:` line and `status_json` a `work_lang` key (resolved via `resolve_work_lang`), so `/status` and the per-turn statusline show the active work language alongside `mode`. **Skill integration** is delivered (#327): the artifact-authoring skills (`file-issue`, `file-directive`, `work-on`, `ship`, `complete-directive`) each carry a `## Work language` note instructing the agent to recast context and author their durable artifact in the work language — co-located with the authoring step (skills-as-environment) so the instruction survives context-narrowing. Enforcement is agent-followed prose, not a hook: the *language of free prose* is not mechanically checkable (#322 success signal 2 delegates it to review); smoke grep-locks only that each skill carries the note.

### 5.8 `/adr <title>`

**ADR vs PR body Decisions**:
- **ADR** — irreversible, system-level. Migrations (DB engine, schema change), external contracts (API spec, wire format), default technology choices (auth scheme, queueing model). High cost to reverse once merged.
- **PR body Decisions** — reversible, PR-scoped trade-offs. Default option choice, algorithm tuning parameters, equivalent-library preference, naming choices. Explainable within this PR and not affecting other PRs.

`/adr` creates `docs/ADRs/NNNN-title.md` (from `.claude/templates/adr.md`). A `→ ADR-NNNN` link is auto-added to the PR body's Decisions section. Authoring is a **reversible doc-write** (the file is editable until merged), so the high-asymmetry reviewer tier (§4.11) does *not* fire here — it fires at the irreversible point, the PR merge carrying the ADR, in `/ship`.

### 5.12 `/activate [<issue-#>]`

Promote a `Proposed` Issue (Directive **or** Execution Issue — `/activate` is type-neutral) to `Active` by removing the `status:proposed` label after a fresh `activation-reviewer` (§4.9) pass on the current body. This is the observer-side gate of the full-symmetry lifecycle (§2.1): every Issue is filed `status:proposed` and becomes actionable only after `/activate` returns `pass`. `/activate` is also the **unblock** path for a Blocked Directive (§5.17): a re-activatable Issue carries `status:proposed` (Proposed) **or** `status:blocked` (an Active-but-Blocked Directive); only when **neither** label is present is the Issue plain Active and `/activate` errors "already Active". On `pass` it removes whichever of `{status:proposed, status:blocked, awaiting-author}` are present (each removal presence-guarded — `gh issue edit --remove-label` errors on an absent label). Batch mode scans `--label status:proposed`, so it does **not** sweep Blocked Directives — unblock is single-mode `/activate <N>` only (auto-unblocking would silently reverse a deliberate maintainer block).

- **Single mode** `/activate <N>`: resolve `<N>`'s type label (tri-state, matching `activation-reviewer`'s §4.9 dispatch — an `initiative`-labelled Issue is **refused**, not activated: Initiatives are consumed via `/consume-initiative` (§5.21) and live outside the Proposed→Active lifecycle (§2.1); a bare `directive`-else-Execution split would otherwise mis-type one as Execution. The guard normally never fires because an Initiative never carries `status:proposed`), invoke `activation-reviewer` with the type context, parse the verdict, and apply it:
  - `pass` → remove the present activation labels (`status:proposed` and/or `status:blocked`, plus `awaiting-author` if present) → Active.
  - `revise` → post the findings as a comment carrying `<!-- activation-verdict: revise -->`, retain `status:proposed`, add `awaiting-author`. After **N=3** `revise` markers on the Issue, escalate to `reject`.
  - `reject` → post the verdict + structured fields with `<!-- activation-verdict: reject -->`, then apply **filer-aware handling** (verdict is content-only; handling keys on `authorAssociation` via `is_trusted_filer`): **trusted** → keep open + `awaiting-author`, never close (composes with `trusted-filer-mutate`, §6.1); **untrusted + `refile-body-draft`** → close (`--reason "not planned"`) + auto-create a `discussion`-tier Issue (§5.19) preserving the draft + a lineage link; **untrusted, no draft** → close + brief comment.
- **Batch mode** `/activate` (no arg): scan all open `status:proposed` Issues and process each. **Loop-safety:** skip an Issue whose latest `<!-- activation-verdict: <verdict> -->` comment marker post-dates its last body/label edit (verdict already delivered, awaiting the author) — independent of the `awaiting-author` label. In `attended` mode `reject` surfaces to the operator; in `unattended` it auto-applies the filer matrix.
- **Reviewer invocation** is robust to the §4.9.3 session-restart caveat: the agent file is on disk and its prompt is self-describing, so a `general-purpose` fallback still behaves as the reviewer.

`/activate-directive <issue-#>` is retained as a **thin alias** delegating to `/activate <N>` for one release cycle (Directive-only ergonomics), then removed. Audit category: `activation` (generalized from `directive-activate`; the append-only log retains historical `directive-activate` entries). The mirror workflow fires on `issues.unlabeled` and updates the Project Item's Status. Escape: `SKIP_HOOKS=directive-review SKIP_REASON='<why>'`. Full procedure lives in `.claude/commands/activate.md`.

### 5.13 `/complete-directive <issue-#>`

Close a Directive Issue with `--reason completed`. `activation-reviewer` (§4.9) evaluates evidence — the linked Execution Issues' close states / AC ticks — against the success signals. **Directive completion is a high-asymmetry decision** (§4.11, `directive-completion`): the evidence evaluation runs as an N=3 independent majority vote of `activation-reviewer`, not a single verdict (a wrong "complete" is irreversible — it closes the Directive). A non-majority leaves the Issue open. `reject` (insufficient evidence) leaves the Issue open. On `pass`, posts a closing comment listing each signal and its evidence, then closes the Issue. The `trusted-filer-mutate` hook matcher (SPEC §1.5, §6.1) allows `--reason completed` on trusted-filer Issues without further confirmation. Audit category: `directive-complete`. Escape: `SKIP_HOOKS=directive-review SKIP_REASON='<why>'` (§7). The mirror workflow fires on `issues.closed` and updates the Project Item's Status. When the Directive's Execution Issues used the topic-branch flow (`/work-on --base <topic-branch>`, §10.5), `/complete-directive` must run **after** the topic branch consolidates to `main` — the reviewer reads closed-Execution-Issue evidence, and still-open `Refs`'d Issues yield insufficient-evidence verdicts. Full procedure lives in `.claude/commands/complete-directive.md`.

Idempotent — re-running on an already-linked pair no-ops with an info-level audit line. ### 5.15 `/reflect [<pr-#>]`

Post a structured reflection comment on a PR's parent Directive (when one exists) summarizing the PR's contribution toward that Directive's success signals. Run after merge; idempotent — classifies the Directive's existing comments by **content marker** (#329): a `<!-- reflect-enriched pr=#N -->` comment → no-op; the hook-enforced floor's `<!-- reflect-stub pr=#N -->` → **enrich in place** (edit that comment by REST comment id); neither → post fresh. (Keying on the marker, not the shared PR URL, is what lets `/reflect` enrich the workflow's at-merge stub instead of no-op'ing against it.) Resolves the parent Directive from each closing Issue's body via the regex `^Parent Directive: #(\d+)$` (the marker `/file-issue --parent` writes on line 1, SPEC §5.2) — **label-aware** (#335): it climbs the marker chain to the first `directive`-labelled ancestor (depth-cap 2, cycle guard, fail-soft) rather than stopping at the first marker, so a nested-umbrella Execution Issue reflects onto the grandparent Directive, not the intervening umbrella. Audit category: `directive-reflect` (decisions: `posted` / `skipped`). Read-only on the linked Execution Issues; writes only the reflection comment on the Directive Issue. The human-readable counterpart to `/ship` step 10.5's bare `directive-exec-count` audit (SPEC §5.7). Full procedure lives in `.claude/commands/reflect.md` (added by tracking #41 child #6, issue #47).

**Hook-enforced floor** (Directive #61): `.github/workflows/dir-mode-post-merge.yml` posts a **stub** reflection automatically on every merged PR with a `Parent Directive:` marker (marked `<!-- reflect-stub pr=#N -->`), so the reflection trail is complete regardless of who merged. The workflow resolves the target Directive via a **sourced helper** `resolve_parent_directive.sh` (shipped alongside the workflow and read after `actions/checkout`) rather than inline run-block bash, so the resolution — the label-aware climb to the first `directive`-labelled ancestor (#335) — is CI-shellchecked and smoke-executed; a resolved-nothing run logs a visible no-op. The helper has three byte-identical copies (canonical `scripts/lib/`, this-repo runtime `.github/workflows/`, and the target-substrate copy installed into onboarded targets), kept in lock-step by a `cmp` smoke assertion (the same discipline that holds the three workflow YAML copies in sync). The same workflow also runs the **bare-`Refs`-to-default check** (`detect_bare_refs_directive.sh`, #337, §10.5): when a merged PR's base is the live default branch, its closing set is empty, and its body carries a directive-scoped `Refs #N`, it posts a single idempotent warning comment on the PR — surfacing the §10.5 leak that would otherwise silently deprive the Directive of a reflection. Warn-only, fail-soft, and (unlike a `gh pr merge` hook) it fires on human UI merges too. When Claude runs `/reflect`, it produces the richer per-signal block by **editing that stub in place** (PATCH by REST comment id), swapping the marker for `<!-- reflect-enriched pr=#N -->`. The two paths dedupe on the **content markers**, not the shared PR URL (#329): a URL-keyed check would make `/reflect` no-op against the workflow's at-merge stub and the enrichment would never happen. Classification: enriched-marker present → no-op; stub-marker present → enrich in place; neither → post fresh.

**`### Learnings for the next Execution` section** (closes Directive #477 signal 3 — the directive-level coding-memory loop). The enriched reflection comment carries a fourth section alongside Contribution / Success signals advanced / Next: distilled **durable coding signal** from the merged PR's Code phase — codebase gotchas, conventions to follow, what to reuse vs avoid — and, when the `/implement` path (§5.28) authored the PR, drawn additionally from the structured-return **discoveries**. **Distillation heuristic**: keep only learning that **outlives this PR** and would help a sibling Execution under the same Directive; the within-Execution **churn** (file reads, abandoned approaches, lint/smoke iterations) is the §1.2 / §3.7 discard boundary and is **not** transcribed. This is the *write* half of the loop `/work-on` step 4.5 (§5.3) reads back. **Idempotency is unchanged**: the section rides **inside** the existing reflection comment under the same `reflect-stub` / `reflect-enriched` marker (a fourth body section, not a new comment), so the step-3 classification and the PATCH-by-REST-id enrich-in-place flow are untouched.

### 5.16 `/revise-directive <issue-#>`

Replace an `Active` Directive Issue's body with a new one when scope or success signals changed, archiving the prior body as a comment for history. Procedure: (1) read the current body; (2) compute its sha256; (3) invoke `activation-reviewer` (§4.9) on the **new** body — the same proposal checks as `/file-directive` (schema / parent-kind alignment / success-signal verifiability / scope / non-goal / active-Directive conflict); (4) on `pass`, post a `## Pre-revision body — archived <ISO date>` comment with the verbatim prior body, then replace the Issue body via `gh issue edit --body-file`; (5) emit `audit_log info directive-revise reconciled "directive=#<issue-#> from-sha=<prior-sha> archive-comment=<url>"`. **No Status change** — the audit-log entry + the archive comment ARE the durable evidence of revision (no transient state needed — see §2.1). On `revise` or `reject`, leave the body unchanged. Escape: `SKIP_HOOKS=directive-review SKIP_REASON='<why>'` (§7). Full procedure lives in `.claude/commands/revise-directive.md`.

### 5.17 `/block-directive <issue-#> --reason <why>`

Mark an `Active` Directive as `Blocked` by adding the `status:blocked` label and a `## Blocked: <reason>` comment. Procedure: (1) require the `--reason <why>` flag — empty or missing reason aborts; (2) post a `## Blocked: <reason>` comment on the Issue (idempotent — skip if the same `## Blocked:` marker is the latest comment AND `status:blocked` label is already present); (3) add the `status:blocked` label via `gh issue edit --add-label`; (4) emit `audit_log info directive-block created "directive: #<issue-#> reason=<short> comment=<url>"`. ### 5.19 Discussion tier (Issue #112 substrate; Directive #109)

A distinct open-Issue tier alongside Directive / Execution / task / bug. **Purpose**: capture deliberately-ambiguous observations ("weird but not a bug", half-formed ideas, smells, RFC-style probes) without forcing premature classification. The tier exists because the other tiers each force a misclassification on such filings: `task` reads as actionable, `bug` claims a defect, and skipping filing loses the observation entirely. Discussion is the honest path. (Raw label-free filings are auto-stamped `status:proposed`+`task` by `auto-status-proposed.yml` (#179) and validated at `/activate`; deliberate ambiguity is filed `discussion` instead.)

**Labeling**: a single `discussion` label (color `#FEF2C0`, light yellow) installed by `scripts/ensure_v3_labels.sh` alongside the other dir-mode labels. Issue Form template at `.github/ISSUE_TEMPLATE/discussion.yml` requires only title + body — no acceptance criteria, no MISSION fit, no rationale-triad prompt; the reduced-required shape is the whole point.

**Lifecycle**: develop in comments. Discussion Issues are NOT parented to Directives (no AC; nothing to roll up). The Issue's comment thread IS the development surface.

**Close paths** — exactly two:

- **`promoted`** — a concrete Issue (bug / task / Directive) was filed; the discussion closes referencing it (`gh issue close <N> --reason completed` plus a `^promoted to #<M>$` comment). The new Issue cross-references back.
- **`no-action`** — the discussion concluded nothing needs doing (`gh issue close <N> --reason "not planned"` plus a one-line comment naming the reason). The `gh` CLI accepts only `{completed|not planned|duplicate}` — the **space** form; `not_planned` (underscore) is rejected by `gh`, so the space form is canonical everywhere the value reaches `gh`.

No third path. Stale discussions surface via `/activate` batch mode (§5.12; relocated from `/triage` by #173); no auto-close timer.

**Tier-relationships table**:

| Tier | Label(s) | AC-bearing? | Parented under Directive? | Close paths |
|---|---|---|---|---|
| Directive | `directive` + `status:*` | No (success signals instead) | No (root of the hierarchy) | `/complete-directive` (§5.13) — Completed via reviewer ship |
| Execution | (none, or template-applied) | Yes | Optional (`--parent`) | `Closes #N` via PR merge |
| task | `task` | Yes | No | `Closes #N` via PR merge or close-as-not-planned |
| bug | `bug` | Yes | No | `Closes #N` via PR merge or close-as-not-planned |
| **discussion** | `discussion` | **No** | **No** | **`promoted` or `no-action` only** |

**Lifecycle implementation** (Issue #116; Directive #109 final slice):

- `/discuss <title>` — friction-free filing; bypasses the rationale triad + `issue-reviewer` gate. Full procedure in `.claude/commands/discuss.md`.
- `/resolve-discussion <N> [--promoted-to <M>] [--no-action <reason>]` — closes via exactly one of the two paths above. Idempotent. Full procedure in `.claude/commands/resolve-discussion.md`.
- Close-path enforcement — hook matcher arm under `trusted-filer-mutate` (§6.1) blocks `gh issue close <N>` on a `discussion`-labeled Issue without `--reason completed` OR `--reason "not planned"`. The matcher tolerates both the gh-valid space form and the legacy underscore so it is never itself the blocker; the canonical emitted form is the space form. Discussion-tier classification is evaluated **first and independent of filer trust** (#236): once an Issue is recognized as `discussion`, the §5.19 two-path rule governs — `--reason completed` OR `--reason "not planned"` is allowed regardless of the filer's `authorAssociation`, and a bare close or `--reason duplicate` is blocked. The trusted-filer block (below) applies only to **non-discussion** Issues. Stage 1 of the close-arm two-stage check (stage 2 = trusted-filer enforcement).
- `/activate` batch mode (§5.12; relocated from `/triage` by #173) — surfaces open `discussion`-labeled Issues older than 14 days as a maintainer-decision queue (promote / dismiss / let-incubate). Mode-independent: AI does not autonomously promote/dismiss even in unattended.
- Smoke §62 — 8 assertions covering skill files, hook close-path enforcement (block + allow paths, the gh-valid `"not planned"` space form + legacy-underscore tolerance + a `duplicate` negative, #216), and `/triage` extension.

**Lockstep with Directive #107**: the `discussion` label is in `/onboard-dir-mode`'s bootstrap-label set (the dir-mode label set — see the canonical count in §1.7's Tier 2 note). #112 (label substrate) merged first; #116 (lifecycle) merged second; #107's `/onboard-dir-mode` consumes both.

### 5.20 `/release <X.Y.Z> [--base <branch>] [--dry-run]`

Cut a versioned release by consolidating per-PR changelog fragments into a new section of `CHANGELOG.md` and opening a `release/X.Y.Z` draft PR. Implements the fragment-contract consumer side of SPEC §18.

Procedure (full body in `.claude/commands/release.md`):

1. **Parse args** — required `<X.Y.Z>` (semver 0.x; reject `MAJOR > 0` with explicit `SPEC §3.5 / §18.2 MAJOR=0 invariant` message), optional `--base <branch>` (default `main`; for maintenance-line cuts per §18.3 / §10.5), optional `--dry-run` (helper stages but does not commit/push/branch).
2. **Preflight** — assert clean working tree on the target base, `git fetch --tags origin`, refuse if `release/X.Y.Z` branch exists on origin OR `vX.Y.Z` tag already exists (exits 0 with `already released` message — idempotent retry).
3. **VERSION write-back** — strip any `-dev` suffix from the top-level `VERSION` and write `X.Y.Z`. Resolution-time queries fall back to `git describe --tags`; write-back is `VERSION`-only per §18.2.
4. **CHANGELOG consolidate** — read every `changelog_unreleased/<category>/<N>.md`, validate (positive-integer filename stem; bullet contains `(#<N>)` matching stem), prepend a new `## [X.Y.Z] — YYYY-MM-DD` section (UTC date) to `CHANGELOG.md` with `### <Category>` subheadings (Keep-a-Changelog), append `[X.Y.Z]: https://github.com/<owner>/<repo>/releases/tag/vX.Y.Z` reference link at the file footer.
5. **Fragment cleanup** — `git rm` every consumed fragment file (leave the six `.gitkeep` placeholders intact).
6. **Branch + draft PR** — create `release/X.Y.Z` (short-lived; covered by the existing `release/*` arm of the protected-branch matcher per §6.1). Commit the VERSION + CHANGELOG + fragment-removal as a single **scopeless** `chore: release X.Y.Z` commit under the documented `SKIP_HOOKS=branch SKIP_REASON='/release initial release-branch commit'` escape (audit-logged; structurally required since the protected-branch matcher blocks all direct commits on `release/X.Y.Z` and this is the one purposeful commit). The subject is scopeless — not `chore(release): …` — because the `commit-format` matcher permits only a `(#N)` scope or no scope, and the `branch` escape does not cover `commit-format`. Push, open a draft PR titled `chore: release X.Y.Z`.
7. **Reviewer gate** — in `unattended` mode, re-run `code-reviewer` (§4.5) against the release diff before `gh pr create` to catch malformed CHANGELOG concatenation / mismatched tag / VERSION drift. Per §1.5 reviewer-substitution. Escape: `SKIP_HOOKS=release-review SKIP_REASON='<why>'`.
8. **On-merge tag + Release** (documented post-merge step; **maintainer-run, not automated** per §18.5 non-goals — this is the one step that sits *past* the `unattended` autonomy ceiling, which terminates at `gh pr merge`/park (§5.7.1); the `unattended` `/release` continuation stops at the merged release PR and does **not** cut the tag/Release): after the release PR merges to the target base, the maintainer runs:
    ```bash
    MERGE_SHA=$(git rev-parse HEAD)  # on target base, post-merge
    awk '/^## \[X\.Y\.Z\]/{f=1; next} /^## \[/{f=0} f' CHANGELOG.md > /tmp/release-notes-vX.Y.Z.md
    gh release create vX.Y.Z --title vX.Y.Z --target "$MERGE_SHA" --notes-file /tmp/release-notes-vX.Y.Z.md
    scripts/release_verify.sh X.Y.Z   # post-merge verify (below)
    ```
    Re-running on an already-released tag is a no-op with a clear message. **No `--verify-tag`** (#448): the flag aborts `gh release create` unless the tag already exists, but the tag is created *together with* the Release here (from `--target`), so it would always abort on a fresh release.

    **Post-merge verify** (#471, Directive #468): `scripts/release_verify.sh X.Y.Z` is a **read-only, fail-open** confirmation run *after* the create — it checks the `vX.Y.Z` tag exists and a GitHub Release with **non-empty notes** is present (one `gh release view --json tagName,body` query), printing a named advisory line on a missing tag / missing-or-empty-notes Release. It **never creates** anything and **exits 0 on every path** (gh error / offline → one advisory line), so it observes the release result without blocking — the verification-only / never-false-block posture (§6.0 cost-asymmetry). The `unattended` `/release` continuation runs it after its post-merge create.

`/release` factors deterministic work into `scripts/release_consolidate.sh` so smoke can exercise it against throwaway repos without booting a Claude Code session (same precedent as `scripts/ac_closeout.sh` ↔ `/ship` step 7.6). Audit category: `release` (decisions: `created`, `idempotent`, `blocked`).

### 5.21 `/consume-initiative <N>` (Issue #257; Directive #256)

The consume flow (SPEC §1.7 "Initiatives — the planning tier the shell consumes"): turn an upstream `initiative`-labelled Issue into Initiative-parented Directive proposals, working backward from its termination condition. A **prompt-driven skill** (`.claude/commands/consume-initiative.md`), script-less like `/file-directive` — the load-bearing work (judging the contract, decomposing the termination condition, authoring Directive bodies) is LLM reasoning, not mechanizable steps. Procedure:

1. **Type/substrate check** — resolve `#N`; require the `initiative` label (else error). Same-repo only (cross-repo Initiatives are out of scope, §1.7).
2. **Contract gate** — invoke `activation-reviewer` (Initiative rulebook, contract-evaluability I1, §4.9). If the termination condition is not evaluable without code knowledge → the Initiative is under-specified: post a plain `gh issue comment <N>` surfacing it upstream and **STOP** (file no Directives). `gh issue comment` is the only Initiative write the flow performs — allowed by `initiative-readonly` (§6.1), which blocks only `edit`/`close`/`reopen`.
3. **Extract (working backward)** — restate the termination condition, decompose it into the sub-outcomes that together satisfy it, and map each to one Directive proposal. The **necessary-and-sufficient** framing is the *faithfulness criterion* the I2 gate checks (necessary → no Directive exceeds the condition; sufficient → the set covers it), **not** a deterministic decomposition algorithm — the partition is judgment. Each proposal carries a line-1 `Parent Initiative: #N` marker and **no `## MISSION fit` field** (the parent-XOR is satisfied by the marker; the Initiative traces to MISSION upstream).
4. **Faithfulness gate** — invoke `activation-reviewer` (extraction-faithfulness I2, §4.9) over the **whole proposal set** (coverage is a set property, not per-Directive); on `revise`, refine and re-invoke (two-round backstop as `/file-directive`).
5. **File** — `gh issue create --label directive --label status:proposed` for each proposal (body via `--body-file` so upstream-authored Initiative text is never interpolated into a `gh` argument position — this covers the injection surface; content-trust is handled by the I1/I2 gates + human activation, not by `--body-file`). The extracted Directives are **never activated** — activation stays human/reviewer-gated via `/activate` (generation-open / decision-gated, §5.7). At create time `label-parent-consistency` is not matched (it is scoped to `--add-label`), so the **I2 gate is the effective parent-XOR check** for the create path (it sees the full proposal bodies before filing).

Audit category: `initiative-consume` (decisions: `extracted` with count=N, `under-specified` (contract gate stopped the flow), `revise`). The command never edits/closes/relabels the Initiative (M1.2 `initiative-readonly`); never activates the extracted Directives; never files a proposal without the `Parent Initiative: #N` marker. Escape (reviewer gate): `SKIP_HOOKS=directive-review`.

### 5.22 `/initiative-feedback <N> {--challenge "<reason>" | --completion}` (Issue #260; Directive #259)

The **upward-feedback** flow (SPEC §1.7 "Feedback is upward and non-deciding") — the execution layer surfaces findings to an Initiative as structured comments, never as strategic decisions. One command, two modes (mirrors `/resolve-discussion`'s two-path shape). Both modes require the `initiative` label on `#N` (else error), operate same-repo, and write **only** a `gh issue comment --body-file` (the sole Initiative write — allowed by `initiative-readonly`; `--body-file` for the injection surface). Each comment leads with a **scannable marker** (`## Initiative challenge` / `## Initiative completion`, modeled on `## AC closeout`) so an upstream consumer could parse it. These markers are projected into the `initiative:challenged` / `initiative:completion-requested` labels by the target-repo `initiative-feedback-label.yml` CI Action (#359, §1.7) — applied by CI, never by eng's runtime, preserving `initiative-readonly`.

- **`--challenge "<reason>"`** — when execution reality contradicts the Initiative (a termination condition that looked measurable but isn't; a real dependency / measurability / implementation cost the planning layer structurally cannot have), post a `## Initiative challenge` comment carrying that **code-derived** information and **requesting re-evaluation**. It **escalates** — it does NOT reject, retire, or close the Initiative (mirrors a reviewer `block` escalating rather than auto-discarding, §4.9.1). The decision stays upstream; the shell does not track what happens next.
- **`--completion`** — when the Directives extracted from the Initiative (`/consume-initiative`, §5.21) have all landed, post a `## Initiative completion` comment carrying the **evidence** (the merged Directives / PRs) and **requesting termination assessment**. It reports the fact and requests assessment; it does NOT assert the termination condition is met and does NOT close the Initiative — asserting completion is a strategic judgment that stays upstream (the same logic that forbids rejection).

Audit category: `initiative-feedback` (decisions: `challenge`, `completion`). The command never edits, closes, reopens, relabels, or retires the Initiative (M1.2 `initiative-readonly` is the runtime backstop); the only write is the structured comment. With §5.22 the milestone-v2 model is **fully live** — recognition (M1.1) + enforcement (M1.2) + reviewer (M2) + consume (M3) + feedback (M4) + derived-view (the mirror's Project `Parent` field for Initiative-parented Directives, #262).

### 5.25 `/recall <topic>` (Issue #422)

The **selective-injection** affordance of the context dual (MISSION "The mechanism") — an on-demand query into durable memory's `active → archived` half. SessionStart restore (§6.5(b)) re-injects only the *current* task's artifacts; `/recall` retrieves *prior, unrelated* decisions from the decision record on demand, answering "have we addressed this shape before?" A thin command file (`.claude/commands/recall.md`) delegating to `helpers/recall.sh::recall_pointers`, mirroring the `/status` → `status.sh` precedent (§5.5).

`recall_pointers <topic>` fans out across the decision record's substrates and merges into one **pointers-only** list:
- closed issues — `gh search issues --state closed`;
- merged PRs — `gh search prs --state closed`;
- ADRs — title/H1 match over `docs/ADRs/*.md`.

**Pointers-only is the load-bearing narrowing safeguard** (MISSION; the same fixed-size-pointer discipline §6.5(d) applies to the friction advisory): each result is one line — a tag (`#` / `PR#` / `ADR-NNNN`) + number + the one-line title — **never a body**. The guarantee is *structural*, not advisory: the `gh` arms project `--json number,title`, so a body is never fetched into reach (a later edit adding `body` to the projection is the one regression path — smoke pins against it). The count is **bounded**: a per-substrate cap `RECALL_LIMIT` (default 5, env-overridable; 15-line worst case) so no substrate starves the others and the worst-case context cost is fixed and documented. **Fail-open**: any `gh`/grep error (offline, rate-limited, unauthenticated) yields a single `recall: decision record unavailable` line — never a block, never a partial body dump.

Out of scope (Issue #422): a vector/embedding store (the v0 substrate is `gh search` + `git grep`); changing §6.5(b) injection; auto-running `/recall` without an explicit invocation (the `/work-on` pre-planning step is a deferred follow-up).

### 5.26 `/replan-check` (Issue #427)

The **in-loop correction** of the selective-injection half — a checkpoint that keeps the Plan (a task-scoped injection manifest, MISSION "The mechanism") reflecting truth as execution proceeds. `planner` is one-shot (then `plan-reviewer`); without this, a Plan the actual diff has outgrown silently stops being a faithful manifest, eroding PR-as-living-doc (§1.4). A thin command (`.claude/commands/replan-check.md`) delegating mechanical fact-gathering to `helpers/replan_check.sh`, with the divergence **judgment** kept in the skill (LLM) — the `/recall`→`recall.sh` split.

`replan_check.sh` surfaces only the mechanical facts, fail-open: the actually-touched files (`git diff --name-only <base>...HEAD`, `<base>` from `gh pr view --json baseRefName`, default `main`), the linked issue's AC checkboxes, and the PR-body `## Plan` / `## Checklist` / `## Key context` / `## Out of scope` (a touched file the Plan declared out-of-scope is a strong structural signal). The skill then applies the **structural-vs-cosmetic discriminator**:

- **Structural → re-invoke `planner`** (mirroring `/work-on` step 6; in `unattended`, a clean `plan-reviewer` re-substitutes for approval): EITHER (a) the diff touches a **load-bearing** file/area the Plan/Checklist/Key-context never anticipated (a new module, a schema/contract surface, an un-named subsystem) — *not* a mechanical incidental; OR (b) an **AC is rendered unreachable** — the approach taken closes off a path a linked-issue acceptance criterion required.
- **Cosmetic → no trigger** (a one-line advisory at most): files that are the **direct mechanical consequence** of a planned change (a regenerated TOC, a changelog fragment, the SSOT-coupling the Plan already implies — e.g. the §8 tree bump for a planned helper), in-scope reordering/renaming, or any change leaving every AC reachable.

The threshold is **asymmetric by design** (§6.0): re-invoking `planner` is expensive and the judgment is fallible, so it **fails toward not-triggering** — when in doubt, emit the advisory, do not re-plan. This is **positive enforcement** (a guiding checkpoint), not a hard gate; a missed nudge is recoverable, a wrong forced re-plan is not. `/sync-pr` (§5.4) references it as a post-sync advisory step (the body was just curated — the natural cadence to compare it against the diff); `/ship` inherits the reference transitively (§5.7 step 5). Trigger is explicit/invoked — the auto-on-every-phase-commit variant is deferred (parallel to §5.25's auto-`/recall` deferral).

### 5.27 `/reconcile-spec` (Issue #464; Directive #455)

The **judgment + action** half of code-vs-SPEC drift reconciliation — the consumer of the measure-first detector `scripts/spec_drift_candidates.sh` (§6.5(d), #462). The detector surfaces code-ahead drift *candidates*; `/reconcile-spec` reads the relevant SPEC section(s) and the drifted commits/code and classifies each candidate into **exactly one of three dispositions, before any SPEC edit**:

- **spec-ahead** — SPEC leads, the code is still being implemented → **track, no edit**.
- **code-ahead-correct** — the code is right and SPEC is stale → **propose a SPEC update to match the code**, applied **only after explicit user approval**.
- **code-wrong** — the code is a bug and SPEC is right → **never edit SPEC**; route to a code fix / file a bug.

**The load-bearing invariant (Directive #455): the shell never silently rewrites SPEC to match possibly-buggy code.** Disposition `code-wrong` *never* edits SPEC, and `code-ahead-correct` edits it *only* after explicit user approval — the correction is human judgment, never the shell's autonomous act. **Operating mode**: in `attended` the command surfaces the classification + proposed edit and pauses for approval before applying; in `unattended` it **never auto-applies** a SPEC correction (a SPEC-to-match-code edit is a content/direction change, and the directing layer never autonomously sets direction — §1.7) — it surfaces/parks the candidate instead. Classification *correctness* is agent judgment (mirroring `/onboard`'s SSOT-drafting half), not a mechanically-verified property.

Each handled candidate is audit-logged under the **`spec-reconcile`** category with its decision: `applied` (a `code-ahead-correct` SPEC edit, after approval), `tracked` (`spec-ahead`), `routed` (`code-wrong` → code fix / bug), or `parked` (surfaced in `unattended`, not applied). Full procedure lives in `.claude/commands/reconcile-spec.md`. Escape: `SKIP_HOOKS=<cat> SKIP_REASON='<why>'` per §7.

### 5.28 `/implement` (Issue #486; Directive #477)

**Manifest contract (input the command assembles).** The subagent has no access to the conversation, so the command gathers the explicit manifest first: the **Plan** (PR-body `## Plan` / `## Checklist`, the planner output §4.1), the **failing Phase-B test** + the exact command to run it, and the **named relevant file paths** (PR-body `## Key context`). ## 6. Hook enforcement

### 6.0 Enforcement style

The action-axis principle of MISSION "The mechanism" — enforcement has a *negative* face (block a wrong action) and a *positive* face (supply the right next one), and both are necessary — lands here as the design rule for every matcher, command, and reviewer the shell adds. The §6.1 **Framing** paragraph below states the negative side's posture (a mistake-prevention guardrail, not a sandbox); this section is the cross-cutting rule that decides, for any given discipline, *which* face to give it and *where* to put it. Of the rules below, **P1, P3, and P4 derive from** that MISSION principle; **P2 is SPEC-local** — an engineering-physics rule about *where* a given enforcement runs (local hook vs remote backstop), with no MISSION parent.

**P1 — Cost-asymmetry selects the face (the criterion).** Choose negative vs positive by the cost of being *wrong*, not by how important the rule feels. Where a wrong *allow* is irreversible or corrupts shared history — a force-push onto a protected branch, a leaked secret, a destructive `rm` — the rule is a hard negative gate that **fails safe toward blocking**; the force-push matcher (§6.1) reasons exactly this way (a wrong allow clobbers shared protected history — irreversible; a wrong block costs naming the branch — recoverable). Where a wrong *nudge* is simply ignored at no cost — a `next:` hint, a scaffolding command — the rule is positive and is laid generously. Corollary: **negative gates are narrow and precise, positive guidance is broad and abundant**, because the two failure costs are asymmetric (the type-aware carve-outs of §6.1 are negative gates being made *more precise* to avoid false blocks, not weaker).

**P2 — Enforcement physics: pick the layer for the bypass model.** A local PreToolUse hook fires pre-action but is **bypassable** (eval, `bash -c`, dynamic construction — see §6.1 Framing); a remote gate (CI, branch protection) fires later but is **unbypassable** and runs regardless of who acts. An irreversible rule therefore cannot rely on the local hook alone — it needs a remote backstop, with the local hook as its **pre-image** (the in-flow, fast-feedback copy of the remote check; e.g. the local changelog presence check is the pre-image of the `check-changelog` CI gate, §18). Reversible / hygiene rules can live at the local layer only.

**P3 — The face is not fixed; it migrates with trust.** A rule is typically born **advisory** (a reviewer or skill recommendation), **hardened** into a hook once the pattern proves out (e.g. #199 promoted the advisory #197 label↔parent layer into the `label-parent-consistency` matcher — §6.1), and **narrowed** again if it starts fighting the prescribed flow (the force-push matcher was relaxed from a blanket block to allow the named-branch force-push of the §13 rebase-pull tail). This migration is the action-axis form of "the shell improves under use" (MISSION). The two directions were historically **not equally observable**: narrowing pressure shows in the audit log (escape clustering on one category/reason — §7; the MISSION success signal "the escape hatch stays narrow"), surfaced by `scripts/narrowing_candidates.sh`. Promotion pressure once left no audit trace — an ignored advisory was invisible — but the reviewer-reject reason-class trail (#361) now records each non-pass reviewer verdict (issue-/plan-/activation-reviewer) with a categorized token, and `scripts/promotion_candidates.sh` aggregates those into advisory→hook promotion candidates above a threshold. Both directions are thus observable; the narrow/promote decision still stays human / reviewer judgment (the consumers **surface** candidates, they do not act — tracking #41 principle #3). **The audit log is a *dual-positive* channel (Directive #356):** a block's in-band message is its immediate positive face (it names the alternative, P4), and the line it appends to `audit.jsonl` is a *deferred* positive face — but that line is only **latently** positive: it becomes an actionable positive channel solely once a consumer is wired to read the aggregate (the two scripts above). An unconsumed log is an inert record, not a positive channel; wiring the consumer is what completes the deferred positive face. **That consumer is now wired (#398, Directive #391):** the SessionStart §6.5(d) friction advisory triggers the two readers (plus the `unattended-park` park-frequency signal bridged in by §5.7.1) once per session, TTL-gated and fail-open, surfacing accumulated friction as a one-line pointer without a human remembering to run a script — so the deferred positive face is no longer latent. A third reader, `scripts/ceremony_candidates.sh` (#401), extends the same surface to **ceremony** mis-sizing — mined from commit history rather than `audit.jsonl` (the ceremony signal is not recorded there) — and is itself **born advisory**: it observes whether ceremony mis-sizing recurs *before* any §1.2 weight model or relaxation gate is hardened against it, the P3 order applied to the device's own roadmap. The narrow/promote decision still stays human / reviewer judgment (the advisory **surfaces**, it does not act — tracking #41 principle #3).

**P4 — Pair the faces; never ship a bare gate.** A negative gate names its positive alternative in the block message (a `status:proposed` branch attempt is blocked with "run `/activate <N>`"; a bare force-push is blocked with "name the target branch") — a block that only says "no" is half a rule. Defaults reinforce the pairing: enforcement **fails open** (a missing helper warns and proceeds, never locks the developer out — see the §6.1 fail-policy table) and the **escape hatch is always open but audited** (§7). A reviewer is the paired form inside one agent — it produces an artifact (positive) and gates on a verdict (negative), a *generative gate* (§1.5). Evaluate any enforcement proposal against this: a hard block with no named alternative, or guidance with no gate behind it, is the one-sided regression MISSION warns of.

### 6.1 PreToolUse — block

**Framing**: Hooks are a **mistake-prevention guardrail**, not a security boundary. An adversarial user can bypass them via shell features the matcher doesn't model (`eval`, `bash -c`, `python -c`, heredoc-spawned shells, dynamic command construction via parameter expansion). The hook's job is to stop honest mistakes — typos, autopilot moments, the wrong tab — not to be a sandbox. Patterns that suggest a bypass attempt are not blocked but are **audit-logged as warnings** so the trail is preserved for review.

Implementation note: command strings are **normalized** before matching — backslash-continued multiline (`\\\n`) collapses to single-space, internal newlines become spaces, multi-space whitespace collapses. So `git \\\n  push \\\n  --force-with-lease` matches the same blocked pattern as the single-line form. Normalization also **re-separates a glued unquoted command separator** (`&&`, `||`, `;`, `|`, and a lone background `&`) into a space-padded segment boundary, so an operator glued to an adjacent token — `git commit -m "x"&&git push --force origin main`, no space before `&&` — does not get folded into a single token by the `parse_env_prefix` shlex round-trip and dissolve the following `git push` verb (the entry-grep keys on `git push`, so a destroyed verb is a **false-negative** on the force-push/protected gate — §6.0 P1, #446). Only separators **outside** quoted string values are split: a `&&`/`;`/`|`/`&` *inside* a quoted `-m`/`--message` value is data, not a boundary, and is never re-separated (no invented boundary — composes with the #440 message-value elision). A lone `&` that is part of a **redirect** (`>&`, `&>`, `N>&M`, `<&`, `&>>` — i.e. immediately adjacent to `>` or `<`) is a redirect operator, not a background separator, and is **never** re-separated, so a legitimate `git push --force origin feature 2>&1` is not over-blocked (#476). `push_segments` splits on the same set (incl. `&`), so a glued `&` cannot collapse a sibling clause's protected-looking token into the `git push` segment (#437 sibling-isolation parity). The glued and spaced forms of every separator are therefore treated identically by every downstream git matcher.

**Git option-prefix tolerance**: every `git <subcommand>` matcher accepts standard git-level options between `git` and the subcommand: `-c <key>=<value>`, `-C <path>`, `--git-dir=<path>`, `--work-tree=<path>`, `--no-pager`, `--literal-pathspecs`, `--exec-path[=<path>]`, `--namespace=<ref>`. So `git -c commit.gpgsign=false commit -m "…"` and `git -C /some/repo push --force` both fire the same downstream checks as the bare `git commit` / `git push` forms — the option prefix can no longer be used (intentionally or not) to slip past the matcher.

**Type-awareness** (introduced alongside dir-mode, §1.7; matchers wired by tracking #41 child #5, issue #46): the engineering-flow matchers below — branch-from-issue, AC-closeout, PR-link / `Closes #N` checks, default `gh issue list` wrappers — **do not apply their engineering-flow gate to a `Type=Directive` Issue**. Directive Issues are gated by `/complete-directive` + `activation-reviewer` (§4.9), not by AC checkboxes or `Closes #N` semantics. (For **AC-closeout** specifically the carve-out is **all-or-nothing**, not per-item: the `gh pr merge` gate is skipped only when **every** closing Issue is a Directive — a mixed PR closing both a Directive and an Execution Issue still gates on the Execution Issue's unchecked AC; see the AC-closeout matcher row below.) The `proposed-protect` matcher in the table is the inverse: it actively blocks engineering-flow actions (most prominently `/work-on`) when the target Issue is **either** `status:proposed` (any type — not-yet-activated work, generalized by #171) **or** a Directive (any status — Directives never branch, §10.5). The two conditions guard distinct invariants (lifecycle vs type) that share one branch-creation trigger; `proposed-protect` subsumes the former `directive-protect` rather than replacing it (a status-only check would let an Active Directive become branchable — a §10.5 regression). The implementation predicates (`is_directive_issue <issue-#>`, cached; and `is_proposed_issue <issue-#>`, uncached because the `status:proposed` label is volatile under `/activate`) and the `safe_source` fail-policy entry live in `helpers/issue_type.sh` (#46, #171).

**Session-restart caveat for hook-spec changes.** Claude Code binds its hook routing from `.claude/settings.json` + the working tree at session start. Adding a *new* helper file under `.claude/hooks/helpers/` (or any path referenced by an existing hook via `.`/`source`) does not retroactively become reachable to a session that started before the file existed. Such a session continues to execute the *updated* `pre_tool_use.sh` (re-read per invocation), but a `. "$SHELL_ROOT/.claude/hooks/helpers/<new-helper>.sh"` line silently fails — and any function defined only in that helper is undefined, so downstream calls return rc=127 and matchers can fall through without ever reaching their block / warn arms. **The fix is to start a fresh `claude-eng` session after pulling a hook-spec change that introduces a new helper file.** The hook-side mitigation is the **`safe_source`** utility (hosted in `.claude/hooks/hookrt.sh` — the "hook runtime" bootstrap that also hosts `audit_log`): every helper source line — both **hook-to-helper** (the 5 hook files under `.claude/hooks/` sourcing helpers) and **helper-to-helper** (a helper under `.claude/hooks/helpers/` sourcing another helper from the same directory, e.g. `branch_guard.sh` → `git_matcher.sh`) — goes through `safe_source <helper-path> <audit-category>`, which presence-checks the file, sources it, and on miss emits `audit_log warn <category> helper-missing "<path>"` and returns 1. The one documented exception is the **bootstrap-to-runtime shim** at `helpers/log.sh:6` → `hookrt.sh`: that source line can't go through `safe_source` because `safe_source` lives in the file being sourced (chicken-and-egg), so it falls back to a primitive `.` and is stderr-only on miss. The scope predicate for "in scope of the `safe_source` invariant" is: the source target is under `.claude/hooks/helpers/`, NOT directly under `.claude/hooks/` (i.e. `hookrt.sh` and the 5 hook entry-point files are exempt — they are the runtime, not helpers). The shape this closes — one site warning while a sibling source-site of the same helper fails silently, hiding the second consumer — was the regression #36 surfaced. The hook continues to **fail-open** (matcher arm short-circuits, command proceeds) so a missing helper is recoverable, not a developer-lockout — but the failure mode is visible in `audit.jsonl` instead of silent. Existing helpers that the running session already loaded (e.g. `helpers/coauthor.sh`) are unaffected — their function bindings persist across invocations of the parent hook. Cross-reference §3.2.1 (hook command path resolution).

**`helper-missing` audit-event shape** — the on-miss line emitted by `safe_source` is:

```json

```

`reason` and `cwd` are JSON-escaped per SPEC §7. `decision` carries the constant token `helper-missing`. **`helper-missing` audit categories are a superset of `SKIP_HOOKS` categories** — for example, `status` is a valid audit category for an informational helper but is NOT a `SKIP_HOOKS` category. The escape-hatch list in §7 stays unchanged; the audit categories enumerate what `safe_source` can emit, which is broader.

**Matcher pass-through audit contract.** PR #32 surfaced a class of bug where a matcher entered (its `grep -qE 'pattern'` matched) but no terminal arm fired — the matcher silently exited with no audit record. This made "matcher entered but produced no audit" indistinguishable from "matcher never matched the command" — a structural blind spot in the audit trail. The contract closes that gap:

> **For every matcher in `pre_tool_use.sh`, if the matcher entered (`grep` matched), `audit.jsonl` MUST gain exactly one record whose `category` field matches the matcher's category in this hook invocation.**

The terminal arm fires that record (or marks it as decided without emission):
- A `block` decision emits `audit_log block <cat> deny <reason>`.
- A `warn` decision emits `audit_log warn <cat> <decision-token> <reason>` (`decision-token` ∈ {`notice`, `helper-missing`, `pass-through`, ...}). Note `bypass-suspect` is a *category* (with decision-token `notice`), not a decision-token.
- A **high-frequency happy-path** arm (e.g., every clean `git commit`, every in-scope `rm -rf`, every local `--amend`) calls `mark_allow <cat>` (in `hookrt.sh`). This sets the caller's `decided` flag with NO audit emission — the matcher's contract is satisfied (it entered, evaluated cleanly, allowed) without per-action audit noise. Distinct from the explicit-`notice` warn path, which is used for low-frequency designed-allow cases where an audit record is desired.
- If no other arm fires by the matcher's tail, the helper `pass_through_trace <cat> "<cmd>"` (also in `hookrt.sh`) emits `audit_log warn <cat> pass-through "<truncated-cmd>"`. This is the anomalous case — the matcher decided to do nothing without naming why, which we treat as a soft regression worth flagging.

Implementation pattern (per Bash-branch matcher in `pre_tool_use.sh`):

```bash
if printf '%s' "$cmd" | grep -qE 'pattern'; then
  decided=
  # ... terminal arms set `decided=1` ...
  [ -z "$decided" ] && pass_through_trace <cat> "$cmd"
fi
```

The `decided=` flag is reset at the head of each matcher block so state cannot leak across matchers. Smoke (`scripts/test/smoke.sh` §39) asserts the invariant:
- **§39a**: for each matcher, a synthesized benign-but-entering command produces exactly one new audit record with the expected category.
- **§39b**: structural awk-grep over `pre_tool_use.sh` — every `if printf '%s' "$cmd" | grep -qE ...; then` block ends with either an explicit `decided=1` audit-emitter or a `pass_through_trace` tail.
- **§39c**: negative — a command that enters no matcher (e.g. `ls -la`) produces zero new audit records.

**Decision-token vocabulary** (`decision` field in `audit_log`):

| Token | Used by | Meaning |
|---|---|---|
| `deny` | `block` | Hook refused the command; `exit 2` follows. |
| `notice` | `audit_log warn` | Soft signal — degraded enforcement (gh timeout, indeterminate state, explicit allow with reason). |
| `helper-missing` | `safe_source` on miss | A hook helper was absent; matcher arm short-circuited. |
| `bypass-suspect` | bypass-suspect matcher | Command looks like an eval/heredoc bypass (warn-only, not a block). |
| `pass-through` | `pass_through_trace` | Matcher entered but no terminal arm fired — anomalous; treat as a soft regression. |
| `skip` | `audit_log escape` | `SKIP_HOOKS=` was honored; categorized as `escape`. |
| `reject` | reviewer-reject audit (#361) | A reviewer subagent returned a non-pass verdict; the `reason` carries `class=<reason-class> issue=#<N>`. |

The token list is intentionally short and stable; future arms add new tokens here, not new `event`-type kinds.

**Event-kind set.** The `event` field is one of `block` / `warn` / `escape` / `info`. `info` is an **informational** record — not a refusal — e.g. the AC-closeout Type=Directive skip (`pre_tool_use.sh`) and the dir-mode lifecycle notices (`audit_log info directive-*`, §5.10+), typically carrying the `notice` decision token. This four-member set is the stable one the "not new `event`-type kinds" rule above refers to: arms add new `decision` tokens, not new event kinds.

**Audit `source` field** (live/test origin discriminator, #361 / Directive #356 signal 1). Every `audit_log` record carries an additive top-level `source` field — `live` for real sessions, `test` for test-harness fixture fires. It is the **last** field in the JSON object (after `cwd`), so it does not perturb the `reason`-anchored §50a format scan or `_audit_validate_format` (which inspects only `category`/`decision`/`reason`, never the record shape). The value resolves from a single harness-owned environment marker: `source=test` **iff** `CLAUDE_ENG_AUDIT_SOURCE` is exactly the token `test`; **any** other value — unset, empty, `smoke`, `live`, or arbitrary — resolves to `live` (fail-safe-to-`live`). The field is additive and append-only-safe: historical lines written before #361 have no `source` key (`.source == null` under `jq`); consumers treat an absent value as legacy/unknown (fail-open on absence).

**Anti-reclassification invariant** (Directive #356 Constraint — a `live` action must never be silently reclassifiable to `test`/`smoke` to dodge a friction signal). The marker is read from the **hook subprocess environment**. Per §7, the live Claude Code Bash tool consumes a leading `VAR=val` as the *spawned command's* own environment — it never lands in `tool_input.command`, and the hook subprocess does not inherit it — and shell state does not persist between Bash-tool calls. So a real action cannot inject `CLAUDE_ENG_AUDIT_SOURCE=test` into the hook subprocess that writes its audit record; only the process that **launches the session** (e.g. `scripts/test/smoke.sh`, which `export`s the marker before driving the hooks directly — mirroring the `ENG_STATE_DIR_OVERRIDE` test seam, §3.2.2) can set it. Smoke pins the invariant: a fixture fire with the marker unset records `source=live`, and one with a *forged* `CLAUDE_ENG_AUDIT_SOURCE=smoke` (a non-`test` value) still records `live`. A query can therefore partition the log by origin (`jq 'select(.source=="live")'` vs the all-records count) to exclude test-fixture noise from the existing backlog — the observability surface Directive #356 signal 1 requires (the forward noise source itself was already removed by #357's whole-run state isolation).

**Reviewer-reject reason-class vocabulary** (#361 / Directive #356 signal 3). When the three reviewer subagents (`issue-reviewer`, `plan-reviewer`, `activation-reviewer`) return a non-pass verdict (`refine`/`revise`/`reject`/`block`), the invoking skill flow emits a categorized audit record via the `reviewer_reject_audit` helper (`helpers/reviewer_audit.sh`): `audit_log warn <category> reject "class=<reason-class> issue=#<N>"`. The `<category>` is the reviewer's audit category — `issue-review` (`/file-issue` → issue-reviewer), `plan-review` (`/work-on` → plan-reviewer), or `activation` (`/activate` and `/complete-directive`, which both drive activation-reviewer). These are **audit-emission** categories, **not** `SKIP_HOOKS` escape categories — they are not reachable via `should_skip` and so are deliberately absent from the §7 escape list (the `helper-missing`/audit-category set is a superset of the escape set). The `<reason-class>` is one of a small controlled vocabulary the skill maps the reviewer's free-text reason onto (nearest token): `schema-incomplete`, `unverifiable-ac`, `scope-bleed`, `mission-misfit`, `conflict`, `evidence-insufficient`. This produces the per-reviewer × reason-class trail that a promotion-candidate report (Directive #356 signal 4, a sibling Execution Issue) aggregates. The mapping is a skill-side judgment (the same free-text→token summarization `/activate` already performs); the helper only formats and emits.

Tests are run by `/ship` (§5.7). During the draft PR's commit/push, only quick lint/format runs. A commit-time test gate would force WIP/share commits to escape every time, polluting the audit log — so we don't add one.

**Lint timeout**: the commit-time lint invocation is bounded by `CLAUDE_ENG_LINT_TIMEOUT` (seconds, default 30) so a misconfigured or slow linter cannot hang the commit. Timeout fires as a `block format` decision, escapable via `SKIP_HOOKS=format`. If neither `timeout(1)` (coreutils) nor `gtimeout(1)` is on `PATH`, the lint runs unbounded with an `audit_log warn lint-timeout-absent` entry (decision token `notice`) recording the missing dependency — the hook neither hangs the user nor silently downgrades enforcement. `lint-timeout-absent` is an **audit-emission** category (like the reviewer-reject categories above), not a `SKIP_HOOKS` escape category, so it is deliberately absent from the §7 escape list.

#### 6.1.1 Conditional issue # enforcement

| commit type | issue # required | Rationale |
|-------------|------------------|-----------|
| `feat`, `fix`, `docs`, `refactor`, `perf` | Required | External surface / behavior change |
| `test`, `style`, `build`, `ci`, `chore`, `revert` | Optional (passes without it) | Internal change |

Structural regex:
```
# required group
^(feat|fix|docs|refactor|perf)(\(#[0-9]+\))!?: .+$
# optional group (issue # optional)
^(test|style|build|ci|chore|revert)(\(#[0-9]+\))?!?: .+$
```

Subject length is measured in **codepoints (1..72)**. Byte-based measurement (`wc -c`, `${#var}`) is forbidden — under multibyte locales it mis-truncates Korean/CJK. The choice of tool is an implementation detail.

**Subject extraction across `-m` forms**: the hook accepts both the plain quoted form and the command-substitution + heredoc form that the harness uses for multi-line bodies (`Co-Authored-By`, etc.).

| `-m` form | Subject |
|-----------|---------|
| `-m "feat(#1): subject"` / `-m 'feat(#1): subject'` | The quoted text (its first line) |
| `-m "feat(#1): subject⏎⏎body"` (embedded newlines, single `-m`) | The **first line** of the value only — extracted from the **raw** command (whitespace normalization flattens the newlines, so a normalized-command parse would capture subject+body as one over-length blob) |
| `-m "subject" -m "body"` (multiple `-m`, git's multi-paragraph form) | The value of the **first** `-m` (not the last) |
| `-m "$(cat <<'TAG'` … `TAG)"` (heredoc body inside the `-m` value) | The first non-blank, non-tag line of the heredoc body — **but only when the value's own first non-blank line is not already a valid Conventional-Commit subject** (line-1 precedence, below) |
| `-m "feat(#1): subject⏎⏎…body mentioning `<<EOF`…"` (normal body that merely contains a `<<TAG` token) | The **line-1 subject** — a `<<TAG` appearing as prose in the body does **not** make it a heredoc message |
| `-F <file>` / `--file <file>` (no inline `-m`/heredoc message) | **Empty** — the message lives in a file the hook cannot read; the format check is **skipped** (fail-open) |

The heredoc tag may be single-quoted (`<<'EOF'`), unquoted (`<<EOF`), or indent-stripping (`<<-EOF`); the extractor returns the first body line that is not blank and not the closing tag, leading whitespace stripped, before the Conventional-Commits regex runs.

**Line-1 precedence** (#383): the heredoc walk is entered **only when the `-m` value's first non-blank line is not itself a valid Conventional-Commit subject**. The genuine substitution form's first line is the `$(cat <<TAG` opener — never a valid subject — so it falls through to the heredoc body as before. But a plain single `-m` whose first line *is* a valid subject and whose body merely mentions a `<<TAG` token (common when the change is *about* shell/heredocs) returns line-1 directly, rather than mis-reading a body line as the subject. This is fail-safe: line-1 is returned only when it passes the Conventional-Commits regex, so a malformed line-1 always falls through to the existing over-extracting (block-leaning) behavior.

**Extraction is bounded to the `git commit` command and operates on the raw command** (before whitespace normalization): it scans from the `commit` token onward, so a heredoc or `-m` belonging to a **sibling** command in a compound invocation — e.g. a `cat > file <<EOF … EOF` redirect or another tool's `-m` chained before the commit — is never mistaken for the subject. A heredoc is treated as the subject **only** when it is the commit's own `-m "$(cat <<TAG)"` message value, not any heredoc elsewhere in the command line. python3 drives the primary parse; the legacy heredoc-walk + single-line `sed` is the fallback when python3 is absent (degrading toward the prior, over-extracting behavior — which fails toward *blocking*, never toward letting a malformed subject through).

Implementations expose a single helper (`extract_commit_subject`) consumed by `pre_tool_use.sh`'s commit gate. An empty return (no `-m`/heredoc message in the commit segment) makes the gate skip the format check (fail-open); the gate only ever blocks a subject it positively extracted and found malformed.

### 6.3 Stop — recommendation after each response

Claude Code's `Stop` hook fires **after every model response**, not at session end. To avoid noise, suggest only once every N responses:

- uncommitted changes + no recent review → suggest `/review`
- recent commit + PR body checklist out of sync → suggest `/sync-pr`

Suggestions print one line to stderr (no blocking). Throttling is a simple modulo of N (default 5, overridable via `$CLAUDE_ENG_STOP_THROTTLE`) — no per-category state file.

Session-level cleanup (branch tidy, temp files) is a **deferred capability** (§0.4) — neither a `SessionEnd` hook nor a `/wrap` command is specified or built today.

### 6.4 UserPromptSubmit — context injection

Before each user message, inject the **compact status** as system context — `status_compact` (`helpers/status.sh`, the same projection §5.5 describes): current branch, the issue / PR one-liners, phase, `next:` hint, mode, shell-root, state, and work-lang. This is a small per-turn refresh, not a full context reload.

The heavier one-time injection — the full current-PR body, the referenced-issue body, and the MISSION summary — is **SessionStart's** job (§6.5(b) Session restore), loaded once when a session resumes in-scope, not re-injected every turn (CLAUDE.md is loaded whole natively by Claude Code, not part of this injection). Keeping the per-turn surface to `status_compact` is the narrowing half of the MISSION mechanism (§1, "as small and relevant as possible"); SessionStart selective-injection is the dual.

### 6.5 SessionStart — shell sync check + session restore + SessionStart banner + friction advisory + SSOT-presence health

At session start, do five things:

**(a) Shell sync check**
- Is the shell repo behind origin/main? If so, warn (no auto-pull; sudden behavior changes are risky).
- The `git fetch` is gated by a stamp file at `.claude/state/last-shell-fetched`. The fetch runs only when the stamp is older than `SESSION_START_FETCH_TTL` seconds (default 21600 = 6 h) or absent. Most sessions therefore skip the network call — the `git rev-list --count HEAD..@{u}` still runs against locally-known refs, so the "behind" warning is correct relative to whatever was last fetched (up to 6 h stale, which matches the warning's polite-not-critical role).
- The fetch is wrapped in `timeout $SESSION_START_FETCH_TIMEOUT` (default 5 s) so a hung network does not stall session startup. If `timeout(1)` / `gtimeout(1)` is unavailable the fetch runs unbounded, matching SPEC §6.1 / `CLAUDE_ENG_LINT_TIMEOUT`'s fallback pattern.

**(b) Session restore** — to seamlessly resume the previous session, inject once:
- The current branch's full PR body (if any)
- Full referenced issue body (if any)
- Target `MISSION.md` head (`head -n 20 MISSION.md`)
- One `/status` snapshot

`.claude/CLAUDE.md` is **not** hook-injected here — Claude Code natively loads it whole every session. Because it is an always-on injected surface it is held to the §9 thin-pointer discipline + 12000-byte budget (smoke §104).

Subsequent turns only refresh the short status summary from §6.4.

**(c) SessionStart banner**

One silent-no-op state evaporates the §0.1 enforcement guarantee with no operator signal: the runtime that hosts enforcement is missing. SessionStart surfaces it via a primitive inline `printf` stderr banner, debounced to one fire per session and independent of `hookrt.sh` / `audit_log` / `safe_source` (the runtime that is itself the missing piece).

**Inject-consistency banner — removed (#318, Directive #311).** This banner is **gone**, not re-conditioned. Its premise died with the #312 binding: it fired when `$CLAUDE_ENG_SHELL_ROOT` was unset and `settings.local.json` was an injected symlink, warning "every hook will no-op". Post-#312 that is the **normal working state** — a plain `claude` in an injected target resolves hooks via the per-project binding symlink `${CLAUDE_PROJECT_DIR}/.claude/eng-shell-root/...` and `session_start.sh` self-locates + back-fills the env (§3.2.1), so enforcement is fully on. The banner only false-fired. And the *genuine* residual no-op (the binding symlink missing/broken **and** the env unset) is **structurally undetectable from SessionStart**: the SessionStart hook command itself traverses that binding (`settings.injected.json`), so a broken binding means `session_start.sh` never runs and the banner inside it can never fire — the banner would sit behind the very link it claims to check. No condition can both detect the failure and execute in it; the honest fix is removal. The forward-looking detector lives at **launch** time instead: `bin/claude-eng <dir>` warns (non-blocking) when an injected target's `.claude/eng-shell-root` binding is missing/dangling, before `exec claude` — it runs *before* the session and resolves the dir directly, so it is the one place that can surface the broken-binding no-op (#334, §3.5).

**Registry-zeroed detector** (#502, Directive #498) — the per-project scope-guard registry (`eng-state/registry.txt`, §3.2.2) going **empty** silently turns ALL enforcement off, because the global `in_scope` gate fails open on an empty/absent registry (by design, §3.2.2 / §773). An audit found this *zeroing* is reachable by an innocuous single command — `: > registry.txt`, `truncate`, or a flagless delete-then-touch — and was previously **traceless**. `session_start.sh` now detects a **present-but-EMPTY** registry (the distinguishable "armed-then-disarmed" signal; an *absent* registry is the normal pre-registration / unregistered-cwd transparent case and does **not** fire) and emits the same once-per-session banner + an `audit_log warn registry-zeroed` record. **This is a §1.4 decision, NOT a fail-open→fail-closed flip:** the empty-registry transparency is the deliberate guardrail-not-sandbox posture (§6.1 Framing) and is unchanged — the detector only makes the disarmed state **observable**, satisfying the "no enforcement-disable that is both innocuous-command-reachable AND traceless" bar. **Documented residuals (accepted, not closed):** (a) the redirect/`truncate` *write itself* is still un-gated at fire time (the §6.1 destructive `find -delete`/`truncate`/`>` residual, #505) — detection is at the *next* SessionStart, not real-time block; (b) the **binding-symlink repoint** — repointing `.claude/eng-shell-root` at an attacker-controlled tree with a neutered `hookrt.sh` disables the session and is **structurally undetectable from within** (this detector, like the whole SessionStart hook, is reached *through* the binding), extending the missing/dangling-binding residual above; the launch-time `bin/claude-eng` check (§3.5) catches a missing/dangling binding but not a repoint-to-a-valid-but-neutered tree. Both residuals are the documented "anyone who can write `eng-state` / repoint the binding can bash around the hook" posture (§6.1 / §7), not a new exposure.

**Hookrt-missing banner** (#37) — `$CLAUDE_ENG_SHELL_ROOT` is set (or self-located), but `.claude/hooks/hookrt.sh` is absent (corrupt checkout, conflict-stashed file, manual `rm`). Each of the 5 hooks has a primitive check for the runtime and emits a per-invocation WARN to stderr before silently exiting 0; that per-invocation line trains operators to ignore "WARN hookrt-missing" as background noise, exactly when they shouldn't. The banner is the once-per-session actionable signal layered on top of that per-invocation diagnostic floor.

- **Trigger**: `$CLAUDE_ENG_SHELL_ROOT` is non-empty **AND** `$CLAUDE_ENG_SHELL_ROOT/.claude/hooks/hookrt.sh` is absent.
- **Action**: emit one stderr line naming the failure mode (`hook enforcement OFF until restored`) and the concrete fix — inspect tree state via `git -C $CLAUDE_ENG_SHELL_ROOT status`, then restore via `git -C $CLAUDE_ENG_SHELL_ROOT checkout -- .claude/hooks/hookrt.sh`. Non-blocking; SessionStart still exits 0 immediately after (no helpers can load without the runtime).
- **Idempotency**: a stamp file under `$TMPDIR/claude-eng-banner-hookrt.${CLAUDE_SESSION_ID:-$PPID}` debounces to one banner per session. The `-hookrt` suffix is retained for stamp-name stability across sessions (it predates the #318 removal of the sibling inject-consistency banner).
- **Diagnostic floor**: the existing per-invocation WARN at `session_start.sh:23` (and equivalent at the other 4 hooks' bootstrap blocks) stays — it is the script-grep-friendly log-scraper-stable diagnostic that survives the banner's stamp-creation failure modes (hostile `$TMPDIR`, low-disk). The banner adds operator-facing actionability without removing the diagnostic floor.
- **Position**: runs **after** the `$CLAUDE_ENG_SHELL_ROOT` resolution (the trigger needs the env var) and **before** the `. "$hookrt"` source line at `session_start.sh:37` — co-located inside the existing `if [ ! -f "$hookrt" ]` block, since it shares the trigger condition and immediate `exit 0` tail.

**(d) Friction-candidate advisory** (#398, Directive #391)

This is the **trigger** that completes the deferred-positive-face loop of §6.0 P3: the friction-candidate readers (`scripts/narrowing_candidates.sh`, `scripts/promotion_candidates.sh`, and the git-history reader `scripts/ceremony_candidates.sh` — #401) plus the new `unattended-park` park-frequency signal (below) are aggregated into a **one-line, non-blocking advisory** so the shell surfaces its own accumulated friction without a human remembering to run a script. It is the always-on counterpart described in MISSION:16 ("the log becomes the channel through which the shell sees its own friction") — a consumer wired to read the aggregate.

- **Face**: **positive, never a block** (§6.0 P1 — a missed friction nudge is ignorable at no cost). It is a single advisory line pointing to `/audit` and the three reader scripts; it **never dumps** their output into context (MISSION:14 narrowing constraint — the per-session surface stays a fixed-size pointer, not a variable-length report).
- **Trigger condition**: emitted only when the readers surface at least one above-threshold candidate cluster, **or** when the audit aggregate contains `unattended-park` records. When nothing clusters, **no line is emitted** (the advisory is falsifiable — smoke §105 pins both the fires-when-candidates and suppressed-when-none arms).
- **Cost control**: gated by a stamp at the `eng_state_dir`-resolved `.../state/last-friction-surfaced` (with the `$CLAUDE_ENG_SHELL_ROOT/.claude/state/...` fallback, parity with the §6.5(a) fetch stamp). The compute runs only when the stamp is older than `SESSION_START_FRICTION_TTL` seconds (default 21600 = 6 h, sharing the §6.5(a) sync-check cadence) or absent, so most sessions skip it. The reader invocation is wrapped in `timeout $SESSION_START_FRICTION_TIMEOUT` (default 3 s; unbounded fallback when `timeout(1)`/`gtimeout(1)` is absent, matching §6.5(a)). The stamp is touched only after a successful compute.
- **Fail-open**: any error (missing/empty audit log, reader non-zero, absent script, timeout) degrades to **silence** — the advisory never blocks, stalls, or fails session startup (SessionStart still reaches the prompt). It reads the per-project audit aggregate via `eng_state_dir` (keyed on `CLAUDE_PROJECT_DIR`, not cwd), so it runs **before** the `in_scope || exit 0` gate like the §6.5(a) sync check.
- **Park-frequency line**: the `unattended-park` records (emitted by `ship_park_pr`, §5.7.1) are read **directly** by the advisory — neither candidate script matches a `warn`/`parked` record (`narrowing_candidates.sh` keys `event==escape`, `promotion_candidates.sh` keys `event==warn`+`decision==reject`), so park frequency is surfaced as its own grep, leaving both scripts untouched.
- **Ceremony-mismatch line** (#401, Directive #401): `scripts/ceremony_candidates.sh` is a **measure-first** reader (§6.0 P3). Unlike the two audit-log readers it mines **commit history** — the ceremony signal (commit `type` + the Doc→Test→Code phase arc + changed-file count) lives in `git log`, not `audit.jsonl`, and the Directive forbids adding an author-facing field to record it there. It groups commits by `#<issue>` and surfaces **both directions**: *under-ceremony* (a `feat` group spanning >1 file with no `test`/`docs` phase commit) and *over-ceremony* (a ≥3-commit phase arc over a single file). It emits the same indented `… | … =N` cluster shape the advisory greps, so it adds **no new surface**, and it runs `git -C` against the project repo through the advisory's `timeout … || true` envelope so a non-repo cwd / missing base degrades to silence. **Honest limits** (surfaced, not corrected): a batched single-commit PR collapses the phase arc and can false-positive as under-ceremony; over-classification is only weakly detectable. **Measure-first**: it observes whether ceremony mis-sizing is real and recurring; hardening into a §1.2 weight model or a relaxation gate is a deferred follow-up Directive **gated on this consumer's evidence** (the P3 born-advisory-then-harden order applied to the device's own roadmap).
- **Spec-drift line** (#462, Directive #455): `scripts/spec_drift_candidates.sh` is a fourth **measure-first** reader (§6.0 P3) joining the family. Like the ceremony reader it mines `git log` (not `audit.jsonl`): it greps `SPEC.md` for the repo paths it references, then flags **code-ahead drift** — a commit that touched a referenced path **without** co-touching `SPEC.md` ("code moved, SPEC didn't"). It emits the same indented `… | … =N` cluster shape (`  <path> | drift-commits=N`), so it adds **no new surface**, and fails open (non-repo cwd / absent or empty `SPEC.md` / no candidates → a sentinel line, exit 0). **Detection only**: the three-way classification (spec-ahead / code-ahead-correct / code-wrong) and the user-gated SPEC correction are the sibling `/reconcile-spec` flow (§5.27). The reader **is wired into the advisory above** (#466): it joins the `_session_friction_advisory` reader set under the same `timeout` envelope + once-per-session `last-friction-surfaced` stamp, and when candidates exist the advisory line reports the outstanding **drift-candidate count** (`N spec-drift candidate(s)`) — the once-per-session surface that makes the drift signal consumed without a human running the script. **Honest limits** (surfaced, not corrected): a batched commit touching both code and `SPEC.md` for unrelated reasons masks a real drift, and the path-token grep is coarse — it over-surfaces by design, to **measure** the real signal before any reconciliation machinery is hardened.

**(e) SSOT-presence health line** (#460, Directive #454)

A once-per-session, zero-network glance at the target's SSOT presence — the recurring counterpart to the one-time `/onboard` SSOT check, so a repo that drifts into (or was never given) a `SPEC.md` is re-surfaced each session rather than flagged only at adoption. `SPEC.md` is the **required behavioural SSOT** for any code-carrying repo (§1.3); this line keeps that expectation visible.

- **Single source**: the presence facts come from `scripts/lib/onboard_checks.sh --dry-run` (filtered to its `ssot:MISSION.md` / `ssot:SPEC.md` lines) — **not** a reimplemented `[ -f SPEC.md ]` (§9 single-source; the same script `/onboard` and `setup.sh` consume).
- **No per-session `gh` call**: `--dry-run` short-circuits `onboard_checks.sh`'s `gh_usable()` **before** any `command -v gh`, so the path is purely filesystem (`ssot:*` are two `[ -f ]` facts). The gh-dependent checks (fork / permission / branch-protection) stay one-time in `/onboard` — this surface is the cheap glance, not the full report.
- **Render**: both present → one unobtrusive line `SSOT: MISSION.md ✓ SPEC.md ✓`; `SPEC.md` absent → a prominent **SPEC-first nudge** pointing the user to author it before other work (the shell scaffolds the slot from `.claude/templates/spec.md`, never authors the contract content — the MISSION boundary).
- **Cadence — every session, not TTL-debounced**: unlike the §6.5(d) friction advisory (whose 6 h TTL debounces an *expensive* compute), the SSOT check is two stats, so it fires **every** session start to satisfy the "once per session" contract — a nudge suppressed for 6 h would under-deliver it. It therefore lives in the in-scope section (alongside the §6.5(b) branch banner, **after** `in_scope`), so it reports only for a registered target's cwd (no false "SPEC.md absent" in an arbitrary directory) and needs no second stamp.
- **Fail-open**: any error (script absent, unreadable output, indeterminate status) degrades to **silence** — never blocks or stalls session startup.

### 6.6 Stack-aware behavior

`helpers/detect_stack.sh` picks an active stack for `/ship`'s test runner, the commit-time lint gate, and the PostToolUse formatter. Detection is by sentinel file at PWD:

| Stack | Sentinel | Test (used by `/ship`) | Lint (commit gate) | Format (PostToolUse) |
|-------|----------|------------------------|--------------------|----------------------|
| node | `package.json` | `npm test` (if `test` script) | `npm run lint` (if `lint` script) | `prettier --write` |
| python | `pyproject.toml` / `setup.py` / `requirements.txt` | `pytest` | `ruff check .` | `ruff format` or `black` |
| go | `go.mod` | `go test ./...` | `go vet ./...` | `gofmt -w` |
| rust | `Cargo.toml` | `cargo test` | `cargo check` | `rustfmt` |
| ruby | `Gemfile` | `bundle exec rspec` | — | — |
| godot | `project.godot` | `godot --headless …` (GUT if present, otherwise parse-only) | `gdlint .` (if installed) | `gdformat` (.gd) |

Notes for godot:
- Test: when `addons/gut/gut_cmdln.gd` is present, `/ship` invokes GUT via `godot --headless --path . -s res://addons/gut/gut_cmdln.gd -gexit`. Otherwise a parse-only check runs (`godot --headless --path . --quit-after 1`) — confirms the project loads.
- Lint / format: `gdlint` and `gdformat` ship in the `gdtoolkit` PyPI package. Helpers only invoke them when on PATH; absence is silent (no false-positive block).
- Scene and resource files (`.tscn`, `.tres`, `.import`) are not auto-formatted — they are editor-managed and any rewrite risks corrupting binary subresources.
- The C# variant of Godot (`project.godot` + `*.csproj`) still resolves to `godot` (script-level tooling). Mixed-language CI is the project's call, not the shell's.

**Formatter invocation safety**: `detect_format_cmd` shell-quotes the file path via `printf '%q'` before interpolating it into the returned command string. The PostToolUse hook can therefore safely `eval` the result — filenames containing spaces, `;`, `$(...)`, backticks, or other shell metacharacters do not change the executed command. (PostToolUse is best-effort and silent on failure; SPEC §6.2.)

A target with no recognized sentinel resolves to `unknown`; `/ship` reports "no test runner detected" but does not block. Adding a stack means extending `detect_stack` plus the corresponding `detect_test_cmd` / `detect_lint_cmd` / `detect_format_cmd` branches.

**Version-field location** (the §18.2 release manifest-match preflight, #469, Directive #468): `detect_version` (in `helpers/detect_stack.sh`) is the **single-source** per-stack locator that reads the project's native version field, for a verification-only `/release` preflight — the field is read, **never written** (write-back stays the shell's own `VERSION`, §18.2):

| Stack | Version field |
|-------|---------------|
| node | `package.json` → `.version` |
| python | `pyproject.toml` → `[project] version` |
| rust | `Cargo.toml` → `[package] version` |
| go / ruby / godot | — (no universal manifest version field → skip) |

An unrecognized stack, an absent manifest, or an unparseable field returns empty → the preflight **degrades to a graceful skip** (never a false block), mirroring `detect_stack`'s `unknown` fail-open. The locator lives only here; `/release` does not duplicate per-stack version knowledge (§9).

## 7. Escape hatch

Three forms, chosen by where the escape needs to be read. The **file-token channel** (below) is the **primary in-agent recourse**; the two in-command forms are verbatim-delivery forms (a real shell, the smoke harness) — they do **not** land in-harness.

> **Observed caveat (this harness, 2026-06; the harness is a moving floor — re-verify, §1.9):** in the live Claude Code Bash tool **both** in-command forms are **stripped before the hook reads `tool_input.command`** — the leading env-prefix is consumed as the spawned subprocess's environment, and the trailing sentinel comment is stripped too. The parsers are correct (they set the tokens when fed the literal string in isolation, e.g. the smoke harness), but the harness does not deliver either to the hook, so **neither in-command form takes effect in-harness** (#478). The working in-agent channel is the **file-based skip token** below (#479): it is written to per-project state and read by the hook *at fire time*, so the harness has nothing to strip. The **fallback** for a sanctioned in-agent guarded op remains a **real terminal** (no PreToolUse hook fires there) or doing the work on a non-protected branch and renaming.

**File-based skip token (primary in-agent channel, #479)** — a one-shot, category-scoped token the hook reads at fire time from per-project state, so it survives the harness (unlike the two in-command forms above):

- **Location**: one file per category at `$(eng_state_dir)/escape/<category>.token` (per-project state dir, §3.2.2; gitignored under `eng-state/`). Per-category isolation makes category-scoping **structural** — a `branch.token` can only ever disarm `branch`.
- **Format**: `KEY=VALUE`, one per line, exactly these four keys — `category`, `reason`, `cmd_fingerprint`, `created` (epoch seconds). **Strict allow-list parse**: an unknown key or a missing key ⇒ malformed ⇒ **not honored**.
- **Reader**: pure bash — **no `python3`/`jq`** — so there is no interpreter-absent failure mode; the reader is strictly fail-safe. It is integrated into `should_skip <cat>`: after the existing `$SKIP_HOOKS` check, `should_skip` consults `<cat>.token`. The hook exports `ESCAPE_BIND_CMD=$raw_cmd` (the current command) for the fingerprint check.
- **Honored only if ALL hold**: the file exists and is readable; all four keys present with no unknown keys; the `category` field equals the requested category equals the filename; `cmd_fingerprint` is non-empty **and a substring of the current command** (`ESCAPE_BIND_CMD`); `created` is numeric **and** `now − created ≤ 60` (the **TTL**, 60 seconds); the bind-cmd is non-empty.
- **On honor**: `audit_log escape <category> skip "<reason>"` (parity with the `SKIP_HOOKS` escape audit) → **consume (delete) the token** → skip that one category.
- **One-shot / consume-on-read**: the token is deleted on honor **and** deleted on a stale/malformed reject (poison cleanup) — a token never survives a `should_skip` call that examined it.
- **Fail-safe-to-block**: **any** doubt — absent, unreadable, malformed, category-mismatch, fingerprint-not-a-substring, stale, future-or-non-numeric `created`, empty bind-cmd — leaves enforcement **armed** (the matcher blocks). Never a spurious skip.
- **Binding weight (read this correctly).** The real AC-5 guards are **consume-once + the 60s TTL**: a token is single-use and self-expires. The `cmd_fingerprint` substring binding and the writer's min-length refusal are **footgun-reducers** — they stop a *stale or pre-written* token from disarming an *unintended* command — **not a security mechanism**. The threat model (§6.1) is unchanged: hooks are mistake-prevention, not a security boundary; anyone who can write `eng-state/` can `bash -c` around the hook, so **no crypto/signature is used or needed**. Choose a **high-entropy distinguishing** fingerprint — the commit subject / version / branch name — **not** a bare verb like `git commit`.
- **Writer affordance**: `scripts/eng_skip.sh <category> <cmd_fingerprint> [reason]` — the fingerprint is **required** (it refuses to write without one, and refuses a fingerprint shorter than 8 chars as a footgun-reducer); the write is atomic, per-category, under `eng_state_dir` (the writer resolves the same project dir the hook reads — deriving it from the git top-level when `CLAUDE_PROJECT_DIR` is unset for a Bash-tool subprocess — so writer and reader always agree, #483). The Write tool to `$(eng_state_dir)/escape/<cat>.token` (e.g. `.claude/eng-state/escape/<cat>.token`) is a documented **fallback** writer (the `.claude/` Edit/Write carve-out, §6.1).
- **Bash-only boundary (R1, by design — not an oversight).** The channel works only on the **Bash** tool path, where `ESCAPE_BIND_CMD=$raw_cmd` is set. The **Edit/Write** protected-branch arm has **no command string**, so it cannot carry a token and the in-agent escape does **not** cover Edit/Write — consistent with §5.0 step 2, which already routes the bootstrap MISSION/README authoring through a Bash `cp` precisely because "there is no command string to carry [a sentinel]."

The two in-command forms below remain defined (they land verbatim in a real shell and the smoke harness); they are no longer the in-agent recourse.

**Trailing sentinel** — read from the trailing edge of the raw command:
```bash
<command>  # claude-eng:skip=<category>[,<category>...] reason=<reason>
```

**Leading env-prefix** — read from the leading edge of `tool_input.command`; works only where the prefix arrives *inside* the command string (a real shell that passes it verbatim, and the smoke harness):
```bash
SKIP_HOOKS=<category>[,<category>...] SKIP_REASON='<reason>' <command>
```

The PreToolUse hook parses the leading `SKIP_HOOKS=…`/`SKIP_REASON=…` tokens from the leading edge of `tool_input.command`, **and** a `# claude-eng:skip=… reason=…` sentinel from the trailing edge of the raw command. Two forms exist as defense in depth across execution environments where one or the other arrives verbatim (a real shell, the smoke harness). **In the live Claude Code Bash tool, however, neither lands in `tool_input.command`:** the leading `VAR=val` is consumed as the spawned subprocess's own environment (the hook subprocess does not inherit it either — §6.1), and the trailing `#`-comment is **stripped before the hook reads the command** too — so in-harness the recourse above (real terminal / non-protected branch + rename) applies, not these in-command forms. Both forms export the recognized tokens into the hook before downstream matchers run, then strip them from the command for the matcher pass (so a `reason=` containing a blockable substring cannot bleed into another matcher); **if both are present, the leading form wins.** Recognized tokens: `SKIP_HOOKS`/`SKIP_REASON` (leading) and `claude-eng:skip=`/`reason=` (trailing) only — any other ALLCAPS `K=V` leading token (`PATH=`, `LD_PRELOAD=`, …) halts the leading scan and is left as an ordinary argument. The sentinel is parsed from the **raw** command before whitespace-normalization (the normalization + `shlex` pass would otherwise mangle the `#`), is namespaced (`claude-eng:skip=`) so an ordinary trailing comment is never mistaken for an escape, and is **one-shot** — it travels with the single command, so there is no persistent bypass state. The sentinel is honored **only when its `#` is a genuine unquoted shell comment token** — a `#` that the executed shell itself treats as the start of a comment. A `#` sitting inside a quoted argument (e.g. `gh pr comment 5 --body "see note # claude-eng:skip=all reason=x"`) is argument text, not a comment: the shell runs the whole command intact, so the hook must **not** read it as an escape (doing so would let ordinary quoted text — a commit message, a PR-body paste, audit output quoting the sentinel — silently disarm every matcher with a falsified audit reason). The comment-token boundary is resolved with the same `python3`/`shlex` machinery the leading env-prefix uses; if `python3` is absent the trailing sentinel is **not** honored — a fail-safe no-op (enforcement stays armed), never a spurious skip. The sentinel must sit on the command's **final line** (the match is confined to a single line via `[[:blank:]]`/non-control-char classes plus a newline guard) so a line-1 sentinel can never strip or disarm a dangerous *later* line — such a command falls through to the matchers and blocks. **Scope:** like the leading env-prefix, the trailing sentinel is **command-scoped** — it disarms its named category (or `all`) across **every** line/clause of the command, not only the line it sits on. It remains **category-scoped**: a tail sentinel naming category X does not disarm a category-Y matcher (that command still blocks). So an `escape` audit record proves *which* category was skipped with a stated reason — read it as "category X was skipped for this command," not "this exact line was approved"; the reason text need not describe the specific operation disarmed. When run in a real terminal where neither form reaches a hook (there is no PreToolUse hook outside the harness), the command simply runs; the escape only matters when a hook is in the path.

Categories:
- `all`
- **eng-flow hook matchers:** `secret`, `branch`, `commit-format`, `format`, `force-push`, `amend`, `no-verify`, `destructive`, `sensitive`, `out-of-scope`, `backmerge`, `ac-closeout`, `merge-strategy`
- **type-aware / dir-mode hook matchers (§6.1):** `proposed-protect`, `trusted-filer-mutate`, `label-parent-consistency`, `initiative-readonly`, `directive-close`
- **command-level reviewer-gate escapes:** `directive-review` (the `activation-reviewer` gate on `/file-directive` / `/activate` / `/revise-directive` / `/complete-directive` / `/consume-initiative`, §4.9/§5.10–§5.22), `release-review` (the `code-reviewer` gate on `/release`, §5.20)

This list is the complete set of `SKIP_HOOKS` categories the runtime honors (the hook matchers via `should_skip <cat>` in `pre_tool_use.sh`; the reviewer-gate escapes via the command/skill prose). It is kept in sync with the matchers/commands by the doc-consistency smoke check (§17 / #267) — a new `should_skip <cat>` or reviewer-gate escape that is not listed here fails smoke.

(`issue-ref` is folded into `commit-format` — not a separate category. `format` is the commit-time lint/format gate. `out-of-scope` covers paths outside the registry (§6.1). `backmerge` covers `git merge <protected-base>` on a feature branch (§6.1). `ac-closeout` covers `gh pr merge` when a linked issue has unchecked AC and no `^## AC closeout` comment — see SPEC §6.1 row + §5.7.1 step 7.6 for the rule (step 7.5 is the unrelated checklist audit); `scripts/ac_closeout.sh <pr-num>` is the canonical fixer. `merge-strategy` covers `gh pr merge` to the default branch without `--merge` — see SPEC §6.1 row + §5.7.1 (squash erases the Doc→Test→Code arc on the trunk); squash is allowed on a non-default base. Every skip is recorded in `.claude/audit/audit.jsonl`:
```json
{"ts":"2026-05-19T...","event":"escape","decision":"skip","category":"format","reason":"flaky lint JIRA-1234","cwd":"...","source":"live"}
```

The `reason` and `cwd` fields are JSON-encoded via `jq -Rs .` (or an inline `\n`/`\r`/`\t`/`\\`/`\"` escape when jq is absent). Newlines and control characters in `SKIP_REASON`, and shell metacharacters in PWD (legal in POSIX paths), therefore do not break the "one JSON object per line" invariant — every `audit.jsonl` is parseable by `jq -c '.'` regardless of input contents. Other fields (`event`, `category`, `decision`, `ts`, `source`) are internal-constant or RFC 3339-shaped and need no encoding (`source` is the `live`/`test` enum, §6.1).

Policy:
- Skips should be temporary.
- A skip with no `SKIP_REASON` is logged as `unspecified` — review target.
- `all` is for emergencies.
- A category that gets skipped repeatedly means the hook is misconfigured — open a fix PR.

Audit logs rotate as appropriate (exact threshold is an implementation choice), but the diff body is never written — only path, line, category metadata.

## 9. Template specifications

Each `.claude/templates/` file is read by a command and variable-substituted at use time. Single source of truth.

**`docs/*.md` and `.claude/CLAUDE.md` are thin pointers, not parallel content.** The human-facing `docs/*.md` digests (e.g. `ENGINEERING_FLOW.md`, `DIR_MODE_FLOW.md`, `SUBAGENTS.md`) exist as a readable on-ramp to a SPEC section — they must **lead with a "Full details/spec in SPEC §X" reference and carry no parallel or detailed contract content** that could silently drift from the SPEC. SPEC is the single source; a digest that restates a contract becomes a second copy to keep in sync by hand (the divergence hazard §1.3's single-file principle exists to prevent). Smoke enforces the lead-with-a-SPEC-reference rule (§91). When a digest would need detail to be useful, link to the SPEC section instead of duplicating it.

The same discipline binds `.claude/CLAUDE.md`. Unlike the `docs/*.md` digests, `.claude/CLAUDE.md` is an **always-on injected surface** — Claude Code natively loads it whole every session — so it must point to SPEC for contract detail and carry no detailed contract content that could drift. It is held to a size budget: `wc -c .claude/CLAUDE.md` must stay **≤ 12000 bytes** (drift-guarded by smoke §104). The always-on injected floor is task-independent context loaded before any task is known, so an unbounded always-on surface is "bloat past the task" (MISSION:20); keeping `CLAUDE.md` a thin pointer-index is the always-on counterpart to §6.4's per-turn `status_compact` narrowing.

## 10. Convention summary

### 10.1 Branch naming

```
<gh-username>/<type>/[<issue#>-]<slug>

# required-issue-# group (feat/fix/docs/refactor/perf) — always includes <issue#>-
ilgyu/feat/42-oidc-login
ilgyu/fix/57-null-pointer
ilgyu/docs/89-update-architecture

# optional group (chore/style/build/ci/test/revert) — issue # may be omitted
ilgyu/chore/typo-readme
ilgyu/chore/89-update-deps    ← if an issue exists, you can include it
```

Types: `feat | fix | docs | style | refactor | perf | test | build | ci | chore | revert`

`/work-on <issue#>` always has an issue, so it always includes `<issue#>-` regardless of group. Issueless ad-hoc branches in the optional group are created by the user with plain `git checkout -b`.

### 10.2 Commit messages

```
<type>(#<issue>)[!]: <subject ≤ 72 chars>

<body — one blank line>

Closes #<N>     ← last commit, or single-PR consolidator
Refs #<N>       ← intermediate commit
Co-Authored-By: Claude <noreply@anthropic.com>   ← optional trailer; toggleable
```

Phase prefixes:
- `docs(#42): describe OIDC callback flow`
- `test(#42): add failing tests for /auth/oidc/callback`
- `feat(#42): implement OIDC callback handler`

**Co-Authored-By trailer toggle**: the `Co-Authored-By: Claude …` line is a convention, not a contract. It is included by default but can be disabled per-target. Resolution priority (high → low, mirroring SPEC §5.7.1's mode pattern):

1. `$CLAUDE_ENG_COAUTHOR` env var (`on` / `off`).
2. `.claude/state/coauthor` per-target file (one of `on` / `off`, no other content).
3. Default `on`.

Unknown values fail-safe to `on` and emit a stderr warning naming the offending source. The helper `.claude/hooks/helpers/coauthor.sh` exposes `coauthor_trailer` (prints the line when enabled, empty when disabled); `/work-on`'s commit template sources it. The line is **version-agnostic** — `Co-Authored-By: Claude <noreply@anthropic.com>`, with no model version (#294) — so it never re-drifts at a model bump (it was previously pinned to `Opus 4.7` and went stale once the model advanced to 4.8). The toggle covers the default behavior of `/work-on` flows only — the trailer remains free to be added or omitted manually in any specific commit.

**`eng_commit` slot-assembly helper** (Issue #436; ADR-0002 GO target): the positive-enforcement face (§6.0) of the commit-subject contract — the agent *fills slots* instead of hand-rolling `git commit -m "<type>(#N): …"`. `.claude/hooks/helpers/eng_commit.sh` exposes `eng_commit <type> <issue> <subject> [body-paragraph…]`: it assembles `<type>(#<issue>): <subject>`, validates it via `check_commit_subject` (format + 1..72 codepoints) **before** committing — on rejection it errors `eng_commit: …` with a nonzero rc and **does not commit** (the failure surfaces with full context, not as an opaque post-hoc hook block). It then commits via a **bash argv array** `git commit -m "<subject>" -m "<body-para>"… ` (one `-m` per body paragraph, `coauthor_trailer` appended as a final `-m` when enabled):

- The **first `-m` carries only the assembled subject**, so the `commit-format` matcher *sees and accepts* it on the happy path — `eng_commit` does **not** rely on the `-F` fail-open bypass (git's `-m`/`-F` are mutually exclusive anyway, and an `-F`-only commit makes the matcher extract an empty subject and skip the check — a bypass, not a guarantee).
- The **array argv** (no `eval`, no string re-quoting) makes multibyte and multi-paragraph bodies round-trip losslessly — the inline-`-m` quoting pitfall is a string-construction hazard, absent when argv is assembled programmatically.

**Offered, not forced**: `eng_commit` is positive guidance with the `commit-format` hook retained as the mandatory net (§6.0 P4); skills *offer* it (e.g. `/work-on`'s commit step) but the hand-roll path remains valid. The caller passes the `Closes #N` / `Refs #N` trailer as a body paragraph (the trailer choice is base-dependent, §10.3/§10.5 — the caller's decision, not the helper's).

**Relationship with Conventional Commits (conscious choice)**: we reuse the standard CC `scope` slot for the issue # marker. Consequences:

- Passes default `commitlint` rules (with `(#42)` allowed in scope).
- Tools like `conventional-changelog` / `semantic-release` parse `(#42)` as scope, so changelog scope columns will show `#42`. We don't carry both a domain scope (e.g. `auth`) and issue # — issue # wins.
- If a domain scope is essential, put it in the commit body via a `Scope: auth` trailer.

### 10.3 PRs

- Always start as draft.
- `Closes #N` or `Refs #N` is required. If a single issue spans multiple PRs, intermediates use `Refs #N` and the final uses `Closes #N`. Checklist distribution is planner/user's call.
- **Base**: PRs default to `main`. For experimental or long-lived feature work, `/work-on --base <branch>` opens the PR against an alternate base — see §10.5. Non-`main`-base PRs default to `Refs` (the issue closes only when the work reaches `main`).
- Body follows `.claude/templates/pr_body.md`.
- Curate the body right after each commit.
- `/ship` transitions to ready after ship-gate verification.

### 10.4 PR ↔ checklist sync

Order: **commit → PR body update**. Commit is the fact; PR body reflects it. Reverse order creates "checklist ticked but commit absent" states. Refetch the remote body right before updating — abort the auto-update if external edits (reviewer, GitHub UI, parallel session) are detected.

### 10.5 Topic-branch workflow

The default flow (`/work-on <#>`) targets `main`. For two scenarios the user can opt into an alternate base via `/work-on <#> --base <branch>`:

1. **Experimental work** — risk too high for `main`. The user creates `experiment/foo` from `main`, iterates with a few PRs whose base is `experiment/foo`, and only consolidates to `main` via one final PR when the experiment stabilizes (or abandons the branch).
2. **Big features** — a multi-PR refactor where the integration point is a long-lived feature branch. Sub-task PRs target the feature branch; the feature branch itself is eventually merged to `main`.

**Conventions:**
- Sub-task PRs use `Refs #N` (not `Closes #N`) so the issue stays open until work reaches `main`. `/work-on` enforces this default for non-`main` bases.
- The alternate base must already exist locally + remotely. `/work-on --base <branch>` does NOT auto-create it; the user creates the base deliberately (e.g. `git checkout main && git pull && git checkout -b experiment/foo && git push -u origin experiment/foo`).
- The alternate base's remote branch-protection settings are the user's responsibility. For sub-task PRs to auto-merge into `experiment/foo`, the GitHub branch-protection rules on `experiment/foo` must permit the same flow as `main` (or be more permissive). `/work-on` does not inspect or modify remote settings.
- The alternate base is NOT in SPEC §6.1's protected list (`main`/`master`/`release/*`). The user is free to commit directly to `experiment/foo` between sub-task PRs — it's an integration point, not the canonical product line.
- The work-branch name (`<gh-user>/<type>/<#>-<slug>`) is unchanged. Stacking multiple work-PRs on the same issue+base is not supported by `/work-on` alone — manual branch creation (`git checkout -b ...`) is the escape hatch for that rare case.

**Use with Directives (dir-mode integration):** scenario 2 ("Big features") is the canonical pattern for Directive-scoped multi-PR work (§1.7, §2.1). The user creates a topic branch from `main` (e.g., `feature/directive-92`); each Execution Issue parented under the Directive (via `/file-issue --parent <directive-#>`, §5.2) runs `/work-on <execution-#> --base feature/directive-92`, producing sub-task PRs with `Refs #N`. The Directive Issue itself is **not** branched — the `proposed-protect` matcher in §6.1 blocks `git checkout -b` against a Directive Issue (any status), by design; the Directive scopes the work, the Execution Issues do the work. Consolidation follows the standard pattern (one final PR head=topic-branch, base=`main`, body `Closes` each Execution Issue), and `/ship`'s Directive-aware audit (§5.7) fires once per parented Execution Issue at consolidation. `/complete-directive <directive-#>` (§5.13) must run **after** consolidation — `activation-reviewer` (§4.9) evaluates closed-Execution-Issue evidence, so running completion while the topic branch is still unconsolidated yields insufficient-evidence and blocks.

**Consolidation back to `main`:** when the topic branch is ready, open ONE PR with head=`experiment/foo` and base=`main`, body `Closes #<#>`. This is a regular `gh pr create` invocation — `/work-on` is not designed for the consolidator step (the branch already exists; there's no planner phase to repeat).

**Enforcement — the bare-`Refs`-to-default anti-pattern (#337):** the convention above requires that directive-scoped work reach the default branch via a PR that `Closes` an Execution Issue; a PR that merges to the default branch carrying only bare `Refs #N` (empty closing set) leaks the work — no Execution Issue closes, so the dir-mode post-merge reflection chain never fires and the parent Directive gets no progress record (the historical #92 failure: PRs #100–105 merged to `main` with bare `Refs`, no `Closes`). The **post-merge workflow** (`.github/workflows/dir-mode-post-merge.yml`, §5.15) enforces this: on a merged PR whose base is the **live default branch** AND whose `closingIssuesReferences` is **empty** AND whose body carries ≥1 `Refs #N` resolving (via the same depth-2 label-aware climb as the reflection resolver, #335) to a **directive-scoped** issue (a Directive, or an Execution Issue parented under one), it posts a single idempotent **warning comment on the merged PR** (marker `<!-- bare-refs-warning pr=#N -->`) naming the violation and the recovery (re-open + `Closes` an Execution Issue, or run `/reflect <pr>`). It is **warn-only** — the merge already happened, so there is no block point — and **fail-soft** (any `gh` resolution failure → no warning, never aborts the workflow). The check lives in the post-merge **workflow**, not a PreToolUse hook, deliberately: the #92 violations were predominantly human GitHub-UI merges, which a `gh pr merge` hook structurally never observes; the merge-triggered workflow fires regardless of who merged. The detection logic is a sourced, smoke-tested helper (`scripts/lib/detect_bare_refs_directive.sh`) with the same three-copy cmp-locked discipline as the resolver.

## 11. CI awareness

- **Right before** ready transition: `gh pr checks --json state` for a one-time check. **Block only on `failure` or `cancelled`**. `pending` / `in_progress` / `queued` pass — the background watch picks up monitoring after ready transition. (§5.7 step 7)
- **Right after** ready transition: `gh pr checks --watch` runs in **background**, returning immediately.
- The background watch's failure detection routes to a PostToolUse hook (§6.2) notification.
- The post-push PostToolUse hook (`Bash` matching `git push`) also warns about PR-less push patterns (detached HEAD, missing draft PR).

This shell repo ships its own CI workflow at `.github/workflows/ci.yml` running on `pull_request` and `push: main`. The `syntax` and `smoke` jobs run on an OS matrix of `ubuntu-latest` and `macos-latest` (the BSD branch is the production path for the primary user; Linux-only CI would leave it unverified); the `mutation` job (§11.1) runs on `ubuntu-latest` only (its logic is platform-independent; the matchers themselves are covered per-OS by `smoke`):
- **syntax** — `bash -n` AND `shellcheck --severity=warning --shell=bash` over `bin/claude-eng`, `scripts/*.sh`, `scripts/lib/*.sh`, `.claude/hooks/*.sh`, `.claude/hooks/helpers/*.sh`. Repo-wide silences (e.g. SC1090 for dynamic sourcing via `BASH_SOURCE`-relative paths and per-test fixture variables) live in `.shellcheckrc`; documented-deliberate patterns (`eval` on jq-emitted `@sh` scripts in `helpers/status.sh`, `printf -v` to dynamic varname in `helpers/escape.sh`) carry inline `# shellcheck disable=SC…` with one-line justification. shellcheck install is OS-conditional: `apt-get install -y shellcheck` on Linux; pre-installed on macOS-runner images (fallback `brew install shellcheck` if absent).
- **smoke** — installs `jq` (apt-get on Linux; pre-installed on macOS) and runs `./scripts/test/smoke.sh`.
- **mutation** — runs `./scripts/test/mutation.sh`, the enforcement-matcher kill-rate guard (§11.1).

`fail-fast: false` on the matrix so per-platform failures surface independently.

### 11.1 Enforcement-matcher mutation harness

The smoke suite asserts that the enforcement matchers *behave* correctly; it does not, on its own, prove those assertions actually **kill** a regression — a silently-weakened matcher could still pass a smoke assertion that no longer exercises the weakened path. `scripts/test/mutation.sh` closes that gap by measuring the smoke suite's **kill-rate** on the critical matchers (#423): it is the systematic, generalized form of the ad-hoc falsifiability premise-locks the suite already carries (the §357 live-sink lock; the §65i release-subject lock).

For each critical matcher the harness applies one **mutation** — a weakened redefinition *appended* to a throwaway **git-worktree copy** of the tree (append-override: bash takes the last definition, so no fragile in-body edit) — then runs the full smoke suite in that worktree with `CLAUDE_ENG_SHELL_ROOT` pointed at it, and asserts smoke **fails** (exit ≠ 0 — the mutant is *killed*). A **surviving** mutant (smoke still green under a weakened guard) is reported as a harness failure: it means the matcher is no longer pinned by any assertion. The git worktree isolates the mutation from the parent working tree and is removed after each case, so the harness never mutates the live tree.

The seeded mutations (extensible) target the matchers whose wrong-*allow* cost is highest (§6.0): `check_commit_subject` → always-accept (commit-format), `scan_staged_secrets` → always-clean (secret), `PROTECTED_BRANCH_PATTERN` → never-match (protected-branch). Per §6.0 this is the cost-asymmetry rationale made testable — the same matchers whose negative gate fails safe toward blocking are the ones whose test coverage most needs a kill-rate proof.

So `/ship`'s `gh pr checks` step has something concrete to watch for on PRs against this repo. Other repos that adopt the shell can copy the workflow as a starting point.

## 13. Failure modes and recovery

| Situation | Handling |
|-----------|----------|
| Issue body is ambiguous | `/work-on` refuses to start; asks the user to clarify the issue |
| Phase A reveals the spec was wrong | Strike through the checklist with reason, re-invoke planner |
| Phase B test passes accidentally | Two branches: (a) Phase A spec is weak or test inputs insufficient → strengthen doc/test. (b) Adding a regression test for already-implemented behavior is legitimate → note in one line in PR body and proceed. Don't always restart from doc. |
| Base (main or `--base` target) advances during work | `git pull --rebase origin <base>` → resolve conflicts → re-push PR. The base is whatever the PR's `baseRefName` is — read via `gh pr view --json baseRefName` if uncertain. | Issue acceptance criteria change mid-work | Update the issue body + comment with the change log; mirror in the PR body |

## 18. Release backbone

Locks the shape of how releases happen: per-PR fragment files, one consolidator skill (`/release`), and a templated CI gate. Adopters consume the substrate via `/onboard-dir-mode` (§18.6). Established by Directive #128; depends on the VERSION + tag surface from Directive #122 (§3.5).

### 18.1 Fragment contract

The release line consumes per-PR **changelog fragments** under `changelog_unreleased/<category>/<N>.md`:

- `<category>` ∈ {`added`, `changed`, `deprecated`, `removed`, `fixed`, `security`} — the six [Keep a Changelog](https://keepachangelog.com/en/1.1.0/) categories. Each is a subdirectory under `changelog_unreleased/`.
- `<N>` is the issue or PR number that introduces the fragment (positive integer; filename stem).
- The fragment body is a **single-line markdown bullet** (one line starting with `- `).
- The bullet **must contain `(#<N>)`** where `<N>` matches the filename stem. CI rejects mismatches (§18.6).
- **No YAML frontmatter, no multi-paragraph prose, no nested headers inside the fragment.** The contract is intentionally minimal — the directory is the category, the filename is the PR/issue link, the bullet is the user-facing line.

`changelog_unreleased/TEMPLATE.md` documents the contract for fragment-authors in project-neutral language. The template ships in the shell and `/onboard-dir-mode` installs it (along with the six empty category subdirectories) into adopting repos.

`/release X.Y.Z` (§5.20) consolidates all present fragments into a new `## [X.Y.Z] — YYYY-MM-DD` section in `CHANGELOG.md` at repo root, grouped under Keep-a-Changelog category subheadings, and deletes the consumed fragments in the same commit.

### 18.2 Version of record

The canonical version-of-record is the top-level `VERSION` file (locked by Directive #122 / §3.5). `/release X.Y.Z`:

- Validates `X.Y.Z` is semver 0.x (`MAJOR=0`).
- Strips any `-dev` suffix from `VERSION`.
- Writes `X.Y.Z` to `VERSION`.
- Falls back to `git describe --tags` for **resolution-time queries only** — never for write-back. The file is authoritative on `main`.

Multi-file version-of-record (`package.json`, `Cargo.toml`, `pyproject.toml`) is deferred. A documented extension point — `.claude/release-config.json` — is **reserved** at this location for projects whose VoR lives elsewhere; the file is currently unimplemented and the reservation prevents the name from being claimed by something else.

**Manifest-match preflight** (verify, not write-back; #469, Directive #468): while multi-file VoR *write-back* is deferred, `/release` runs a **verification-only** preflight — it resolves the detected stack's version field via `detect_version` (§6.6) and **refuses** (non-zero, naming both the manifest version and the `X.Y.Z` being cut, with the fix: bump the manifest) when they contradict, so a release cannot tag a version the project's own manifest does not reflect. This is the **confident-mismatch** arm only; an unknown stack / absent or unparseable field is the **uncertain** arm and degrades to a graceful skip with a note (never a false block — the diverse-project posture, §6.0 cost-asymmetry). The manifest is **read, never written** — write-back remains `VERSION`-only. Stack-aware bump *automation* is an explicit deferred non-goal (Directive #468).

### 18.3 Trunk branch model

Default development line is `main`. `/release` operates on `main` unless `--base <branch>` targets a maintenance branch (e.g., `0.3-maintenance`). Maintenance branches are **not auto-created by the shell** — adopters who need LTS lines create them manually. See §10.5 for the topic-branch / alternate-base flow.

The CI fragment-gate (§18.6) targets `main` and `*-maintenance` by default; configurable per project at workflow-install time.

### 18.4 Tag format

Tags follow `v` + semver 0.x — `v0.1.0`, `v0.2.0`, etc. Locked by Directive #122 (CLOSED); preserved verbatim here so downstream tooling reads one canonical clause. Tag-freshness is **not** enforced as a merge gate per #122's non-goal — tag-state drift is drift, not breakage. `/release` creates the tag (`gh release create vX.Y.Z --target <merge-sha>`) at the release-PR merge boundary; pre-merge release-PRs are tagless.

### 18.5 Non-goals

Locked by Directive #128 (closed):

- **No `/prepare-release` skill.** VERSION's `-dev` suffix replaces the scope-locking ceremony of a prepare PR.
- **No `/changelog-check` *lint* skill.** CI workflow + optional local hook arm cover the *validation* surface; a manual re-validation skill is a third redundant surface. This forbids a skill that **re-checks** an existing fragment's shape — it does **not** forbid an **authoring** affordance: `/changelog` (§5.23) *writes* a fragment or applies the `skip-changelog` label and delegates all validation to the CI gate (§18.6), and the `/ship` pre-ready arm (§5.7, step 7.7) is exactly the "optional local hook arm" sanctioned in this clause. Authoring the artifact plus one in-flow presence check of the single contract is permitted; re-validating that contract a second time is not.
- **No tag-push merge gate, no tag-freshness enforcement** (preserves Directive #122's explicit non-goal).
- **No automated maintenance-branch creation, no backport / port-PR flow.** Deferred to a successor Directive when at least one adopter requests LTS.
- **No release-please / external-service integration.** The shell ships a ~30-line CI script under shell control. Adopters who already use release-please keep it; the shell does not displace it but also does not configure it.
- **No automated distribution** (npm, PyPI, Docker, Homebrew, binary upload). Per-project extension surface.
- **No release cadence enforcement.** When to cut a release is a human / per-project call.
- **No breaking-change policy enforcement.** Shell stays at `0.x` per #122 (`MAJOR=0` invariant); downstream projects at `1.x+` define their own compat policy.
- **No multi-file version-of-record bump.** `VERSION` is canonical; `.claude/release-config.json` is reserved but currently unimplemented.
- **No fragment frontmatter or template enforcement beyond filename/PR-number consistency.**
- **No major-rollover archive logic.** Shell stays at `0.x` per #122; revisit when `1.0.0` is on the table.
- **No retroactive per-milestone CHANGELOG entries** for pre-`v0.1.0` history. The inaugural section covers `init..v0.1.0` as one block, not split per historical sub-milestone.

### 18.6 Cross-project surface

`/onboard-dir-mode` installs the release-backbone substrate into adopting target repos as part of its tier-3 PR:

- `.github/workflows/check-changelog.yml` — the per-PR fragment-gate workflow.
- `changelog_unreleased/TEMPLATE.md` + the six empty category subdirectories with `.gitkeep` placeholders.

The workflow validates the following rules on every PR targeting a protected branch (`main` and `*-maintenance` by default). Rule 1 is a presence check; rules 2–4 are applied to **every** added fragment, and if **any** added fragment fails the gate fails — a single valid fragment does **not** mask a malformed sibling (the all-valid contract, matching `scripts/release_consolidate.sh`):

1. The PR diff adds at least one file under `changelog_unreleased/<category>/<N>.md` where `<category>` is one of the six Keep-a-Changelog subdirectories.
2. The filename stem `<N>` is a positive integer.
3. The file contains `(#<N>)` where `<N>` matches the filename stem.
4. The bullet begins with `- ` (the single-line markdown bullet form, per `changelog_unreleased/TEMPLATE.md`).

Failure (no fragment, malformed filename, mismatched PR ref, or a non-bullet line) blocks the PR with an error referencing this section. Rules 2–4 mirror `release_consolidate.sh`'s fragment validation so the per-PR gate and the release-time validator enforce one contract rather than diverging.

**`skip-changelog` opt-out**: a PR labeled `skip-changelog` bypasses the fragment check. The label is auditable on the PR; the workflow logs its application. Intended for PRs with no observable end-user behavior change — pure docs-fix typos, dependency-only chores, internal refactors. **PRs carrying the label produce no fragment and are not represented in the eventual CHANGELOG section** — by design, since there is nothing for a user to read about. Workflow trigger events include `labeled` and `unlabeled` so the opt-out re-runs the check.

Adopters consume the backbone with zero in-repo configuration beyond the substrate install plus their own `VERSION` file and a `CHANGELOG.md` at repo root. `/release X.Y.Z` runs against any repo where these three artifacts are present.

**Workflow does not self-trigger on its own landing PR**: per [GitHub's `pull_request` security policy](https://docs.github.com/en/actions/using-workflows/triggering-a-workflow#using-event-types), workflows modified inside a PR do **not** run on that PR (otherwise a fork could exfiltrate secrets via a self-modified workflow). The first PR after `check-changelog.yml` lands on `main` is the first empirical observation of the fragment-gate; the landing PR itself only validates the workflow's structure via smoke §66c-e + the `code-reviewer` walk-through. Adopters running `/onboard-dir-mode` see the same: the substrate-install PR opens the workflow but the workflow fires from the next PR onward.

### 18.7 Skip criterion (the `skip-changelog` decision)

Every PR satisfies the fragment gate (§18.6) one of two ways: it **adds a fragment**, or it carries the **`skip-changelog`** label. Which one applies is a content judgment, and this clause is its SSOT — `changelog_unreleased/TEMPLATE.md`, the `/changelog` skill (§5.23), and the `/ship` pre-ready arm (§5.7) reference this criterion rather than restate it:

- **Fragment required** when the PR changes user- or adopter-observable surface: a behaviour/contract change, a CLI/API/schema/protocol change, a SPEC contract change, or anything that alters how someone uses or integrates the project.
- **`skip-changelog` eligible** when the PR has no such surface: an internal `refactor` with unchanged external behaviour, `test`-only, `chore`/`build`/`ci`, `style`, or a docs-internal edit that changes no contract.

When in doubt, write the fragment — a redundant one-line entry is cheaper than a silently missing one. This criterion governs the *decision* only; the *shape* of any fragment written is validated solely by §18.6, and neither the skill nor the `/ship` arm re-validates shape (§18.5).
