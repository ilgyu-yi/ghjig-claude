## 0. Intent and scope

### 0.1 What this is

An operating shell for Claude Code that runs on top of the standard GitHub workflow. Target repos are cloned into the shell's `workspace/` directory and worked on from there. The shell guarantees:

1. **Adherence to the GitHub-standard flow** вҖ” issue вҶ’ branch вҶ’ draft PR вҶ’ checklist commits вҶ’ ready PR вҶ’ merge.
2. **Doc вҶ’ Test вҶ’ Code work order** вҖ” every code change goes through docs first, then tests, then implementation.
3. **Active SSOT maintenance** вҖ” documents are updated alongside code. Code-only changes with stale docs are prevented.
4. **Aggressive use of subagents** вҖ” output quality doesn't rely on a single model and single context. Specialized per-phase agents are composed.
5. **Hook enforcement of engineering discipline** вҖ” direct commits to protected branches, force pushes, secret exposure, malformed commits, untested merges, etc. are actually blocked.

### 0.2 What this is not

- Not a general dotfiles framework. The shell is scoped to `workspace/`.
- Not an absolute enforcer. No policy is set in stone. A conscious escape hatch is always open, and every bypass is recorded in the audit log.
- Doesn't own the target repo's knowledge SSOT. Updates are made by people / Claude inside the target.

### 0.3 Assumptions

- GitHub-standard, but **upstream-only**. Not a fork workflow. All branches are created on upstream; PRs go branch вҶ’ main within the same repo.
- Hosted in a git repo, shareable across a team/org. Changes ship via PR.

## 1. Core principles

These principles must be visible throughout the shell. CLAUDE.md, agents, commands, and hook messages should all point to the same principles.

### 1.1 GH-standard is the backbone

The flow comes before the principles. Each principle is an attribute attached to some stage of the flow.

### 1.2 Doc вҶ’ Test вҶ’ Code order

A code change has three phases:

**Phase A вҖ” Doc**: write the behavior / contract / decision into the doc first. Whichever of the target SSOTs applies (MISSION, README, CLAUDE.md, ARCHITECTURE, ADR, API doc).

**Phase B вҖ” Test**: translate the doc from A into a test. The test must fail at this stage. Confirm the failure message is the intended picture.

**Phase C вҖ” Code**: minimal implementation to make B pass. Run adjacent regression checks afterward.

Each phase should be a separate commit when possible. The PR's commit graph should show the doc вҶ’ test вҶ’ code arc.

Effect of this order:
- Intent is externalized before code, enabling self-consistency checking.
- Tests stay closer to spec than when written alongside code.
- Re-entering the work later, you immediately know "which phase am I in?"

**Exception categories** вҖ” the following types relax the phase order. Note the reason in one line in the PR Plan.

| Type | Relaxation | Rationale |
|------|-----------|-----------|
| `fix` | failing test reproducing вҶ’ code вҶ’ (if needed) doc | Bug reproduction plays the spec role |
| `refactor` | when external behavior is unchanged, doc can be skipped; existing tests suffice | New doc/tests would be fake commits |
| `perf` | measure вҶ’ code вҶ’ doc (report results) | Hypothesis validation comes first |
| spike | free on a separate branch, restructured into a proper PR before merge | Protect exploration cost |
| one-line typo / obvious fix | single commit, keep the phase prefix | Splitting cost > benefit |

`feat`, `docs`, and changes to external contracts / API / schema apply the phase order strictly.

### 1.3 Active SSOT maintenance

Two SSOTs coexist:

| Layer | Location | Question it answers |
|-------|----------|---------------------|
| Process SSOT | claude-eng-shell git | **How** we work |
| Project SSOT | each target repo git | **What** / **why** we're building |

Target SSOT structure:
- `MISSION.md` вҖ” the project's ground truth. Why it exists, success criteria, explicit non-goals.
- `README.md` вҖ” for external readers.
- `CLAUDE.md` вҖ” directory index, run commands, domain vocabulary, conventions.
- `docs/ARCHITECTURE.md` вҖ” module boundaries, data flow.
- `docs/ADRs/NNNN-*.md` вҖ” record of irreversible decisions.

A code change commits together with the SSOT items it invalidates or updates. If a doc is intentionally omitted, the commit body states `Docs: n/a вҖ” <short reason>`.

### 1.4 PR-as-living-doc

The PR body simultaneously serves three roles:
- **The work item's ground truth** вҖ” what this PR is for.
- **Cross-session memory** вҖ” Claude's context is volatile but the PR body is durable. The next session auto-loads it to restore context.
- **Execution progress display** вҖ” the checklist renders as a GitHub progress bar.

Principles:
- The PR body is **editorial, not append-only**. Curate right after each commit. Tidy stale items; if history is needed, keep one line per entry in a separate "Changelog" section.
- **commit вҶ’ PR sync** order. Fact (commit) first, then reflection (PR body). Reversing it creates false states.
- Before updating, refetch the remote body once. If it differs from what the shell last saw вҖ” checked against the persistent cache at `.claude/state/pr-cache/<owner>__<repo>__pr-<n>.json` via `helpers/pr_cache.sh` (В§5.4) вҖ” abort the automatic update and inform the user. Don't build a merge logic beyond that вҖ” conflicts are rare, and a human should be in the loop when they happen.
- **No stale unchecked items at merge.** A merged PR body must reflect truth at the moment of merge. Every `- [ ]` line is in one of three terminal states: ticked (`[x]`) because it was done, marked `[~] N/A вҖ” <reason>` because it intentionally won't be done, or removed from the body. "Pending, but oh well" is not acceptable вҖ” closed/merged PRs are loaded by SessionStart (В§6.5) into future context, so stale uncheckedness propagates as misinformation. The same rule applies to **issue bodies on auto-close**: every acceptance-criterion item is ticked, N/A'd with reason, or struck through with a closing comment that names the disposition.

### 1.5 Aggressive use of subagents

Quality benefits from not loading every burden onto a single context.

| Situation | Subagent |
|-----------|---------|
| Wide read-only exploration | `explorer` |
| 3+ file changes, API/schema/migration | `planner` |
| Doc writing / updating | `doc-writer` |
| Test writing (Phase B) | `test-writer` |
| Pre-commit / pre-PR review | `code-reviewer` |
| Auth / input / deps / crypto changes | `security-reviewer` |
| Rationale check on a proposed issue body | `issue-reviewer` |
| Approach / alternatives check on a planner output | `plan-reviewer` |

Independent investigations that can run in parallel: call multiple agents at once.

You don't need to call all eight on a single PR. Use them per the table; the main assistant shouldn't re-run in `explorer` what it already explored. No call-count limits or tracking.

**Operating-mode coupling**: `code-reviewer`, `security-reviewer`, `issue-reviewer`, and `plan-reviewer` all substitute for human review at different points in the flow. In `attended` mode they surface findings to the user, who decides; in `unattended` mode their verdict gates the next step directly (`ship` вҶ’ proceed; `refine` вҶ’ re-produce with feedback; `block` вҶ’ stop with the reason and either ask the user, or вҖ” when no human is available вҖ” park the work). This pattern is what makes the `unattended` mode operationally complete: every human-confirm checkpoint has a subagent fallback.

### 1.6 GitHub standard (upstream-only)

- No forks. The clone's origin is always upstream.
- All branches are created on upstream.
- PRs are branch вҶ’ **base** within the same repo. **Default base is `main`**; `/work-on --base <branch>` (В§5.3) opts into an alternate base for experimental work or long-lived feature branches. See В§10.5 "Topic-branch workflow."
- Every change starts from an issue. issue вҶ’ PR is 1:N (intermediate PRs use `Refs #N`, final uses `Closes #N`). For PRs whose base is non-`main`, `Refs` is the right default вҖ” the issue closes only when work reaches `main`.
- Branch naming: `<gh-username>/<type>/[<issue#>-]<slug>`. The required-issue-# group (В§6.1.1) always includes `<issue#>-`; the optional group (`chore`/`style`/`build`/`ci`/`test`/`revert`) may omit it. Examples: `ilgyu/feat/42-oidc-login`, `ilgyu/chore/typo-readme`.

### 3.2 Injection mechanism

`scripts/clone-into.sh` runs the following right after `git clone`. **The target's existing `.claude/` content is never overwritten.** If a same-named asset already exists, that asset is not injected вҖ” only a warning is printed. The target's choice wins.

```bash
mkdir -p "$TARGET/.claude" "$TARGET/.claude/agents" "$TARGET/.claude/commands"

# settings.local.json: symlink to the shell's settings.json (Claude Code's 'this clone only' slot).
if [ -e "$TARGET/.claude/settings.local.json" ] && [ ! -L "$TARGET/.claude/settings.local.json" ]; then
  echo "WARN: $TARGET/.claude/settings.local.json already exists. Shell settings not injected; user merge needed." >&2
else
  ln -sfn "$SHELL_ROOT/.claude/settings.json" "$TARGET/.claude/settings.local.json"
fi

# agents/commands: skip and warn if a same-named asset exists.
for kind in agents commands; do
  for src in "$SHELL_ROOT/.claude/$kind"/*.md; do
    [ -e "$src" ] || continue
    dest="$TARGET/.claude/$kind/$(basename "$src")"
    if [ -e "$dest" ] && [ ! -L "$dest" ]; then
      echo "WARN: $dest already exists. Shell asset injection skipped." >&2
    else
      ln -sfn "$src" "$dest"
    fi
  done
done

# Add the shell injection (symlink) to the target's .git/info/exclude so target git doesn't see it.
grep -qxF '.claude/settings.local.json' "$TARGET/.git/info/exclude" 2>/dev/null \
  || printf '.claude/settings.local.json\n' >> "$TARGET/.git/info/exclude"
```

Rules:
- If the target has a same-named asset, the shell's version is not injected (conflict avoidance). If you want to use both, rename or move your own asset.
- Post-hoc registration of a directory the user already cloned: `scripts/register.sh <path>` runs the same inject logic.
- The target's `CLAUDE.md`, `MISSION.md`, `docs/` are **not touched**. The shell's CLAUDE.md (process norms) is combined into context via the UserPromptSubmit hook.

#### 3.2.1 Hook command path resolution

All `command` entries in the `settings.json` pointed to by the symlinked `settings.local.json` must use **absolute paths based on `$CLAUDE_ENG_SHELL_ROOT`**. When Claude Code runs a hook, `$CLAUDE_PROJECT_DIR` points to the target, not the shell root. Relative paths or `$CLAUDE_PROJECT_DIR` would resolve to an empty location in the target and fail.

```json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "hooks": [{
        "type": "command",
        "command": "$CLAUDE_ENG_SHELL_ROOT/.claude/hooks/pre_tool_use.sh"
      }]
    }]
  }
}
```

`$CLAUDE_ENG_SHELL_ROOT` is exported by the `bin/claude-eng` wrapper on entry:

```bash
#!/usr/bin/env bash
set -euo pipefail
export CLAUDE_ENG_SHELL_ROOT="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd -P)"
exec claude "$@"
```

Inside hook scripts, shell-side assets (helpers, audit) are accessed via `$CLAUDE_ENG_SHELL_ROOT`, while target files are accessed via `$CLAUDE_PROJECT_DIR` вҖ” keep the two cleanly separated.

### 3.3 Scope guard вҖ” cwd-based

All hook scripts prelude:

```bash
# Match physical PWD against registered (physical) paths in the registry.
# The registry is written by register.sh / clone-into.sh; clones inside workspace/ are recorded the same way.
REGISTRY="$CLAUDE_ENG_SHELL_ROOT/.claude/state/registry.txt"
PWD_REAL=$(cd "$PWD" 2>/dev/null && pwd -P) || exit 0
[ -f "$REGISTRY" ] || exit 0
in_scope=0
while IFS= read -r entry; do
  [ -z "$entry" ] && continue
  case "$PWD_REAL/" in "$entry"/*) in_scope=1; break ;; esac
  [ "$PWD_REAL" = "$entry" ] && { in_scope=1; break; }
done < "$REGISTRY"
[ "$in_scope" = "1" ] || exit 0
```

вҶ’ If `claude-eng` is invoked from an unregistered directory, hooks pass through transparently. Works regardless of logical vs physical `$PWD` (we use PWD_REAL). External path registration is covered by the same mechanism.

`.claude/state/registry.txt` format: one physical path (`pwd -P` output) per line. Comments and blank lines are ignored.

### 3.4 User-global isolation

The shell's installation and operation leave no changes in the user-global environment. The only way to "disable" the shell is to stop invoking `claude-eng`. No deactivation procedure or cleanup script is needed.

**Surfaces never touched**:
- `~/.claude` (user-global Claude settings)
- shell rc/login files (`~/.zshrc`, `~/.bashrc`, `~/.bash_profile`, `~/.profile`, `~/.config/zsh/*`, `~/.config/fish/*`, etc.)
- global git config (`~/.gitconfig`, `~/.config/git/*`)
- PATH auto-registration вҖ” the user adds `<shell-root>/bin` to PATH or uses absolute path / alias themselves. The shell never injects an export line into rc files.
- LaunchAgents, systemd user units, login items, or any auto-start registration

`bootstrap.sh` boundary:
- Allowed: check dependencies (`gh`, `git`, `python3`, `jq`), run `gh auth status`, create `.claude/audit/` and `.claude/state/` inside the shell repo.
- Forbidden: any modification of the surfaces above, package manager invocations, sudo.

Target repos are the shell's work вҖ” they fall outside this isolation rule (the injection in В§3.2 is intended behavior).

### 3.5 Entry point

`claude-eng` is meant to start at the root of `workspace/<repo>/`. Subdirectories work too (walk-up), but starting at the root avoids edge cases in monorepos where subprojects may carry their own `.claude/`.

**External path support**: a target repo can live outside the shell's `workspace/` (e.g. `~/code/<repo>`). `scripts/register.sh <abs-path>` (1) creates a `workspace/<basename>` symlink, (2) adds the target's physical path (`pwd -P`) to `.claude/state/registry.txt`, (3) runs the inject logic of В§3.2. The shell's `.gitignore` whitelists `workspace/*` so the shell git tree doesn't pick up target trees even in the external-path case. Direct clones into `workspace/` are also recorded in the registry вҖ” the guard is always registry-based.

**Auto-register prompt**: when `claude-eng <path>` is called with an unregistered external path, prompt once before entering the session:
```
$ claude-eng ~/code/foo
? '/Users/.../code/foo' is not registered. Register as workspace/foo and enter? [Y/n]
```
yes вҶ’ run `scripts/register.sh` вҶ’ inject logic (В§3.2) вҶ’ start session. no вҶ’ exit.

**Monorepo subprojects**: clone-into injects **only at the repo root**. Subprojects with their own `.claude/` are not recursively injected. When the user starts `claude-eng` in a subproject, the walk-up finds the root's shell injection. The subproject's `.claude/CLAUDE.md` is merged via Claude Code's default walk-up behavior.

### 3.6 Shell self-registration

The shell repo itself must appear in `.claude/state/registry.txt` so that `cwd_guard.in_scope` (В§3.3) returns true when `claude-eng` is invoked from the shell root. Without it, the shell's own hooks short-circuit on `in_scope || exit 0` and the dogfooding promise of В§17 is empty вҖ” the shell can't enforce its own discipline on itself.

`scripts/bootstrap.sh` ensures `$SHELL_ROOT` is present in the registry (idempotent вҖ” repeated runs do not duplicate). Self-registration is **registry-only**: no `workspace/<basename>` symlink and no `inject_into` are performed against the shell, because the shell IS the injection source and the symlink would loop into `workspace/` contained within `$SHELL_ROOT`.

`scripts/register.sh "$SHELL_ROOT"` (or any path that physical-resolves to the shell root) recognizes this case and:
- skips the `workspace/<basename>` symlink (would otherwise create a directory loop);
- skips `inject_into` (the source files are the destinations);
- still ensures the registry entry exists.

For target repos this is unchanged: В§3.2 inject + registry entry, both performed by `clone-into.sh` / `register.sh` as before.

## 4. Subagent specifications

Each agent is defined in `.claude/agents/<name>.md` as YAML frontmatter + body.

### 4.1 planner

**When**: before any code edit. Required for 3+ file changes, migrations, schema changes, API changes, or external contract changes.

**Input**: user request, linked issue body, MISSION.md, target CLAUDE.md.

**When MISSION is absent**: planner proceeds with `issue body + README + CLAUDE.md` alone. The "How this serves the mission" section is labeled `(MISSION.md absent вҖ” review /onboard stub suggestion)`. Don't synthesize a MISSION via inference.

**Output** (markdown that drops straight into the PR body):

```
## How this serves the mission
<Which MISSION.md item this contributes to>

## Plan
<Why this approach, trade-offs, decision rationale>

## Target base
<branch name; default `main`. For non-`main`, name the topic-branch / feature-branch the PR will merge into. See SPEC В§10.5.>

## Alternatives considered
- <Alternative 1>: <one-line why-not>
- <Alternative 2>: <one-line why-not>
(If the choice is forced вҖ” single viable approach вҖ” say so explicitly with a one-line justification rather than fabricating alternatives.)

## Key context
<related file:line>

## Checklist
- [ ] **Doc**: <doc update item 1>
- [ ] **Doc**: <doc update item 2>
- [ ] **Test**: <failing test item>
- [ ] **Code**: <implementation item>
- [ ] **Code**: <implementation item>

## Out of scope
<What this PR will NOT do>

## Risks
<Compatibility, performance, data, security>

## Open questions
<To be decided before proceeding>
```

Checklist order: **Doc first, Test next, Code last**. Granularity: one item вүҲ one commit вүҲ 30 min to 2 hours.

**Alternatives section is mandatory** вҖ” even when the choice feels obvious. A "no alternatives вҖ” forced choice" entry with a one-line justification is acceptable; an empty section is not. The section exists so reviewers see what was rejected and why, and so future-you can revisit the trade-off without re-running the analysis.

### 4.2 explorer

**When**: locating code, symbol lookups, "where is X defined", "who calls Y" вҖ” read-only wide search. Protects the main context.

**Constraints**: no code edits. No dumping large files. Always summarize.

**Output**:
- Definition location (`path:line`)
- Up to 5 main references (`path:line` + one-line context)
- Related modules
- If nothing found: say "not found" and suggest next searches.

### 4.3 doc-writer

**When**: Phase A. Any change that alters external surface.

**Responsibilities**:
- Identify affected SSOT docs (MISSION, README, CLAUDE.md, ARCHITECTURE, ADR, API doc).
- Patch each doc preserving its existing format and tone.
- For absent docs, propose a minimal stub only after user confirmation.
- When an irreversible decision is detected, recommend writing an ADR (`docs/ADRs/NNNN-title.md`).

**Forbidden**: inventing target SSOT content (humans must review and sign off). Only stub proposals when absent.

### 4.4 test-writer

**When**: Phase B, right after doc.

**Responsibilities**:
- Take Phase A docs / spec as input; write a failing test.
- Reuse existing test patterns, tools, fixtures.
- Cover intended behavior + at least one boundary / exception / error path.
- Run it and confirm it **fails** (intentionally). If it passes, either Phase A is wrong or the test is too weak.

**Principles**:
- Suspect over-mocking.
- One test = one assertion.
- If tooling is absent (e.g. pytest not installed), report and stop. Don't guess.

### 4.5 code-reviewer

**When**: before commit / PR. Auto-invoked by `/review`, `/ship`.

**Premise**: assume no knowledge of the main assistant's discussion or reasoning. Input is:
- The full PR diff. Prefer `gh pr diff <num>` (or `--web` equivalent) over a literal `git diff main...HEAD` вҖ” the PR's `baseRefName` may be non-`main` for topic-branch / experimental work (В§10.5), and `gh pr diff` honors that automatically. The principle is "review what the PR actually changes against its declared base," not "everything since main."
- Read-only adjacent context for changed files (for call-graph tracing, as needed)
- Target `MISSION.md`, referenced issue body (for MISSION fit + acceptance criteria evaluation)
- Full PR body (for declared plan + checklist verification)

Nothing outside this input.

**Check**:
- Consistency (coherent changes, missing adjacent callers)
- Tests (alignment with Phase B, regression risk)
- Error handling (at boundaries only, no defensive code creep)
- Security surface
- Obvious performance traps
- Readability, naming
- Scope (extraneous changes?)
- **Doc sync (Phase A reflected?)**
- **MISSION fit (contributing to or violating which item?)**
- **Issue acceptance criteria met?**

**Output**: ship / ship after fix / block. Each finding cites `path:line`.

### 4.6 security-reviewer

**When**: PRs touching auth / authz / sessions / external input / new deps / crypto / hashing / randomness / new IO boundaries.

**Check areas**: authentication / authorization, injection (SQL / shell / HTML / path), sensitive data exposure, weak crypto, CSRF / CORS / headers, dependency risk.

**Output**: High / Medium / Low / Info. Each finding: risk + exploit scenario + remediation.

### 4.7 issue-reviewer

**When**: invoked by `/file-issue` before `gh issue create`. Validates the rationale triad of SPEC В§5.2 вҖ” every issue passes through this gate.

**Input**:
- Proposed issue body (title + body + label).
- Target `MISSION.md` (if absent, label as `(MISSION.md absent вҖ” review /onboard stub suggestion)`).
- Open-issues list (`gh issue list --state open`).

**Check**:
- **MISSION fit**: does the body cite a specific MISSION item? "It's general improvement" or "tidy" answers fail.
- **Why now**: does the body explain what changes if this waits? Vague "would be nice" fails.
- **Existing-coverage**: scan open-issues / open-PRs; does another issue cover this, or would a comment on an open PR suffice? If yes вҶ’ suggest the cheaper path.
- **Acceptance criteria**: explicit, testable, вүҘ1 item.

**Output**: one of three verdicts on stdout's last line, each with a one-line reason:
- `ship` вҖ” file as-is.
- `refine: <one-line why>` вҖ” return for body revision before filing.
- `block: <one-line why>` вҖ” don't file at all (existing-coverage hit, scope problem, or non-MISSION work).

Inside `attended` mode the verdict surfaces to the user; inside `unattended` mode it gates the next step directly. Hook into `attended` is non-skippable вҖ” Auto mode does not suppress this gate (SPEC В§5.2).

### 4.8 plan-reviewer

**When**: invoked by `/work-on` between `planner` and the plan-approval step. Validates that the alternatives section is real and that the chosen approach actually beats them.

**Input**:
- `planner` output (markdown).
- Issue body (the one the planner read).
- Target `MISSION.md`.

**Check**:
- **Alternatives section is real**: вүҘ1 alternative with a substantive why-not, or an explicit "no alternatives вҖ” forced choice" with a justification. "Just because" or empty fails.
- **Chosen approach beats alternatives**: read the "Plan" rationale; does it engage with the alternatives it lists, or just describe the chosen path?
- **Scope hygiene**: items in `Checklist` map to `Acceptance criteria` of the issue; anything orphaned belongs in `Out of scope` or a follow-up issue.
- **Phase-order sanity**: Doc вҶ’ Test вҶ’ Code as per SPEC В§1.2; relaxation reasons noted if applicable.
- **Target base fit**: the `Target base` field (В§4.1) names a branch. If non-`main`, the plan should engage with the topic-branch's constraints (see В§10.5). A `main`-default plan that quietly proposes work belonging on an experimental base вҖ” or vice versa вҖ” fails this check.

**Output**: same three verdicts as `issue-reviewer`. `refine` causes `/work-on` to re-invoke `planner` with the feedback; `block` stops with a comment on the issue naming the structural problem.

## 5. Slash commands

### 5.1 `/onboard`

**When**: once, right after a target repo is first cloned.

**Steps**:

1. **Upstream check** вҖ” `gh repo view --json isFork,parent`. Fork вҶ’ block.
2. **Permission check** вҖ” push permission required. Missing вҶ’ advise and block.
3. **Target SSOT check**:
   - `MISSION.md` present? Absent вҶ’ strong warning + draft proposal from `.claude/templates/mission.md` (no auto-create).
   - Report `README.md`, `CLAUDE.md`, `docs/ARCHITECTURE.md` presence.
4. **`.github/` check**:
   - `ISSUE_TEMPLATE/` absent вҶ’ suggest shell template.
   - `PULL_REQUEST_TEMPLATE.md` absent вҶ’ suggest shell template.
   - `CODEOWNERS` absent вҶ’ recommend.
5. **Branch protection check** вҖ” `gh api repos/{owner}/{repo}/branches/main/protection`. Report missing items (PR required, review required, status checks required, force push blocked). Setup requires admin, so just print commands.
6. **CI presence** вҖ” check `.github/workflows/`. Absent вҶ’ recommend appropriate workflows.

No automatic changes. All recommendations require human review and execution.

### 5.2 `/file-issue [--quick] <description>`

**Steps**:
- Confirm title, body, label with the user.
- Force MISSION reference and acceptance criteria in the body.
- **Rationale check** (before `gh issue create`): surface (a) which MISSION item this serves, (b) why now vs later, (c) whether an existing issue or an open-PR follow-up note would suffice. If any of (a)вҖ“(c) is weak, refine the issue or drop the request instead of filing. The check is mandatory вҖ” Auto / unattended mode does not skip it; a poorly-justified issue propagates downstream into planner output and merged code, where the cost of redirection is highest.
- **Reviewer gate**: invoke `issue-reviewer` (В§4.7) on the proposed body before `gh issue create`. In `attended` mode the verdict surfaces to the user (who may override); in `unattended` mode the verdict gates the next step (`ship` вҶ’ proceed; `refine` вҶ’ re-author the body with the feedback and re-invoke; `block` вҶ’ stop and surface the reason). The reviewer fills the human-confirm slot when no human is present.
- `--quick`: one-line issue (label `chore`) for fast creation. Rationale check + reviewer gate still required, in compressed form (one-line "serves X mission item, blocks Y" suffices).
- Call `gh issue create`, return the number.

### 5.3 `/work-on <issue#> [--base <branch>]`

**Steps**:

1. `gh issue view <#> --json title,body,labels,comments` вҖ” read the issue.
2. Validate body вҖ” if acceptance criteria are missing or vague, ask the user to fix it. **Don't start work with an ambiguous goal.**
3. Resolve the **target base** (default `main`; override via `--base <branch>`). `git fetch origin && git checkout <base> && git pull --ff-only`. If `<base>` is not `main` and is not reachable locally or remotely, fail with a clear message вҖ” `/work-on` does NOT auto-create alternate bases (see В§10.5).
4. Create branch:
   ```
   USER=$(gh api user --jq .login)
   TYPE=<detected type>
   SLUG=<kebab of issue title>
   git checkout -b "${USER}/${TYPE}/${ISSUE}-${SLUG}"
   ```
   If the branch already exists on remote: `git pull --ff-only origin <branch>`.
5. Invoke `planner` agent (with issue body + MISSION + target CLAUDE.md context + the resolved `<base>`). The planner's output must include the mandatory `## Alternatives considered` AND `## Target base` sections (В§4.1). The `Target base` value must equal the `<base>` passed in.
6. **Reviewer gate**: invoke `plan-reviewer` (В§4.8) on the planner output. Parse the verdict line (`^VERDICT: (ship|refine|block)`). In `attended` mode the verdict surfaces to the user; in `unattended` mode it gates the next step (`ship` вҶ’ step 7; `refine` вҶ’ re-invoke `planner` with the feedback, then re-review; `block` вҶ’ stop with a comment on the issue naming the structural problem). The reviewer substitutes for the human reviewer when no human is present.
7. **Wait for user approval of the plan.** Approval requires an **approach check**: confirm with the user that the chosen approach beats the alternatives the planner surfaced, not just that the plan exists. If the alternatives section is weak (single option without a forced-choice justification), refine before approval. In `unattended` mode, a clean `plan-reviewer` verdict from step 6 counts as approval.
8. **Write and commit Phase A (Doc)** вҖ” the first commit on the branch is real Doc work, never an empty seed. The branch graph begins with content the PR Plan describes, not a placeholder.
   ```
   # вҖҰedit the SSOTs identified by the plan (MISSION, README, CLAUDE.md, ARCHITECTURE, ADR)вҖҰ
   git add <changed SSOTs>
   git commit -m "<type>(#<#>): <subject>

   {Closes|Refs} #<#>"
   git push -u origin HEAD
   gh pr create --draft --base "<base>" --title "<type>(#<#>): <subject>" --body "<template>"
   ```
   First-line trailer: use `Closes #<#>` when `<base> == main`; otherwise use `Refs #<#>` (per В§10.4 вҖ” the issue closes only when work reaches `main`).

   If the planner's Phase A isn't ready to commit (e.g. blocked on clarification), defer the PR вҖ” keep the issue + plan in conversation. **Never fall back to an empty seed commit just to open the draft PR.**
9. Insert plan + checklist + ship gate sections in the PR body.
10. Sync internal TaskList with the checklist.
11. Continue with the next phase (Test).

### 5.4 `/sync-pr`

Explicit entry point for updating the PR body. The main assistant does this automatically; this command lets the user trigger it on demand.

Refetch the remote body once before updating. The shell stores the SHA-256 of the last body it saw at `.claude/state/pr-cache/<owner>%2F<repo>__pr-<n>.json` (the `/` between owner and repo is URL-encoded as `%2F` so `<owner>/<repo>` slug boundaries survive the filename); the file holds at least `last_seen_body_sha256`, `last_synced_head`, and `last_synced_at`. If the remote body's SHA differs from `last_seen_body_sha256`, an external edit happened (reviewer, parallel session, GitHub UI) вҖ” the automatic update aborts with a stderr message naming the conflict, and merging is the human's call. After a successful sync the cache is rewritten. A corrupt cache file (unparseable / missing key) also aborts the sync rather than silently treating as "first sync."

The cache lives under `.claude/state/`, which is gitignored вҖ” it's per-clone state, not part of the repo's history. The helper `.claude/hooks/helpers/pr_cache.sh` exports `pr_cache_read`, `pr_cache_write`, and `pr_cache_check`. The repo slug is derived from `gh repo view` of the cwd, overridable via `$PR_CACHE_REPO` (used by the smoke test suite). `/ship` reuses the cache transitively by routing its body curation through `/sync-pr` (В§5.7 step 5). `/work-on` doesn't update the body after PR creation, so it doesn't touch the cache directly; later edits go through `/sync-pr`.

### 5.5 `/status`

`/status` and the `UserPromptSubmit` hook both delegate to the canonical helper at `.claude/hooks/helpers/status.sh`. Two output modes:

- `status_compact` вҖ” multi-line plaintext block, the standard turn-context surface.
- `status_json` вҖ” same fields as stable JSON for diff-comparison or piping.

Fields (both modes):
- Branch + dirty status
- Linked issue (#, title) вҖ” extracted from `Closes #N` / `Refs #N` in the PR body
- PR (#, draft/ready, checklist progress `[x/y]`, CI status) вҖ” `gh pr view`, `gh pr checks`
- Next unchecked checklist item
- Current phase (Doc / Test / Code / Review / Ship), inferred from the next unchecked item's section heading
- Active mode (attended / unattended) per В§5.7.1

`UserPromptSubmit` (В§6.4) emits `status_compact` instead of a hand-rolled branch summary. Keeping the source single ensures `/status` and per-turn injection never drift.

The UserPromptSubmit hook reuses this output for the short per-turn summary.

**Cache**: status results are cached at `.claude/state/status-cache/<branch>.json` with a TTL of `STATUS_CACHE_TTL` seconds (default 5; env-overridable). The cache key is the branch name, so a `git checkout` to a different branch transparently loads a different cache file. Hits skip every `gh` round-trip (today: `gh pr view`, `gh issue view`, `gh pr checks` вҖ” 3 calls per uncached invocation). The 5-second TTL is short enough that real PR/issue/CI changes surface within one user turn (turns typically exceed 5 s) and long enough to dedupe a flurry of prompts. Cache files live under `.claude/state/` (gitignored).

### 5.6 `/review [--staged | <base>]`

Invokes `code-reviewer`. If the diff touches a security surface, also invokes `security-reviewer`. No auto-fix unless explicitly asked.

### 5.7 `/ship`

Runs the 8-step ship sequence from В§2. `/ship` runs the full test suite (step 4) before transitioning to ready (step 8). Tests are `/ship`'s responsibility; the hook does not add a separate test gate on ready transition (В§6.1).

CI monitoring: snapshot `gh pr checks --json state` right before ready transition вҖ” already failing вҶ’ block. Right after the transition, run `gh pr checks --watch` in background and return immediately. On failure, the PostToolUse hook notifies the user.

**Checklist audit (mandatory step 7.5, between CI snapshot and `gh pr ready`)**: scan the PR body for any `- [ ]` line. Each unchecked item must be resolved before transition вҖ” ticked if done, marked `[~] N/A вҖ” <reason>` if intentionally skipped, or removed if obsolete. The audit applies to every checklist in the body (Plan, Test plan, Docs touched, Ship gate). Per SPEC В§1.4, stale uncheckedness at merge propagates as misinformation into future context.

When mode is `unattended` (see В§5.7.1), `/ship` continues past step 8 into the CI-wait вҶ’ merge / park branch.

### 5.7.1 Operating modes (attended / unattended)

`/ship` has two terminal behaviors selected by **mode**:

- **`attended`** (default): `/ship` stops at "ready" and hands off to a human reviewer.
- **`unattended`**: after "ready", `/ship` waits on CI / required reviews and either merges or parks the PR for human attention.

**Resolution priority** (high вҶ’ low):

1. `--mode=<attended|unattended>` flag on `/ship`
2. `$CLAUDE_ENG_SHELL_MODE` env var
3. `.claude/state/mode` per-target file (one of `attended` / `unattended`, no other content)
4. default `attended`

Unknown values fail closed to `attended` and emit a stderr warning naming the offending source. `settings.json` is intentionally **not** a layer вҖ” that file is symlinked across targets (В§3.2) and a mode key would leak between unrelated work.

**Blocker taxonomy** вҖ” `ship_classify_blocker` returns one of:

| State | Criterion |
|-------|-----------|
| `clean` | All required CI checks green; required reviewers (if any) satisfied; branch protection rules satisfied. Eligible for auto-merge. |
| `soft` | CI is `pending` / `in_progress` / `queued`, **or** there is a CI failure that is attributable to the agent's own last commit and is fixable without new product decisions. One self-fix attempt is permitted (see below). |
| `hard` | Required-review with no available reviewer, undiagnosable / non-attributable CI failure, branch-protection rule the agent cannot satisfy, or a `soft` state that has already consumed its one self-fix attempt. |

**Soft-blocker behavior**: at most **one** self-fix-and-push attempt per `/ship` invocation. The retry pushes a `fix(#N): ...` commit and returns to the CI-wait state. A second `soft` outcome immediately escalates to `hard` вҖ” no further fix attempts.

**Hard-blocker behavior** (parking) вҖ” all four actions, in order, and idempotent:

1. Post a `gh pr comment` with a deterministic state summary (blocker reason + current SHA + CI snapshot).
2. Apply label `unattended-parked` (create the label on demand if missing). If the label is already present from a previous park on the same PR, `ship_park_pr` writes a single `park-suppressed: <reason>` line to the log instead of reposting the comment, and emits no comment to stdout. The original park comment remains canonical; the label remains applied.
3. Append one line to `${SHIP_PARK_LOG_PATH:-$CLAUDE_ENG_SHELL_ROOT/.claude/state/unattended-park.log}` (shell-side, gitignored вҖ” not the target's MEMORY.md, not `audit.jsonl`). The override exists for smoke-test isolation. On suppressed repeat, only the `park-suppressed: <reason>` line is appended.
4. Stop. Do not toggle draft or self-assign.

**Classifier robustness**: `ship_classify_blocker` fails closed to `hard` in two failure modes and emits a one-line stderr warning naming the cause:

- empty stdin вҶ’ `ship_classify_blocker: no input on stdin вҖ” defaulting to hard`
- missing `jq` dependency вҶ’ `ship_classify_blocker: jq not found вҖ” defaulting to hard`

The intent is to make overnight runs debuggable without changing the safe fail-closed default. Either condition resolves the classification to `hard` and the dispatch path proceeds to park.

**Merge mechanism**: `gh pr merge --auto --merge --delete-branch` (no-fast-forward merge commit; PR branch's commits are preserved on `main`). This requires repo-level auto-merge to be enabled at GitHub. If the API call fails because auto-merge is disabled, fall back to the park path (log + label + comment) with reason `auto-merge-disabled`.

**Why `--merge` rather than `--squash`**: SPEC В§1.2 makes the Doc вҶ’ Test вҶ’ Code commit graph load-bearing вҖ” the arc is the artifact that proves the discipline happened. Squash erases the arc on merge (the PR branch is deleted; only a single new commit lands on `main`). `--merge` keeps both: the individual phase commits stay on `main`, plus a merge commit anchors the PR-as-a-unit so `git log --first-parent main` gives a clean PR-level view. The visible cost is a busier `git log`; the audit-trail philosophy in В§6/В§7 trades that for the preserved arc.

**Composition with branch protection**: server-side rules in GitHub are always authoritative. `unattended` does not bypass them. A required-reviews rule with no available reviewer resolves to `hard` and parks вҖ” the shell never self-approves or auto-requests review against policy.

**Composition with Auto mode**: Auto mode (single-conversation clarification suppression) is orthogonal to operating mode. Any combination is valid: `attended + Auto`, `unattended + non-Auto`, etc. Mode controls the `/ship` terminal behavior; Auto controls in-turn prompting.

Note: В§6.1's "direct push to protected branch blocked" hook is **not** tripped by the merge step вҖ” `gh pr merge --auto` is a GitHub API call, not a `git push` to `main`.

### 5.8 `/adr <title>`

**ADR vs PR body Decisions**:
- **ADR** вҖ” irreversible, system-level. Migrations (DB engine, schema change), external contracts (API spec, wire format), default technology choices (auth scheme, queueing model). High cost to reverse once merged.
- **PR body Decisions** вҖ” reversible, PR-scoped trade-offs. Default option choice, algorithm tuning parameters, equivalent-library preference, naming choices. Explainable within this PR and not affecting other PRs.

`/adr` creates `docs/ADRs/NNNN-title.md` (from `.claude/templates/adr.md`). A `вҶ’ ADR-NNNN` link is auto-added to the PR body's Decisions section.

### 5.9 `/audit [<filter>]`

Query `.claude/audit/audit.jsonl`. Recent blocks / escapes / warns. A frequently-escaped category is a signal to tune the hook.

## 6. Hook enforcement

### 6.1 PreToolUse вҖ” block

**Framing**: Hooks are a **mistake-prevention guardrail**, not a security boundary. An adversarial user can bypass them via shell features the matcher doesn't model (`eval`, `bash -c`, `python -c`, heredoc-spawned shells, dynamic command construction via parameter expansion). The hook's job is to stop honest mistakes вҖ” typos, autopilot moments, the wrong tab вҖ” not to be a sandbox. Patterns that suggest a bypass attempt are not blocked but are **audit-logged as warnings** so the trail is preserved for review.

Implementation note: command strings are **normalized** before matching вҖ” backslash-continued multiline (`\\\n`) collapses to single-space, internal newlines become spaces, multi-space whitespace collapses. So `git \\\n  push \\\n  --force-with-lease` matches the same blocked pattern as the single-line form.

**Git option-prefix tolerance**: every `git <subcommand>` matcher accepts standard git-level options between `git` and the subcommand: `-c <key>=<value>`, `-C <path>`, `--git-dir=<path>`, `--work-tree=<path>`, `--no-pager`, `--literal-pathspecs`, `--exec-path[=<path>]`, `--namespace=<ref>`. So `git -c commit.gpgsign=false commit -m "вҖҰ"` and `git -C /some/repo push --force` both fire the same downstream checks as the bare `git commit` / `git push` forms вҖ” the option prefix can no longer be used (intentionally or not) to slip past the matcher.

| Trigger | Check | Escape category |
|---------|-------|----------------|
| `git commit` pattern | If on a protected branch (`main`/`master`/`release/*`), block. Detached HEAD on the tip commit of any protected branch is treated the same вҖ” the matcher resolves `HEAD` against `refs/heads/{main,master}` and `refs/heads/release/*` and blocks when they share a SHA. | `branch` |
| `git commit -m "..."` | Conventional Commits format. Issue # required per type (В§6.1.1) | `commit-format` |
| `git commit` | Secret scan over staged diff | `secret` |
| `git commit` | Quick lint/format check, bounded by `CLAUDE_ENG_LINT_TIMEOUT` (default 30 s; "quick" enforced) | `format` |
| `git commit --amend` | **Block only when the commit is already on upstream.** Local amend allowed | `amend` |
| `git commit --no-verify` | Block | `no-verify` |
| `git push` | Force variants (`--force`/`-f`/`--force-with-lease`) blocked | `force-push` |
| `git push origin main`, etc. | Direct push to protected branch blocked | `branch` |
| `rm -rf`, `mv -f`, `cp -f` args pointing outside registry (e.g. `rm -rf ~/.claude`, `rm -rf /`, `rm -rf ${HOME}/x`) | Block. Arg parsing is shell-aware (quotes preserved, no glob expansion via the hook); recognizes `~`, `$HOME`, and `${HOME}` forms; bare relative names are resolved against PWD and routed through `path_in_scope`. | `out-of-scope` |
| `eval`, `bash -c`, `sh -c`, `python -c`, heredoc-spawned shells (`bash <<EOF`) | Don't block. Audit-log as `warn` with category `bypass-suspect` so the trail exists. Hooks are guardrails, not sandboxes. | (warn-only) |
| `git reset --hard` | Block | `destructive` |
| `git clean -f` | Block | `destructive` |
| `git merge <protected>` (e.g. `git merge main`, `git merge origin/main`, `git merge release/v1`) on a non-protected branch | Block as a **backmerge**. The recommended motion when the base advances is `git pull --rebase origin <base>` per В§13; merging the base into the feature branch makes a noisy bubble that muddies the per-PR commit graph. Allowed when currently on the protected branch itself (e.g. merging a PR locally on `main`). | `backmerge` |
| Edit/Write target path outside the registry (except `$CLAUDE_ENG_SHELL_ROOT/` prefix, which is shell self-modification) | Block | `out-of-scope` |
| Edit/Write on `.env`, `*.pem`, `credentials*` | Block | `sensitive` |
| Edit/Write on a protected branch (except when the file path is under `$CLAUDE_ENG_SHELL_ROOT/`, which is shell self-modification) | Block | `branch` |

Tests are run by `/ship` (В§5.7). During the draft PR's commit/push, only quick lint/format runs. A commit-time test gate would force WIP/share commits to escape every time, polluting the audit log вҖ” so we don't add one.

**Lint timeout**: the commit-time lint invocation is bounded by `CLAUDE_ENG_LINT_TIMEOUT` (seconds, default 30) so a misconfigured or slow linter cannot hang the commit. Timeout fires as a `block format` decision, escapable via `SKIP_HOOKS=format`. If `timeout(1)` (coreutils) is not on `PATH`, the lint runs unbounded with an `audit_log warn` entry noting the missing dependency вҖ” the hook neither hangs the user nor silently downgrades enforcement.

#### 6.1.1 Conditional issue # enforcement

| commit type | issue # required | Rationale |
|-------------|------------------|-----------|
| `feat`, `fix`, `docs`, `refactor`, `perf` | Required | External surface / behavior change |
| `test`, `style`, `build`, `ci`, `chore`, `revert` | Optional (passes without it) | Internal change |

Structural regex:
```
# required group
^(feat|fix|docs|refactor|perf)(\(#[0-9]+\))!?: .+$
# optional group (issue # optional)
^(test|style|build|ci|chore|revert)(\(#[0-9]+\))?!?: .+$
```

Subject length is measured in **codepoints (1..72)**. Byte-based measurement (`wc -c`, `${#var}`) is forbidden вҖ” under multibyte locales it mis-truncates Korean/CJK. The choice of tool is an implementation detail.

**Subject extraction across `-m` forms**: the hook accepts both the plain quoted form and the command-substitution + heredoc form that the harness uses for multi-line bodies (`Co-Authored-By`, etc.).

| `-m` form | Subject |
|-----------|---------|
| `-m "feat(#1): subject"` / `-m 'feat(#1): subject'` | The quoted text |
| `-m "$(cat <<'TAG'` вҖҰ `TAG)"` (heredoc) | The first non-blank line of the heredoc body |

The heredoc tag may be single-quoted (`<<'EOF'`), unquoted (`<<EOF`), or indent-stripping (`<<-EOF`). The extractor walks the raw command (before whitespace normalization), locates the heredoc opener, and returns the first line of body content that is not blank and not the closing tag. Leading whitespace on that line is stripped before the Conventional-Commits regex runs.

Implementations that need both the heredoc and plain extraction should expose a single helper (`extract_commit_subject`) consumed by `pre_tool_use.sh`'s commit gate.

### 6.2 PostToolUse вҖ” supplementary

| Trigger | Action |
|---------|--------|
| After `Edit/Write/MultiEdit` | Auto-format the changed file with the detected formatter (failures don't block) |
| `Bash` matches `git commit` | Print a "remember to update the matching PR body checklist item" reminder (warn-only) |
| `Bash` matches `git push` | If a PR exists, check `gh pr checks` and notify the user |

### 6.3 Stop вҖ” recommendation after each response

Claude Code's `Stop` hook fires **after every model response**, not at session end. To avoid noise, suggest only once every N responses:

- uncommitted changes + no recent review вҶ’ suggest `/review`
- recent commit + PR body checklist out of sync вҶ’ suggest `/sync-pr`

Suggestions print one line to stderr (no blocking). Throttling is a simple modulo of N (default 5, overridable via `$CLAUDE_ENG_STOP_THROTTLE`) вҖ” no per-category state file.

Session-level cleanup (branch tidy, temp files) belongs to a separate `SessionEnd` hook (where supported) or a user-invoked `/wrap`.

### 6.4 UserPromptSubmit вҖ” context injection

Before each user message, inject as system context:

- Current branch
- `/status` summary (1вҖ“3 lines)
- Current PR body (if any, within length limit)
- Referenced issue body (if any, within length limit)
- 5-line summary of the shell's CLAUDE.md core principles

Length limit: combined output вүӨ 4 KB (4096 UTF-8 bytes). On overflow, PR body shrinks to checklist only, issue body shrinks to acceptance criteria only.

### 6.5 SessionStart вҖ” shell sync check + session restore

At session start, do two things:

**(a) Shell sync check**
- Is the shell repo behind origin/main? If so, warn (no auto-pull; sudden behavior changes are risky).
- The `git fetch` is gated by a stamp file at `.claude/state/last-shell-fetched`. The fetch runs only when the stamp is older than `SESSION_START_FETCH_TTL` seconds (default 21600 = 6 h) or absent. Most sessions therefore skip the network call вҖ” the `git rev-list --count HEAD..@{u}` still runs against locally-known refs, so the "behind" warning is correct relative to whatever was last fetched (up to 6 h stale, which matches the warning's polite-not-critical role).
- The fetch is wrapped in `timeout $SESSION_START_FETCH_TIMEOUT` (default 5 s) so a hung network does not stall session startup. If `timeout(1)` / `gtimeout(1)` is unavailable the fetch runs unbounded, matching SPEC В§6.1 / `CLAUDE_ENG_LINT_TIMEOUT`'s fallback pattern.

**(b) Session restore** вҖ” to seamlessly resume the previous session, inject once:
- The current branch's full PR body (if any)
- Full referenced issue body (if any)
- Target `MISSION.md` core summary (3вҖ“5 lines)
- Target `CLAUDE.md` directory index + domain vocabulary summary
- One `/status` snapshot

Subsequent turns only refresh the short status summary from В§6.4.

### 6.6 Stack-aware behavior

`helpers/detect_stack.sh` picks an active stack for `/ship`'s test runner, the commit-time lint gate, and the PostToolUse formatter. Detection is by sentinel file at PWD:

| Stack | Sentinel | Test (used by `/ship`) | Lint (commit gate) | Format (PostToolUse) |
|-------|----------|------------------------|--------------------|----------------------|
| node | `package.json` | `npm test` (if `test` script) | `npm run lint` (if `lint` script) | `prettier --write` |
| python | `pyproject.toml` / `setup.py` / `requirements.txt` | `pytest` | `ruff check .` | `ruff format` or `black` |
| go | `go.mod` | `go test ./...` | `go vet ./...` | `gofmt -w` |
| rust | `Cargo.toml` | `cargo test` | `cargo check` | `rustfmt` |
| ruby | `Gemfile` | `bundle exec rspec` | вҖ” | вҖ” |
| godot | `project.godot` | `godot --headless вҖҰ` (GUT if present, otherwise parse-only) | `gdlint .` (if installed) | `gdformat` (.gd) |

Notes for godot:
- Test: when `addons/gut/gut_cmdln.gd` is present, `/ship` invokes GUT via `godot --headless --path . -s res://addons/gut/gut_cmdln.gd -gexit`. Otherwise a parse-only check runs (`godot --headless --path . --quit-after 1`) вҖ” confirms the project loads.
- Lint / format: `gdlint` and `gdformat` ship in the `gdtoolkit` PyPI package. Helpers only invoke them when on PATH; absence is silent (no false-positive block).
- Scene and resource files (`.tscn`, `.tres`, `.import`) are not auto-formatted вҖ” they are editor-managed and any rewrite risks corrupting binary subresources.
- The C# variant of Godot (`project.godot` + `*.csproj`) still resolves to `godot` (script-level tooling). Mixed-language CI is the project's call, not the shell's.

**Formatter invocation safety**: `detect_format_cmd` shell-quotes the file path via `printf '%q'` before interpolating it into the returned command string. The PostToolUse hook can therefore safely `eval` the result вҖ” filenames containing spaces, `;`, `$(...)`, backticks, or other shell metacharacters do not change the executed command. (PostToolUse is best-effort and silent on failure; SPEC В§6.2.)

A target with no recognized sentinel resolves to `unknown`; `/ship` reports "no test runner detected" but does not block. Adding a stack means extending `detect_stack` plus the corresponding `detect_test_cmd` / `detect_lint_cmd` / `detect_format_cmd` branches.

## 7. Escape hatch

```bash
SKIP_HOOKS=<category>[,<category>...] SKIP_REASON='<reason>' <command>
```

The PreToolUse hook parses the `SKIP_HOOKS=вҖҰ` and `SKIP_REASON=вҖҰ` tokens from the leading edge of the `tool_input.command` string itself вҖ” it does not rely on those names being exported in the hook's own environment (Claude Code's hook subprocess does not inherit command-line env-prefixes). Tokens are matched left-to-right; **only `SKIP_HOOKS` and `SKIP_REASON` are recognized**, and any other ALLCAPS `K=V` token (including the documented-elsewhere `PATH=`, `LD_PRELOAD=`, etc.) halts the scan and is left in place as an ordinary command argument. The recognized tokens are exported into the hook before downstream matchers run, then stripped from the command for the matcher pass; the actual command-as-executed still receives the original env-prefix from its own shell. The user-facing syntax stays exactly as documented above; the parsing is an implementation detail.

Categories:
- `all`
- `secret`, `branch`, `commit-format`, `format`, `force-push`, `amend`, `no-verify`, `destructive`, `sensitive`, `out-of-scope`, `backmerge`

(`issue-ref` is folded into `commit-format` вҖ” not a separate category. `format` is the commit-time lint/format gate. `out-of-scope` covers paths outside the registry (В§6.1). `backmerge` covers `git merge <protected-base>` on a feature branch (В§6.1). Tests are run by `/ship`, not a hook, so there's no `tests` category.)

Every skip is recorded in `.claude/audit/audit.jsonl`:
```json
{"ts":"2026-05-19T...","event":"escape","decision":"skip","category":"format","reason":"flaky lint JIRA-1234","cwd":"..."}
```

The `reason` and `cwd` fields are JSON-encoded via `jq -Rs .` (or an inline `\n`/`\r`/`\t`/`\\`/`\"` escape when jq is absent). Newlines and control characters in `SKIP_REASON`, and shell metacharacters in PWD (legal in POSIX paths), therefore do not break the "one JSON object per line" invariant вҖ” every `audit.jsonl` is parseable by `jq -c '.'` regardless of input contents. Other fields (`event`, `category`, `decision`, `ts`) are internal-constant or RFC 3339-shaped and need no encoding.

Policy:
- Skips should be temporary.
- A skip with no `SKIP_REASON` is logged as `unspecified` вҖ” review target.
- `all` is for emergencies.
- A category that gets skipped repeatedly means the hook is misconfigured вҖ” open a fix PR.

Audit logs rotate as appropriate (exact threshold is an implementation choice), but the diff body is never written вҖ” only path, line, category metadata.

## 10. Convention summary

### 10.1 Branch naming

```
<gh-username>/<type>/[<issue#>-]<slug>

# required-issue-# group (feat/fix/docs/refactor/perf) вҖ” always includes <issue#>-
ilgyu/feat/42-oidc-login
ilgyu/fix/57-null-pointer
ilgyu/docs/89-update-architecture

# optional group (chore/style/build/ci/test/revert) вҖ” issue # may be omitted
ilgyu/chore/typo-readme
ilgyu/chore/89-update-deps    вҶҗ if an issue exists, you can include it
```

Types: `feat | fix | docs | style | refactor | perf | test | build | ci | chore | revert`

`/work-on <issue#>` always has an issue, so it always includes `<issue#>-` regardless of group. Issueless ad-hoc branches in the optional group are created by the user with plain `git checkout -b`.

### 10.2 Commit messages

```
<type>(#<issue>)[!]: <subject вүӨ 72 chars>

<body вҖ” one blank line>

Closes #<N>     вҶҗ last commit, or single-PR consolidator
Refs #<N>       вҶҗ intermediate commit
Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>   вҶҗ optional trailer; toggleable
```

Phase prefixes:
- `docs(#42): describe OIDC callback flow`
- `test(#42): add failing tests for /auth/oidc/callback`
- `feat(#42): implement OIDC callback handler`

**Co-Authored-By trailer toggle**: the `Co-Authored-By: Claude вҖҰ` line is a convention, not a contract. It is included by default but can be disabled per-target. Resolution priority (high вҶ’ low, mirroring SPEC В§5.7.1's mode pattern):

1. `$CLAUDE_ENG_COAUTHOR` env var (`on` / `off`).
2. `.claude/state/coauthor` per-target file (one of `on` / `off`, no other content).
3. Default `on`.

Unknown values fail-safe to `on` and emit a stderr warning naming the offending source. The helper `.claude/hooks/helpers/coauthor.sh` exposes `coauthor_trailer` (prints the line when enabled, empty when disabled); `/work-on`'s commit template sources it. The toggle covers the default behavior of `/work-on` flows only вҖ” the trailer remains free to be added or omitted manually in any specific commit.

**Relationship with Conventional Commits (conscious choice)**: we reuse the standard CC `scope` slot for the issue # marker. Consequences:

- Passes default `commitlint` rules (with `(#42)` allowed in scope).
- Tools like `conventional-changelog` / `semantic-release` parse `(#42)` as scope, so changelog scope columns will show `#42`. We don't carry both a domain scope (e.g. `auth`) and issue # вҖ” issue # wins.
- If a domain scope is essential, put it in the commit body via a `Scope: auth` trailer.

### 10.3 PRs

- Always start as draft.
- `Closes #N` or `Refs #N` is required. If a single issue spans multiple PRs, intermediates use `Refs #N` and the final uses `Closes #N`. Checklist distribution is planner/user's call.
- **Base**: PRs default to `main`. For experimental or long-lived feature work, `/work-on --base <branch>` opens the PR against an alternate base вҖ” see В§10.5. Non-`main`-base PRs default to `Refs` (the issue closes only when the work reaches `main`).
- Body follows `.claude/templates/pr_body.md`.
- Curate the body right after each commit.
- `/ship` transitions to ready after ship-gate verification.

### 10.4 PR вҶ” checklist sync

Order: **commit вҶ’ PR body update**. Commit is the fact; PR body reflects it. Reverse order creates "checklist ticked but commit absent" states. Refetch the remote body right before updating вҖ” abort the auto-update if external edits (reviewer, GitHub UI, parallel session) are detected.

### 10.5 Topic-branch workflow

The default flow (`/work-on <#>`) targets `main`. For two scenarios the user can opt into an alternate base via `/work-on <#> --base <branch>`:

1. **Experimental work** вҖ” risk too high for `main`. The user creates `experiment/foo` from `main`, iterates with a few PRs whose base is `experiment/foo`, and only consolidates to `main` via one final PR when the experiment stabilizes (or abandons the branch).
2. **Big features** вҖ” a multi-PR refactor where the integration point is a long-lived feature branch. Sub-task PRs target the feature branch; the feature branch itself is eventually merged to `main`.

**Conventions:**
- Sub-task PRs use `Refs #N` (not `Closes #N`) so the issue stays open until work reaches `main`. `/work-on` enforces this default for non-`main` bases.
- The alternate base must already exist locally + remotely. `/work-on --base <branch>` does NOT auto-create it; the user creates the base deliberately (e.g. `git checkout main && git pull && git checkout -b experiment/foo && git push -u origin experiment/foo`).
- The alternate base's remote branch-protection settings are the user's responsibility. For sub-task PRs to auto-merge into `experiment/foo`, the GitHub branch-protection rules on `experiment/foo` must permit the same flow as `main` (or be more permissive). `/work-on` does not inspect or modify remote settings.
- The alternate base is NOT in SPEC В§6.1's protected list (`main`/`master`/`release/*`). The user is free to commit directly to `experiment/foo` between sub-task PRs вҖ” it's an integration point, not the canonical product line.
- The work-branch name (`<gh-user>/<type>/<#>-<slug>`) is unchanged. Stacking multiple work-PRs on the same issue+base is not supported by `/work-on` alone вҖ” manual branch creation (`git checkout -b ...`) is the escape hatch for that rare case.

**Consolidation back to `main`:** when the topic branch is ready, open ONE PR with head=`experiment/foo` and base=`main`, body `Closes #<#>`. This is a regular `gh pr create` invocation вҖ” `/work-on` is not designed for the consolidator step (the branch already exists; there's no planner phase to repeat).

## 11. CI awareness

- **Right before** ready transition: `gh pr checks --json state` for a one-time check. **Block only on `failure` or `cancelled`**. `pending` / `in_progress` / `queued` pass вҖ” the background watch picks up monitoring after ready transition. (В§5.7 step 7)
- **Right after** ready transition: `gh pr checks --watch` runs in **background**, returning immediately.
- The background watch's failure detection routes to a PostToolUse hook (В§6.2) notification.
- The post-push PostToolUse hook (`Bash` matching `git push`) also warns about PR-less push patterns (detached HEAD, missing draft PR).

This shell repo ships its own CI workflow at `.github/workflows/ci.yml` running on `pull_request` and `push: main`. Two jobs:
- **syntax** вҖ” `bash -n` over `bin/claude-eng`, `scripts/*.sh`, `scripts/lib/*.sh`, `.claude/hooks/*.sh`, `.claude/hooks/helpers/*.sh`.
- **smoke** вҖ” installs `jq` via `apt-get`, runs `./scripts/test/smoke.sh`.

So `/ship`'s `gh pr checks` step has something concrete to watch for on PRs against this repo. Other repos that adopt the shell can copy the workflow as a starting point.

## 13. Failure modes and recovery

| Situation | Handling |
|-----------|----------|
| Issue body is ambiguous | `/work-on` refuses to start; asks the user to clarify the issue |
| Phase A reveals the spec was wrong | Strike through the checklist with reason, re-invoke planner |
| Phase B test passes accidentally | Two branches: (a) Phase A spec is weak or test inputs insufficient вҶ’ strengthen doc/test. (b) Adding a regression test for already-implemented behavior is legitimate вҶ’ note in one line in PR body and proceed. Don't always restart from doc. |
| Base (main or `--base` target) advances during work | `git pull --rebase origin <base>` вҶ’ resolve conflicts вҶ’ re-push PR. The base is whatever the PR's `baseRefName` is вҖ” read via `gh pr view --json baseRefName` if uncertain. Do NOT `git merge <base>` into the feature branch вҖ” the В§6.1 `backmerge` matcher blocks that motion specifically to keep feature branches rebase-clean (escape `SKIP_HOOKS=backmerge` exists for exceptional cases but should be rare). |
| Another contributor touches the same branch | `--ff-only pull`, escalate to user on conflict |
| Can't finish in one PR | Intermediate PRs use `Refs #N`, the final uses `Closes #N` |
| Issue acceptance criteria change mid-work | Update the issue body + comment with the change log; mirror in the PR body |
| CI fails while local passes | `/ship` blocks. CI logs first, then hotfix commit + re-push |
| Hook misfires | Use escape hatch + audit it + file a fix PR for the hook |
| Same escape category repeats in audit | Sign the hook is wrong вҖ” fix PR takes priority |

## 14. Security / sensitive items

- Direct editing of `.env`, `credentials.*`, `*.pem`, `id_rsa*`, etc. is blocked by hook.
- Pattern detection over staged diff for known secret formats (AWS, GitHub tokens, GitLab tokens, OpenAI/Anthropic keys, Slack tokens, password literals, etc.).
- Shell repo audit logs never contain secrets вҖ” the diff body is not written to the log.
- bootstrap checks `gh auth status`. Claude creates issues/PRs under the human's name, so commits include a `Co-Authored-By` trailer.

## 16. Build order (recommended)

When starting from an empty directory:

1. Directory skeleton + `.gitignore`. The shell repo's `.gitignore`:
   ```gitignore
   workspace/*
   !workspace/.gitkeep
   !workspace/README.md
   .claude/audit/
   .claude/state/
   ```
   The `workspace/` directory itself is tracked via `.gitkeep`; clones and symlinks inside are ignored, except for the explanatory `workspace/README.md`. `.claude/audit/` is for runtime logs; `.claude/state/` is for runtime state like the registry. `.claude/settings.local.json` exists only on the target side (as a symlink slot), so it's not in the shell repo's `.gitignore`.
2. `.claude/audit/` and `.claude/state/` empty dirs with `.gitkeep` вҖ” at runtime, `audit.jsonl` (audit/) and `registry.txt` (state/) are created here.
3. `.claude/templates/*` first вҖ” every command and agent consumes these.
4. `.claude/CLAUDE.md` вҖ” shell norms (a summary of В§1, В§2 of this document).
5. `.claude/agents/*` (six of them).
6. `.claude/hooks/helpers/*` (log, escape, cwd_guard, detect_stack, git_matcher, branch_guard, conventional_commit, secret_scan, tests, gh_state). `git_matcher` must precede `branch_guard` вҖ” branch_guard sources it for the `PROTECTED_BRANCH_PATTERN` SSOT.
7. `.claude/hooks/{pre_tool_use, post_tool_use, stop, user_prompt_submit, session_start}.sh`.
8. `.claude/settings.json` вҖ” registers the hooks above.
9. `.claude/commands/*` (onboard, file-issue, work-on, sync-pr, status, review, ship, adr, audit).
10. `bin/claude-eng` wrapper вҖ” including the auto-register prompt for unregistered external paths.
11. `scripts/bootstrap.sh`, `scripts/clone-into.sh`, `scripts/register.sh` (sharing inject logic via a helper).
12. `scripts/test/smoke.sh` вҖ” single-entry-point reproduction of the clone-into вҶ’ /onboard вҶ’ /file-issue вҶ’ /work-on вҶ’ D-T-C cycle вҶ’ /ship simulation (В§17's primary verification).
13. `docs/*`, `README.md`.
14. Run the smoke test once to confirm passing.
15. First-commit the shell repo on itself (bootstrap exception). All subsequent commits self-enforce the flow.
