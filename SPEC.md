## 0. Intent and scope

### 0.1 What this is

An operating shell for Claude Code that runs on top of the standard GitHub workflow. Target repos are cloned into the shell's `workspace/` directory and worked on from there.

The shell covers two layers of the same workflow:

- **Engineering execution** ‚Äî issue ‚Üí branch ‚Üí draft PR ‚Üí checklist commits ‚Üí ready PR ‚Üí merge. Per the GitHub-standard flow.
- **Directing maintenance** ‚Äî `MISSION.md` ‚Üí Directive Issue ‚Üí Execution Issue, where a Directive is a medium-term directional context that scopes several issues without being directly executable.

Both layers use the same generate ‚Üí review ‚Üí mode-based-approval ‚Üí audit pattern (¬ß1.5). See ¬ß1.7 for the eng-mode / dir-mode split, ¬ß2.1 for the Directive lifecycle, ¬ß4.9 for the dir-mode reviewer.

The shell guarantees:

1. **Adherence to the GitHub-standard flow** ‚Äî issue ‚Üí branch ‚Üí draft PR ‚Üí checklist commits ‚Üí ready PR ‚Üí merge.
2. **Doc ‚Üí Test ‚Üí Code work order** ‚Äî every code change goes through docs first, then tests, then implementation.
3. **Active SSOT maintenance** ‚Äî documents are updated alongside code. Code-only changes with stale docs are prevented.
4. **Aggressive use of subagents** ‚Äî output quality doesn't rely on a single model and single context. Specialized per-phase agents are composed.
5. **Hook enforcement of engineering discipline** ‚Äî direct commits to protected branches, force pushes, secret exposure, malformed commits, untested merges, etc. are actually blocked.
6. **Type-aware separation of dir-mode and eng-mode artifacts** ‚Äî Directives and Execution Issues share storage but follow distinct workflows; hooks, commands, and listings discriminate by the `Type` custom field (¬ß1.7).

### 0.2 What this is not

- Not a general dotfiles framework. The shell is scoped to `workspace/`.
- Not an absolute enforcer. No policy is set in stone. A conscious escape hatch is always open, and every bypass is recorded in the audit log.
- Doesn't own the target repo's knowledge SSOT. Updates are made by people / Claude inside the target.

### 0.3 Assumptions

- GitHub-standard, but **upstream-only**. Not a fork workflow. All branches are created on upstream; PRs go branch ‚Üí main within the same repo.
- Hosted in a git repo, shareable across a team/org. Changes ship via PR.

### 0.4 Deferred capabilities

The following directional-layer capabilities are deferred ‚Äî listed so current design choices preserve the option to add them later, not because they are scheduled work. Operating experience motivates each addition; see MISSION's "Success looks like" ‚Üí orchestrator criterion for the framing.

- Orchestrator / automatic switching between eng-mode and dir-mode workflows.
- `/derive-next-directive`, `/reassess` commands. (`/block-directive` and `/revise-directive` were promoted from this list under Directive #75 ‚Äî see ¬ß5.16 and ¬ß5.17.)
- Auto-confidence calculation for Directives.
- Auto-detection of Directive completion based on success-signal evaluation.
- Auto-revision of Directives based on engineering outcomes.
- Directive dependency graph or visualization.
- Multi-MISSION support (multiple canonical-direction docs per shell instance).
- External Director-tool integration interface (e.g., webhook receivers).
- Stop conditions / kill switches / budget controls (deferred with the orchestrator).
- `roadmap-reviewer` subagent (deferred until operating experience surfaces real friction).

## 1. Core principles

These principles must be visible throughout the shell. CLAUDE.md, agents, commands, and hook messages should all point to the same principles.

### 1.1 GH-standard is the backbone

The flow comes before the principles. Each principle is an attribute attached to some stage of the flow.

### 1.2 Doc ‚Üí Test ‚Üí Code order

A code change has three phases:

**Phase A ‚Äî Doc**: write the behavior / contract / decision into the doc first. Whichever of the target SSOTs applies (MISSION, README, CLAUDE.md, ARCHITECTURE, ADR, API doc).

**Phase B ‚Äî Test**: translate the doc from A into a test. The test must fail at this stage. Confirm the failure message is the intended picture.

**Phase C ‚Äî Code**: minimal implementation to make B pass. Run adjacent regression checks afterward.

Each phase should be a separate commit when possible. The PR's commit graph should show the doc ‚Üí test ‚Üí code arc.

Effect of this order:
- Intent is externalized before code, enabling self-consistency checking.
- Tests stay closer to spec than when written alongside code.
- Re-entering the work later, you immediately know "which phase am I in?"

**Exception categories** ‚Äî the following types relax the phase order. Note the reason in one line in the PR Plan.

| Type | Relaxation | Rationale |
|------|-----------|-----------|
| `fix` | failing test reproducing ‚Üí code ‚Üí (if needed) doc | Bug reproduction plays the spec role |
| `refactor` | when external behavior is unchanged, doc can be skipped; existing tests suffice | New doc/tests would be fake commits |
| `perf` | measure ‚Üí code ‚Üí doc (report results) | Hypothesis validation comes first |
| spike | free on a separate branch, restructured into a proper PR before merge | Protect exploration cost |
| one-line typo / obvious fix | single commit, keep the phase prefix | Splitting cost > benefit |

`feat`, `docs`, and changes to external contracts / API / schema apply the phase order strictly.

### 1.3 Active SSOT maintenance

Two SSOTs coexist:

| Layer | Location | Question it answers |
|-------|----------|---------------------|
| Process SSOT | claude-eng-shell git | **How** we work |
| Project SSOT | each target repo git | **What** / **why** we're building |

Target SSOT structure:
- `MISSION.md` ‚Äî the project's ground truth. Why it exists, success criteria, explicit non-goals.
- `README.md` ‚Äî for external readers.
- `CLAUDE.md` ‚Äî directory index, run commands, domain vocabulary, conventions.
- `docs/ARCHITECTURE.md` ‚Äî module boundaries, data flow.
- `docs/ADRs/NNNN-*.md` ‚Äî record of irreversible decisions.

A code change commits together with the SSOT items it invalidates or updates. If a doc is intentionally omitted, the commit body states `Docs: n/a ‚Äî <short reason>`.

### 1.4 PR-as-living-doc

The PR body simultaneously serves three roles:
- **The work item's ground truth** ‚Äî what this PR is for.
- **Cross-session memory** ‚Äî Claude's context is volatile but the PR body is durable. The next session auto-loads it to restore context.
- **Execution progress display** ‚Äî the checklist renders as a GitHub progress bar.

Principles:
- The PR body is **editorial, not append-only**. Curate right after each commit. Tidy stale items; if history is needed, keep one line per entry in a separate "Changelog" section.
- **commit ‚Üí PR sync** order. Fact (commit) first, then reflection (PR body). Reversing it creates false states.
- Before updating, refetch the remote body once. If it differs from what the shell last saw ‚Äî checked against the persistent cache at `.claude/state/pr-cache/<owner>__<repo>__pr-<n>.json` via `helpers/pr_cache.sh` (¬ß5.4) ‚Äî abort the automatic update and inform the user. Don't build a merge logic beyond that ‚Äî conflicts are rare, and a human should be in the loop when they happen.
- **No stale unchecked items at merge.** A merged PR body must reflect truth at the moment of merge. Every `- [ ]` line is in one of three terminal states: ticked (`[x]`) because it was done, marked `[~] N/A ‚Äî <reason>` because it intentionally won't be done, or removed from the body. "Pending, but oh well" is not acceptable ‚Äî closed/merged PRs are loaded by SessionStart (¬ß6.5) into future context, so stale uncheckedness propagates as misinformation. The same rule applies to **issue bodies on auto-close**: every acceptance-criterion item is ticked, N/A'd with reason, or struck through with a closing comment that names the disposition.

### 1.5 Aggressive use of subagents

Quality benefits from not loading every burden onto a single context.

| Situation | Subagent |
|-----------|---------|
| Wide read-only exploration | `explorer` |
| 3+ file changes, API/schema/migration | `planner` |
| Doc writing / updating | `doc-writer` |
| Test writing (Phase B) | `test-writer` |
| Pre-commit / pre-PR review | `code-reviewer` |
| Auth / input / deps / crypto changes | `security-reviewer` |
| Rationale check on a proposed issue body | `issue-reviewer` |
| Approach / alternatives check on a planner output | `plan-reviewer` |
| Quality check on a proposed Directive or completion claim | `activation-reviewer` (¬ß4.9) |

Independent investigations that can run in parallel: call multiple agents at once.

You don't need to call all nine on a single PR. Use them per the table; the main assistant shouldn't re-run in `explorer` what it already explored. No call-count limits or tracking.

**Operating-mode coupling**: `code-reviewer`, `security-reviewer`, `issue-reviewer`, `plan-reviewer`, and `activation-reviewer` all substitute for human review at different points in the flow. In `attended` mode they surface findings to the user, who decides; in `unattended` mode their verdict gates the next step directly (`ship` ‚Üí proceed; `refine` ‚Üí re-produce with feedback; `block` ‚Üí stop with the reason and either ask the user, or ‚Äî when no human is available ‚Äî park the work). (`activation-reviewer` uses the equivalent `pass` / `revise` / `reject` vocabulary, ¬ß4.9.1.) This pattern is what makes the `unattended` mode operationally complete: every human-confirm checkpoint has a subagent fallback.

**Filer-aware invariants** (added by Directive #92 ‚Äî Issue #95): a layer of protections **above** the attended/unattended axis, keyed on the Issue's `authorAssociation` per GitHub's role taxonomy (`OWNER` / `MEMBER` / `MAINTAINER` / `COLLABORATOR` ‚Üí trusted; everything else ‚Üí untrusted). Independent of operating mode ‚Äî both attended and unattended honor the same filer-aware rules. Two protections enforced by the `trusted-filer-mutate` hook matcher (¬ß6.1):

- **AI never auto-closes-as-not-planned on a trusted-filer Issue.** Closing a trusted-filer Issue with `--reason completed` is allowed (evidence-backed); closing as `not planned` or `duplicate` requires human confirm. Untrusted-filer Issues continue to follow the regular attended/unattended close rules.
- **Removing the `directive` label declassifies a Directive ‚Äî human confirm required always, regardless of filer trust tier.** The `directive` label is the type-awareness key (¬ß1.7, ¬ß6.1 `proposed-protect`); silently dropping it would bypass dir-mode review for an Issue still positioned as a Directive.

Both invariants escape via `SKIP_HOOKS=trusted-filer-mutate SKIP_REASON='<why>'` (¬ß7). Helper `is_trusted_filer <issue#>` lives at `.claude/hooks/helpers/issue_filer.sh`; caches per-session under `.claude/state/issue-filer-cache/`. Bot-filed Issues (Dependabot, GitHub Actions) carry `authorAssociation="NONE"` and are currently treated as untrusted ‚Äî documented limitation per the helper's notes (Directive #92 brief ¬ß11).

### 1.6 GitHub standard (upstream-only)

- No forks. The clone's origin is always upstream.
- All branches are created on upstream.
- PRs are branch ‚Üí **base** within the same repo. **Default base is `main`**; `/work-on --base <branch>` (¬ß5.3) opts into an alternate base for experimental work or long-lived feature branches. See ¬ß10.5 "Topic-branch workflow."
- Every change starts from an issue. issue ‚Üí PR is 1:N (intermediate PRs use `Refs #N`, final uses `Closes #N`). For PRs whose base is non-`main`, `Refs` is the right default ‚Äî the issue closes only when work reaches `main`.
- Branch naming: `<gh-username>/<type>/[<issue#>-]<slug>`. The required-issue-# group (¬ß6.1.1) always includes `<issue#>-`; the optional group (`chore`/`style`/`build`/`ci`/`test`/`revert`) may omit it. Examples: `ilgyu/feat/42-oidc-login`, `ilgyu/chore/typo-readme`.

### 1.7 Artifact tiers and the subagent shape they share

The shell operates over two **artifact tiers** above the commit/PR layer: a Directive Issue (medium-term directional context) and an Execution Issue (a concrete unit of work). Both tiers follow the same workflow pattern from ¬ß1.5 (generate ‚Üí review ‚Üí approval ‚Üí audit); they differ in which subagents author and which review at each step. The terms `eng-mode` and `dir-mode` name the workflow each tier exercises, but the substrate, hooks, audit log, mode toggle (¬ß5.7.1), and state directory are shared.

| Tier (workflow name) | Artifact type | Generate-subagent(s) | Review-subagent | Approval rule |
|---------------------|---------------|----------------------|------------------|---------------|
| **Execution Issue** (eng-mode) | Execution Issue, PR, Commit | `planner`, `doc-writer`, `test-writer`, `explorer` | `code-reviewer`, `security-reviewer`, `issue-reviewer`, `plan-reviewer` | attended: human / unattended: reviewer verdict (¬ß1.5) |
| **Directive Issue** (dir-mode) | Directive Issue | `planner`, `doc-writer`, `explorer` | `activation-reviewer` (¬ß4.9) | attended: human / unattended: `activation-reviewer` verdict |

**Hierarchy of artifacts** that the two modes act on:

```
MISSION.md                 ‚Üê human-defined long-term anchor (doc-as-code)
  ‚îÇ            ‚Üì
  ‚îÇ       Initiative        ‚Üê planning-tier strategic commitment, consumed from
  ‚îÇ          ‚îÇ                upstream ‚Äî never authored by the shell (live; the
  ‚îÇ          ‚îÇ                tier the shell consumes ‚Äî see below)
  ‚îî‚îÄ‚îÄ‚Üí Directive Issue ‚Üê‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îò  directing context (parent: a MISSION section OR an
       ‚îÇ                         Initiative), revisable, medium-term (dir-mode)
       ‚Üì
     Execution Issue        ‚Üê concrete unit of work (eng-mode)
       ‚Üì
     Pull Request           ‚Üê implementation evidence (eng-mode)
       ‚Üì
     Reflection             ‚Üê engineering result summary
```

Dir-mode operates on the Directive layer; eng-mode operates on Execution Issue and below. The Initiative tier above the Directive is consumed from upstream, never authored by the shell ‚Äî see "Initiatives ‚Äî the planning tier the shell consumes" below.

**Workflow selection is manual.** Users invoke `/file-directive`, `/activate`, `/complete-directive`, `/list-directives`, `/link-directive`, `/revise-directive`, `/block-directive`, `/consume-initiative`, `/initiative-feedback` (¬ß5.10‚Äì¬ß5.22) for directing work and `/file-issue`, `/work-on`, `/ship` for engineering work. (`/activate-directive` and `/triage` are deprecated aliases of `/activate`, retained one release cycle ‚Äî ¬ß5.12/¬ß5.18.) No orchestrator decides between them; an automatic selector is a future addition that operating experience may motivate (see MISSION's "Success looks like" ‚Üí orchestrator criterion).

**Strategic judgment remains external.** AI may generate, draft, propose, and review at any level. Activation, completion, and revision decisions pass through either a human (attended) or `activation-reviewer` (unattended) ‚Äî the shell never autonomously decides direction.

**Canonical direction.** The repo's `MISSION.md` is the canonical long-term anchor. Directives reference its sections in their `## MISSION fit` field; `activation-reviewer` checks alignment with MISSION rather than a separate Goal artifact. MISSION.md as doc-as-code beats an Issue-or-Project Goal artifact because the latter creates two representations of the same canonical direction ‚Äî revising one without the other surfaces drift.

**Substrate.** All dir-mode state lives on GitHub Issues. The rationale for Issues-as-SSOT over Project-fields-as-SSOT: Project access is owner-scoped while repo access is broader (storing state on Project fields gates writes by access the contributor may not have); dual-recording across labels, fields, and body invites silent sync drift; and Issues are the OSS-standard proposal channel that external contributors expect.

- **`directive`** label ‚Äî marks an Issue as a Directive. The single Type-awareness key.
- **`initiative`** label (#249) ‚Äî marks an Issue as an Initiative (the planning tier above a Directive, consumed from upstream ‚Äî see "Initiatives ‚Äî the planning tier the shell consumes" below). The single Type-awareness key for that tier, symmetric to and mutually exclusive with `directive`; resolved by the `is_initiative_issue` predicate (`helpers/issue_type.sh`).
- **`status:proposed`** label ‚Äî Status=Proposed (filed but not yet activated).
- **`status:blocked`** label ‚Äî Status=Blocked (active but cannot proceed without external input).
- Open + `directive` + no `status:*` label ‚Üí Status=**Active**.
- Closed + `directive` + state-reason=completed ‚Üí Status=**Completed**.
- **`P0`/`P1`/`P2`/`P3`** label ‚Äî Priority.
- **`execution`** label (#186) ‚Äî positive type label for an Execution Issue (a unit of work parented under a Directive). Makes Execution a first-class queryable type rather than label-absence; the `Parent Directive: #N` marker is then purely the parent pointer. Distinct from `task` (standalone, not parented) and `bug`.
- **Issue body** ‚Äî Objective, Success signals, Non-goals, Constraints, MISSION fit (Confidence is documented in body, not a field).
- **Body line 1 `Parent Directive: #N`** ‚Äî Parent reference for Execution Issues; consumed by `/reflect`, the dir-mode-post-merge workflow, AND the issues-to-project-mirror workflow.

The GitHub Project named `<repo-name> roadmap` (override via `$CLAUDE_ENG_PROJECT_NAME`) is a **derived view**, not the SSOT. Setup script `scripts/setup_project.sh` ensures the four mirrored fields exist: `Type / Status / Priority / Parent` (SINGLE_SELECT for the first three, TEXT for the last). `Iteration` is owner-managed via the GH UI; the mirror workflow does NOT mirror it (Iteration is owner-level cadence, not repo-level deliverable grouping ‚Äî those concerns split between Project Iteration and GitHub Milestone). Confidence and Success Signals live in the Issue body, not Project fields. The Project Item is populated by `.github/workflows/issues-to-project-mirror.yml` on every `issues.{opened,edited,labeled,unlabeled,closed,reopened,milestoned,demilestoned}` event. Dir-mode commands write to Issues only; they never write to the Project (the mirror handles the derivation). The two-layer idempotency contract on `setup_project.sh` (field existence + SINGLE_SELECT option-set correctness, from issue #76) carries forward to the 4-state Status schema: missing declared options are added via `updateProjectV2Field` GraphQL mutation; user-added options outside the declared set are preserved.

**Filer-aware invariants** (¬ß1.5, cluster C / issue #95): the `trusted-filer-mutate` hook matcher protects trusted-filer (OWNER / MEMBER / MAINTAINER / COLLABORATOR per `authorAssociation`) Issues from AI auto-close-as-not-planned and protects the `directive` label from declassification on any filer. Mode-independent ‚Äî applies in both `attended` and `unattended`.

**Audit-line format guard** (issue #87): `audit_log` in `.claude/hooks/hookrt.sh` enforces strict format on `directive-file/created` lines pre-write ‚Äî malformed calls are rejected and written as `decision=format-error` warn lines instead. Smoke ¬ß50a remains as defense-in-depth + historical-line grandfathering.

**Initiatives ‚Äî the planning tier the shell consumes** *(fully live as of milestone `v2 consuming initiatives`: the **recognition layer** (the `initiative` label, the `is_initiative_issue` predicate, `Parent Initiative: #N` marker reading ‚Äî #249, M1.1), the **enforcement layer** (label mutual-exclusivity, the Directive parent-XOR, Initiative read-only-except-comments ‚Äî #251, M1.2), the **reviewer layer** (`activation-reviewer` parent-kind branching + the contract-evaluability and extraction-faithfulness checks ‚Äî #254, M2), the **consumption** flow (`/consume-initiative` extraction ‚Äî #257, M3), and the **feedback** flow (`/initiative-feedback` structured challenge + completion ‚Äî #260, M4) are all live.)* An **Initiative** is a planning-tier strategic commitment sitting one tier above a Directive. It arrives from **outside the shell** as a GitHub Issue bearing the `initiative` label ‚Äî the single Type-awareness key for the tier, symmetric to `directive` and mutually exclusive with it. The shell **consumes** Initiatives and never authors them; an Initiative is **read-only to the shell except for appended comments** (the shell never edits its body, labels, or state). Consequently an Initiative is **outside the 4-state Directive lifecycle and the `status:*` taxonomy entirely** (¬ß2.1): the shell never applies `status:proposed`/`status:blocked`/`status:*` to it, it is never "Activated" or "Completed" or "Blocked," and `/activate` refuses it (¬ß5.12). Its real progress lives upstream and surfaces only through `/initiative-feedback` (¬ß5.22); the Project mirror may render an open Initiative as `Active` purely as a derived-view artifact of the shared Status field (¬ß2.1), not a lifecycle claim. The origin of an Initiative (a person, a planning tool) is immaterial ‚Äî the shell relies only on the contract below, not on how the Initiative was produced.

- **Parent generalization.** A Directive has exactly one parent, of one of two mutually-exclusive kinds: a **MISSION section** (named in the Directive's `## MISSION fit` body field per ¬ß2.1 ‚Äî the default) or an **Initiative** (a `Parent Initiative: #N` marker on body line 1, symmetric to `Parent Directive: #N`). The two parent kinds have **different shapes** ‚Äî a body *field* naming a section vs. a line-1 *marker* ‚Äî so the parent-XOR check (below) tests "a MISSION-fit field and no Initiative marker" XOR "an Initiative marker and no MISSION-fit field," not two parallel markers. An Initiative is introduced only when a strategic commitment genuinely bundles several Directives and is worth reviewing and evolving in its own right; the default stays MISSION-parented, so the tier does not become ceremony. This **generalizes** the hierarchy above rather than replacing it ‚Äî every existing MISSION-parented Directive remains valid unchanged.
  - **Create-path split.** The two parent kinds are born from **different commands**: `/file-directive` (¬ß5.10) authors a `## MISSION fit` field only, so it produces **MISSION-parented** Directives exclusively; `/consume-initiative` (¬ß5.21) is the **sole** producer of `Parent Initiative`-marked Directives. There is deliberately no `/file-directive` path to an Initiative parent ‚Äî a hand-added `Parent Initiative` marker on a `/file-directive` body would carry a MISSION-fit field too and trip the parent-XOR. To parent a Directive under an Initiative, consume the Initiative; do not hand-author the marker.
- **The contract (load-bearing).** An Initiative must carry a **termination condition evaluable without knowledge of the code.** The shell extracts Directives by working backward from it. A *complete* Directive needs code knowledge ‚Äî its Success signals and Constraints reference code reality, which is why the shell, not the upstream, extracts Directives ‚Äî but the Initiative's termination condition sits above code and so can and must be stated without it. A condition that cannot be (one that smuggles in execution detail) marks the Initiative under-specified; the shell surfaces that upstream rather than guessing. This is the planning/execution boundary: above the point where a termination condition can be set without the code is the upstream's job; below it is the shell's.
- **Feedback is upward and non-deciding.** The shell sends findings to the Initiative as comments, a structured **challenge** when execution reality contradicts the Initiative, and a **completion** signal when the extracted Directives land ‚Äî carrying code-derived information the planning layer structurally cannot have (real dependencies, measurability, implementation cost). It never rejects, retires, or asserts completion of an Initiative: discarding or closing a strategic commitment is a strategic judgment that stays upstream. This mirrors the reviewer pattern ‚Äî a `block`/challenge escalates rather than auto-discards (¬ß4.9.1, ¬ß5.7.1).
- **Location.** An Initiative is a GitHub Issue **in the same target repo**, located by the `initiative` label. The mechanism of its arrival is out of scope; **cross-repo Initiatives are out of scope** (they would reintroduce the cross-repo-trust surface of ¬ß6.1's `trusted-filer-mutate` close arm).
- **Mechanisms.** **Live (M1.1, #249):** the `is_initiative_issue` predicate (symmetric to `is_directive_issue`, `helpers/issue_type.sh`; cached, fail-open per ¬ß6.1) and `Parent Initiative: #N` marker reading (the `issue_has_initiative_parent_marker` tri-state resolver, distinct from the `Parent Directive` resolver that backs `label-parent-consistency`). **Live (M1.2, #251):** three enforcement matchers ‚Äî the `initiative`/`directive` label mutual-exclusivity and the Directive parent-XOR (exactly one parent kind present: a `## MISSION fit` field **or** a line-1 `Parent Initiative:` marker, never both and never neither (different shapes ‚Äî see Parent generalization), enforced via `issue_has_mission_fit_field`/`issue_has_initiative_parent_marker`), both folded into the `label-parent-consistency` matcher on the `--add-label` path; and `initiative-readonly` (mutating `gh issue edit`/`close`/`reopen` on an `initiative` Issue blocked, `gh issue comment` allowed). All fail-open per ¬ß6.1. **Live (M2, #254):** reviewer branching ‚Äî `activation-reviewer` (¬ß4.9, extended in place rather than via a new subagent, per the ¬ß4.9.3 caveat) recognizes the `initiative` label, branches a Directive's alignment check on its parent kind (MISSION section vs. parent Initiative's termination condition), and gains a contract-evaluability check (Initiative intake) + an extraction-faithfulness check (post-extraction), both invoked by the M3 consume flow. **Live (M3, #257):** the `/consume-initiative <N>` command (¬ß5.21) ‚Äî the consume flow: contract gate (I1) ‚Üí working-backward extraction ‚Üí faithfulness gate (I2) ‚Üí file the extracted Directives as `status:proposed` `Parent Initiative`-parented proposals (never activated; the only Initiative write is a plain under-specification comment). **Live (M4, #260):** the `/initiative-feedback <N>` command (¬ß5.22) ‚Äî the upward-feedback flow: a `## Initiative challenge` comment (code-derived contradiction ‚Üí request re-evaluation, escalate-not-reject) and a `## Initiative completion` comment (merged-Directive evidence ‚Üí request termination assessment, never assert-done/close); comments only, never mutating the Initiative. **The milestone-`v2` model is now fully live** ‚Äî recognition + enforcement + reviewer + consume + feedback.

**Out of scope** (Initiative tier): authoring Initiatives; knowing their origin; tracking a challenge's outcome once escalated; rejecting/retiring an Initiative from within the shell; nested Initiative tiers (the tier is flat above the Directive); automatic orchestration between planning-mode and execution-mode work; cross-repo Initiatives.

#### Substrate-in-target contract (Issue #114; Directive #107)

Dir-mode's substrate is installed into target repos under `workspace/` via the `/onboard-dir-mode` skill. The boundary broadening ‚Äî shell now mutates target `.github/` files + labels in addition to its `.claude/` symlinks ‚Äî is acceptable only because each install step is detectable, individually revertable, and additive (never replaces target-owned artifacts). The reversibility contract below makes adoption low-risk: a target can fully unwind the shell via one PR plus a few `gh` commands, with no load-bearing dependency on installed files.

**Three-tier feature model.** Targets adopt the shell at one of three tiers; each is a strict superset of the previous and tier transitions are mechanical:

- **Tier 1: eng-mode** ‚Äî no substrate required. `/file-issue`, `/work-on`, `/ship`, secret scan, AC closeout, conventional-commit enforcement.
- **Tier 2: dir-mode-with-labels** ‚Äî tier 1 + the 12-label dir-mode set (`directive`, `initiative`, `status:proposed`, `status:blocked`, `awaiting-author`, `execution`, `task`, `discussion`, `P0`, `P1`, `P2`, `P3`), installed via `scripts/ensure_v3_labels.sh` + inline `directive` and `initiative` creates. (Count note: these 12 enumerate the dir-mode lifecycle/priority/type labels; the tier-2 *installer* reports **13 total** because `ensure_v3_labels.sh` creates 11 labels that include the release-backbone `skip-changelog` (#133) but not the two tier type-keys, and `onboard_target.sh` then adds `directive` and `initiative` (#249) inline ‚Äî i.e. 11 ensure + 2 inline = 13, traceable to `onboard_target.sh`'s count message and smoke ¬ß63g.) Unlocks `/file-directive`, `/activate`, `/complete-directive` writing to Issues directly. No Project mirror, no Issue templates. (`awaiting-author` ‚Äî #172 ‚Äî marks the handoff state after a `revise` or trusted-filer `reject`; it is an action-signal label without a `status:` prefix, orthogonal to the lifecycle stage. The former raw-filing triage label was retired by #179 ‚Äî raw filings now receive `status:proposed`+`task` directly.)
- **Tier 3: full substrate** ‚Äî tier 2 + `.github/ISSUE_TEMPLATE/*.yml` + `.github/workflows/{auto-status-proposed,auto-clear-awaiting-author,issues-to-project-mirror,dir-mode-post-merge,check-changelog}.yml` (5 workflows, matching the canonical source `target-substrate/workflows/` and smoke ¬ß63a) + GitHub Project (`<repo> roadmap`) with the dir-mode field schema. Unlocks the template chooser for external contributors and the Project-as-derived-view. **Blank issues are disabled** (`blank_issues_enabled: false`, #186) ‚Äî every UI filing must use a typed template (directive/execution/task/bug/discussion), so it is always classified and (for the four work-types) `status:proposed`-gated; the `discussion` template (¬ß5.19) is the sanctioned low-friction path, so blank issues are not needed. (`auto-status-proposed.yml` ‚Äî repurposed by #179 ‚Äî now serves as **defense-in-depth** for deliberate non-UI label-free creations (API / `gh issue create --no-label`): it stamps `status:proposed`+`task`. `/triage` is a deprecated alias for `/activate`, ¬ß5.12/¬ß5.18.)

**Bootstrap path.** `/onboard-dir-mode` (deferred) installs the tier-N artifacts via a **PR** to the target ‚Äî never direct-push. Labels + Project install directly (not git-tracked); maintainer reverts via `gh label delete` / `gh project delete` per the reversibility contract below.

**Graceful-degradation principle.** Dir-mode commands check their substrate prerequisites before acting. **Hard-refusal only when the SPECIFIC dependency for the command is missing AND the command's contract cannot be met without it** ‚Äî otherwise run in degraded mode with a one-line warning. Per-command judgment, not global. Test anchors: `/file-issue` works at tier 1; `/file-directive` hard-refuses at tier 1 (no `directive` label), degrades-with-warning at tier 2 (no Project mirror), works fully at tier 3; `/activate` (which `/triage` now aliases, ¬ß5.12/¬ß5.18) requires tier 2 (the `status:proposed` label); `activation-reviewer` at tier 2 falls back to the existing bootstrap allowance for repos without MISSION.md.

**Reversibility contract.** Every install step is detectable + revertable: files via `git rm` in a normal PR; labels via `gh label delete`; Project via out-of-band `gh project delete` (owner-state, not scripted). The shell can be fully unwound from a target via one PR (file removals) + one-liner per label + an out-of-band Project decision. Per-command preflight implementation is deferred to a subsequent Execution Issue under Directive #107.

### 2.1 Directive lifecycle

A Directive moves through **four states** (Proposed / Active / Blocked / Completed). All transitions are entered via explicit dir-mode commands (¬ß5.10‚Äì¬ß5.18). Status is encoded as **labels on the Issue**; the Project Item's Status field is mirrored by `.github/workflows/issues-to-project-mirror.yml`. A transient revision state was considered and dropped ‚Äî the audit-log entry + the archived prior-body comment are the durable evidence of revision (see `/revise-directive`, ¬ß5.16).

**Scope of the 4-state machine.** The four states + the `Blocked`/`Completed` transitions are **Directive-specific**. Execution Issues (`execution`/`task`/`bug`) share only the `Proposed ‚Üí Active` gate (the full-symmetry lifecycle below), not `Blocked`/`Completed`. **`discussion`** Issues (¬ß5.19) and **`initiative`** Issues (¬ß1.7) are **outside this lifecycle entirely** ‚Äî a discussion closes via `/resolve-discussion`, and an Initiative is read-only/consumed (the shell never sets its `status:*`). Because the mirror derives the Project `Status` field from the open/no-`status:*` label shape, an open `discussion`/`initiative` Issue surfaces in the Project as `Active` by default; that is a derived-view artifact of the shared field, **not** a claim that they run the Directive 4-state machine ‚Äî read their real state from their own path (resolve/consume), not the mirrored `Status`.

| State | Encoding on Issue | Entry transition | Reviewer-gated? |
|-------|-------------------|------------------|-----------------|

**Lifecycle symmetry for Execution Issues** (#172, Directive #167). Under **full symmetry**, the `Proposed ‚Üí Active` gate applies to **all** Issues, not just Directives: `/file-issue` stamps `status:proposed` on every new Execution Issue, and `/activate <N>` (¬ß5.12) removes it after an `activation-reviewer` pass ‚Äî the same generate ‚Üí review ‚Üí gate shape the Directive path uses. `status:proposed` Execution Issues are Issue-ops-only and cannot enter the PR/branch flow (the `proposed-protect` hook, ¬ß6.1, blocks branch creation). `issue-reviewer` (author-side, at `/file-issue`) and `activation-reviewer` (observer-side, at `/activate`) are complementary, mirroring how `/file-directive` and `/activate` both gate the Directive path. The `awaiting-author` label (¬ß1.7) marks the handoff state when the reviewer returned `revise` or a trusted-filer `reject` and the author's action is pending.

**Reviewer-gating contract** (per ¬ß1.5 operating-mode coupling):

- `attended` mode: human reviews `activation-reviewer`'s output and decides.
- `unattended` mode: `activation-reviewer`'s VERDICT line (`pass` / `revise` / `reject`) gates directly. `reject` on activation applies the filer-aware handling (¬ß5.12: trusted ‚Üí keep open + `awaiting-author`; untrusted + draft ‚Üí close + auto-discussion; untrusted no-draft ‚Üí close); `reject`/insufficient-evidence on completion leaves the Directive `Active`; `reject` on revision aborts the body replacement (no Status change, no audit `reconciled` line). `revise` retains `status:proposed` + `awaiting-author`, escalating to `reject` after N=3 rounds.

**Escape hatch.** `SKIP_HOOKS=directive-review SKIP_REASON='<why>' /complete-directive <id>` (and the equivalent on `/file-directive`, `/activate`, `/revise-directive`) bypasses the reviewer; the bypass is audit-logged per ¬ß7. Use is reserved for cases where the reviewer's verdict is wrong and a human accepts the recorded responsibility ‚Äî not a normalized routing. `/block-directive` is not reviewer-gated (annotation-only ‚Äî no body change) and therefore has no review-skip variant.

**Audit categories** for dir-mode events: `directive-file`, `directive-activate`, `activation` (the `/activate` category, generalized from `directive-activate`; the log retains historical `directive-activate` entries), `directive-complete`, `directive-revise`, `directive-block`, `directive-link`, `directive-reflect` (`/reflect` posts/skips), `directive-exec-count` (`/ship`'s post-merge directive-aware audit, ¬ß5.7), `directive-review`, `proposed-protect` (generalized from `directive-protect` by #171; the append-only log retains historical `directive-protect` entries), `triage` (decisions: `decided` with action=accept|reject, `refiled`, `rejected-skip`), `trusted-filer-mutate` (decisions: `deny`, `allow`), `label-parent-consistency` (decisions: `block`, `allow`; enforces label‚Üîparent-marker consistency on `gh issue edit --add-label` ‚Äî the `Parent Directive` marker for execution/task/bug (#199), plus the `initiative`/`directive` tier-key mutual-exclusivity and the Directive parent-XOR (#251), ¬ß6.1), `initiative-readonly` (decisions: `block`, `allow`; blocks mutating `gh issue edit`/`close`/`reopen` on an `initiative` Issue ‚Äî the shell consumes Initiatives, never mutates them; #251, ¬ß6.1), `initiative-consume` (decisions: `extracted` with count=N, `under-specified`, `revise`; emitted by `/consume-initiative` ¬ß5.21, #257), `initiative-feedback` (decisions: `challenge`, `completion`; emitted by `/initiative-feedback` ¬ß5.22, #260), `mirror-stale-warn` (reserved for `issues-to-project-mirror.yml` failures; currently emits nothing). All append to `.claude/audit/audit.jsonl` per ¬ß6/¬ß7.

### 3.2 Injection mechanism

`scripts/clone-into.sh` runs the following right after `git clone`. **The target's existing `.claude/` content is never overwritten.** If a same-named asset already exists, that asset is not injected ‚Äî only a warning is printed. The target's choice wins.

```bash
mkdir -p "$TARGET/.claude" "$TARGET/.claude/agents" "$TARGET/.claude/commands"

# settings.local.json: symlink to the shell's settings.json (Claude Code's 'this clone only' slot).
if [ -e "$TARGET/.claude/settings.local.json" ] && [ ! -L "$TARGET/.claude/settings.local.json" ]; then
  echo "WARN: $TARGET/.claude/settings.local.json already exists. Shell settings not injected; user merge needed." >&2
else
  ln -sfn "$SHELL_ROOT/.claude/settings.json" "$TARGET/.claude/settings.local.json"
fi

# agents/commands: skip and warn if a same-named asset exists.
for kind in agents commands; do
  for src in "$SHELL_ROOT/.claude/$kind"/*.md; do
    [ -e "$src" ] || continue
    dest="$TARGET/.claude/$kind/$(basename "$src")"
    if [ -e "$dest" ] && [ ! -L "$dest" ]; then
      echo "WARN: $dest already exists. Shell asset injection skipped." >&2
    else
      ln -sfn "$src" "$dest"
    fi
  done
done

# Add the shell injection (symlink) to the target's .git/info/exclude so target git doesn't see it.
grep -qxF '.claude/settings.local.json' "$TARGET/.git/info/exclude" 2>/dev/null \
  || printf '.claude/settings.local.json\n' >> "$TARGET/.git/info/exclude"
```

Rules:
- If the target has a same-named asset, the shell's version is not injected (conflict avoidance). If you want to use both, rename or move your own asset.
- Post-hoc registration of a directory the user already cloned: `scripts/register.sh <path>` runs the same inject logic.
- The target's `CLAUDE.md`, `MISSION.md`, `docs/` are **not touched**. The shell's CLAUDE.md (process norms) is combined into context via the UserPromptSubmit hook.

#### 3.2.1 Hook command path resolution

All `command` entries in the `settings.json` pointed to by the symlinked `settings.local.json` must use **absolute paths based on `$CLAUDE_ENG_SHELL_ROOT`**. When Claude Code runs a hook, `$CLAUDE_PROJECT_DIR` points to the target, not the shell root. Relative paths or `$CLAUDE_PROJECT_DIR` would resolve to an empty location in the target and fail.

```json
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "hooks": [{
        "type": "command",
        "command": "$CLAUDE_ENG_SHELL_ROOT/.claude/hooks/pre_tool_use.sh"
      }]
    }]
  }
}
```

`$CLAUDE_ENG_SHELL_ROOT` is exported by the `bin/claude-eng` wrapper on entry:

```bash
#!/usr/bin/env bash
set -euo pipefail
export CLAUDE_ENG_SHELL_ROOT="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd -P)"
exec claude "$@"
```

Inside hook scripts, shell-side assets (helpers, audit) are accessed via `$CLAUDE_ENG_SHELL_ROOT`, while target files are accessed via `$CLAUDE_PROJECT_DIR` ‚Äî keep the two cleanly separated.

### 3.3 Scope guard ‚Äî cwd-based

All hook scripts prelude:

```bash
# Match physical PWD against registered (physical) paths in the registry.
# The registry is written by register.sh / clone-into.sh; clones inside workspace/ are recorded the same way.
REGISTRY="$CLAUDE_ENG_SHELL_ROOT/.claude/state/registry.txt"
PWD_REAL=$(cd "$PWD" 2>/dev/null && pwd -P) || exit 0
[ -f "$REGISTRY" ] || exit 0
in_scope=0
while IFS= read -r entry; do
  [ -z "$entry" ] && continue
  case "$PWD_REAL/" in "$entry"/*) in_scope=1; break ;; esac
  [ "$PWD_REAL" = "$entry" ] && { in_scope=1; break; }
done < "$REGISTRY"
[ "$in_scope" = "1" ] || exit 0
```

‚Üí If `claude-eng` is invoked from an unregistered directory, hooks pass through transparently. Works regardless of logical vs physical `$PWD` (we use PWD_REAL). External path registration is covered by the same mechanism.

`.claude/state/registry.txt` format: one physical path (`pwd -P` output) per line. Comments and blank lines are ignored.

### 3.4 User-global isolation

The shell's installation and operation leave no changes in the user-global environment. The only way to "disable" the shell is to stop invoking `claude-eng`. No deactivation procedure or cleanup script is needed.

**Surfaces never touched**:
- `~/.claude` (user-global Claude settings)
- shell rc/login files (`~/.zshrc`, `~/.bashrc`, `~/.bash_profile`, `~/.profile`, `~/.config/zsh/*`, `~/.config/fish/*`, etc.)
- global git config (`~/.gitconfig`, `~/.config/git/*`)
- PATH auto-registration ‚Äî the user adds `<shell-root>/bin` to PATH or uses absolute path / alias themselves. The shell never injects an export line into rc files.
- LaunchAgents, systemd user units, login items, or any auto-start registration

`bootstrap.sh` boundary:
- Allowed: check dependencies (`gh`, `git`, `python3`, `jq`), run `gh auth status`, create `.claude/audit/` and `.claude/state/` inside the shell repo.
- Forbidden: any modification of the surfaces above, package manager invocations, sudo.

Target repos are the shell's work ‚Äî they fall outside this isolation rule (the injection in ¬ß3.2 is intended behavior).

### 3.5 Entry point

`claude-eng` is meant to start at the root of `workspace/<repo>/`. Subdirectories work too (walk-up), but starting at the root avoids edge cases in monorepos where subprojects may carry their own `.claude/`.

**External path support**: a target repo can live outside the shell's `workspace/` (e.g. `~/code/<repo>`). `scripts/register.sh <abs-path>` (1) creates a `workspace/<basename>` symlink, (2) adds the target's physical path (`pwd -P`) to `.claude/state/registry.txt`, (3) runs the inject logic of ¬ß3.2. The shell's `.gitignore` whitelists `workspace/*` so the shell git tree doesn't pick up target trees even in the external-path case. Direct clones into `workspace/` are also recorded in the registry ‚Äî the guard is always registry-based.

**Auto-register prompt**: when `claude-eng <path>` is called with an unregistered external path, prompt once before entering the session:
```
$ claude-eng ~/code/foo
? '/Users/.../code/foo' is not registered. Register as workspace/foo and enter? [Y/n]
```
yes ‚Üí run `scripts/register.sh` ‚Üí inject logic (¬ß3.2) ‚Üí start session. no ‚Üí exit.

**Monorepo subprojects**: clone-into injects **only at the repo root**. Subprojects with their own `.claude/` are not recursively injected. When the user starts `claude-eng` in a subproject, the walk-up finds the root's shell injection. The subproject's `.claude/CLAUDE.md` is merged via Claude Code's default walk-up behavior.

**Version surface** (Issue #123, Directive #122): `claude-eng --version` reads the top-level `VERSION` file (single line, semver 0.x format ‚Äî `MAJOR.MINOR.PATCH` with `MAJOR=0` per Directive #122 constraint #1) and exits 0 with that string on stdout. If `VERSION` is missing or empty, the handler falls back to `git -C "$SHELL_ROOT" describe --tags --always`; if both fail, a one-line diagnostic is printed to stderr and the process exits 1. The `--version` short-circuit runs immediately after `SHELL_ROOT` resolution and before the cwd-based registry guard (¬ß3.3), so it works from any cwd including unregistered paths. The flag has no side effects: no audit log, no registry mutation, no cwd change. Tags follow `v` + semver (e.g., `v0.2.0`) and are pushed manually post-merge ‚Äî `gh pr merge` does not push tags, and the merge gate does not assert tag-state freshness (explicit Directive #122 non-goal).

### 3.6 Shell self-registration

The shell repo itself must appear in `.claude/state/registry.txt` so that `cwd_guard.in_scope` (¬ß3.3) returns true when `claude-eng` is invoked from the shell root. Without it, the shell's own hooks short-circuit on `in_scope || exit 0` and the dogfooding promise of ¬ß17 is empty ‚Äî the shell can't enforce its own discipline on itself.

`scripts/bootstrap.sh` ensures `$SHELL_ROOT` is present in the registry (idempotent ‚Äî repeated runs do not duplicate). Self-registration is **registry-only**: no `workspace/<basename>` symlink and no `inject_into` are performed against the shell, because the shell IS the injection source and the symlink would loop into `workspace/` contained within `$SHELL_ROOT`.

`scripts/register.sh "$SHELL_ROOT"` (or any path that physical-resolves to the shell root) recognizes this case and:
- skips the `workspace/<basename>` symlink (would otherwise create a directory loop);
- skips `inject_into` (the source files are the destinations);
- still ensures the registry entry exists.

For target repos this is unchanged: ¬ß3.2 inject + registry entry, both performed by `clone-into.sh` / `register.sh` as before.

## 4. Subagent specifications

Each agent is defined in `.claude/agents/<name>.md` as YAML frontmatter + body.

### 4.1 planner

**When**: before any code edit. Required for 3+ file changes, migrations, schema changes, API changes, or external contract changes.

**Input**: user request, linked issue body, MISSION.md, target CLAUDE.md.

**When MISSION is absent**: planner proceeds with `issue body + README + CLAUDE.md` alone. The "How this serves the mission" section is labeled `(MISSION.md absent ‚Äî review /onboard stub suggestion)`. Don't synthesize a MISSION via inference.

**Output** (markdown that drops straight into the PR body):

```
## How this serves the mission
<Which MISSION.md item this contributes to>

## Plan
<Why this approach, trade-offs, decision rationale>

## Target base
<branch name; default `main`. For non-`main`, name the topic-branch / feature-branch the PR will merge into. See SPEC ¬ß10.5.>

## Alternatives considered
- <Alternative 1>: <one-line why-not>
- <Alternative 2>: <one-line why-not>
(If the choice is forced ‚Äî single viable approach ‚Äî say so explicitly with a one-line justification rather than fabricating alternatives.)

## Key context
<related file:line>

## Checklist
- [ ] **Doc**: <doc update item 1>
- [ ] **Doc**: <doc update item 2>
- [ ] **Test**: <failing test item>
- [ ] **Code**: <implementation item>
- [ ] **Code**: <implementation item>

## Out of scope
<What this PR will NOT do>

## Risks
<Compatibility, performance, data, security>

## Open questions
<To be decided before proceeding>
```

Checklist order: **Doc first, Test next, Code last**. Granularity: one item ‚âà one commit ‚âà 30 min to 2 hours.

**Alternatives section is mandatory** ‚Äî even when the choice feels obvious. A "no alternatives ‚Äî forced choice" entry with a one-line justification is acceptable; an empty section is not. The section exists so reviewers see what was rejected and why, and so future-you can revisit the trade-off without re-running the analysis.

### 4.2 explorer

**When**: locating code, symbol lookups, "where is X defined", "who calls Y" ‚Äî read-only wide search. Protects the main context.

**Constraints**: no code edits. No dumping large files. Always summarize.

**Output**:
- Definition location (`path:line`)
- Up to 5 main references (`path:line` + one-line context)
- Related modules
- If nothing found: say "not found" and suggest next searches.

### 4.3 doc-writer

**When**: Phase A. Any change that alters external surface.

**Responsibilities**:
- Identify affected SSOT docs (MISSION, README, CLAUDE.md, ARCHITECTURE, ADR, API doc).
- Patch each doc preserving its existing format and tone.
- For absent docs, propose a minimal stub only after user confirmation.
- When an irreversible decision is detected, recommend writing an ADR (`docs/ADRs/NNNN-title.md`).

**Forbidden**: inventing target SSOT content (humans must review and sign off). Only stub proposals when absent.

### 4.4 test-writer

**When**: Phase B, right after doc.

**Responsibilities**:
- Take Phase A docs / spec as input; write a failing test.
- Reuse existing test patterns, tools, fixtures.
- Cover intended behavior + at least one boundary / exception / error path.
- Run it and confirm it **fails** (intentionally). If it passes, either Phase A is wrong or the test is too weak.

**Principles**:
- Suspect over-mocking.
- One test = one assertion.
- If tooling is absent (e.g. pytest not installed), report and stop. Don't guess.

### 4.5 code-reviewer

**When**: before commit / PR. Auto-invoked by `/review`, `/ship`.

**Premise**: assume no knowledge of the main assistant's discussion or reasoning. Input is:
- The full PR diff. Prefer `gh pr diff <num>` (or `--web` equivalent) over a literal `git diff main...HEAD` ‚Äî the PR's `baseRefName` may be non-`main` for topic-branch / experimental work (¬ß10.5), and `gh pr diff` honors that automatically. The principle is "review what the PR actually changes against its declared base," not "everything since main."
- Read-only adjacent context for changed files (for call-graph tracing, as needed)
- Target `MISSION.md`, referenced issue body (for MISSION fit + acceptance criteria evaluation)
- Full PR body (for declared plan + checklist verification)

Nothing outside this input.

**Check**:
- Consistency (coherent changes, missing adjacent callers)
- Tests (alignment with Phase B, regression risk)
- Error handling (at boundaries only, no defensive code creep)
- Security surface
- Obvious performance traps
- Readability, naming
- Scope (extraneous changes?)
- **Doc sync (Phase A reflected?)**
- **MISSION fit (contributing to or violating which item?)**
- **Issue acceptance criteria met?**

**Output**: ship / ship after fix / block. Each finding cites `path:line`.

### 4.6 security-reviewer

**When**: PRs touching auth / authz / sessions / external input / new deps / crypto / hashing / randomness / new IO boundaries.

**Check areas**: authentication / authorization, injection (SQL / shell / HTML / path), sensitive data exposure, weak crypto, CSRF / CORS / headers, dependency risk.

**Output**: High / Medium / Low / Info. Each finding: risk + exploit scenario + remediation.

### 4.7 issue-reviewer

**When**: invoked by `/file-issue` before `gh issue create`. Validates the rationale triad of SPEC ¬ß5.2 ‚Äî every issue passes through this gate.

**Input**:
- Proposed issue body (title + body + label).
- Target `MISSION.md` (if absent, label as `(MISSION.md absent ‚Äî review /onboard stub suggestion)`).
- Open-issues list (`gh issue list --state open`).

**Check**:
- **MISSION fit**: does the body cite a specific MISSION item? "It's general improvement" or "tidy" answers fail.
- **Why now**: does the body explain what changes if this waits? Vague "would be nice" fails.
- **Existing-coverage**: scan open-issues / open-PRs; does another issue cover this, or would a comment on an open PR suffice? If yes ‚Üí suggest the cheaper path.
- **Acceptance criteria**: explicit, testable, ‚â•1 item.

**Output**: one of three verdicts on stdout's last line, each with a one-line reason:
- `ship` ‚Äî file as-is.
- `refine: <one-line why>` ‚Äî return for body revision before filing.
- `block: <one-line why>` ‚Äî don't file at all (existing-coverage hit, scope problem, or non-MISSION work).

Inside `attended` mode the verdict surfaces to the user; inside `unattended` mode it gates the next step directly. Hook into `attended` is non-skippable ‚Äî Auto mode does not suppress this gate (SPEC ¬ß5.2).

### 4.8 plan-reviewer

**When**: invoked by `/work-on` between `planner` and the plan-approval step. Validates that the alternatives section is real and that the chosen approach actually beats them.

**Input**:
- `planner` output (markdown).
- Issue body (the one the planner read).
- Target `MISSION.md`.

**Check**:
- **Alternatives section is real**: ‚â•1 alternative with a substantive why-not, or an explicit "no alternatives ‚Äî forced choice" with a justification. "Just because" or empty fails.
- **Chosen approach beats alternatives**: read the "Plan" rationale; does it engage with the alternatives it lists, or just describe the chosen path?
- **Scope hygiene**: items in `Checklist` map to `Acceptance criteria` of the issue; anything orphaned belongs in `Out of scope` or a follow-up issue.
- **Phase-order sanity**: Doc ‚Üí Test ‚Üí Code as per SPEC ¬ß1.2; relaxation reasons noted if applicable.
- **Target base fit**: the `Target base` field (¬ß4.1) names a branch. If non-`main`, the plan should engage with the topic-branch's constraints (see ¬ß10.5). A `main`-default plan that quietly proposes work belonging on an experimental base ‚Äî or vice versa ‚Äî fails this check.

**Output**: same three verdicts as `issue-reviewer`. `refine` causes `/work-on` to re-invoke `planner` with the feedback; `block` stops with a comment on the issue naming the structural problem.

#### 4.9.1 activation-reviewer

**When**: invoked by `/file-directive` (proposed Directive body, ¬ß5.10), `/activate` (re-check on the possibly-edited body before Proposed‚ÜíActive, and on unblock, ¬ß5.12), `/revise-directive` (the new body before replacement, ¬ß5.16), and `/complete-directive` (completion claim + linked-Execution-Issue evidence, ¬ß5.13); and ‚Äî via the Initiative rulebook ‚Äî by `/consume-initiative` (contract-evaluability at Initiative intake, extraction-faithfulness on the proposed Directive set, ¬ß5.21). - **For completion review:** the Directive's body (success signals as written at activation), the list of linked Execution Issues (Issues with `Parent` = this Directive), and their close states / AC ticks.

**Checks**:

- **Schema completeness** ‚Äî does the body cover objective / success signals / non-goals / constraints / MISSION fit (or, for an Initiative-parented Directive, a `Parent Initiative: #N` marker)? Missing any ‚Üí `revise`.
- **Parent-kind alignment** (#254) ‚Äî a Directive has exactly one parent kind; judge alignment against it: a `## MISSION fit` parent against the named MISSION section, a line-1 `Parent Initiative: #N` parent against that Initiative's termination condition. - **Success-signal verifiability** ‚Äî are signals objectively testable, or vague aspiration? "Engineering reviews go faster" without a metric ‚Üí `revise`.
- **Scope clarity** ‚Äî is the objective bounded? "Improve the codebase" ‚Üí `reject`.
- **Non-goal clarity** ‚Äî are exclusions explicit or implied? Vague exclusions invite scope creep.
- **Active-Directive conflict** ‚Äî does the proposed Directive overlap with an existing `Status=Active` Directive? If yes ‚Üí `revise` or `reject` with a pointer.
- **Evidence sufficiency (completion only)** ‚Äî do the linked Execution Issues collectively satisfy each success signal? A signal with no linked Execution Issue ‚Üí `revise` ("need engineering evidence"). A signal whose Execution Issues are open or stalled ‚Üí `revise`.

**Output** ‚Äî three verdicts: `pass` / `revise` / `reject` (introduced by #172; these replaced `ship` / `refine` / `block` 1:1). On `reject` the verdict additionally carries the structured refile fields (`verdict` / `reason` / `refile-target-type` / `refile-target-parent` / `refile-body-draft`) and follows the type-mismatch / parent-mismatch matrices. The full output contract ‚Äî verdict meanings, the `revise`-vs-`reject` boundary rule, the structured fields, both matrices, the `<!-- activation-verdict: <verdict> -->` marker, and the N=3 `revise`‚Üí`reject` escalation ‚Äî is specified in `.claude/agents/activation-reviewer.md` (the contract SSOT). Callers apply the post-verdict handling per ¬ß2.1 (Directive lifecycle) and ¬ß5.12 (`/activate`).

**Rules**: do not author body content for a `revise` (matches ¬ß4.7 rule; the one exception is the `reject` `refile-body-draft` field ‚Äî drafting there IS the preserve signal). Verdict is content-only; filer identity affects only the caller's post-verdict handling. Judge from the inputs above only. Auto-approving without scanning active-Directive conflicts is a regression against the ¬ß1.5 reviewer-pattern integrity.

#### 4.9.3 Session-restart caveat for new reviewer subagents

Claude Code enumerates available `subagent_type` values **at session start** by scanning `.claude/agents/*.md`. A reviewer added mid-session is **not routable** as a first-class subagent until the next session start: `Agent({subagent_type: "<new-name>"})` falls back to `general-purpose`, which loses the Type-aware specialized prompt.

This caveat is the parallel of ¬ß6.1's "Session-restart caveat for hook-spec changes" ‚Äî Claude Code binds its routing surfaces (hook configuration AND subagent enumeration) at session start. Both bindings are stable for the session's duration; both are picked up automatically at the next session.

Empirical confirmation (Directive #62 / PR #67-followup): in the dogfood session, `.claude/agents/directive-reviewer.md` landed via PR #50 mid-session (this reviewer was later renamed to `activation-reviewer` by Issue #170 under Directive #167; the historical filename is retained here because it records the actual PR #50 event). Every subsequent `Agent({subagent_type: "directive-reviewer"})` invocation routed through `general-purpose` because the harness's `subagent_type` enumeration had been frozen against the pre-#50 state. A session restart resolves this automatically ‚Äî no `.claude/settings.json` registration needed, no harness-config change. The same mechanism applies to the Issue #170 rename itself: until a session restart, `subagent_type: "activation-reviewer"` falls back to `general-purpose`.

**Mitigation for the in-session window**: the falling-back-to-`general-purpose` path is functionally complete ‚Äî the agent's prompt instructs `general-purpose` to behave as the new reviewer. The cost is the loss of Type-aware specialization (the harness's per-subagent prompt + tool restriction). Acceptable for one session; restart at session end is canonical.

**Hook-side note**: the smoke `¬ß42` structural assertions (PR #50) lock the agent file's shape, so the routing-layer regression is the only failure mode this caveat names. The behavioral surface that catches the routing regression is **smoke ¬ß42e** (issue #69, under Directive #62; extended for the type-neutral reviewer by #170): three assertions gated behind `CLAUDE_ENG_BEHAVIORAL_SMOKE=1` invoke `activation-reviewer` with a synthetic minimal-but-valid Directive body (asserting `^VERDICT: pass`), a synthetic Directive body missing the `## Success signals` section (asserting `^VERDICT: (revise|reject)`), and a synthetic minimal-but-valid Execution Issue body exercising the Execution-rulebook dispatch (asserting `^VERDICT: pass`). ### 5.2 `/file-issue [--quick] <description>`

**Steps**:
- Confirm title, body, label with the user.
- Force MISSION reference and acceptance criteria in the body.
- **Rationale check** (before `gh issue create`): surface (a) which MISSION item this serves, (b) why now vs later, (c) whether an existing issue or an open-PR follow-up note would suffice. If any of (a)‚Äì(c) is weak, refine the issue or drop the request instead of filing. The check is mandatory ‚Äî Auto / unattended mode does not skip it; a poorly-justified issue propagates downstream into planner output and merged code, where the cost of redirection is highest.
- **Reviewer gate**: invoke `issue-reviewer` (¬ß4.7) on the proposed body before `gh issue create`. In `attended` mode the verdict surfaces to the user (who may override); in `unattended` mode the verdict gates the next step (`ship` ‚Üí proceed; `refine` ‚Üí re-author the body with the feedback and re-invoke; `block` ‚Üí stop and surface the reason). The reviewer fills the human-confirm slot when no human is present.
- `--quick`: one-line issue (label `bug`) for fast creation. Rationale check + reviewer gate still required, in compressed form (one-line "serves X mission item, blocks Y" suffices). The `bug` label matches the `bug-report.yml` Issue Form template.
- **Directive parenting** (added by tracking #41 child #6, issue #47): when the new Issue is parented to a `Status=Active` Directive (via `--parent <directive-id>` or interactive prompt listing active Directives, ¬ß1.7), prepend the `Parent Directive: #N` body marker (line 1); the Project `Parent` field is **derived** from that marker by `issues-to-project-mirror.yml` ‚Äî `/file-issue` writes the Issue only, never the Project directly (¬ß1.7 Issues-as-SSOT). `--no-parent` opts out. Rationale check + `issue-reviewer` gate apply unchanged ‚Äî parenting is metadata, not part of the rationale triad.
- **Full-symmetry stamp** (#172): `gh issue create` applies the `status:proposed` label so every new Execution Issue enters the lifecycle as Proposed and requires `/activate <N>` (¬ß5.12) before it is actionable. This is why `issue-reviewer` (author-side, here) and `activation-reviewer` (observer-side, at `/activate`) are complementary, not redundant (¬ß2.1). The output `Next:` line points at `/activate <N>`. `/work-on` against a `status:proposed` Issue is blocked by `proposed-protect` (¬ß6.1) until activation.
- Call `gh issue create`, return the number.

### 5.3 `/work-on <issue#> [--base <branch>]`

**Steps**:

1. `gh issue view <#> --json title,body,labels,comments` ‚Äî read the issue.
2. Validate body ‚Äî if acceptance criteria are missing or vague, ask the user to fix it. **Don't start work with an ambiguous goal.**
3. Resolve the **target base** (default `main`; override via `--base <branch>`). `git fetch origin && git checkout <base> && git pull --ff-only`. If `<base>` is not `main` and is not reachable locally or remotely, fail with a clear message ‚Äî `/work-on` does NOT auto-create alternate bases (see ¬ß10.5).
4. Create branch:
   ```
   USER=$(gh api user --jq .login)
   TYPE=<detected type>
   SLUG=<kebab of issue title>
   git checkout -b "${USER}/${TYPE}/${ISSUE}-${SLUG}"
   ```
   If the branch already exists on remote: `git pull --ff-only origin <branch>`.
5. Invoke `planner` agent (with issue body + MISSION + target CLAUDE.md context + the resolved `<base>`). The planner's output must include the mandatory `## Alternatives considered` AND `## Target base` sections (¬ß4.1). The `Target base` value must equal the `<base>` passed in.
6. **Reviewer gate**: invoke `plan-reviewer` (¬ß4.8) on the planner output. Parse the verdict line (`^VERDICT: (ship|refine|block)`). In `attended` mode the verdict surfaces to the user; in `unattended` mode it gates the next step (`ship` ‚Üí step 7; `refine` ‚Üí re-invoke `planner` with the feedback, then re-review; `block` ‚Üí stop with a comment on the issue naming the structural problem). The reviewer substitutes for the human reviewer when no human is present.
7. **Wait for user approval of the plan.** Approval requires an **approach check**: confirm with the user that the chosen approach beats the alternatives the planner surfaced, not just that the plan exists. If the alternatives section is weak (single option without a forced-choice justification), refine before approval. In `unattended` mode, a clean `plan-reviewer` verdict from step 6 counts as approval.
8. **Write and commit Phase A (Doc)** ‚Äî the first commit on the branch is real Doc work, never an empty seed. The branch graph begins with content the PR Plan describes, not a placeholder.
   ```
   # ‚Ä¶edit the SSOTs identified by the plan (MISSION, README, CLAUDE.md, ARCHITECTURE, ADR)‚Ä¶
   git add <changed SSOTs>
   git commit -m "<type>(#<#>): <subject>

   {Closes|Refs} #<#>"
   git push -u origin HEAD
   gh pr create --draft --base "<base>" --title "<type>(#<#>): <subject>" --body "<template>"
   ```
   First-line trailer: use `Closes #<#>` when `<base> == main`; otherwise use `Refs #<#>` (per ¬ß10.4 ‚Äî the issue closes only when work reaches `main`).

   If the planner's Phase A isn't ready to commit (e.g. blocked on clarification), defer the PR ‚Äî keep the issue + plan in conversation. **Never fall back to an empty seed commit just to open the draft PR.**
9. Insert plan + checklist + ship gate sections in the PR body.
10. Sync internal TaskList with the checklist.
11. Continue with the next phase (Test).

### 5.4 `/sync-pr`

Explicit entry point for updating the PR body. The main assistant does this automatically; this command lets the user trigger it on demand.

Refetch the remote body once before updating. The shell stores the SHA-256 of the last body it saw at `.claude/state/pr-cache/<owner>%2F<repo>__pr-<n>.json` (the `/` between owner and repo is URL-encoded as `%2F` so `<owner>/<repo>` slug boundaries survive the filename); the file holds at least `last_seen_body_sha256`, `last_synced_head`, and `last_synced_at`. If the remote body's SHA differs from `last_seen_body_sha256`, an external edit happened (reviewer, parallel session, GitHub UI) ‚Äî the automatic update aborts with a stderr message naming the conflict, and merging is the human's call. After a successful sync the cache is rewritten. A corrupt cache file (unparseable / missing key) also aborts the sync rather than silently treating as "first sync."

The cache lives under `.claude/state/`, which is gitignored ‚Äî it's per-clone state, not part of the repo's history. The helper `.claude/hooks/helpers/pr_cache.sh` exports `pr_cache_read`, `pr_cache_write`, and `pr_cache_check`. The repo slug is derived from `gh repo view` of the cwd, overridable via `$PR_CACHE_REPO` (used by the smoke test suite). `/ship` reuses the cache transitively by routing its body curation through `/sync-pr` (¬ß5.7 step 5). `/work-on` doesn't update the body after PR creation, so it doesn't touch the cache directly; later edits go through `/sync-pr`.

### 5.5 `/status`

`/status` and the `UserPromptSubmit` hook both delegate to the canonical helper at `.claude/hooks/helpers/status.sh`. Two output modes:

- `status_compact` ‚Äî multi-line plaintext block, the standard turn-context surface.
- `status_json` ‚Äî same fields as stable JSON for diff-comparison or piping.

Fields (both modes):
- Branch + dirty status
- Linked issue (#, title) ‚Äî extracted from `Closes #N` / `Refs #N` in the PR body
- PR (#, draft/ready, checklist progress `[x/y]`, CI status) ‚Äî `gh pr view`, `gh pr checks`
- Next unchecked checklist item
- Current phase (Doc / Test / Code / Review / Ship), inferred from the next unchecked item's section heading
- Active mode (attended / unattended) per ¬ß5.7.1

`UserPromptSubmit` (¬ß6.4) emits `status_compact` instead of a hand-rolled branch summary. Keeping the source single ensures `/status` and per-turn injection never drift.

The UserPromptSubmit hook reuses this output for the short per-turn summary.

**Cache**: status results are cached at `.claude/state/status-cache/<branch>.json` with a TTL of `STATUS_CACHE_TTL` seconds (default 5; env-overridable). The cache key is the branch name, so a `git checkout` to a different branch transparently loads a different cache file. Hits skip every `gh` round-trip (today: `gh pr view`, `gh issue view`, `gh pr checks` ‚Äî 3 calls per uncached invocation). The 5-second TTL is short enough that real PR/issue/CI changes surface within one user turn (turns typically exceed 5 s) and long enough to dedupe a flurry of prompts. Cache files live under `.claude/state/` (gitignored).

### 5.7 `/ship`

Runs the 8-step ship sequence from ¬ß2. `/ship` runs the full test suite (step 4) before transitioning to ready (step 8). Tests are `/ship`'s responsibility; the hook does not add a separate test gate on ready transition (¬ß6.1).

CI monitoring: snapshot `gh pr checks --json state` right before ready transition ‚Äî already failing ‚Üí block. Right after the transition, run `gh pr checks --watch` in background and return immediately. On failure, the PostToolUse hook notifies the user.

**Checklist audit (mandatory step 7.5, between CI snapshot and `gh pr ready`)**: scan the PR body for any `- [ ]` line. Each unchecked item must be resolved before transition ‚Äî ticked if done, marked `[~] N/A ‚Äî <reason>` if intentionally skipped, or removed if obsolete. The audit applies to every checklist in the body (Plan, Test plan, Docs touched, Ship gate). Per SPEC ¬ß1.4, stale uncheckedness at merge propagates as misinformation into future context.

When mode is `unattended` (see ¬ß5.7.1), `/ship` continues past step 8 into the CI-wait ‚Üí merge / park branch.

### 5.7.1 Operating modes (attended / unattended)

`/ship` has two terminal behaviors selected by **mode**:

- **`attended`** (default): `/ship` stops at "ready" and hands off to a human reviewer.
- **`unattended`**: after "ready", `/ship` waits on CI / required reviews and either merges or parks the PR for human attention.

**Resolution priority** (high ‚Üí low):

1. `--mode=<attended|unattended>` flag on `/ship`
2. `$CLAUDE_ENG_SHELL_MODE` env var
3. `.claude/state/mode` per-target file (one of `attended` / `unattended`, no other content)
4. default `attended`

Unknown values fail closed to `attended` and emit a stderr warning naming the offending source. `settings.json` is intentionally **not** a layer ‚Äî that file is symlinked across targets (¬ß3.2) and a mode key would leak between unrelated work.

**Blocker taxonomy** ‚Äî `ship_classify_blocker` returns one of:

| State | Criterion |
|-------|-----------|
| `clean` | All required CI checks green; required reviewers (if any) satisfied; branch protection rules satisfied. Eligible for auto-merge. |
| `soft` | CI is `pending` / `in_progress` / `queued`, **or** there is a CI failure that is attributable to the agent's own last commit and is fixable without new product decisions. One self-fix attempt is permitted (see below). |
| `hard` | Required-review with no available reviewer, undiagnosable / non-attributable CI failure, branch-protection rule the agent cannot satisfy, or a `soft` state that has already consumed its one self-fix attempt. |

**Soft-blocker behavior**: at most **one** self-fix-and-push attempt per `/ship` invocation. The retry pushes a `fix(#N): ...` commit and returns to the CI-wait state. A second `soft` outcome immediately escalates to `hard` ‚Äî no further fix attempts.

**Hard-blocker behavior** (parking) ‚Äî all four actions, in order, and idempotent:

1. Post a `gh pr comment` with a deterministic state summary (blocker reason + current SHA + CI snapshot).
2. Apply label `unattended-parked` (create the label on demand if missing). If the label is already present from a previous park on the same PR, `ship_park_pr` writes a single `park-suppressed: <reason>` line to the log instead of reposting the comment, and emits no comment to stdout. The original park comment remains canonical; the label remains applied.
3. Append one line to `${SHIP_PARK_LOG_PATH:-$CLAUDE_ENG_SHELL_ROOT/.claude/state/unattended-park.log}` (shell-side, gitignored ‚Äî not the target's MEMORY.md, not `audit.jsonl`). The override exists for smoke-test isolation. On suppressed repeat, only the `park-suppressed: <reason>` line is appended.
4. Stop. Do not toggle draft or self-assign.

**Classifier robustness**: `ship_classify_blocker` fails closed to `hard` in two failure modes and emits a one-line stderr warning naming the cause:

- empty stdin ‚Üí `ship_classify_blocker: no input on stdin ‚Äî defaulting to hard`
- missing `jq` dependency ‚Üí `ship_classify_blocker: jq not found ‚Äî defaulting to hard`

The intent is to make overnight runs debuggable without changing the safe fail-closed default. Either condition resolves the classification to `hard` and the dispatch path proceeds to park.

**Merge mechanism**: `gh pr merge --auto --merge --delete-branch` (no-fast-forward merge commit; PR branch's commits are preserved on `main`). This requires repo-level auto-merge to be enabled at GitHub. If the API call fails because auto-merge is disabled, fall back to the park path (log + label + comment) with reason `auto-merge-disabled`.

**Why `--merge` rather than `--squash`**: SPEC ¬ß1.2 makes the Doc ‚Üí Test ‚Üí Code commit graph load-bearing ‚Äî the arc is the artifact that proves the discipline happened. Squash erases the arc on merge (the PR branch is deleted; only a single new commit lands on `main`). `--merge` keeps both: the individual phase commits stay on `main`, plus a merge commit anchors the PR-as-a-unit so `git log --first-parent main` gives a clean PR-level view. The visible cost is a busier `git log`; the audit-trail philosophy in ¬ß6/¬ß7 trades that for the preserved arc.

**Composition with branch protection**: server-side rules in GitHub are always authoritative. `unattended` does not bypass them. A required-reviews rule with no available reviewer resolves to `hard` and parks ‚Äî the shell never self-approves or auto-requests review against policy.

**Composition with Auto mode**: Auto mode (single-conversation clarification suppression) is orthogonal to operating mode. Any combination is valid: `attended + Auto`, `unattended + non-Auto`, etc. Mode controls the `/ship` terminal behavior; Auto controls in-turn prompting.

Note: ¬ß6.1's "direct push to protected branch blocked" hook is **not** tripped by the merge step ‚Äî `gh pr merge --auto` is a GitHub API call, not a `git push` to `main`.

**Directive-aware audit** (added by tracking #41 child #6, issue #47): when the merged PR's linked Issue carries a `Parent Directive: #N` marker (¬ß5.2), `/ship` appends one audit line `directive-exec-count directive=#N pr=#M issue=#K` to `audit.jsonl`. The audit reads `closingIssuesReferences` ‚Äî so in topic-branch flow (¬ß10.5), sub-task PRs (which use `Refs #N`) emit no audit line, and the audit fires only on the **consolidator PR** that closes each parented Execution Issue at the moment work reaches `main`. Informational only ‚Äî `/ship` does **not** auto-mark the Directive `Completed`. Completion is `/complete-directive`'s job, gated by `activation-reviewer` (¬ß4.9). This preserves the ¬ß1.5 invariant: generation is open, decision is gated.

**Canonical hook-enforced path** (Directive #61): `.github/workflows/dir-mode-post-merge.yml` (installed from `.claude/templates/dir-mode-post-merge.yml`) fires on every `pull_request.closed && merged == true` event, regardless of who performed the merge (Claude-in-loop, human via UI, CI auto-merge). The Action emits an equivalent reflection comment on the parent Directive ‚Äî the comment surface IS the audit surface for hook-enforced events, since the workflow runner cannot write to a per-clone `audit.jsonl`. The Markdown procedure above stays valid for Claude-in-loop runs (defense in depth); the existing-URL idempotency check on the parent Directive's comments dedupes across both paths.

### 5.8 `/adr <title>`

**ADR vs PR body Decisions**:
- **ADR** ‚Äî irreversible, system-level. Migrations (DB engine, schema change), external contracts (API spec, wire format), default technology choices (auth scheme, queueing model). High cost to reverse once merged.
- **PR body Decisions** ‚Äî reversible, PR-scoped trade-offs. Default option choice, algorithm tuning parameters, equivalent-library preference, naming choices. Explainable within this PR and not affecting other PRs.

`/adr` creates `docs/ADRs/NNNN-title.md` (from `.claude/templates/adr.md`). A `‚Üí ADR-NNNN` link is auto-added to the PR body's Decisions section.

### 5.12 `/activate [<issue-#>]`

Promote a `Proposed` Issue (Directive **or** Execution Issue ‚Äî `/activate` is type-neutral) to `Active` by removing the `status:proposed` label after a fresh `activation-reviewer` (¬ß4.9) pass on the current body. This is the observer-side gate of the full-symmetry lifecycle (¬ß2.1): every Issue is filed `status:proposed` and becomes actionable only after `/activate` returns `pass`. `/activate` is also the **unblock** path for a Blocked Directive (¬ß5.17): a re-activatable Issue carries `status:proposed` (Proposed) **or** `status:blocked` (an Active-but-Blocked Directive); only when **neither** label is present is the Issue plain Active and `/activate` errors "already Active". On `pass` it removes whichever of `{status:proposed, status:blocked, awaiting-author}` are present (each removal presence-guarded ‚Äî `gh issue edit --remove-label` errors on an absent label). Batch mode scans `--label status:proposed`, so it does **not** sweep Blocked Directives ‚Äî unblock is single-mode `/activate <N>` only (auto-unblocking would silently reverse a deliberate maintainer block).

- **Single mode** `/activate <N>`: resolve `<N>`'s type label (tri-state, matching `activation-reviewer`'s ¬ß4.9 dispatch ‚Äî an `initiative`-labelled Issue is **refused**, not activated: Initiatives are consumed via `/consume-initiative` (¬ß5.21) and live outside the Proposed‚ÜíActive lifecycle (¬ß2.1); a bare `directive`-else-Execution split would otherwise mis-type one as Execution. The guard normally never fires because an Initiative never carries `status:proposed`), invoke `activation-reviewer` with the type context, parse the verdict, and apply it:
  - `pass` ‚Üí remove the present activation labels (`status:proposed` and/or `status:blocked`, plus `awaiting-author` if present) ‚Üí Active.
  - `revise` ‚Üí post the findings as a comment carrying `<!-- activation-verdict: revise -->`, retain `status:proposed`, add `awaiting-author`. After **N=3** `revise` markers on the Issue, escalate to `reject`.
  - `reject` ‚Üí post the verdict + structured fields with `<!-- activation-verdict: reject -->`, then apply **filer-aware handling** (verdict is content-only; handling keys on `authorAssociation` via `is_trusted_filer`): **trusted** ‚Üí keep open + `awaiting-author`, never close (composes with `trusted-filer-mutate`, ¬ß6.1); **untrusted + `refile-body-draft`** ‚Üí close (`--reason "not planned"`) + auto-create a `discussion`-tier Issue (¬ß5.19) preserving the draft + a lineage link; **untrusted, no draft** ‚Üí close + brief comment.
- **Batch mode** `/activate` (no arg): scan all open `status:proposed` Issues and process each. **Loop-safety:** skip an Issue whose latest `<!-- activation-verdict: <verdict> -->` comment marker post-dates its last body/label edit (verdict already delivered, awaiting the author) ‚Äî independent of the `awaiting-author` label. In `attended` mode `reject` surfaces to the operator; in `unattended` it auto-applies the filer matrix.
- **Reviewer invocation** is robust to the ¬ß4.9.3 session-restart caveat: the agent file is on disk and its prompt is self-describing, so a `general-purpose` fallback still behaves as the reviewer.

`/activate-directive <issue-#>` is retained as a **thin alias** delegating to `/activate <N>` for one release cycle (Directive-only ergonomics), then removed. Audit category: `activation` (generalized from `directive-activate`; the append-only log retains historical `directive-activate` entries). The mirror workflow fires on `issues.unlabeled` and updates the Project Item's Status. Escape: `SKIP_HOOKS=directive-review SKIP_REASON='<why>'`. Full procedure lives in `.claude/commands/activate.md`.

### 5.13 `/complete-directive <issue-#>`

Close a Directive Issue with `--reason completed`. `activation-reviewer` (¬ß4.9) evaluates evidence ‚Äî the linked Execution Issues' close states / AC ticks ‚Äî against the success signals. `reject` (insufficient evidence) leaves the Issue open. On `pass`, posts a closing comment listing each signal and its evidence, then closes the Issue. The `trusted-filer-mutate` hook matcher (SPEC ¬ß1.5, ¬ß6.1) allows `--reason completed` on trusted-filer Issues without further confirmation. Audit category: `directive-complete`. Escape: `SKIP_HOOKS=directive-review SKIP_REASON='<why>'` (¬ß7). The mirror workflow fires on `issues.closed` and updates the Project Item's Status. When the Directive's Execution Issues used the topic-branch flow (`/work-on --base <topic-branch>`, ¬ß10.5), `/complete-directive` must run **after** the topic branch consolidates to `main` ‚Äî the reviewer reads closed-Execution-Issue evidence, and still-open `Refs`'d Issues yield insufficient-evidence verdicts. Full procedure lives in `.claude/commands/complete-directive.md`.

### 5.15 `/reflect [<pr-#>]`

Post a structured reflection comment on a PR's parent Directive (when one exists) summarizing the PR's contribution toward that Directive's success signals. Run after merge; idempotent ‚Äî scans the Directive's existing comments for the merged PR's URL before posting and no-ops if found. Resolves the parent Directive from each closing Issue's body via the regex `^Parent Directive: #(\d+)$` (the marker `/file-issue --parent` writes on line 1, SPEC ¬ß5.2). Audit category: `directive-reflect` (decisions: `posted` / `skipped`). Read-only on the linked Execution Issues; writes only the new comment on the Directive Issue. The human-readable counterpart to `/ship` step 10.5's bare `directive-exec-count` audit (SPEC ¬ß5.7). Full procedure lives in `.claude/commands/reflect.md` (added by tracking #41 child #6, issue #47).

**Hook-enforced floor** (Directive #61): `.github/workflows/dir-mode-post-merge.yml` posts an equivalent reflection comment automatically on every merged PR with a `Parent Directive:` marker, so the reflection trail is complete regardless of who merged. The Markdown procedure above produces a richer per-signal evidence block when Claude runs it; the workflow's auto-generated stub satisfies the contract minimally. Both paths use the same existing-URL idempotency check on the parent Directive's comments.

### 5.16 `/revise-directive <issue-#>`

Replace an `Active` Directive Issue's body with a new one when scope or success signals changed, archiving the prior body as a comment for history. Procedure: (1) read the current body; (2) compute its sha256; (3) invoke `activation-reviewer` (¬ß4.9) on the **new** body ‚Äî the same proposal checks as `/file-directive` (schema / parent-kind alignment / success-signal verifiability / scope / non-goal / active-Directive conflict); (4) on `pass`, post a `## Pre-revision body ‚Äî archived <ISO date>` comment with the verbatim prior body, then replace the Issue body via `gh issue edit --body-file`; (5) emit `audit_log info directive-revise reconciled "directive=#<issue-#> from-sha=<prior-sha> archive-comment=<url>"`. **No Status change** ‚Äî the audit-log entry + the archive comment ARE the durable evidence of revision (no transient state needed ‚Äî see ¬ß2.1). On `revise` or `reject`, leave the body unchanged. Escape: `SKIP_HOOKS=directive-review SKIP_REASON='<why>'` (¬ß7). Full procedure lives in `.claude/commands/revise-directive.md`.

### 5.17 `/block-directive <issue-#> --reason <why>`

Mark an `Active` Directive as `Blocked` by adding the `status:blocked` label and a `## Blocked: <reason>` comment. Procedure: (1) require the `--reason <why>` flag ‚Äî empty or missing reason aborts; (2) post a `## Blocked: <reason>` comment on the Issue (idempotent ‚Äî skip if the same `## Blocked:` marker is the latest comment AND `status:blocked` label is already present); (3) add the `status:blocked` label via `gh issue edit --add-label`; (4) emit `audit_log info directive-block created "directive: #<issue-#> reason=<short> comment=<url>"`. The Issue stays **open** on GitHub ‚Äî closing the Directive Issue continues to require `/complete-directive` (¬ß5.13). Unblock by running `/activate <issue-#>` (the `/activate-directive` alias delegates to it): it treats `status:blocked` as a re-activatable precondition, re-invokes `activation-reviewer` on the current body, and on `pass` removes the `status:blocked` label (¬ß5.12). Not reviewer-gated ‚Äî blocking is an annotation, not a body change. Audit category: `directive-block`. The mirror workflow fires on `issues.labeled` and updates the Project Item's Status. Full procedure lives in `.claude/commands/block-directive.md`.

### 5.19 Discussion tier (Issue #112 substrate; Directive #109)

A distinct open-Issue tier alongside Directive / Execution / task / bug. **Purpose**: capture deliberately-ambiguous observations ("weird but not a bug", half-formed ideas, smells, RFC-style probes) without forcing premature classification. The tier exists because the other tiers each force a misclassification on such filings: `task` reads as actionable, `bug` claims a defect, and skipping filing loses the observation entirely. Discussion is the honest path. (Raw label-free filings are auto-stamped `status:proposed`+`task` by `auto-status-proposed.yml` (#179) and validated at `/activate`; deliberate ambiguity is filed `discussion` instead.)

**Labeling**: a single `discussion` label (color `#FEF2C0`, light yellow) installed by `scripts/ensure_v3_labels.sh` alongside the other dir-mode labels. Issue Form template at `.github/ISSUE_TEMPLATE/discussion.yml` requires only title + body ‚Äî no acceptance criteria, no MISSION fit, no rationale-triad prompt; the reduced-required shape is the whole point.

**Lifecycle**: develop in comments. Discussion Issues are NOT parented to Directives (no AC; nothing to roll up). The Issue's comment thread IS the development surface.

**Close paths** ‚Äî exactly two:

- **`promoted`** ‚Äî a concrete Issue (bug / task / Directive) was filed; the discussion closes referencing it (`gh issue close <N> --reason completed` plus a `^promoted to #<M>$` comment). The new Issue cross-references back.
- **`no-action`** ‚Äî the discussion concluded nothing needs doing (`gh issue close <N> --reason "not planned"` plus a one-line comment naming the reason). The `gh` CLI accepts only `{completed|not planned|duplicate}` ‚Äî the **space** form; `not_planned` (underscore) is rejected by `gh`, so the space form is canonical everywhere the value reaches `gh`.

No third path. Stale discussions surface via `/activate` batch mode (¬ß5.12; relocated from `/triage` by #173); no auto-close timer.

**Tier-relationships table**:

| Tier | Label(s) | AC-bearing? | Parented under Directive? | Close paths |
|---|---|---|---|---|
| Directive | `directive` + `status:*` | No (success signals instead) | No (root of the hierarchy) | `/complete-directive` (¬ß5.13) ‚Äî Completed via reviewer ship |
| Execution | (none, or template-applied) | Yes | Optional (`--parent`) | `Closes #N` via PR merge |
| task | `task` | Yes | No | `Closes #N` via PR merge or close-as-not-planned |
| bug | `bug` | Yes | No | `Closes #N` via PR merge or close-as-not-planned |
| **discussion** | `discussion` | **No** | **No** | **`promoted` or `no-action` only** |

**Lifecycle implementation** (Issue #116; Directive #109 final slice):

- `/discuss <title>` ‚Äî friction-free filing; bypasses the rationale triad + `issue-reviewer` gate. Full procedure in `.claude/commands/discuss.md`.
- `/resolve-discussion <N> [--promoted-to <M>] [--no-action <reason>]` ‚Äî closes via exactly one of the two paths above. Idempotent. Full procedure in `.claude/commands/resolve-discussion.md`.
- Close-path enforcement ‚Äî hook matcher arm under `trusted-filer-mutate` (¬ß6.1) blocks `gh issue close <N>` on a `discussion`-labeled Issue without `--reason completed` OR `--reason "not planned"`. The matcher tolerates both the gh-valid space form and the legacy underscore so it is never itself the blocker; the canonical emitted form is the space form. Discussion-tier classification is evaluated **first and independent of filer trust** (#236): once an Issue is recognized as `discussion`, the ¬ß5.19 two-path rule governs ‚Äî `--reason completed` OR `--reason "not planned"` is allowed regardless of the filer's `authorAssociation`, and a bare close or `--reason duplicate` is blocked. The trusted-filer block (below) applies only to **non-discussion** Issues. Stage 1 of the close-arm two-stage check (stage 2 = trusted-filer enforcement).
- `/activate` batch mode (¬ß5.12; relocated from `/triage` by #173) ‚Äî surfaces open `discussion`-labeled Issues older than 14 days as a maintainer-decision queue (promote / dismiss / let-incubate). Mode-independent: AI does not autonomously promote/dismiss even in unattended.
- Smoke ¬ß62 ‚Äî 8 assertions covering skill files, hook close-path enforcement (block + allow paths, the gh-valid `"not planned"` space form + legacy-underscore tolerance + a `duplicate` negative, #216), and `/triage` extension.

**Lockstep with Directive #107**: the `discussion` label is in `/onboard-dir-mode`'s bootstrap-label set (the dir-mode label set ‚Äî see the canonical count in ¬ß1.7's Tier 2 note). #112 (label substrate) merged first; #116 (lifecycle) merged second; #107's `/onboard-dir-mode` consumes both.

### 5.20 `/release <X.Y.Z> [--base <branch>] [--dry-run]`

Cut a versioned release by consolidating per-PR changelog fragments into a new section of `CHANGELOG.md` and opening a `release/X.Y.Z` draft PR. Implements the fragment-contract consumer side of SPEC ¬ß18.

Procedure (full body in `.claude/commands/release.md`):

1. **Parse args** ‚Äî required `<X.Y.Z>` (semver 0.x; reject `MAJOR > 0` with explicit `SPEC ¬ß3.5 / ¬ß18.2 MAJOR=0 invariant` message), optional `--base <branch>` (default `main`; for maintenance-line cuts per ¬ß18.3 / ¬ß10.5), optional `--dry-run` (helper stages but does not commit/push/branch).
2. **Preflight** ‚Äî assert clean working tree on the target base, `git fetch --tags origin`, refuse if `release/X.Y.Z` branch exists on origin OR `vX.Y.Z` tag already exists (exits 0 with `already released` message ‚Äî idempotent retry).
3. **VERSION write-back** ‚Äî strip any `-dev` suffix from the top-level `VERSION` and write `X.Y.Z`. Resolution-time queries fall back to `git describe --tags`; write-back is `VERSION`-only per ¬ß18.2.
4. **CHANGELOG consolidate** ‚Äî read every `changelog_unreleased/<category>/<N>.md`, validate (positive-integer filename stem; bullet contains `(#<N>)` matching stem), prepend a new `## [X.Y.Z] ‚Äî YYYY-MM-DD` section (UTC date) to `CHANGELOG.md` with `### <Category>` subheadings (Keep-a-Changelog), append `[X.Y.Z]: https://github.com/<owner>/<repo>/releases/tag/vX.Y.Z` reference link at the file footer.
5. **Fragment cleanup** ‚Äî `git rm` every consumed fragment file (leave the six `.gitkeep` placeholders intact).
6. **Branch + draft PR** ‚Äî create `release/X.Y.Z` (short-lived; covered by the existing `release/*` arm of the protected-branch matcher per ¬ß6.1). Commit the VERSION + CHANGELOG + fragment-removal as a single `chore(release): X.Y.Z` commit under the documented `SKIP_HOOKS=branch SKIP_REASON='/release initial release-branch commit'` escape (audit-logged; structurally required since the protected-branch matcher blocks all direct commits on `release/X.Y.Z` and this is the one purposeful commit). Push, open a draft PR titled `chore(release): X.Y.Z`.
7. **Reviewer gate** ‚Äî in `unattended` mode, re-run `code-reviewer` (¬ß4.5) against the release diff before `gh pr create` to catch malformed CHANGELOG concatenation / mismatched tag / VERSION drift. Per ¬ß1.5 reviewer-substitution. Escape: `SKIP_HOOKS=release-review SKIP_REASON='<why>'`.
8. **On-merge tag + Release** (documented post-merge step; **maintainer-run, not automated** per ¬ß18.5 non-goals ‚Äî this is the one step that sits *past* the `unattended` autonomy ceiling, which terminates at `gh pr merge`/park (¬ß5.7.1); the `unattended` `/release` continuation stops at the merged release PR and does **not** cut the tag/Release): after the release PR merges to the target base, the maintainer runs:
    ```bash
    MERGE_SHA=$(git rev-parse HEAD)  # on target base, post-merge
    awk '/^## \[X\.Y\.Z\]/{f=1; next} /^## \[/{f=0} f' CHANGELOG.md > /tmp/release-notes-vX.Y.Z.md
    gh release create vX.Y.Z --title vX.Y.Z --target "$MERGE_SHA" --notes-file /tmp/release-notes-vX.Y.Z.md --verify-tag
    ```
    Re-running on an already-released tag is a no-op with a clear message.

`/release` factors deterministic work into `scripts/release_consolidate.sh` so smoke can exercise it against throwaway repos without booting a Claude Code session (same precedent as `scripts/ac_closeout.sh` ‚Üî `/ship` step 7.6). Audit category: `release` (decisions: `created`, `idempotent`, `blocked`).

### 5.21 `/consume-initiative <N>` (Issue #257; Directive #256 / milestone v2)

The consume flow (SPEC ¬ß1.7 "Initiatives ‚Äî the planning tier the shell consumes"): turn an upstream `initiative`-labelled Issue into Initiative-parented Directive proposals, working backward from its termination condition. A **prompt-driven skill** (`.claude/commands/consume-initiative.md`), script-less like `/file-directive` ‚Äî the load-bearing work (judging the contract, decomposing the termination condition, authoring Directive bodies) is LLM reasoning, not mechanizable steps. Procedure:

1. **Type/substrate check** ‚Äî resolve `#N`; require the `initiative` label (else error). Same-repo only (cross-repo Initiatives are out of scope, ¬ß1.7).
2. **Contract gate** ‚Äî invoke `activation-reviewer` (Initiative rulebook, contract-evaluability I1, ¬ß4.9). If the termination condition is not evaluable without code knowledge ‚Üí the Initiative is under-specified: post a plain `gh issue comment <N>` surfacing it upstream and **STOP** (file no Directives). `gh issue comment` is the only Initiative write the flow performs ‚Äî allowed by `initiative-readonly` (¬ß6.1), which blocks only `edit`/`close`/`reopen`.
3. **Extract (working backward)** ‚Äî restate the termination condition, decompose it into the sub-outcomes that together satisfy it, and map each to one Directive proposal. The **necessary-and-sufficient** framing is the *faithfulness criterion* the I2 gate checks (necessary ‚Üí no Directive exceeds the condition; sufficient ‚Üí the set covers it), **not** a deterministic decomposition algorithm ‚Äî the partition is judgment. Each proposal carries a line-1 `Parent Initiative: #N` marker and **no `## MISSION fit` field** (the parent-XOR is satisfied by the marker; the Initiative traces to MISSION upstream).
4. **Faithfulness gate** ‚Äî invoke `activation-reviewer` (extraction-faithfulness I2, ¬ß4.9) over the **whole proposal set** (coverage is a set property, not per-Directive); on `revise`, refine and re-invoke (two-round backstop as `/file-directive`).
5. **File** ‚Äî `gh issue create --label directive --label status:proposed` for each proposal (body via `--body-file` so upstream-authored Initiative text is never interpolated into a `gh` argument position ‚Äî this covers the injection surface; content-trust is handled by the I1/I2 gates + human activation, not by `--body-file`). The extracted Directives are **never activated** ‚Äî activation stays human/reviewer-gated via `/activate` (generation-open / decision-gated, ¬ß5.7). At create time `label-parent-consistency` is not matched (it is scoped to `--add-label`), so the **I2 gate is the effective parent-XOR check** for the create path (it sees the full proposal bodies before filing).

Audit category: `initiative-consume` (decisions: `extracted` with count=N, `under-specified` (contract gate stopped the flow), `revise`). The command never edits/closes/relabels the Initiative (M1.2 `initiative-readonly`); never activates the extracted Directives; never files a proposal without the `Parent Initiative: #N` marker. Escape (reviewer gate): `SKIP_HOOKS=directive-review`.

### 5.22 `/initiative-feedback <N> {--challenge "<reason>" | --completion}` (Issue #260; Directive #259 / milestone v2)

The **upward-feedback** flow (SPEC ¬ß1.7 "Feedback is upward and non-deciding") ‚Äî the execution layer surfaces findings to an Initiative as structured comments, never as strategic decisions. One command, two modes (mirrors `/resolve-discussion`'s two-path shape). Both modes require the `initiative` label on `#N` (else error), operate same-repo, and write **only** a `gh issue comment --body-file` (the sole Initiative write ‚Äî allowed by `initiative-readonly`; `--body-file` for the injection surface). Each comment leads with a **scannable marker** (`## Initiative challenge` / `## Initiative completion`, modeled on `## AC closeout`) so an upstream consumer could parse it.

- **`--challenge "<reason>"`** ‚Äî when execution reality contradicts the Initiative (a termination condition that looked measurable but isn't; a real dependency / measurability / implementation cost the planning layer structurally cannot have), post a `## Initiative challenge` comment carrying that **code-derived** information and **requesting re-evaluation**. It **escalates** ‚Äî it does NOT reject, retire, or close the Initiative (mirrors a reviewer `block` escalating rather than auto-discarding, ¬ß4.9.1). The decision stays upstream; the shell does not track what happens next.
- **`--completion`** ‚Äî when the Directives extracted from the Initiative (`/consume-initiative`, ¬ß5.21) have all landed, post a `## Initiative completion` comment carrying the **evidence** (the merged Directives / PRs) and **requesting termination assessment**. It reports the fact and requests assessment; it does NOT assert the termination condition is met and does NOT close the Initiative ‚Äî asserting completion is a strategic judgment that stays upstream (the same logic that forbids rejection).

Audit category: `initiative-feedback` (decisions: `challenge`, `completion`). The command never edits, closes, reopens, relabels, or retires the Initiative (M1.2 `initiative-readonly` is the runtime backstop); the only write is the structured comment. With ¬ß5.22 the milestone-v2 model is **fully live** ‚Äî recognition (M1.1) + enforcement (M1.2) + reviewer (M2) + consume (M3) + feedback (M4).

## 6. Hook enforcement

### 6.1 PreToolUse ‚Äî block

**Framing**: Hooks are a **mistake-prevention guardrail**, not a security boundary. An adversarial user can bypass them via shell features the matcher doesn't model (`eval`, `bash -c`, `python -c`, heredoc-spawned shells, dynamic command construction via parameter expansion). The hook's job is to stop honest mistakes ‚Äî typos, autopilot moments, the wrong tab ‚Äî not to be a sandbox. Patterns that suggest a bypass attempt are not blocked but are **audit-logged as warnings** so the trail is preserved for review.

Implementation note: command strings are **normalized** before matching ‚Äî backslash-continued multiline (`\\\n`) collapses to single-space, internal newlines become spaces, multi-space whitespace collapses. So `git \\\n  push \\\n  --force-with-lease` matches the same blocked pattern as the single-line form.

**Git option-prefix tolerance**: every `git <subcommand>` matcher accepts standard git-level options between `git` and the subcommand: `-c <key>=<value>`, `-C <path>`, `--git-dir=<path>`, `--work-tree=<path>`, `--no-pager`, `--literal-pathspecs`, `--exec-path[=<path>]`, `--namespace=<ref>`. So `git -c commit.gpgsign=false commit -m "‚Ä¶"` and `git -C /some/repo push --force` both fire the same downstream checks as the bare `git commit` / `git push` forms ‚Äî the option prefix can no longer be used (intentionally or not) to slip past the matcher.

**Type-awareness** (introduced alongside dir-mode, ¬ß1.7; matchers wired by tracking #41 child #5, issue #46): the engineering-flow matchers below ‚Äî branch-from-issue, AC-closeout, PR-link / `Closes #N` checks, default `gh issue list` wrappers ‚Äî **do not apply their engineering-flow gate to a `Type=Directive` Issue**. Directive Issues are gated by `/complete-directive` + `activation-reviewer` (¬ß4.9), not by AC checkboxes or `Closes #N` semantics. (For **AC-closeout** specifically the carve-out is **all-or-nothing**, not per-item: the `gh pr merge` gate is skipped only when **every** closing Issue is a Directive ‚Äî a mixed PR closing both a Directive and an Execution Issue still gates on the Execution Issue's unchecked AC; see the AC-closeout matcher row below.) The `proposed-protect` matcher in the table is the inverse: it actively blocks engineering-flow actions (most prominently `/work-on`) when the target Issue is **either** `status:proposed` (any type ‚Äî not-yet-activated work, generalized by #171) **or** a Directive (any status ‚Äî Directives never branch, ¬ß10.5). The two conditions guard distinct invariants (lifecycle vs type) that share one branch-creation trigger; `proposed-protect` subsumes the former `directive-protect` rather than replacing it (a status-only check would let an Active Directive become branchable ‚Äî a ¬ß10.5 regression). The implementation predicates (`is_directive_issue <issue-#>`, cached; and `is_proposed_issue <issue-#>`, uncached because the `status:proposed` label is volatile under `/activate`) and the `safe_source` fail-policy entry live in `helpers/issue_type.sh` (#46, #171).

**Session-restart caveat for hook-spec changes.** Claude Code binds its hook routing from `.claude/settings.json` + the working tree at session start. Adding a *new* helper file under `.claude/hooks/helpers/` (or any path referenced by an existing hook via `.`/`source`) does not retroactively become reachable to a session that started before the file existed. Such a session continues to execute the *updated* `pre_tool_use.sh` (re-read per invocation), but a `. "$SHELL_ROOT/.claude/hooks/helpers/<new-helper>.sh"` line silently fails ‚Äî and any function defined only in that helper is undefined, so downstream calls return rc=127 and matchers can fall through without ever reaching their block / warn arms. **The fix is to start a fresh `claude-eng` session after pulling a hook-spec change that introduces a new helper file.** The hook-side mitigation is the **`safe_source`** utility (hosted in `.claude/hooks/hookrt.sh` ‚Äî the "hook runtime" bootstrap that also hosts `audit_log`): every helper source line ‚Äî both **hook-to-helper** (the 5 hook files under `.claude/hooks/` sourcing helpers) and **helper-to-helper** (a helper under `.claude/hooks/helpers/` sourcing another helper from the same directory, e.g. `branch_guard.sh` ‚Üí `git_matcher.sh`) ‚Äî goes through `safe_source <helper-path> <audit-category>`, which presence-checks the file, sources it, and on miss emits `audit_log warn <category> helper-missing "<path>"` and returns 1. The one documented exception is the **bootstrap-to-runtime shim** at `helpers/log.sh:6` ‚Üí `hookrt.sh`: that source line can't go through `safe_source` because `safe_source` lives in the file being sourced (chicken-and-egg), so it falls back to a primitive `.` and is stderr-only on miss. The scope predicate for "in scope of the `safe_source` invariant" is: the source target is under `.claude/hooks/helpers/`, NOT directly under `.claude/hooks/` (i.e. `hookrt.sh` and the 5 hook entry-point files are exempt ‚Äî they are the runtime, not helpers). The shape this closes ‚Äî one site warning while a sibling source-site of the same helper fails silently, hiding the second consumer ‚Äî was the regression #36 surfaced. The hook continues to **fail-open** (matcher arm short-circuits, command proceeds) so a missing helper is recoverable, not a developer-lockout ‚Äî but the failure mode is visible in `audit.jsonl` instead of silent. Existing helpers that the running session already loaded (e.g. `helpers/coauthor.sh`) are unaffected ‚Äî their function bindings persist across invocations of the parent hook. Cross-reference ¬ß3.2.1 (hook command path resolution).

**`helper-missing` audit-event shape** ‚Äî the on-miss line emitted by `safe_source` is:

```json
{"ts":"<RFC3339>","event":"warn","category":"<from-table>","decision":"helper-missing","reason":"<absolute-path-to-missing-helper>","cwd":"<pwd>"}
```

`reason` and `cwd` are JSON-escaped per SPEC ¬ß7. `decision` carries the constant token `helper-missing`. **`helper-missing` audit categories are a superset of `SKIP_HOOKS` categories** ‚Äî for example, `status` is a valid audit category for an informational helper but is NOT a `SKIP_HOOKS` category. The escape-hatch list in ¬ß7 stays unchanged; the audit categories enumerate what `safe_source` can emit, which is broader.

**Matcher pass-through audit contract.** PR #32 surfaced a class of bug where a matcher entered (its `grep -qE 'pattern'` matched) but no terminal arm fired ‚Äî the matcher silently exited with no audit record. This made "matcher entered but produced no audit" indistinguishable from "matcher never matched the command" ‚Äî a structural blind spot in the audit trail. The contract closes that gap:

> **For every matcher in `pre_tool_use.sh`, if the matcher entered (`grep` matched), `audit.jsonl` MUST gain exactly one record whose `category` field matches the matcher's category in this hook invocation.**

The terminal arm fires that record (or marks it as decided without emission):
- A `block` decision emits `audit_log block <cat> deny <reason>`.
- A `warn` decision emits `audit_log warn <cat> <decision-token> <reason>` (`decision-token` ‚àà {`notice`, `helper-missing`, `pass-through`, ...}). Note `bypass-suspect` is a *category* (with decision-token `notice`), not a decision-token.
- A **high-frequency happy-path** arm (e.g., every clean `git commit`, every in-scope `rm -rf`, every local `--amend`) calls `mark_allow <cat>` (in `hookrt.sh`). This sets the caller's `decided` flag with NO audit emission ‚Äî the matcher's contract is satisfied (it entered, evaluated cleanly, allowed) without per-action audit noise. Distinct from the explicit-`notice` warn path, which is used for low-frequency designed-allow cases where an audit record is desired.
- If no other arm fires by the matcher's tail, the helper `pass_through_trace <cat> "<cmd>"` (also in `hookrt.sh`) emits `audit_log warn <cat> pass-through "<truncated-cmd>"`. This is the anomalous case ‚Äî the matcher decided to do nothing without naming why, which we treat as a soft regression worth flagging.

Implementation pattern (per Bash-branch matcher in `pre_tool_use.sh`):

```bash
if printf '%s' "$cmd" | grep -qE 'pattern'; then
  decided=
  # ... terminal arms set `decided=1` ...
  [ -z "$decided" ] && pass_through_trace <cat> "$cmd"
fi
```

The `decided=` flag is reset at the head of each matcher block so state cannot leak across matchers. Smoke (`scripts/test/smoke.sh` ¬ß39) asserts the invariant:
- **¬ß39a**: for each matcher, a synthesized benign-but-entering command produces exactly one new audit record with the expected category.
- **¬ß39b**: structural awk-grep over `pre_tool_use.sh` ‚Äî every `if printf '%s' "$cmd" | grep -qE ...; then` block ends with either an explicit `decided=1` audit-emitter or a `pass_through_trace` tail.
- **¬ß39c**: negative ‚Äî a command that enters no matcher (e.g. `ls -la`) produces zero new audit records.

**Decision-token vocabulary** (`decision` field in `audit_log`):

| Token | Used by | Meaning |
|---|---|---|
| `deny` | `block` | Hook refused the command; `exit 2` follows. |
| `notice` | `audit_log warn` | Soft signal ‚Äî degraded enforcement (gh timeout, indeterminate state, explicit allow with reason). |
| `helper-missing` | `safe_source` on miss | A hook helper was absent; matcher arm short-circuited. |
| `bypass-suspect` | bypass-suspect matcher | Command looks like an eval/heredoc bypass (warn-only, not a block). |
| `pass-through` | `pass_through_trace` | Matcher entered but no terminal arm fired ‚Äî anomalous; treat as a soft regression. |
| `skip` | `audit_log escape` | `SKIP_HOOKS=` was honored; categorized as `escape`. |

The token list is intentionally short and stable; future arms add new tokens here, not new `event`-type kinds.

Tests are run by `/ship` (¬ß5.7). During the draft PR's commit/push, only quick lint/format runs. A commit-time test gate would force WIP/share commits to escape every time, polluting the audit log ‚Äî so we don't add one.

**Lint timeout**: the commit-time lint invocation is bounded by `CLAUDE_ENG_LINT_TIMEOUT` (seconds, default 30) so a misconfigured or slow linter cannot hang the commit. Timeout fires as a `block format` decision, escapable via `SKIP_HOOKS=format`. If `timeout(1)` (coreutils) is not on `PATH`, the lint runs unbounded with an `audit_log warn` entry noting the missing dependency ‚Äî the hook neither hangs the user nor silently downgrades enforcement.

#### 6.1.1 Conditional issue # enforcement

| commit type | issue # required | Rationale |
|-------------|------------------|-----------|
| `feat`, `fix`, `docs`, `refactor`, `perf` | Required | External surface / behavior change |
| `test`, `style`, `build`, `ci`, `chore`, `revert` | Optional (passes without it) | Internal change |

Structural regex:
```
# required group
^(feat|fix|docs|refactor|perf)(\(#[0-9]+\))!?: .+$
# optional group (issue # optional)
^(test|style|build|ci|chore|revert)(\(#[0-9]+\))?!?: .+$
```

Subject length is measured in **codepoints (1..72)**. Byte-based measurement (`wc -c`, `${#var}`) is forbidden ‚Äî under multibyte locales it mis-truncates Korean/CJK. The choice of tool is an implementation detail.

**Subject extraction across `-m` forms**: the hook accepts both the plain quoted form and the command-substitution + heredoc form that the harness uses for multi-line bodies (`Co-Authored-By`, etc.).

| `-m` form | Subject |
|-----------|---------|
| `-m "feat(#1): subject"` / `-m 'feat(#1): subject'` | The quoted text |
| `-m "$(cat <<'TAG'` ‚Ä¶ `TAG)"` (heredoc) | The first non-blank line of the heredoc body |

The heredoc tag may be single-quoted (`<<'EOF'`), unquoted (`<<EOF`), or indent-stripping (`<<-EOF`). The extractor walks the raw command (before whitespace normalization), locates the heredoc opener, and returns the first line of body content that is not blank and not the closing tag. Leading whitespace on that line is stripped before the Conventional-Commits regex runs.

Implementations that need both the heredoc and plain extraction should expose a single helper (`extract_commit_subject`) consumed by `pre_tool_use.sh`'s commit gate.

### 6.5 SessionStart ‚Äî shell sync check + session restore + SessionStart banners

At session start, do three things:

**(a) Shell sync check**
- Is the shell repo behind origin/main? If so, warn (no auto-pull; sudden behavior changes are risky).
- The `git fetch` is gated by a stamp file at `.claude/state/last-shell-fetched`. The fetch runs only when the stamp is older than `SESSION_START_FETCH_TTL` seconds (default 21600 = 6 h) or absent. Most sessions therefore skip the network call ‚Äî the `git rev-list --count HEAD..@{u}` still runs against locally-known refs, so the "behind" warning is correct relative to whatever was last fetched (up to 6 h stale, which matches the warning's polite-not-critical role).
- The fetch is wrapped in `timeout $SESSION_START_FETCH_TIMEOUT` (default 5 s) so a hung network does not stall session startup. If `timeout(1)` / `gtimeout(1)` is unavailable the fetch runs unbounded, matching SPEC ¬ß6.1 / `CLAUDE_ENG_LINT_TIMEOUT`'s fallback pattern.

**(b) Session restore** ‚Äî to seamlessly resume the previous session, inject once:
- The current branch's full PR body (if any)
- Full referenced issue body (if any)
- Target `MISSION.md` core summary (3‚Äì5 lines)
- Target `CLAUDE.md` directory index + domain vocabulary summary
- One `/status` snapshot

Subsequent turns only refresh the short status summary from ¬ß6.4.

**(c) SessionStart banners**

Two silent-no-op states evaporate the ¬ß0.1 enforcement guarantee with no operator signal. SessionStart surfaces both via primitive inline `printf` stderr banners, each debounced to one fire per session and each independent of `hookrt.sh` / `audit_log` / `safe_source` (the runtime that may itself be the missing piece).

**Inject-consistency banner** ‚Äî a user who enters a workspace via plain `claude` (not `claude-eng`) gets every hook silently `exit 0` because `$CLAUDE_ENG_SHELL_ROOT` is unset.

- **Trigger**: `$CLAUDE_ENG_SHELL_ROOT` is empty **AND** `$PWD/.claude/settings.local.json` is a symlink (the canonical "shell was injected" marker per `scripts/lib/inject.sh`).
- **Action**: emit one stderr line naming the two fixes ‚Äî `invoke 'claude-eng' instead of 'claude'`, or `export CLAUDE_ENG_SHELL_ROOT=<shell repo path>`. Non-blocking; the rest of SessionStart still runs (which means ¬ß(a) and ¬ß(b) skip ‚Äî there's nothing for them to do without the env var ‚Äî but the banner makes the silence visible).
- **Idempotency**: a stamp file under `$TMPDIR/claude-eng-banner.${CLAUDE_SESSION_ID:-$PPID}` debounces to one banner per session. `$CLAUDE_SESSION_ID` is set by Claude Code; `$PPID` is the same-session fallback. A missed banner (PID reuse across sessions) costs at most one undetected mis-invocation, strictly better than the prior silent-no-op state.
- **No false positives on non-injected dirs**: a workspace that never had `inject_into` applied has no symlink and the banner stays quiet. A real-file `settings.local.json` (the inject-skipped case at `inject.sh:14`) also stays quiet.
- **Position**: runs **before** the `$CLAUDE_ENG_SHELL_ROOT` guard at the top of `session_start.sh`, since the whole point is to surface a state where that var is empty.

**Hookrt-missing banner** (#37) ‚Äî `$CLAUDE_ENG_SHELL_ROOT` is set, but `.claude/hooks/hookrt.sh` is absent (corrupt checkout, conflict-stashed file, manual `rm`). Each of the 5 hooks has a primitive check for the runtime and emits a per-invocation WARN to stderr before silently exiting 0; that per-invocation line trains operators to ignore "WARN hookrt-missing" as background noise, exactly when they shouldn't. The banner is the once-per-session actionable signal layered on top of that per-invocation diagnostic floor.

- **Trigger**: `$CLAUDE_ENG_SHELL_ROOT` is non-empty **AND** `$CLAUDE_ENG_SHELL_ROOT/.claude/hooks/hookrt.sh` is absent.
- **Action**: emit one stderr line naming the failure mode (`hook enforcement OFF until restored`) and the concrete fix ‚Äî inspect tree state via `git -C $CLAUDE_ENG_SHELL_ROOT status`, then restore via `git -C $CLAUDE_ENG_SHELL_ROOT checkout -- .claude/hooks/hookrt.sh`. Non-blocking; SessionStart still exits 0 immediately after (no helpers can load without the runtime).
- **Idempotency**: a stamp file under `$TMPDIR/claude-eng-banner-hookrt.${CLAUDE_SESSION_ID:-$PPID}` debounces to one banner per session. Distinct suffix (`-hookrt`) prevents collision with the inject-consistency stamp; the two banners' triggers are mutually exclusive (one requires `SHELL_ROOT` unset, the other set), but distinct stamps make the independence explicit.
- **Diagnostic floor**: the existing per-invocation WARN at `session_start.sh:34` (and equivalent at the other 4 hooks' bootstrap blocks) stays ‚Äî it is the script-grep-friendly log-scraper-stable diagnostic that survives the banner's stamp-creation failure modes (hostile `$TMPDIR`, low-disk). The banner adds operator-facing actionability without removing the diagnostic floor.
- **Position**: runs **after** the `$CLAUDE_ENG_SHELL_ROOT` resolution (the trigger needs the env var) and **before** the `. "$hookrt"` source line at `session_start.sh:38` ‚Äî co-located inside the existing `if [ ! -f "$hookrt" ]` block, since it shares the trigger condition and immediate `exit 0` tail.

## 7. Escape hatch

Two forms, chosen by where the escape needs to be read:

**Trailing sentinel** ‚Äî the form that survives the **live Claude Code Bash tool** (use this in-harness):
```bash
<command>  # claude-eng:skip=<category>[,<category>...] reason=<reason>
```

**Leading env-prefix** ‚Äî works only where the prefix arrives *inside* the command string (a real shell that passes it verbatim, and the smoke harness):
```bash
SKIP_HOOKS=<category>[,<category>...] SKIP_REASON='<reason>' <command>
```

The PreToolUse hook parses the leading `SKIP_HOOKS=‚Ä¶`/`SKIP_REASON=‚Ä¶` tokens from the leading edge of `tool_input.command`, **and** a `# claude-eng:skip=‚Ä¶ reason=‚Ä¶` sentinel from the trailing edge of the raw command. Two forms exist because the leading env-prefix does **not** reliably reach the hook through every execution environment: the live Claude Code Bash tool consumes a leading `VAR=val` as the spawned subprocess's own environment, so it never lands in `tool_input.command` and the hook cannot see it (the hook subprocess does not inherit it either ‚Äî ¬ß6.1). A trailing `#`-comment, by contrast, stays inside `tool_input.command` and is ignored by the executed shell, so it is the reliable in-harness escape. Both forms export the recognized tokens into the hook before downstream matchers run, then strip them from the command for the matcher pass (so a `reason=` containing a blockable substring cannot bleed into another matcher); **if both are present, the leading form wins.** Recognized tokens: `SKIP_HOOKS`/`SKIP_REASON` (leading) and `claude-eng:skip=`/`reason=` (trailing) only ‚Äî any other ALLCAPS `K=V` leading token (`PATH=`, `LD_PRELOAD=`, ‚Ä¶) halts the leading scan and is left as an ordinary argument. The sentinel is parsed from the **raw** command before whitespace-normalization (the normalization + `shlex` pass would otherwise mangle the `#`), is namespaced (`claude-eng:skip=`) so an ordinary trailing comment is never mistaken for an escape, and is **one-shot** ‚Äî it travels with the single command, so there is no persistent bypass state. The sentinel is honored **only when its `#` is a genuine unquoted shell comment token** ‚Äî a `#` that the executed shell itself treats as the start of a comment. A `#` sitting inside a quoted argument (e.g. `gh pr comment 5 --body "see note # claude-eng:skip=all reason=x"`) is argument text, not a comment: the shell runs the whole command intact, so the hook must **not** read it as an escape (doing so would let ordinary quoted text ‚Äî a commit message, a PR-body paste, audit output quoting the sentinel ‚Äî silently disarm every matcher with a falsified audit reason). The comment-token boundary is resolved with the same `python3`/`shlex` machinery the leading env-prefix uses; if `python3` is absent the trailing sentinel is **not** honored ‚Äî a fail-safe no-op (enforcement stays armed), never a spurious skip. The sentinel must sit on the command's **final line** (the match is confined to a single line via `[[:blank:]]`/non-control-char classes plus a newline guard) so a line-1 sentinel can never strip or disarm a dangerous *later* line ‚Äî such a command falls through to the matchers and blocks. **Scope:** like the leading env-prefix, the trailing sentinel is **command-scoped** ‚Äî it disarms its named category (or `all`) across **every** line/clause of the command, not only the line it sits on. It remains **category-scoped**: a tail sentinel naming category X does not disarm a category-Y matcher (that command still blocks). So an `escape` audit record proves *which* category was skipped with a stated reason ‚Äî read it as "category X was skipped for this command," not "this exact line was approved"; the reason text need not describe the specific operation disarmed. When run in a real terminal where neither form reaches a hook (there is no PreToolUse hook outside the harness), the command simply runs; the escape only matters when a hook is in the path.

Categories:
- `all`
- **eng-flow hook matchers:** `secret`, `branch`, `commit-format`, `format`, `force-push`, `amend`, `no-verify`, `destructive`, `sensitive`, `out-of-scope`, `backmerge`, `ac-closeout`
- **type-aware / dir-mode hook matchers (¬ß6.1):** `proposed-protect`, `trusted-filer-mutate`, `label-parent-consistency`, `initiative-readonly`
- **command-level reviewer-gate escapes:** `directive-review` (the `activation-reviewer` gate on `/file-directive` / `/activate` / `/revise-directive` / `/complete-directive` / `/consume-initiative`, ¬ß4.9/¬ß5.10‚Äì¬ß5.22), `release-review` (the `code-reviewer` gate on `/release`, ¬ß5.20)

This list is the complete set of `SKIP_HOOKS` categories the runtime honors (the hook matchers via `should_skip <cat>` in `pre_tool_use.sh`; the reviewer-gate escapes via the command/skill prose). It is kept in sync with the matchers/commands by the doc-consistency smoke check (¬ß17 / #267) ‚Äî a new `should_skip <cat>` or reviewer-gate escape that is not listed here fails smoke.

(`issue-ref` is folded into `commit-format` ‚Äî not a separate category. `format` is the commit-time lint/format gate. `out-of-scope` covers paths outside the registry (¬ß6.1). `backmerge` covers `git merge <protected-base>` on a feature branch (¬ß6.1). `ac-closeout` covers `gh pr merge` when a linked issue has unchecked AC and no `^## AC closeout` comment ‚Äî see SPEC ¬ß6.1 row + ¬ß5.7.1 step 7.6 for the rule (step 7.5 is the unrelated checklist audit); `scripts/ac_closeout.sh <pr-num>` is the canonical fixer. Tests are run by `/ship`, not a hook, so there's no `tests` category.)

Every skip is recorded in `.claude/audit/audit.jsonl`:
```json
{"ts":"2026-05-19T...","event":"escape","decision":"skip","category":"format","reason":"flaky lint JIRA-1234","cwd":"..."}
```

The `reason` and `cwd` fields are JSON-encoded via `jq -Rs .` (or an inline `\n`/`\r`/`\t`/`\\`/`\"` escape when jq is absent). Newlines and control characters in `SKIP_REASON`, and shell metacharacters in PWD (legal in POSIX paths), therefore do not break the "one JSON object per line" invariant ‚Äî every `audit.jsonl` is parseable by `jq -c '.'` regardless of input contents. Other fields (`event`, `category`, `decision`, `ts`) are internal-constant or RFC 3339-shaped and need no encoding.

Policy:
- Skips should be temporary.
- A skip with no `SKIP_REASON` is logged as `unspecified` ‚Äî review target.
- `all` is for emergencies.
- A category that gets skipped repeatedly means the hook is misconfigured ‚Äî open a fix PR.

Audit logs rotate as appropriate (exact threshold is an implementation choice), but the diff body is never written ‚Äî only path, line, category metadata.

## 10. Convention summary

### 10.1 Branch naming

```
<gh-username>/<type>/[<issue#>-]<slug>

# required-issue-# group (feat/fix/docs/refactor/perf) ‚Äî always includes <issue#>-
ilgyu/feat/42-oidc-login
ilgyu/fix/57-null-pointer
ilgyu/docs/89-update-architecture

# optional group (chore/style/build/ci/test/revert) ‚Äî issue # may be omitted
ilgyu/chore/typo-readme
ilgyu/chore/89-update-deps    ‚Üê if an issue exists, you can include it
```

Types: `feat | fix | docs | style | refactor | perf | test | build | ci | chore | revert`

`/work-on <issue#>` always has an issue, so it always includes `<issue#>-` regardless of group. Issueless ad-hoc branches in the optional group are created by the user with plain `git checkout -b`.

### 10.2 Commit messages

```
<type>(#<issue>)[!]: <subject ‚â§ 72 chars>

<body ‚Äî one blank line>

Closes #<N>     ‚Üê last commit, or single-PR consolidator
Refs #<N>       ‚Üê intermediate commit
Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>   ‚Üê optional trailer; toggleable
```

Phase prefixes:
- `docs(#42): describe OIDC callback flow`
- `test(#42): add failing tests for /auth/oidc/callback`
- `feat(#42): implement OIDC callback handler`

**Co-Authored-By trailer toggle**: the `Co-Authored-By: Claude ‚Ä¶` line is a convention, not a contract. It is included by default but can be disabled per-target. Resolution priority (high ‚Üí low, mirroring SPEC ¬ß5.7.1's mode pattern):

1. `$CLAUDE_ENG_COAUTHOR` env var (`on` / `off`).
2. `.claude/state/coauthor` per-target file (one of `on` / `off`, no other content).
3. Default `on`.

Unknown values fail-safe to `on` and emit a stderr warning naming the offending source. The helper `.claude/hooks/helpers/coauthor.sh` exposes `coauthor_trailer` (prints the line when enabled, empty when disabled); `/work-on`'s commit template sources it. The toggle covers the default behavior of `/work-on` flows only ‚Äî the trailer remains free to be added or omitted manually in any specific commit.

**Relationship with Conventional Commits (conscious choice)**: we reuse the standard CC `scope` slot for the issue # marker. Consequences:

- Passes default `commitlint` rules (with `(#42)` allowed in scope).
- Tools like `conventional-changelog` / `semantic-release` parse `(#42)` as scope, so changelog scope columns will show `#42`. We don't carry both a domain scope (e.g. `auth`) and issue # ‚Äî issue # wins.
- If a domain scope is essential, put it in the commit body via a `Scope: auth` trailer.

### 10.3 PRs

- Always start as draft.
- `Closes #N` or `Refs #N` is required. If a single issue spans multiple PRs, intermediates use `Refs #N` and the final uses `Closes #N`. Checklist distribution is planner/user's call.
- **Base**: PRs default to `main`. For experimental or long-lived feature work, `/work-on --base <branch>` opens the PR against an alternate base ‚Äî see ¬ß10.5. Non-`main`-base PRs default to `Refs` (the issue closes only when the work reaches `main`).
- Body follows `.claude/templates/pr_body.md`.
- Curate the body right after each commit.
- `/ship` transitions to ready after ship-gate verification.

### 10.4 PR ‚Üî checklist sync

Order: **commit ‚Üí PR body update**. Commit is the fact; PR body reflects it. Reverse order creates "checklist ticked but commit absent" states. Refetch the remote body right before updating ‚Äî abort the auto-update if external edits (reviewer, GitHub UI, parallel session) are detected.

### 10.5 Topic-branch workflow

The default flow (`/work-on <#>`) targets `main`. For two scenarios the user can opt into an alternate base via `/work-on <#> --base <branch>`:

1. **Experimental work** ‚Äî risk too high for `main`. The user creates `experiment/foo` from `main`, iterates with a few PRs whose base is `experiment/foo`, and only consolidates to `main` via one final PR when the experiment stabilizes (or abandons the branch).
2. **Big features** ‚Äî a multi-PR refactor where the integration point is a long-lived feature branch. Sub-task PRs target the feature branch; the feature branch itself is eventually merged to `main`.

**Conventions:**
- Sub-task PRs use `Refs #N` (not `Closes #N`) so the issue stays open until work reaches `main`. `/work-on` enforces this default for non-`main` bases.
- The alternate base must already exist locally + remotely. `/work-on --base <branch>` does NOT auto-create it; the user creates the base deliberately (e.g. `git checkout main && git pull && git checkout -b experiment/foo && git push -u origin experiment/foo`).
- The alternate base's remote branch-protection settings are the user's responsibility. For sub-task PRs to auto-merge into `experiment/foo`, the GitHub branch-protection rules on `experiment/foo` must permit the same flow as `main` (or be more permissive). `/work-on` does not inspect or modify remote settings.
- The alternate base is NOT in SPEC ¬ß6.1's protected list (`main`/`master`/`release/*`). The user is free to commit directly to `experiment/foo` between sub-task PRs ‚Äî it's an integration point, not the canonical product line.
- The work-branch name (`<gh-user>/<type>/<#>-<slug>`) is unchanged. Stacking multiple work-PRs on the same issue+base is not supported by `/work-on` alone ‚Äî manual branch creation (`git checkout -b ...`) is the escape hatch for that rare case.

**Use with Directives (dir-mode integration):** scenario 2 ("Big features") is the canonical pattern for Directive-scoped multi-PR work (¬ß1.7, ¬ß2.1). The user creates a topic branch from `main` (e.g., `feature/directive-92`); each Execution Issue parented under the Directive (via `/file-issue --parent <directive-#>`, ¬ß5.2) runs `/work-on <execution-#> --base feature/directive-92`, producing sub-task PRs with `Refs #N`. The Directive Issue itself is **not** branched ‚Äî the `proposed-protect` matcher in ¬ß6.1 blocks `git checkout -b` against a Directive Issue (any status), by design; the Directive scopes the work, the Execution Issues do the work. Consolidation follows the standard pattern (one final PR head=topic-branch, base=`main`, body `Closes` each Execution Issue), and `/ship`'s Directive-aware audit (¬ß5.7) fires once per parented Execution Issue at consolidation. `/complete-directive <directive-#>` (¬ß5.13) must run **after** consolidation ‚Äî `activation-reviewer` (¬ß4.9) evaluates closed-Execution-Issue evidence, so running completion while the topic branch is still unconsolidated yields insufficient-evidence and blocks.

**Consolidation back to `main`:** when the topic branch is ready, open ONE PR with head=`experiment/foo` and base=`main`, body `Closes #<#>`. This is a regular `gh pr create` invocation ‚Äî `/work-on` is not designed for the consolidator step (the branch already exists; there's no planner phase to repeat).

## 11. CI awareness

- **Right before** ready transition: `gh pr checks --json state` for a one-time check. **Block only on `failure` or `cancelled`**. `pending` / `in_progress` / `queued` pass ‚Äî the background watch picks up monitoring after ready transition. (¬ß5.7 step 7)
- **Right after** ready transition: `gh pr checks --watch` runs in **background**, returning immediately.
- The background watch's failure detection routes to a PostToolUse hook (¬ß6.2) notification.
- The post-push PostToolUse hook (`Bash` matching `git push`) also warns about PR-less push patterns (detached HEAD, missing draft PR).

This shell repo ships its own CI workflow at `.github/workflows/ci.yml` running on `pull_request` and `push: main`. Two jobs, each on an OS matrix of `ubuntu-latest` and `macos-latest` (the BSD branch is the production path for the primary user; Linux-only CI would leave it unverified):
- **syntax** ‚Äî `bash -n` AND `shellcheck --severity=warning --shell=bash` over `bin/claude-eng`, `scripts/*.sh`, `scripts/lib/*.sh`, `.claude/hooks/*.sh`, `.claude/hooks/helpers/*.sh`. Repo-wide silences (e.g. SC1090 for dynamic sourcing via `BASH_SOURCE`-relative paths and per-test fixture variables) live in `.shellcheckrc`; documented-deliberate patterns (`eval` on jq-emitted `@sh` scripts in `helpers/status.sh`, `printf -v` to dynamic varname in `helpers/escape.sh`) carry inline `# shellcheck disable=SC‚Ä¶` with one-line justification. shellcheck install is OS-conditional: `apt-get install -y shellcheck` on Linux; pre-installed on macOS-runner images (fallback `brew install shellcheck` if absent).
- **smoke** ‚Äî installs `jq` (apt-get on Linux; pre-installed on macOS) and runs `./scripts/test/smoke.sh`.

`fail-fast: false` on the matrix so per-platform failures surface independently.

So `/ship`'s `gh pr checks` step has something concrete to watch for on PRs against this repo. Other repos that adopt the shell can copy the workflow as a starting point.

## 13. Failure modes and recovery

| Situation | Handling |
|-----------|----------|
| Issue body is ambiguous | `/work-on` refuses to start; asks the user to clarify the issue |
| Phase A reveals the spec was wrong | Strike through the checklist with reason, re-invoke planner |
| Phase B test passes accidentally | Two branches: (a) Phase A spec is weak or test inputs insufficient ‚Üí strengthen doc/test. (b) Adding a regression test for already-implemented behavior is legitimate ‚Üí note in one line in PR body and proceed. Don't always restart from doc. |
| Base (main or `--base` target) advances during work | `git pull --rebase origin <base>` ‚Üí resolve conflicts ‚Üí re-push PR. The base is whatever the PR's `baseRefName` is ‚Äî read via `gh pr view --json baseRefName` if uncertain. Do NOT `git merge <base>` into the feature branch ‚Äî the ¬ß6.1 `backmerge` matcher blocks that motion specifically to keep feature branches rebase-clean (escape `SKIP_HOOKS=backmerge` exists for exceptional cases but should be rare). |
| Another contributor touches the same branch | `--ff-only pull`, escalate to user on conflict |
| Can't finish in one PR | Intermediate PRs use `Refs #N`, the final uses `Closes #N` |
| Issue acceptance criteria change mid-work | Update the issue body + comment with the change log; mirror in the PR body |
| CI fails while local passes | `/ship` blocks. CI logs first, then hotfix commit + re-push |
| Hook misfires | Use escape hatch + audit it + file a fix PR for the hook |
| Same escape category repeats in audit | Sign the hook is wrong ‚Äî fix PR takes priority |

## 18. Release backbone

Locks the shape of how releases happen: per-PR fragment files, one consolidator skill (`/release`), and a templated CI gate. Adopters consume the substrate via `/onboard-dir-mode` (¬ß18.6). Established by Directive #128; depends on the VERSION + tag surface from Directive #122 (¬ß3.5).

### 18.1 Fragment contract

The release line consumes per-PR **changelog fragments** under `changelog_unreleased/<category>/<N>.md`:

- `<category>` ‚àà {`added`, `changed`, `deprecated`, `removed`, `fixed`, `security`} ‚Äî the six [Keep a Changelog](https://keepachangelog.com/en/1.1.0/) categories. Each is a subdirectory under `changelog_unreleased/`.
- `<N>` is the issue or PR number that introduces the fragment (positive integer; filename stem).
- The fragment body is a **single-line markdown bullet** (one line starting with `- `).
- The bullet **must contain `(#<N>)`** where `<N>` matches the filename stem. CI rejects mismatches (¬ß18.6).
- **No YAML frontmatter, no multi-paragraph prose, no nested headers inside the fragment.** The contract is intentionally minimal ‚Äî the directory is the category, the filename is the PR/issue link, the bullet is the user-facing line.

`changelog_unreleased/TEMPLATE.md` documents the contract for fragment-authors in project-neutral language. The template ships in the shell and `/onboard-dir-mode` installs it (along with the six empty category subdirectories) into adopting repos.

`/release X.Y.Z` (¬ß5.20) consolidates all present fragments into a new `## [X.Y.Z] ‚Äî YYYY-MM-DD` section in `CHANGELOG.md` at repo root, grouped under Keep-a-Changelog category subheadings, and deletes the consumed fragments in the same commit.

### 18.2 Version of record

The canonical version-of-record is the top-level `VERSION` file (locked by Directive #122 / ¬ß3.5). `/release X.Y.Z`:

- Validates `X.Y.Z` is semver 0.x (`MAJOR=0`).
- Strips any `-dev` suffix from `VERSION`.
- Writes `X.Y.Z` to `VERSION`.
- Falls back to `git describe --tags` for **resolution-time queries only** ‚Äî never for write-back. The file is authoritative on `main`.

Multi-file version-of-record (`package.json`, `Cargo.toml`, `pyproject.toml`) is deferred. A documented extension point ‚Äî `.claude/release-config.json` ‚Äî is **reserved** at this location for projects whose VoR lives elsewhere; the file is currently unimplemented and the reservation prevents the name from being claimed by something else.

### 18.3 Trunk branch model

Default development line is `main`. `/release` operates on `main` unless `--base <branch>` targets a maintenance branch (e.g., `0.3-maintenance`). Maintenance branches are **not auto-created by the shell** ‚Äî adopters who need LTS lines create them manually. See ¬ß10.5 for the topic-branch / alternate-base flow.

The CI fragment-gate (¬ß18.6) targets `main` and `*-maintenance` by default; configurable per project at workflow-install time.

### 18.4 Tag format

Tags follow `v` + semver 0.x ‚Äî `v0.1.0`, `v0.2.0`, etc. Locked by Directive #122 (CLOSED); preserved verbatim here so downstream tooling reads one canonical clause. Tag-freshness is **not** enforced as a merge gate per #122's non-goal ‚Äî tag-state drift is drift, not breakage. `/release` creates the tag (`gh release create vX.Y.Z --target <merge-sha>`) at the release-PR merge boundary; pre-merge release-PRs are tagless.

### 18.5 Non-goals

Locked by Directive #128 (closed):

- **No `/prepare-release` skill.** VERSION's `-dev` suffix replaces the scope-locking ceremony of a prepare PR.
- **No `/changelog-check` skill.** CI workflow + optional local hook arm cover the enforcement surface; a manual lint skill is a third redundant surface.
- **No tag-push merge gate, no tag-freshness enforcement** (preserves Directive #122's explicit non-goal).
- **No automated maintenance-branch creation, no backport / port-PR flow.** Deferred to a successor Directive when at least one adopter requests LTS.
- **No release-please / external-service integration.** The shell ships a ~30-line CI script under shell control. Adopters who already use release-please keep it; the shell does not displace it but also does not configure it.
- **No automated distribution** (npm, PyPI, Docker, Homebrew, binary upload). Per-project extension surface.
- **No release cadence enforcement.** When to cut a release is a human / per-project call.
- **No breaking-change policy enforcement.** Shell stays at `0.x` per #122 (`MAJOR=0` invariant); downstream projects at `1.x+` define their own compat policy.
- **No multi-file version-of-record bump.** `VERSION` is canonical; `.claude/release-config.json` is reserved but currently unimplemented.
- **No fragment frontmatter or template enforcement beyond filename/PR-number consistency.**
- **No major-rollover archive logic.** Shell stays at `0.x` per #122; revisit when `1.0.0` is on the table.
- **No retroactive per-milestone CHANGELOG entries** for pre-`v0.1.0` history. The inaugural section covers `init..v0.1.0` as one block, not split per historical sub-milestone.

### 18.6 Cross-project surface

`/onboard-dir-mode` installs the release-backbone substrate into adopting target repos as part of its tier-3 PR:

- `.github/workflows/check-changelog.yml` ‚Äî the per-PR fragment-gate workflow.
- `changelog_unreleased/TEMPLATE.md` + the six empty category subdirectories with `.gitkeep` placeholders.

The workflow validates three rules on every PR targeting a protected branch (`main` and `*-maintenance` by default):

1. The PR diff adds at least one file under `changelog_unreleased/<category>/<N>.md` where `<category>` is one of the six Keep-a-Changelog subdirectories.
2. The filename stem `<N>` is a positive integer.
3. The file contains `(#<N>)` where `<N>` matches the filename stem.

Failure (no fragment, malformed filename, mismatched PR ref) blocks the PR with an error referencing this section.

**`skip-changelog` opt-out**: a PR labeled `skip-changelog` bypasses the fragment check. The label is auditable on the PR; the workflow logs its application. Intended for PRs with no observable end-user behavior change ‚Äî pure docs-fix typos, dependency-only chores, internal refactors. **PRs carrying the label produce no fragment and are not represented in the eventual CHANGELOG section** ‚Äî by design, since there is nothing for a user to read about. Workflow trigger events include `labeled` and `unlabeled` so the opt-out re-runs the check.

Adopters consume the backbone with zero in-repo configuration beyond the substrate install plus their own `VERSION` file and a `CHANGELOG.md` at repo root. `/release X.Y.Z` runs against any repo where these three artifacts are present.

**Workflow does not self-trigger on its own landing PR**: per [GitHub's `pull_request` security policy](https://docs.github.com/en/actions/using-workflows/triggering-a-workflow#using-event-types), workflows modified inside a PR do **not** run on that PR (otherwise a fork could exfiltrate secrets via a self-modified workflow). The first PR after `check-changelog.yml` lands on `main` is the first empirical observation of the fragment-gate; the landing PR itself only validates the workflow's structure via smoke ¬ß66c-e + the `code-reviewer` walk-through. Adopters running `/onboard-dir-mode` see the same: the substrate-install PR opens the workflow but the workflow fires from the next PR onward.
